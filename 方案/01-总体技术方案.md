# BlueNote 总体技术方案

版本：v0.1  
状态：草案  
更新时间：2026-06-03

## 1. 项目背景

BlueNote 是一个简易版小红书项目，面向移动端用户，核心能力包括用户登录、笔记发布、点赞收藏、关注关系、关注页 Feed、排行榜、评论、IM，以及第一阶段可扩展的神券秒杀订单能力。

本项目不是一次性做完全部功能的 MVP 原型，而是按照真实企业项目标准进行架构设计。第一阶段会控制部署复杂度和资源消耗，但代码结构、服务边界、数据一致性、可观测性、安全认证、消息事件规范等需要从一开始按长期演进来设计。

## 2. 设计目标

1. 支持移动端 App/H5 的主要业务流程。
2. 后端采用 Java 微服务体系，服务边界清晰。
3. 核心业务链路具备可扩展、可观测、可恢复能力。
4. 支持有限资源部署：一部分部署在 2 核 4G 云服务器，一部分部署在本地旧电脑并通过内网穿透访问。
5. 不过早引入复杂集群，基础设施以单机或主从为主。
6. 对神券秒杀订单、支付、推荐、搜索、审核、数据分析等能力保留扩展空间。

## 3. 非目标范围

当前阶段暂不追求以下目标：

1. 不设计大规模 Kubernetes 集群。
2. 不引入 Kafka 作为主消息队列。
3. 不引入复杂分布式事务框架，例如 Seata。
4. 不在第一阶段引入 Elasticsearch，搜索能力后续单独评估。
5. 不做多机房容灾架构。
6. 不做完整推荐系统，只保留事件和数据基础。

## 4. 业务模块概览

当前规划的核心业务模块包括：

1. 登录服务
2. 笔记发布服务
3. 计数服务
4. 用户关系服务
5. Timeline Feed 服务
6. 排行榜服务
7. 评论服务
8. IM 服务
9. 订单服务，第一阶段聚焦神券秒杀订单

## 5. 总体架构

BlueNote 采用移动端 + API 网关 + Java 微服务 + 基础中间件的架构。

整体调用链路：

```text
移动端
  |
  v
API 网关
  |
  +--> 登录服务
  +--> 用户服务
  +--> 笔记发布服务
  +--> 计数服务
  +--> 用户关系服务
  +--> Timeline Feed 服务
  +--> 排行榜服务
  +--> 评论服务
  +--> IM 服务
  +--> 订单服务

基础设施：
MySQL / Redis / RocketMQ / Nacos / MinIO（第一阶段）/ 云 OSS（后续演进）/ Prometheus / Grafana / Loki
```

核心原则：

1. 网关统一入口，内部服务不直接暴露给移动端。
2. 服务之间优先通过同步 RPC 完成查询类和强即时结果类操作。
3. 状态变更后的扩散、计数、通知、Feed 构建、排行榜更新等通过 RocketMQ 异步解耦。
4. 数据库作为最终事实来源，Redis 和 MQ 不作为唯一状态依据。
5. 所有异步消费必须具备幂等能力。

## 6. 技术栈选型

### 6.1 移动端技术栈

| 类型 | 技术 |
|---|---|
| 开发框架 | uni-app |
| UI 框架 | Vue 3 |
| 开发语言 | TypeScript |
| 状态管理 | Pinia |
| 请求库 | uni.request 封装 |
| 构建工具 | Vite，随 uni-app 体系 |
| 目标平台 | Android / iOS / H5，后续可扩展小程序 |

选择原因：

1. 适合移动端优先项目。
2. Vue 生态学习和开发成本较低。
3. 后续可以同时支持 App、H5、小程序等多端。
4. 相比原生双端开发，初期人力成本更低。

### 6.2 后端技术栈

| 类型 | 技术 |
|---|---|
| 开发语言 | Java 21 LTS |
| 核心框架 | Spring Boot 3.5.x |
| 微服务体系 | Spring Cloud 2025.0.x |
| Web 框架 | Spring MVC |
| 安全框架 | Spring Security |
| ORM | MyBatis-Plus + MyBatis XML |
| 参数校验 | Jakarta Validation |
| 接口文档 | OpenAPI / Knife4j |
| 构建工具 | Maven |

选择原因：

1. Java + Spring 体系适合长期维护的企业级后端项目。
2. Spring Boot 3.5.x 比 Spring Boot 4.x 更稳，第三方生态兼容性更好。
3. Java 21 LTS 在当前阶段足够成熟，后续可评估升级 Java 25 LTS。
4. MyBatis-Plus 适合业务系统 CRUD，复杂 SQL 保留 XML 可控性。

### 6.3 网关与服务治理

| 类型 | 技术 |
|---|---|
| API 网关 | Spring Cloud Gateway |
| 服务注册 | Nacos |
| 配置中心 | Nacos |
| 负载均衡 | Spring Cloud LoadBalancer |
| 限流 | Gateway 限流 + Redis |
| 灰度能力 | 第一阶段不做完整灰度，只预留路由能力 |

网关职责：

1. 统一认证鉴权。
2. 请求路由。
3. 基础限流。
4. 请求日志和追踪 ID 注入。
5. 对移动端隐藏内部服务结构。

### 6.4 数据存储技术栈

| 类型 | 技术 |
|---|---|
| 主数据库 | MySQL 8.x |
| 缓存 | Redis 7.x |
| 对象存储 | MinIO（第一阶段全环境）/ 云 OSS/COS/R2（后续演进） |
| 本地开发数据库 | Docker Compose MySQL |

数据设计原则：

1. 第一阶段按服务拆 schema，不直接做物理分库分表。
2. 每个服务只直接访问自己的数据库 schema。
3. 跨服务数据通过接口或事件同步，不跨库 join。
4. Redis 主要承担缓存、计数、Feed 缓冲、排行榜、分布式锁等职责。
5. 图片和视频不存 MySQL，只存对象存储地址和元数据。

### 6.5 消息队列技术栈

| 类型 | 技术 |
|---|---|
| 主消息队列 | RocketMQ 5.x |
| 消息类型 | 普通消息、顺序消息、延时消息、事务消息 |
| 使用场景 | 计数异步、Feed 事件、排行榜更新、通知生成、推送请求、IM 内部消息、神券秒杀异步下单、订单超时关单 |

选择 RocketMQ 的原因：

1. 原生支持延时消息，适合订单超时关单、任务超时处理。
2. 支持事务消息，适合本地事务和消息发送最终一致。
3. 支持顺序消息，适合 IM 会话消息、订单状态流转等局部有序场景。
4. 更贴近业务消息场景，当前阶段比 Kafka 更适合作为主 MQ。

Kafka 暂不引入。后续如果需要大规模埋点、实时数据分析、推荐系统、用户行为流处理，再单独评估 Kafka。

### 6.6 推送与实时投递技术栈

| 类型 | 技术 |
|---|---|
| 在线实时投递 | Netty + WebSocket |
| 离线系统 Push | uni-push 2.0 / 厂商 Push 通道 |
| 设备与偏好存储 | MySQL |
| 在线状态与连接路由 | Redis |
| 投递请求 | RocketMQ |
| 通道配置 | Nacos |

推送服务第一阶段重点保证：

1. 设备注册、解绑和多设备绑定。
2. App 前台在线时的轻量 WebSocket 投递。
3. App 后台或离线时的系统通知栏提醒。
4. 用户推送偏好、IM 消息预览开关和免打扰设置。
5. 通知、IM、订单等服务通过统一接口或事件请求投递。
6. 推送失败不影响业务事实，客户端通过通知、IM、订单接口拉取补偿。

### 6.7 IM 技术栈

| 类型 | 技术 |
|---|---|
| 连接鉴权 | JWT / 访问令牌 |
| 消息投递 | RocketMQ + bluenote-push |
| 消息存储 | MySQL，后续可拆冷热存储 |
| 未读数 | Redis + MySQL 兜底 |

IM 第一阶段重点保证：

1. 企业级单聊基础能力。
2. 文本、图片、笔记卡片消息。
3. `clientMsgId` 发送幂等。
4. 消息持久化。
5. 会话内 `conversationSeq` 顺序。
6. 服务端 ACK。
7. 客户端接收 ACK 和已读位置。
8. 离线消息拉取。
9. 会话未读数。
10. 使用推送服务做在线或离线提醒。

### 6.8 部署与运维技术栈

| 类型 | 技术 |
|---|---|
| 容器化 | Docker |
| 编排 | Docker Compose |
| 反向代理 | Nginx 或 Caddy |
| HTTPS | Caddy 自动证书，或 Nginx + Certbot |
| 日志 | Loki |
| 指标 | Prometheus |
| 面板 | Grafana |
| 健康检查 | Spring Boot Actuator |
| 链路追踪 | Micrometer Tracing，后续接 Tempo 或 Zipkin |

暂不引入 Kubernetes。当前资源规模下，Docker Compose 更容易部署、排查和维护。

## 8. 核心基础设施

### 8.1 MySQL

MySQL 作为核心业务事实库。

建议：

1. 开发期单机部署。
2. 正式环境至少开启定时备份。
3. 后续可做主从复制。
4. 每个服务独立 schema。
5. 所有核心表必须包含 `id`、`created_at`、`updated_at`、`deleted` 或等价字段。

### 8.2 Redis

Redis 用于高频读写和临时状态。

主要场景：

1. 登录态和刷新令牌辅助校验。
2. 热点笔记缓存。
3. 计数缓存。
4. Feed 收件箱。
5. 排行榜 ZSet。
6. IM 在线状态。
7. 神券秒杀库存、用户限购、秒杀 token。
8. 分布式锁。
9. 幂等去重。

Redis 数据不能作为唯一事实来源，关键状态必须可从 MySQL 恢复。

### 8.3 RocketMQ

RocketMQ 用于服务间异步事件。

主要 topic 初步规划：

| Topic | 用途 |
|---|---|
| note-event | 笔记发布、删除、状态变更 |
| interaction-event | 点赞、取消点赞、收藏、取消收藏 |
| relation-event | 关注、取关 |
| comment-event | 评论创建、删除、点赞 |
| feed-event | Feed 构建事件 |
| rank-event | 排行榜更新事件 |
| im-message-event | IM 消息投递事件 |
| order-seckill-task-event | 神券秒杀预扣成功后的异步下单任务 |
| order-event | 神券订单创建、支付、关闭、发券 |
| order-timeout-event | 付费神券订单超时关单延时消息 |

消息设计原则：

1. 事件命名使用过去式，例如 `NotePublished`、`UserFollowed`。
2. 消息体必须包含 `eventId`、`eventType`、`occurredAt`、`traceId`。
3. 消费端必须用 `eventId` 或业务唯一键做幂等。
4. 消息失败进入重试，超过重试次数进入死信或人工补偿流程。
5. 不通过 MQ 传输大文件，只传文件 ID 或对象存储地址。

### 8.4 对象存储

图片、视频、头像等文件进入对象存储。

第一阶段开发、测试、正式环境统一使用 MinIO。正式环境 MinIO 部署在 2 核 4G 云服务器上，通过 HTTPS 域名或反向代理对外提供文件访问，并定期备份到本地旧电脑。后续当文件数量、访问量、带宽或可用性要求提高后，再迁移到云对象存储 + CDN。

文件上传建议流程：

1. 移动端向文件服务申请上传凭证。
2. 移动端直传对象存储。
3. 上传完成后回调或主动确认。
4. 业务服务保存文件元数据。

## 9. 数据一致性策略

总体原则：避免强分布式事务，采用本地事务 + 事件驱动 + 幂等消费 + 补偿任务。

典型模式：

1. 单服务内部使用本地事务。
2. 跨服务状态同步通过 RocketMQ 事件。
3. 关键消息发送使用 RocketMQ 事务消息或 Outbox 表。
4. 消费端保证幂等。
5. 定时任务做数据校验和补偿。

示例：

1. 用户点赞笔记：互动记录落库后发送 `NoteLiked` 事件，计数服务异步更新计数。
2. 用户发布笔记：笔记服务落库后发送 `NotePublished` 事件，Feed、排行榜、审核、通知等服务订阅处理。
3. 神券秒杀订单：Redis Lua 预扣成功后发送 RocketMQ 异步下单任务，订单消费者落 MySQL；付费神券订单创建后发送延时消息，到期后查询订单状态并 CAS 关闭订单，同时定时任务兜底扫描。

## 10. 缓存与计数策略

计数类数据包括：

1. 笔记点赞数
2. 笔记收藏数
3. 笔记评论数
4. 用户关注数
5. 用户粉丝数

策略：

1. 高频计数优先写 Redis。
2. 通过 RocketMQ 异步聚合落 MySQL。
3. MySQL 保留最终计数快照。
4. Redis 丢失时可以从 MySQL 和行为明细重建。
5. 点赞、收藏、关注必须有明细表，不能只存计数。

不建议每次点赞都同步更新多个 MySQL 计数字段，否则高峰时写放大会比较明显。

## 11. Feed 与排行榜策略

### 11.1 Timeline Feed

关注页 Feed 初步采用推拉结合：

1. 普通用户发布笔记后，将笔记推送到粉丝的 Feed 收件箱。
2. 大 V 用户不全量推送，用户刷新时拉取补充。
3. Feed 收件箱可使用 Redis ZSet。
4. MySQL 保留必要的持久化记录或可重建依据。

具体推拉阈值在 Feed 服务详细设计中确定。

### 11.2 排行榜

排行榜第一阶段设计两个榜单：

1. 本周热门笔记榜：面向内容发现，展示 Top 100，使用 Redis ZSet 做精确排名。
2. 本年创作者成长榜：面向创作者激励，展示 Top 100，并为 Top 100 外创作者提供粗估排名。

榜单分数第一阶段基于点赞、评论、收藏的周期内净增量计算。排行榜不是强一致场景，可以接受秒级到分钟级延迟。后续可以扩展时间衰减、内容质量权重、分享数、浏览数和反作弊权重。

## 12. 安全认证方案

认证方式：

1. 移动端登录后获取 Access Token 和 Refresh Token。
2. Access Token 短有效期。
3. Refresh Token 长有效期，支持设备维度管理。
4. 网关统一校验 Access Token。
5. 内部服务通过用户上下文 Header 获取用户 ID、设备 ID、请求 ID 等信息。

安全要求：

1. 密码使用强哈希算法，不明文存储。
2. 敏感接口增加限流。
3. 登录、注册、验证码、刷新 Token 需要风控策略。
4. 文件上传限制类型、大小和频率。
5. 管理端和移动端权限体系分离。

## 13. 部署架构

第一阶段部署目标是可运行、可维护、可恢复，不追求复杂集群。

建议云服务器部署：

1. Nginx 或 Caddy
2. API 网关
3. 登录服务
4. 用户服务
5. 笔记服务
6. 计数服务
7. MySQL
8. Redis
9. RocketMQ 核心组件，视资源情况决定

建议本地旧电脑部署：

1. Grafana
2. Prometheus
3. Loki
4. MinIO 备份副本、测试环境对象存储
5. 低优先级异步任务
6. 开发/测试环境
7. 备份任务

关键原则：

1. 用户主链路不能强依赖本地旧电脑。
2. 内网穿透只适合非核心链路、管理端、测试环境或备份入口。
3. 第一阶段正式图片访问使用云服务器 MinIO；后续根据访问量和成本迁移云对象存储或 CDN。
4. 数据库必须有自动备份和恢复演练。

## 14. 监控、日志与告警

### 14.1 指标监控

使用 Spring Boot Actuator + Micrometer 暴露指标，由 Prometheus 采集，Grafana 展示。

核心指标：

1. HTTP QPS
2. HTTP 错误率
3. HTTP 延迟
4. JVM 内存
5. JVM GC
6. 线程池状态
7. 数据库连接池状态
8. Redis 延迟
9. RocketMQ 消费堆积
10. IM 在线连接数

### 14.2 日志

日志进入 Loki。

日志要求：

1. 所有请求带 `traceId`。
2. 网关生成或透传 `traceId`。
3. 服务日志包含服务名、实例、用户 ID、traceId。
4. 异步消息消费日志包含 eventId、topic、consumerGroup。

### 14.3 告警

第一阶段至少需要：

1. 服务不可用告警。
2. HTTP 5xx 错误率告警。
3. RocketMQ 消费堆积告警。
4. MySQL 磁盘空间告警。
5. Redis 内存告警。
6. IM 连接数异常告警。

  bluenote-note/
  bluenote-counter/

  bluenote-feed/

## 16. 技术风险与演进方向

| 风险 | 说明 | 应对 |
|---|---|---|
| 服务器资源不足 | 多个 Java 服务和中间件会占用较多内存 | 允许服务合并部署，限制 JVM 内存 |
| RocketMQ 运维成本 | 比 RabbitMQ 更重 | 第一阶段控制 topic 和 consumer 数量，保留文档化运维流程 |
| Redis 数据丢失 | 计数、Feed、在线状态依赖 Redis | MySQL 保留事实数据，支持重建 |
| IM 长连接复杂度 | 连接管理、ACK、离线消息都有状态 | IM 服务单独详细设计，不和普通 HTTP 服务混在一起 |
| 内网穿透不稳定 | 本地旧电脑不可作为核心依赖 | 主链路部署在云服务器 |
| Feed 扩散成本 | 粉丝量增加后推送压力变大 | 推拉结合，大 V 特殊处理 |

后续演进方向：

1. 引入 Elasticsearch 支持搜索。
2. 引入 Kafka 支持用户行为流和实时分析。
3. 引入 Flink 或其他实时计算框架支持推荐和实时榜单。
4. MySQL 从单机演进到主从复制。
5. Redis 从单机演进到哨兵或集群。
6. RocketMQ 从单机演进到主从或 DLedger 高可用。
7. 部署方式从 Docker Compose 演进到 Kubernetes。
