# BlueNote 服务划分方案

版本：v0.1  
状态：草案  
更新时间：2026-06-03

## 1. 文档目标

本文档用于定义 BlueNote 的后端服务边界、职责归属、数据归属、服务依赖、事件流转和阶段性部署策略。

本文不展开具体接口字段、数据库表结构、Redis Key、消息体字段细节。这些内容会在各服务详细设计文档中继续设计。

## 2. 划分原则

### 2.1 业务边界优先

服务划分首先按照业务领域和数据所有权进行，而不是按照 Controller、Service、Mapper 等代码分层进行。

例如：

1. 登录、Token、设备会话归登录服务。
2. 关注、取关、粉丝列表归用户关系服务。
3. 点赞数、收藏数、粉丝数等聚合计数归计数服务。
4. 关注页 Feed 收件箱归 Timeline Feed 服务。

### 2.2 数据归属明确

每个服务拥有自己的数据 schema。其他服务不能直接跨库查询或修改该服务的数据。

跨服务访问只允许两种方式：

1. 同步接口调用。
2. 订阅领域事件后维护自己的读模型。

### 2.3 写操作归属唯一

一个业务对象只能有一个主写服务。

例如：

1. 笔记正文只能由笔记发布服务写。
2. 关注关系只能由用户关系服务写。
3. 评论只能由评论服务写。
4. IM 消息只能由 IM 服务写。
5. 订单状态只能由订单服务写。

其他服务需要感知变化时，通过事件同步，不直接修改源服务数据。

### 2.4 查询可以适度冗余

为了提升移动端查询性能，允许部分服务维护冗余读模型。

例如：

1. Feed 服务可以保存笔记 ID、作者 ID、发布时间等 Feed 展示所需的轻量快照。
2. 排行榜服务可以保存热度分和榜单快照。
3. 评论服务可以冗余评论用户昵称、头像快照。

冗余数据不作为最终事实来源，必须能通过源服务事件或补偿任务修正。

### 2.5 异步优先解耦

以下场景优先使用 RocketMQ 异步处理：

1. 计数更新。
2. Feed 扩散。
3. 排行榜更新。
4. 站内通知。
5. IM 内部跨节点投递。
6. 订单超时关单。
7. 用户行为后续分析。

### 2.6 逻辑服务与物理部署分离

逻辑上按照微服务设计，物理部署上允许阶段性合并。

原因：

1. 当前目标部署资源有限。
2. 多个 Java 进程会带来较高内存占用。
3. 企业项目标准关注的是边界、协议、数据归属和演进能力，不等于第一天必须把所有服务独立进程部署。

## 3. 服务总览

### 3.1 逻辑服务清单

| 服务 | 服务名 | 核心职责 | 第一阶段优先级 |
|---|---|---|---|
| API 网关 | bluenote-gateway | 统一入口、鉴权、路由、限流、追踪 ID | P0 |
| 登录服务 | bluenote-auth | 登录注册、Token、设备会话、安全校验 | P0 |
| 用户服务 | bluenote-user | 用户资料、主页信息、账号基础信息 | P0 |
| 文件服务 | bluenote-file | 上传凭证、文件元数据、对象存储接入 | P0 |
| 笔记发布服务 | bluenote-note | 笔记发布、编辑、删除、详情、状态流转 | P0 |
| 计数服务 | bluenote-counter | 点赞、收藏、评论、关注、粉丝等计数 | P0 |
| 用户关系服务 | bluenote-relation | 关注、取关、关注列表、粉丝列表 | P0 |
| Timeline Feed 服务 | bluenote-feed | 关注页 Feed 流、收件箱、Feed 读模型 | P0 |
| 评论服务 | bluenote-comment | 一级评论、二级评论、评论删除、评论点赞 | P0 |
| 排行榜服务 | bluenote-rank | 本周热门笔记榜、本年创作者成长榜、榜单快照、分数计算 | P1 |
| 推送与实时投递服务 | bluenote-push | 设备注册、WebSocket 在线投递、离线 Push、推送偏好 | P1 |
| 通知服务 | bluenote-notification | 站内通知、通知聚合、已读未读、请求 Push | P1 |
| IM 服务 | bluenote-im | 会话、消息、ACK、离线消息、聊天未读 | P1 |
| 订单服务 | bluenote-order | 神券秒杀活动、抢券订单、支付状态、超时关单、用户券 | P2 |

优先级说明：

1. P0：核心内容社区闭环必须具备。
2. P1：提升互动体验或运营能力，可在 P0 稳定后实现。
3. P2：后续商业化能力，当前只预留架构位置。

### 3.2 推荐服务分组

按业务域可以分为：

| 业务域 | 服务 |
|---|---|
| 入口与身份域 | bluenote-gateway、bluenote-auth |
| 用户域 | bluenote-user、bluenote-relation |
| 内容域 | bluenote-note、bluenote-file、bluenote-comment |
| 互动与分发域 | bluenote-counter、bluenote-feed、bluenote-rank、bluenote-notification |
| 实时投递与通信域 | bluenote-push、bluenote-im |
| 交易域 | bluenote-order |

## 4. 服务详细划分

### 4.1 API 网关

服务名：`bluenote-gateway`

#### 职责

1. 移动端 API 统一入口。
2. 路由到后端服务。
3. Access Token 校验。
4. 注入用户上下文，例如 `userId`、`deviceId`、`traceId`。
5. 基础限流和黑名单拦截。
6. 统一跨域、请求大小限制、超时控制。
7. 网关访问日志。

#### 不负责

1. 不处理具体业务逻辑。
2. 不直接访问业务数据库。
3. 不保存用户核心状态。

#### 主要依赖

| 依赖 | 用途 |
|---|---|
| Nacos | 服务发现、配置 |
| Redis | 限流、黑名单、Token 辅助校验 |
| bluenote-auth | Token 校验规则或公钥配置同步 |

#### 输出

网关不发布业务领域事件，只负责生成或透传 `traceId`。

### 4.2 登录服务

服务名：`bluenote-auth`

#### 职责

1. 用户注册。
2. 手机号、密码、验证码等登录方式。
3. Access Token 签发。
4. Refresh Token 签发和轮换。
5. 设备会话管理。
6. 登出。
7. 登录失败次数控制。
8. 账号安全策略。

#### 数据归属

| 数据 | 说明 |
|---|---|
| 用户登录凭证 | 密码哈希、手机号绑定、第三方登录绑定 |
| 设备会话 | 设备 ID、登录时间、刷新令牌状态 |
| 登录安全记录 | 登录失败、异常登录、封禁状态 |

#### 不负责

1. 不负责用户主页资料展示。
2. 不负责关注关系。
3. 不负责用户计数。

#### 同步依赖

| 依赖服务 | 用途 |
|---|---|
| bluenote-user | 注册成功后创建用户基础资料，或查询用户基础状态 |

#### 发布事件

| 事件 | 触发时机 |
|---|---|
| UserRegistered | 用户注册成功 |
| UserLoggedIn | 用户登录成功 |
| UserLoggedOut | 用户登出 |
| UserDeviceRevoked | 用户设备会话失效 |

### 4.3 用户服务

服务名：`bluenote-user`

#### 职责

1. 用户基础资料。
2. 用户主页信息。
3. 昵称、头像、简介、性别、地区等资料维护。
4. 用户状态，例如正常、禁用、注销。
5. 对其他服务提供用户轻量信息查询。

#### 数据归属

| 数据 | 说明 |
|---|---|
| 用户基础资料 | 昵称、头像、简介、地区等 |
| 用户状态 | 正常、禁用、注销 |
| 用户主页配置 | 背景图、展示设置等 |

#### 不负责

1. 不保存密码和登录凭证。
2. 不保存关注关系明细。
3. 不直接维护粉丝数、关注数最终计数。
4. 不负责用户发布的笔记内容。

#### 同步依赖

用户服务原则上依赖较少。主页聚合展示可以由 BFF 或网关聚合，也可以由用户服务调用计数服务、笔记服务获取轻量数据，具体在用户服务详细设计中确定。

#### 发布事件

| 事件 | 触发时机 |
|---|---|
| UserProfileUpdated | 用户资料更新 |
| UserStatusChanged | 用户状态变化 |

### 4.4 文件服务

服务名：`bluenote-file`

#### 职责

1. 生成对象存储上传凭证。
2. 保存文件元数据。
3. 管理图片、视频、头像等文件资源。
4. 文件访问地址转换。
5. 文件类型、大小、数量限制。
6. 后续接入图片压缩、视频转码、内容审核。

#### 数据归属

| 数据 | 说明 |
|---|---|
| 文件元数据 | fileId、bucket、objectKey、size、mimeType、ownerId |
| 文件业务绑定 | 文件和笔记、头像等业务对象的绑定关系，可按后续设计确定 |
| 上传记录 | 上传状态、上传时间、校验信息 |

#### 不负责

1. 不保存笔记正文。
2. 不判断笔记是否发布成功。
3. 不直接承载公网大流量文件下载，正式环境应交给 OSS/CDN。

#### 同步依赖

| 依赖 | 用途 |
|---|---|
| 对象存储 | 文件上传和读取 |

#### 发布事件

| 事件 | 触发时机 |
|---|---|
| FileUploaded | 文件上传确认成功 |
| FileDeleted | 文件删除或标记删除 |
| MediaProcessCompleted | 媒体处理完成，后续扩展 |

### 4.5 笔记发布服务

服务名：`bluenote-note`

#### 职责

1. 笔记发布。
2. 笔记编辑。
3. 笔记删除。
4. 笔记上下架或状态流转。
5. 笔记详情查询。
6. 笔记媒体资源绑定。
7. 作者笔记列表。
8. 笔记可见性控制。

#### 数据归属

| 数据 | 说明 |
|---|---|
| 笔记主体 | 标题、正文、作者、状态、发布时间 |
| 笔记媒体 | 图片、视频 fileId 列表和展示顺序 |
| 笔记标签 | 标签、话题，第一阶段可简单设计 |
| 笔记状态 | 草稿、已发布、删除、审核中、审核失败等 |

#### 不负责

1. 不负责点赞、收藏、评论计数的最终聚合。
2. 不负责 Feed 扩散。
3. 不负责排行榜计算。
4. 不负责评论内容存储。
5. 不负责文件二进制存储。

#### 同步依赖

| 依赖服务 | 用途 |
|---|---|
| bluenote-user | 校验作者状态、查询作者轻量信息 |
| bluenote-file | 校验文件归属、绑定媒体文件 |
| bluenote-counter | 查询笔记计数，用于详情页聚合 |

#### 发布事件

| 事件 | 触发时机 |
|---|---|
| NotePublished | 笔记发布成功 |
| NoteUpdated | 笔记编辑成功 |
| NoteDeleted | 笔记删除 |
| NoteVisibilityChanged | 可见性变化 |
| NoteStatusChanged | 审核或上下架状态变化 |

#### 订阅事件

| 事件 | 用途 |
|---|---|
| FileUploaded | 可选，用于确认文件状态 |
| UserStatusChanged | 作者被禁用时处理笔记状态，后续扩展 |

### 4.6 计数服务

服务名：`bluenote-counter`

#### 职责

1. 笔记点赞数。
2. 笔记收藏数。
3. 笔记评论数。
4. 用户关注数。
5. 用户粉丝数。
6. 用户获赞数。
7. 高频计数 Redis 聚合。
8. 计数异步落库。
9. 计数校准和重建。

#### 数据归属

| 数据 | 说明 |
|---|---|
| 计数快照 | 各业务对象当前计数 |
| 计数变更流水 | 可选，用于审计、修正和重放 |
| Redis 计数缓存 | 高频读写计数 |

#### 不负责

1. 不负责点赞、收藏、关注的行为明细最终归属。
2. 不判断用户是否已经点赞或收藏。
3. 不负责评论内容。
4. 不负责 Feed 排序。

#### 同步依赖

计数服务原则上不强依赖其他业务服务。对外提供批量计数查询接口。

#### 订阅事件

| 事件 | 来源 | 用途 |
|---|---|---|
| NotePublished | bluenote-note | 增加用户公开作品数 |
| NoteDeleted | bluenote-note | 减少用户公开作品数 |
| NoteVisibilityChanged | bluenote-note | 根据公开性变化调整用户作品数 |
| NoteStatusChanged | bluenote-note | 根据下架、恢复调整用户作品数 |
| NoteLiked | 互动行为来源服务，后续可归属笔记或互动模块 | 增加笔记点赞数 |
| NoteUnliked | 互动行为来源服务 | 减少笔记点赞数 |
| NoteCollected | 互动行为来源服务 | 增加笔记收藏数 |
| NoteUncollected | 互动行为来源服务 | 减少笔记收藏数 |
| CommentCreated | bluenote-comment | 增加笔记评论数 |
| CommentDeleted | bluenote-comment | 减少笔记评论数 |
| UserFollowed | bluenote-relation | 增加关注数和粉丝数 |
| UserUnfollowed | bluenote-relation | 减少关注数和粉丝数 |

#### 发布事件

| 事件 | 触发时机 |
|---|---|
| CounterChanged | 重要计数变化 |
| CounterRebuilt | 计数重建完成 |

### 4.7 用户关系服务

服务名：`bluenote-relation`

#### 职责

1. 关注。
2. 取关。
3. 关注列表。
4. 粉丝列表。
5. 关注关系查询。
6. 黑名单关系，后续扩展。
7. 好友关系，后续可由关注互相关系推导。

#### 数据归属

| 数据 | 说明 |
|---|---|
| 关注关系明细 | followerId、followeeId、状态、时间 |
| 关系变更记录 | 关注、取关流水 |
| 黑名单关系 | 后续扩展 |

#### 不负责

1. 不直接维护粉丝数和关注数快照。
2. 不负责 Feed 写入。
3. 不负责通知创建。
4. 不负责用户资料。

#### 同步依赖

| 依赖服务 | 用途 |
|---|---|
| bluenote-user | 校验用户是否存在、状态是否正常 |

#### 发布事件

| 事件 | 触发时机 |
|---|---|
| UserFollowed | 关注成功 |
| UserUnfollowed | 取关成功 |
| UserBlocked | 拉黑成功，后续扩展 |
| UserUnblocked | 取消拉黑，后续扩展 |

### 4.8 Timeline Feed 服务

服务名：`bluenote-feed`

#### 职责

1. 关注页 Feed 收件箱。
2. 新笔记扩散。
3. Feed 列表分页。
4. Feed 去重。
5. 大 V 用户推拉结合策略。
6. Feed 读模型维护。
7. Feed 失效和重建。

#### 数据归属

| 数据 | 说明 |
|---|---|
| Feed 收件箱 | 用户维度的笔记列表，Redis ZSet 为主 |
| Feed 持久化快照 | 可选，根据重建成本决定 |
| Feed 游标 | 用户分页游标或时间游标 |
| 作者分发策略 | 大 V 标记、推送阈值等 |

#### 不负责

1. 不保存笔记正文的最终版本。
2. 不负责关注关系明细。
3. 不负责排行榜。
4. 不负责推荐页 Feed，后续推荐系统单独设计。

#### 同步依赖

| 依赖服务 | 用途 |
|---|---|
| bluenote-note | 拉取笔记详情或批量笔记摘要 |
| bluenote-relation | 查询关注关系，重建 Feed 时使用 |
| bluenote-user | 获取作者轻量信息，必要时 |
| bluenote-counter | 批量获取计数，Feed 展示用 |

#### 订阅事件

| 事件 | 来源 | 用途 |
|---|---|---|
| NotePublished | bluenote-note | 向粉丝 Feed 扩散 |
| NoteDeleted | bluenote-note | 从 Feed 中移除或标记不可见 |
| NoteVisibilityChanged | bluenote-note | 更新 Feed 可见性 |
| UserFollowed | bluenote-relation | 可选，补充最近笔记 |
| UserUnfollowed | bluenote-relation | 可选，清理对应作者笔记 |

#### 发布事件

| 事件 | 触发时机 |
|---|---|
| FeedDelivered | 笔记成功写入用户 Feed，必要时 |
| FeedRebuilt | 用户 Feed 重建完成 |

### 4.9 排行榜服务

服务名：`bluenote-rank`

1. 本周热门笔记榜。

2. 不负责点赞收藏行为明细。
3. 不负责 Feed 个性化排序。
4. 不负责搜索。

#### 同步依赖

| 依赖服务 | 用途 |
|---|---|

| 事件 | 来源 | 用途 |

| NoteVisibilityChanged | bluenote-note | 公开性变化导致贡献启停 |

### 4.10 评论服务

#### 职责

2. 发布二级评论。
3. 删除评论。
4. 评论列表分页。
5. 二级评论分页。
6. 评论点赞，是否独立为互动服务后续评估。
7. 评论状态管理。

#### 数据归属

| 数据 | 说明 |

| 事件 | 触发时机 |
|---|---|

| 事件 | 用途 |
|---|---|
| CouponSeckillAccepted | RocketMQ 内部异步下单 |
| OrderTimeoutCheck | RocketMQ 延时消息触发关单检查 |

#### 订单超时关单原则

1. 免费神券抢到后直接发券，不需要关单。
2. 付费神券订单创建后发送 RocketMQ 延时消息。
3. 消费时查询订单当前状态。
4. 只有订单仍为待支付且已过期，才允许关闭。
5. 关闭订单使用 CAS 更新。
6. 关闭后回补 MySQL 库存，并通过 Lua 回补 Redis 库存和用户参与标记。
7. 定时任务扫描数据库兜底。

## 5. 互动行为归属说明

BlueNote 中点赞、收藏这类互动行为比较特殊，既影响计数，也影响通知、排行榜、个人记录。

第一阶段建议采用以下归属：

| 行为 | 明细归属 | 计数归属 | 事件 |
|---|---|---|---|
| 笔记点赞 | bluenote-note 或后续 bluenote-interaction | bluenote-counter | NoteLiked / NoteUnliked |
| 笔记收藏 | bluenote-note 或后续 bluenote-interaction | bluenote-counter | NoteCollected / NoteUncollected |
| 评论点赞 | bluenote-comment | bluenote-counter 或评论服务内计数 | CommentLiked / CommentUnliked |
| 关注 | bluenote-relation | bluenote-counter | UserFollowed / UserUnfollowed |

建议暂时不单独拆 `bluenote-interaction` 服务，避免服务数量过多。点赞和收藏明细可以先放在笔记服务内，等互动规模变大后再拆出互动服务。

拆分条件：

1. 点赞、收藏、浏览、转发等互动行为明显增多。
2. 互动数据需要单独支撑推荐、风控、数据分析。
3. 笔记服务写压力明显受到互动行为影响。

## 6. 服务依赖关系

### 6.1 同步调用关系

```text
移动端
  |
  v
bluenote-gateway
  |
  +--> bluenote-auth
  +--> bluenote-user
  +--> bluenote-note
  +--> bluenote-comment
  +--> bluenote-relation
  +--> bluenote-feed
  +--> bluenote-rank
  +--> bluenote-im
  +--> bluenote-order

bluenote-note
  +--> bluenote-user
  +--> bluenote-file
  +--> bluenote-counter

bluenote-comment
  +--> bluenote-note
  +--> bluenote-user

bluenote-feed
  +--> bluenote-note
  +--> bluenote-counter
  +--> bluenote-user

bluenote-rank
  +--> bluenote-note
  +--> bluenote-counter

bluenote-im
  +--> bluenote-auth
  +--> bluenote-user
  +--> bluenote-relation
```

同步调用原则：

1. 只在需要即时结果时调用。
2. 不允许形成循环强依赖。
3. 读接口优先支持批量查询，避免 N+1 调用。
4. 写操作尽量只写本服务数据库。
5. 调用失败时要有降级策略，尤其是 Feed 和排行榜展示场景。

### 6.2 异步事件关系

```text
bluenote-note
  -- NotePublished --> bluenote-feed
  -- NotePublished --> bluenote-rank
  -- NotePublished --> bluenote-notification

bluenote-relation
  -- UserFollowed --> bluenote-counter
  -- UserFollowed --> bluenote-feed
  -- UserFollowed --> bluenote-notification

bluenote-comment
  -- CommentCreated --> bluenote-counter
  -- CommentCreated --> bluenote-notification

互动行为来源服务
  -- NoteLiked --> bluenote-counter
  -- NoteLiked --> bluenote-notification

bluenote-counter
  -- CounterChanged --> bluenote-rank

bluenote-order
  -- CouponSeckillAccepted --> 订单服务内部异步下单消费者
  -- OrderCreated --> 通知、运营后续
  -- CouponIssued --> 通知、运营后续
  -- OrderTimeoutCheck --> 订单服务内部关单消费者
  -- OrderPaid --> 通知、运营后续
  -- OrderClosed --> 通知、运营后续
  -- PushSendRequested --> bluenote-push

bluenote-im
  -- ImMessageSent --> IM 领域事件订阅方
  -- PushSendRequested --> bluenote-push
```

异步事件原则：

1. 事件表示已经发生的事实。
2. 消费者不能假设事件只投递一次。
3. 消费者必须幂等。
4. 消费失败必须可重试。
5. 核心事件需要有补偿或重放能力。

## 8. API 边界

### 8.1 外部 API

外部 API 只允许移动端通过 `bluenote-gateway` 访问。

示例：

```text
/api/auth/**
/api/users/**
/api/notes/**
/api/comments/**
/api/relations/**
/api/feed/**
/api/rank/**
/api/im/**
/api/order/**
```

### 8.2 内部 API

内部 API 用于服务间调用，不直接暴露给移动端。

示例：

```text
/internal/users/batch
/internal/notes/batch
/internal/counters/batch
/internal/relations/check
```

内部 API 要求：

1. 只能内网访问。
2. 需要服务间认证或网关隔离。
3. 支持批量查询。
4. 返回结构稳定，避免频繁破坏消费者。

## 9. 公共模块划分

建议保留 `bluenote-common`，但必须限制职责，避免变成大杂烩。

允许放入：

1. 通用响应结构。
2. 通用错误码基础定义。
3. 通用异常类型。
4. 分页对象。
5. traceId 工具。
6. 时间工具。
7. 基础安全上下文对象。
8. 通用事件基础结构。

不允许放入：

1. 具体业务逻辑。
2. 具体 Mapper。
3. 具体业务表实体。
4. 跨服务直接访问数据库的工具。
5. 会导致服务强耦合的巨型 DTO。

公共模块原则：

1. 只能放稳定基础能力。
2. 业务 DTO 优先由服务自己维护。
3. 事件契约可以单独抽出 `bluenote-event-contract`，后续评估。

## 10. 阶段性部署策略

### 10.1 逻辑拆分

逻辑上仍按独立服务设计：

```text
gateway / auth / user / file / note / counter / relation / feed / rank / comment / im / notification / order
```

### 10.2 第一阶段物理部署建议

考虑 2 核 4G 云服务器资源，第一阶段可以减少 Java 进程数量。

推荐物理部署分组：

| 物理应用 | 包含逻辑服务 | 原因 |
|---|---|---|
| bluenote-gateway-app | gateway | 网关独立，便于限流、路由和公网入口管理 |
| bluenote-member-app | auth、user | 登录和用户资料关联紧密，初期流量较小 |
| bluenote-content-app | note、file、comment | 内容发布、文件元数据、评论可先合并部署 |
| bluenote-social-app | relation、counter、feed、rank、notification | 社交互动和分发事件联系紧密 |
| bluenote-im-app | im | 消息顺序、消息链和写库事务模型特殊，应独立进程 |
| bluenote-order-app | order | 神券秒杀涉及库存、支付、关单和补偿，交易域独立 |

注意：合并部署不等于合并业务边界。即使放在同一个应用内，也应保持模块包结构、数据 schema 和接口契约清晰。

### 10.3 后续拆分条件

当出现以下情况时，应拆成独立服务进程：

1. 某个模块 CPU、内存、数据库连接或 MQ 消费压力明显高于其他模块。
2. 某个模块发布频率明显更高。
3. 某个模块故障会影响其他核心链路。
4. 某个模块需要独立扩容。
5. 某个模块需要独立安全隔离。

优先拆分顺序：

1. IM 服务独立。
2. Feed 服务独立。
3. 计数服务独立。
4. 评论服务独立。
5. 文件服务独立。
6. 排行榜服务独立。

1. 一个服务只写自己的 schema。
2. 不允许跨 schema join。
3. 需要跨服务数据时，优先接口查询或事件冗余。
4. 后续物理拆库时，schema 边界可以直接迁移。

## 12. RocketMQ 消费组建议

| Consumer Group | 订阅 Topic | 归属服务 |
|---|---|---|
| counter-consumer-group | note-event、interaction-event、relation-event、comment-event | bluenote-counter |
| counter-delta-updater-group | counter-delta-event | bluenote-counter |
| feed-consumer-group | note-event、relation-event | bluenote-feed |
| feed-fanout-executor-group | feed-fanout-task-event | bluenote-feed |
| rank-consumer-group | note-event、counter-event | bluenote-rank |
| notification-consumer-group | interaction-event、relation-event、comment-event | bluenote-notification |
| push-consumer-group | push-request-event | bluenote-push |
| im-message-consumer-group | im-message-event | bluenote-im |
| order-seckill-consumer-group | order-seckill-task-event | bluenote-order |
| order-timeout-consumer-group | order-timeout-event | bluenote-order |

消费组原则：

1. 一个服务一个或多个消费组。
2. 不同业务语义不要混用同一个消费组。
3. 消费者必须记录幂等处理结果。
4. 消费失败要区分可重试错误和不可重试错误。

## 13. 典型业务链路

### 13.1 用户发布笔记

```text
移动端
  -> 网关
  -> 笔记服务
  -> 校验用户状态
  -> 校验文件归属
  -> 保存笔记
  -> 发布 NotePublished 事件
  -> Feed 服务扩散
  -> 排行榜服务加入候选
  -> 通知服务后续按场景生成通知
  -> 如需提醒则请求推送服务投递
```

### 13.2 用户关注作者

```text
移动端
  -> 网关
  -> 用户关系服务
  -> 校验双方用户状态
  -> 写入关注关系
  -> 发布 UserFollowed 事件
  -> 计数服务更新关注数和粉丝数
  -> Feed 服务可补充作者近期笔记
  -> 通知服务生成关注通知
```

### 13.3 用户评论笔记

```text
移动端
  -> 网关
  -> 评论服务
  -> 校验笔记可评论
  -> 保存评论
  -> 发布 CommentCreated 事件
  -> 计数服务更新评论数
  -> 计数服务发布 CounterChanged
  -> 排行榜服务根据 CounterChanged 更新热度分
  -> 通知服务生成评论通知
```

### 13.4 IM 发送消息

```text
移动端 A
  -> 网关
  -> IM 服务发送消息
  -> 鉴权和会话校验
  -> 消息入库
  -> 发布 ImMessageSent 事件
  -> 发布 PushSendRequested
  -> 推送服务对用户 B 在线 WebSocket 或离线系统 Push 提醒
  -> 用户 B 打开会话后从 IM 服务拉取消息详情
  -> 客户端 ACK 后由 IM 服务更新消息状态
```

### 13.5 神券秒杀订单

```text
移动端
  -> 网关
  -> 订单服务
  -> 获取秒杀 token
  -> Redis Lua 校验 token、库存和用户限购
  -> 预扣 Redis 库存
  -> 发送 CouponSeckillAccepted 到 RocketMQ
  -> 返回抢券处理中
  -> 订单消费者扣 MySQL 库存并创建订单
  -> 免费券直接发放 user_coupon
  -> 付费券创建 WAIT_PAY 订单并发送 OrderTimeoutCheck 延时消息
  -> 支付成功后发券，或超时后 CAS 关单并回补库存
  -> 发布 OrderCreated、OrderPaid、CouponIssued、OrderClosed 等事件
```

## 14. 不建议第一阶段拆出的服务

以下能力暂不建议第一阶段独立成服务：

| 能力 | 原因 | 第一阶段归属 |
|---|---|---|
| 互动服务 | 点赞收藏规模未验证，单独拆分会增加复杂度 | 笔记服务，后续可拆 |
| 搜索服务 | 暂不引入 Elasticsearch | 笔记服务简单搜索或后续独立 |
| 审核服务 | 内容审核流程未展开 | 文件服务/笔记服务预留状态 |
| 推荐服务 | 推荐系统不是第一阶段目标 | 暂不实现，只保留行为事件 |
| 管理后台服务 | 当前只讨论移动端和后端 | 后续单独设计 |
| 支付服务 | 第一阶段只有神券付费订单，支付复杂度可控 | 订单服务内部模块，后续可拆 |

## 16. 待决策问题

当前仍需后续确认的问题：

1. 用户主页聚合由用户服务完成，还是由网关/BFF 聚合。
2. Feed 收件箱是否需要 MySQL 持久化，还是只依赖 Redis 并支持重建。
3. 神券付费订单第一阶段接真实支付渠道，还是先使用测试支付通道。

已在后续设计或 ADR 中明确的事项：

2. 文件服务第一阶段开发、测试、正式环境统一使用 MinIO，后续根据容量、带宽和可用性要求再迁移云 OSS/COS/R2。
3. 第一阶段新增独立推送与实时投递服务，统一承接通知、IM、订单等服务的在线和离线提醒。
4. 通知服务第一阶段支持互动、关注、系统通知；点赞和收藏聚合，评论、回复、关注保留明细，通知未读数由通知服务维护。
5. IM 服务第一阶段实现企业级单聊，支持文本、图片、笔记卡片消息；群聊、撤回、完整已读回执和 WebSocket 上行协议后续扩展。
6. 订单服务第一阶段聚焦神券秒杀订单，支持 Redis Lua 预扣、RocketMQ 异步下单、付费券延时关单和免费券直接发券。

这些问题不影响当前总体服务边界，但会影响后续服务详细设计。
