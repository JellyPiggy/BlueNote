# BlueNote 安全与权限方案

版本：v0.1  
状态：草案  
更新时间：2026-06-03

## 1. 文档目标

本文档定义 BlueNote 的安全与权限总体方案，包括认证、授权、Token、设备会话、网关安全、服务间访问、数据权限、接口防护、文件上传安全、敏感数据保护和审计策略。

本文不展开每个接口的权限规则。具体接口级权限会在各服务详细设计和接口文档中补充。

## 2. 安全目标

BlueNote 第一阶段安全目标：

1. 用户身份可信。
2. 接口访问可控。
3. 用户只能访问和修改自己有权限的数据。
4. 敏感数据不明文泄露。
5. 文件上传不能成为攻击入口。
6. 服务间调用有边界。
7. 关键行为可审计。
8. 异常访问可限流和封禁。

## 3. 信任边界

### 3.1 外部不可信

以下输入全部视为不可信：

1. 移动端请求参数。
2. Header。
3. Token。
4. 文件名。
5. 文件 MIME 类型。
6. IP 地址。
7. 设备信息。
8. 第三方回调内容。

服务端必须校验，不信任客户端声明。

### 3.2 网关是外部入口

移动端只能访问 API 网关。

网关职责：

1. Access Token 校验。
2. 基础限流。
3. 黑名单拦截。
4. traceId 注入。
5. 用户上下文透传。
6. 请求大小限制。
7. 基础安全 Header。

业务服务不直接暴露公网。

### 3.3 内部服务也不是完全可信

内部服务间调用仍需要安全边界。

要求：

1. 内部接口只在内网访问。
2. 内部接口和外部接口路径分离。
3. 内部服务不能伪造用户越权操作。
4. 关键内部调用需要服务身份标识。

## 4. 认证方案

### 4.1 登录方式

第一阶段建议支持：

1. 手机号 + 验证码登录。
2. 手机号 + 密码登录，后续可选。
3. 账号密码登录，开发测试可用。

后续扩展：

1. 微信登录。
2. Apple 登录。
3. OAuth 第三方登录。

### 4.2 Token 方案

采用 Access Token + Refresh Token。

| Token | 用途 | 生命周期 |
|---|---|---|
| Access Token | 访问业务接口 | 短有效期 |
| Refresh Token | 刷新 Access Token | 长有效期 |

建议：

1. Access Token 使用 JWT。
2. Refresh Token 使用随机高熵字符串。
3. Refresh Token 服务端保存哈希或可撤销标识。
4. Access Token 不保存敏感信息。
5. Token 绑定用户、设备和登录会话。

### 4.3 Access Token 内容

Access Token 建议包含：

```json
{
  "sub": "userId",
  "deviceId": "deviceId",
  "sessionId": "sessionId",
  "roles": ["USER"],
  "iat": 1780000000,
  "exp": 1780003600
}
```

不建议包含：

1. 手机号。
2. 密码哈希。
3. Refresh Token。
4. 支付信息。
5. 大量用户资料。

### 4.4 Refresh Token 轮换

刷新 Access Token 时建议轮换 Refresh Token。

流程：

```text
移动端提交 Refresh Token
  -> 登录服务校验 token 哈希和设备会话
  -> 旧 Refresh Token 失效
  -> 生成新的 Access Token
  -> 生成新的 Refresh Token
  -> 返回移动端
```

好处：

1. 降低 Refresh Token 泄露后的长期风险。
2. 支持设备级登出。
3. 支持异常刷新检测。

### 4.5 登出

登出要求：

1. 当前设备 Refresh Token 失效。
2. 可选将 Access Token 加入短期黑名单。
3. 清理设备会话状态。
4. 发布 UserLoggedOut 事件，必要时。

Access Token 有效期较短时，可以不强依赖全量黑名单；高风险场景需要黑名单。

## 5. 设备会话

### 5.1 设备标识

移动端需要生成或获取设备 ID。

设备 ID 要求：

1. 不使用容易变化的临时值。
2. 不直接使用敏感硬件标识。
3. 用户卸载重装后变化可接受。
4. 服务端不完全信任设备 ID。

### 5.2 会话管理

登录服务保存设备会话。

会话字段建议：

1. sessionId
2. userId
3. deviceId
4. deviceName
5. appVersion
6. loginIp
7. loginAt
8. refreshTokenHash
9. status
10. lastActiveAt

### 5.3 多设备策略

第一阶段建议允许多设备登录。

后续可支持：

1. 查看已登录设备。
2. 踢出指定设备。
3. 密码修改后全部设备失效。
4. 异常设备二次验证。

## 6. 授权模型

### 6.1 角色模型

移动端第一阶段角色简单：

| 角色 | 说明 |
|---|---|
| GUEST | 未登录用户 |
| USER | 普通登录用户 |
| DISABLED | 被禁用用户 |
| ADMIN | 后续管理端使用 |

当前移动端主要使用 `USER`。

### 6.2 接口权限

接口分为：

| 类型 | 示例 | 权限 |
|---|---|---|
| 公开接口 | 登录、注册、公开笔记详情 | 不需要登录或部分限流 |
| 登录接口 | 发布笔记、点赞、评论、关注 | 必须登录 |
| 本人接口 | 编辑资料、查看设备、查看订单 | 用户只能访问自己的数据 |
| 作者接口 | 编辑笔记、删除笔记 | 只能作者操作 |
| 管理接口 | 审核、封禁、运营配置 | 后续管理端权限 |

### 6.3 数据权限

数据权限必须在业务服务校验，不能只依赖前端隐藏按钮。

规则示例：

1. 用户只能修改自己的资料。
2. 用户只能删除自己的笔记。
3. 笔记作者可以按规则管理自己笔记下的评论。
4. 用户只能取消自己的点赞和收藏。
5. 用户只能查看自己的 IM 会话。
6. 用户只能查看自己的订单。
7. 被拉黑后私信和评论能力按业务规则限制。

### 6.4 状态权限

用户状态会影响权限。

| 状态 | 允许行为 |
|---|---|
| NORMAL | 正常使用 |
| DISABLED | 禁止登录或禁止发内容 |
| MUTED | 禁止发布笔记、评论、私信 |
| DELETED | 不允许登录 |

具体状态在用户服务详细设计中确定。

## 7. 网关安全

### 7.1 Token 校验

网关校验 Access Token。

通过后向下游透传：

```text
X-User-Id
X-Device-Id
X-Session-Id
X-Trace-Id
```

要求：

1. 下游服务不信任外部传入的用户 Header。
2. 用户 Header 只能由网关注入。
3. 反向代理需要清理客户端伪造的内部 Header。

### 7.2 限流

网关限流维度：

1. IP。
2. userId。
3. deviceId。
4. API 路径。
5. 登录手机号。

高风险接口：

1. 登录。
2. 注册。
3. 验证码发送。
4. 刷新 Token。
5. 上传凭证申请。
6. 发布笔记。
7. 评论。
8. 私信发送。

### 7.3 请求限制

要求：

1. 限制请求体大小。
2. 限制 Header 大小。
3. 限制上传文件大小。
4. JSON 参数做长度限制。
5. 防止超大数组参数。

### 7.4 安全 Header

H5 场景建议配置：

1. HSTS。
2. X-Content-Type-Options。
3. X-Frame-Options 或 CSP frame-ancestors。
4. Content-Security-Policy，后续根据 H5 场景配置。

移动 App API 主要关注 HTTPS 和接口安全。

## 8. 服务间安全

### 8.1 内部接口隔离

内部接口路径建议：

```text
/internal/**
```

规则：

1. 外部网关不暴露内部接口。
2. 内部接口只允许服务网段访问。
3. 管理接口和内部接口分离。
4. 内部接口日志记录调用方服务名。

### 8.2 服务身份

第一阶段可通过网络隔离 + 内部 Header 控制。

后续演进：

1. 服务间签名。
2. mTLS。
3. API 网关内外网分离。
4. Service Mesh，后续 Kubernetes 阶段评估。

### 8.3 用户上下文传递

服务间调用如果代表用户执行操作，需要传递用户上下文。

要求：

1. 明确调用是用户行为还是系统行为。
2. 系统行为不能伪造成普通用户行为。
3. 审计日志记录真实触发来源。

## 9. 密码与验证码安全

### 9.1 密码存储

要求：

1. 不存明文密码。
2. 使用强哈希算法，例如 BCrypt、Argon2 或 Spring Security 推荐方案。
3. 每个密码独立 salt。
4. 支持密码哈希算法升级。

### 9.2 登录失败控制

策略：

1. 同一账号连续失败限制。
2. 同一 IP 高频失败限制。
3. 同一设备 高频失败限制。
4. 高风险时需要验证码。

### 9.3 短信验证码

要求：

1. 验证码短有效期。
2. 同手机号发送频率限制。
3. 同 IP 发送频率限制。
4. 验证失败次数限制。
5. 验证码不写入日志。
6. 测试环境和正式环境短信通道隔离。

## 10. 文件上传安全

### 10.1 上传限制

文件服务必须限制：

1. 文件类型。
2. 文件大小。
3. 文件数量。
4. 用户上传频率。
5. 单篇笔记媒体数量。
6. 文件扩展名和 MIME 类型一致性。

### 10.2 文件名处理

规则：

1. 不信任客户端文件名。
2. 服务端生成 objectKey。
3. objectKey 不包含用户原始文件名。
4. 防止路径穿越。
5. 防止覆盖已有文件。

### 10.3 访问控制

建议：

1. 上传使用临时凭证或预签名 URL。
2. 私有文件使用签名 URL。
3. 公开图片可以走 CDN。
4. 删除文件采用延迟物理删除。

### 10.4 内容安全

后续需要接入：

1. 图片审核。
2. 文本审核。
3. 视频审核。
4. 违规内容下架。
5. 用户处罚策略。

第一阶段至少要预留笔记状态和审核状态。

## 11. 接口防护

### 11.1 参数校验

要求：

1. Controller 层使用 Jakarta Validation。
2. 字符串长度限制。
3. 数组大小限制。
4. 枚举值白名单。
5. 数字范围限制。
6. 游标和分页大小限制。

### 11.2 幂等

需要幂等的接口：

1. 发布笔记。
2. 点赞。
3. 取消点赞。
4. 收藏。
5. 取消收藏。
6. 关注。
7. 取关。
8. 发布评论。
9. 发送 IM 消息。
10. 创建订单。
11. 支付回调。

幂等方式：

1. 唯一索引。
2. clientRequestId。
3. 业务状态机。
4. Redis 短期幂等 key。

### 11.3 防重放

高风险接口后续可增加：

1. 时间戳。
2. nonce。
3. 请求签名。
4. 设备绑定。

第一阶段重点保护登录、刷新 Token、支付回调。

## 12. 敏感数据保护

### 12.1 数据库存储

敏感字段：

1. 手机号。
2. 密码哈希。
3. Refresh Token 哈希。
4. 第三方账号标识。
5. 支付流水号。
6. IP。
7. 设备信息。

要求：

1. 密码强哈希。
2. Refresh Token 不明文保存。
3. 手机号展示脱敏。
4. 支付相关数据限制访问。
5. 数据库账号权限最小化。

### 12.2 日志脱敏

日志不得打印：

1. 密码。
2. 验证码。
3. 完整手机号。
4. Refresh Token。
5. Access Token。
6. 支付密钥。
7. 对象存储密钥。

可以打印：

1. userId。
2. traceId。
3. eventId。
4. 脱敏手机号。
5. 业务对象 ID。

### 12.3 配置密钥

密钥不得提交到 Git。

包括：

1. 数据库密码。
2. Redis 密码。
3. JWT 私钥。
4. 对象存储 AK/SK。
5. 短信平台密钥。
6. 支付平台密钥。

第一阶段使用环境变量或宿主机安全文件管理，后续可接入专门密钥管理服务。

## 13. 支付和订单安全，后续扩展

订单功能上线时必须补充：

1. 支付回调签名校验。
2. 回调幂等。
3. 金额服务端校验。
4. 订单状态机。
5. 支付流水唯一约束。
6. 订单超时关单和支付成功并发控制。
7. 退款权限。
8. 对账任务。

原则：

1. 不信任客户端传入金额。
2. 不信任未验签回调。
3. 不通过前端支付成功结果直接改订单状态。
4. 订单状态以服务端和支付渠道回调为准。

## 14. IM 安全

IM 服务要求：

1. WebSocket 建连必须鉴权。
2. 连接绑定 userId、deviceId、sessionId。
3. 发送消息时校验会话权限。
4. 陌生人私信策略在关系服务中查询。
5. 拉黑后限制消息投递。
6. 消息内容长度限制。
7. 消息发送频率限制。
8. 客户端 ACK 必须校验消息归属。

后续需要补充：

1. 图片消息安全。
2. 消息撤回权限。
3. 已读回执权限。
4. 敏感词和内容审核。

## 15. 审计日志

需要审计的行为：

1. 登录成功和失败。
2. 登出。
3. Refresh Token 刷新。
4. 修改密码。
5. 修改手机号。
6. 设备踢出。
7. 发布、删除笔记。
8. 删除评论。
9. 拉黑、关注、取关。
10. 订单支付、关闭、取消。
11. 管理端操作，后续。

审计日志字段：

1. auditId
2. userId
3. action
4. targetType
5. targetId
6. ip
7. deviceId
8. traceId
9. result
10. occurredAt

审计日志不能依赖普通应用日志替代。关键审计数据建议落库。

## 16. 安全告警

建议告警：

1. 登录失败率异常。
2. 验证码发送异常。
3. Token 刷新异常。
4. 单用户高频发布。
5. 单用户高频评论。
6. 单用户高频私信。
7. 文件上传异常。
8. 支付回调验签失败，后续。
9. 管理端登录异常，后续。

## 17. 风险与应对

| 风险 | 说明 | 应对 |
|---|---|---|
| Token 泄露 | 移动端环境无法完全避免 | Access Token 短有效期、Refresh Token 轮换 |
| Header 伪造 | 客户端伪造 X-User-Id | 反向代理清理 Header，只信任网关注入 |
| 越权访问 | 用户访问他人数据 | 服务层数据权限校验 |
| 上传攻击 | 恶意文件、超大文件 | 类型、大小、频率限制，后续内容审核 |
| 短信轰炸 | 验证码接口被刷 | 多维限流、验证码、黑名单 |
| 日志泄露 | 打印 token 和手机号 | 日志脱敏规范 |
| 内部接口暴露 | internal 接口被公网访问 | 网络隔离、网关路由限制 |
| 支付伪造 | 客户端伪造支付成功 | 服务端验签和回调幂等 |
