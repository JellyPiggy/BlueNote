# BlueNote 落地实施方案

## 1. 文档目标

本文用于把 BlueNote 已完成的总体方案、服务划分方案、基础设施方案、数据存储方案、消息队列方案、安全方案、可观测性方案和各服务详细设计，收敛成可以进入开发、联调、部署和验收的实施路线。

本文不再重新设计业务边界，不替代已有服务方案。它重点回答：

1. 当前方案离真实落地还差哪些工程产物。
2. 后端、移动端、基础设施、数据库、MQ、可观测性应该按什么顺序落地。
3. 每个阶段的交付物和验收标准是什么。
4. 哪些内容必须先冻结，哪些内容可以边开发边补齐。
5. 在 2 核 4G 云服务器和本地旧电脑的资源约束下，如何避免一开始就做成无法运行的重方案。

## 3. 落地标准

### 3.1 什么叫可以开工

一个服务进入编码前，至少应满足以下条件：

| 类别 | 必须具备的内容 |
|---|---|
| 服务边界 | 本服务负责什么、不负责什么、依赖哪些服务已经明确 |
| 接口契约 | 移动端接口、内部接口、请求响应、错误码、分页结构明确 |
| 数据设计 | MySQL DDL、索引、状态枚举、唯一约束、审计字段明确 |
| Redis 设计 | Key、数据结构、TTL、写入时机、失效和重建策略明确 |
| MQ 设计 | 生产事件、消费事件、payload、幂等键、重试、死信处理明确 |
| 安全规则 | 鉴权、权限、限流、幂等、防重放、敏感字段脱敏明确 |
| 可观测性 | 核心日志、指标、告警点、traceId、eventId 规范明确 |
| 测试范围 | 正常流程、异常流程、幂等、一致性、权限、缓存、MQ 测试点明确 |

不是所有服务都必须一次性细化到 100% 才能开始项目，但同一个迭代要开发的服务必须先完成以上收口。

### 3.2 什么叫可以联调

一个功能链路进入联调前，必须满足：

1. 网关路由已配置。
2. 移动端接口路径和后端 OpenAPI 契约一致。
3. MySQL 初始化脚本可重复执行。
4. Redis Key 前缀、TTL 和环境隔离已配置。
5. RocketMQ topic 和 consumer group 已创建或可自动初始化。
6. Nacos 配置已按 local、test、prod 分环境拆分。
7. 服务健康检查可用。
8. 核心日志包含 `traceId`。
9. 失败场景有明确错误码。
10. 本地 Docker Compose 环境可以启动依赖组件。

### 3.3 什么叫可以上线

第一阶段正式环境上线前，必须满足：

1. 云服务器和本地旧电脑的部署边界明确，用户主链路不依赖本地旧电脑。
2. MySQL、Redis、RocketMQ、MinIO 已有基础备份方案。
3. Nginx 或 Caddy 已配置 HTTPS。
4. 生产环境密钥不写入 Git。
5. 管理端、Grafana、MinIO Console 等管理入口不直接公网裸露。
6. 核心接口具备限流和登录态校验。
7. 订单支付回调具备验签、金额校验和幂等。
8. 推送、IM、MQ 消费失败有日志和告警。
9. 有发布前检查清单和回滚步骤。
10. 至少完成一次恢复演练：MySQL 备份恢复、MinIO 文件恢复、Redis 重建演练。

## 4. 总体实施路线

### 4.1 实施原则

BlueNote 的实施不按“一个服务完全做完再做下一个服务”的方式推进，而是按业务链路和基础能力分阶段落地。

原因是：

1. 登录、用户、文件、笔记是后续所有能力的基础。
2. 评论、关系、计数、Feed、排行榜之间有大量事件协作，适合在同一阶段联调。
3. 推送、通知、IM、订单都有较强异步链路，需要在 MQ 和可观测性基础稳定后开发。
4. Docker Compose、Nacos、MySQL、Redis、RocketMQ、MinIO 必须早于业务服务准备好。

### 4.2 推荐阶段

| 阶段 | 名称 | 目标 | 主要交付物 |
|---|---|---|---|
| M0 | 工程基础阶段 | 建立后端、移动端、部署目录和公共规范 | 单仓多模块、公共包、统一响应、异常、日志、配置、CI 基础 |
| M1 | 本地基础设施阶段 | 本地能启动完整依赖 | Docker Compose、MySQL、Redis、RocketMQ、Nacos、MinIO、Prometheus、Loki |
| M2 | 账号与用户基础阶段 | 完成登录、用户资料、文件上传基础闭环 | 登录服务、用户服务、文件服务、Token、MinIO 上传 |
| M3 | 内容发布阶段 | 完成笔记发布、详情、评论、互动基础链路 | 笔记、评论、点赞、收藏、笔记状态、事件发送 |
| M4 | 社交分发阶段 | 完成关注关系、计数、Feed、排行榜 | 关系、计数聚合、关注页 Feed、本周热门榜、本年创作者成长榜 |
| M5 | 触达与 IM 阶段 | 完成通知中心、推送投递、单聊 | 站内通知、Push 请求、WebSocket 下发、IM 单聊 |
| M6 | 神券秒杀订单阶段 | 完成神券秒杀、异步下单、支付状态和超时关单 | Redis Lua、RocketMQ 异步下单、订单状态机、用户券 |
| M7 | 生产部署阶段 | 完成 test/prod 部署和基础运维 | Nginx/Caddy、HTTPS、Nacos prod 配置、备份、告警、发布回滚 |
| M8 | 验收与压测阶段 | 做真实可用性验证 | 接口测试、链路压测、MQ 堆积演练、故障恢复演练 |

阶段之间可以局部并行，但基础设施、公共规范和契约收口不能跳过。

## 5. 代码仓库与工程结构

### 5.1 推荐仓库策略

第一阶段建议采用一个总仓库，内部拆分后端、移动端、部署和文档：

```text
bluenote/
  backend/
  mobile/
  deploy/
  docs/
  scripts/
```

原因：

1. 项目规模还不适合拆成多个仓库。
2. 前后端、部署、数据库脚本需要频繁协同。
3. 单仓库更容易保证接口契约、DDL、配置和代码同步演进。
4. 后续如果某个模块变大，可以再拆独立仓库。

### 5.2 后端多模块结构

后端建议使用 Maven 多模块：

```text
backend/
  pom.xml
  bluenote-dependencies/
  bluenote-common/
    bluenote-common-core/
    bluenote-common-web/
    bluenote-common-security/
    bluenote-common-mybatis/
    bluenote-common-redis/
    bluenote-common-mq/
    bluenote-common-observability/
  bluenote-gateway-app/
  bluenote-member-app/
    bluenote-auth-module/
    bluenote-user-module/
  bluenote-content-app/
    bluenote-file-module/
    bluenote-note-module/
    bluenote-comment-module/
  bluenote-social-app/
    bluenote-relation-module/
    bluenote-counter-module/
    bluenote-feed-module/
    bluenote-rank-module/
    bluenote-notification-module/
    bluenote-push-module/
  bluenote-im-app/
  bluenote-order-app/
```

说明：

1. 物理应用按 `02-服务划分方案.md` 合并，降低 2 核 4G 环境的 JVM 进程数量。
2. 逻辑模块仍保持独立包、独立表、独立接口和独立事件契约。
3. `bluenote-common-*` 只放通用技术能力，不放业务逻辑。
4. 合并部署不代表跨模块直接访问对方数据库表。

### 5.3 移动端结构

移动端建议使用 uni-app + Vue 3 + TypeScript：

```text
mobile/
  src/
    api/
    pages/
    components/
    stores/
    router/
    utils/
    assets/
    styles/
    types/
```

移动端落地原则：

1. 所有接口调用集中在 `src/api/`，不在页面中散写请求路径。
2. 登录态、用户信息、设备信息使用 Pinia 管理。
3. 图片上传统一调用文件服务预签名或上传接口。
4. IM、推送、通知相关状态独立封装，不和普通页面状态混在一起。
5. API 类型优先从 OpenAPI 或统一类型文件生成，减少前后端字段不一致。

### 5.4 部署目录结构

```text
deploy/
  compose/
    compose.base.yml
    compose.local.yml
    compose.test.yml
    compose.prod.yml
  env/
    local.env
    test.env
    prod.env.example
  nacos/
    local/
    test/
    prod/
  nginx/
  prometheus/
  grafana/
  loki/
  rocketmq/
  minio/
```

`prod.env` 不提交仓库，只在服务器上维护。`prod.env.example` 只保留变量名、格式和示例值。

## 6. 契约收口计划

### 6.1 API 契约

API 契约建议按以下顺序收口：

| 顺序 | 范围 | 说明 |
|---|---|---|
| 1 | 全局响应结构 | `code`、`message`、`data`、`traceId`、分页结构 |
| 2 | 全局错误码 | 系统、认证、权限、参数、幂等、限流、下游失败 |
| 3 | 登录与用户接口 | 注册、登录、刷新 Token、登出、用户资料、主页 |
| 4 | 文件与笔记接口 | 上传、文件确认、笔记发布、编辑、详情、列表 |
| 5 | 评论与关系接口 | 评论发布、回复、删除、关注、取关、关系查询 |
| 6 | 计数、Feed、排行接口 | 批量计数、关注 Feed、榜单查询 |
| 7 | 推送、通知、IM 接口 | 设备注册、通知列表、WebSocket、消息发送、会话列表 |
| 8 | 订单接口 | 秒杀活动、秒杀令牌、抢券、支付、订单查询、用户券 |

接口契约落地后，应形成：

```text
docs/contracts/api/
  common-response.md
  error-code.md
  auth-api.md
  user-api.md
  file-api.md
  note-api.md
  comment-api.md
  relation-api.md
  counter-api.md
  feed-api.md
  rank-api.md
  push-api.md
  notification-api.md
  im-api.md
  order-api.md
```

后端实现后由 OpenAPI / Knife4j 输出机器可读契约，移动端以该契约为准。

### 6.2 错误码规范

错误码建议使用分段编码：

| 段位 | 范围 | 含义 |
|---|---|---|
| 0 | `0` | 成功 |
| 10xxx | 系统通用错误 | 参数错误、未知错误、限流、幂等冲突 |
| 20xxx | 登录认证错误 | Token 无效、会话过期、密码错误、账号锁定 |
| 21xxx | 用户服务错误 | 用户不存在、昵称非法、资料状态异常 |
| 22xxx | 文件服务错误 | 文件不存在、类型不支持、大小超限、上传未完成 |
| 23xxx | 笔记服务错误 | 笔记不存在、状态不允许、作者无权限 |
| 24xxx | 评论服务错误 | 评论不存在、层级非法、删除权限不足 |
| 25xxx | 关系服务错误 | 已关注、不能关注自己、关系不存在 |
| 26xxx | 计数服务错误 | 计数对象非法、聚合任务异常 |
| 27xxx | Feed/排行错误 | 游标非法、榜单不存在、榜单未生成 |
| 28xxx | 推送/通知错误 | 设备不存在、通知不存在、投递失败 |
| 29xxx | IM 错误 | 会话不存在、消息重复、被拉黑、频率超限 |
| 30xxx | 订单错误 | 活动不存在、库存不足、重复抢券、订单状态不允许、支付校验失败 |

错误码需要做到：

1. 移动端可以根据错误码做用户提示或跳转登录。
2. 后端日志必须记录详细内部原因。
3. 对用户返回的信息不暴露敏感内部细节。

### 6.3 内部接口契约

内部接口必须单独收口，不能默认复用移动端接口。

| 内部接口 | 调用方 | 被调用方 | 目的 |
|---|---|---|---|
| 批量查询用户摘要 | 笔记、评论、Feed、IM、通知 | 用户服务 | 展示头像、昵称、主页信息 |
| 批量查询文件元数据 | 笔记、IM | 文件服务 | 校验媒体归属、获取访问地址 |
| 批量查询笔记摘要 | Feed、排行、通知、IM | 笔记服务 | 展示笔记卡片 |
| 查询关系状态 | 笔记、IM、用户服务 | 关系服务 | 判断关注、拉黑、私信权限 |
| 查询计数快照 | 笔记、用户、排行 | 计数服务 | 展示点赞数、粉丝数、评论数 |
| 发送推送请求 | 通知、IM、订单 | 推送服务 | 在线下发或离线 Push |
| 支付回调处理 | 支付渠道适配器 | 订单服务 | 更新订单支付状态 |

内部接口应支持批量查询，避免移动端列表页触发 N+1 调用。

## 7. 数据库落地计划

### 7.1 DDL 产出顺序

DDL 建议按依赖顺序产出：

| 顺序 | Schema | 说明 |
|---|---|---|
| 1 | `bluenote_auth` | 账号、凭证、会话、登录日志 |
| 2 | `bluenote_user` | 用户资料、用户设置、主页统计冗余 |
| 3 | `bluenote_file` | 文件元数据、上传会话、文件引用 |
| 4 | `bluenote_note` | 笔记、笔记媒体、互动事实、状态流转 |
| 5 | `bluenote_comment` | 一级评论、二级评论、评论状态 |
| 6 | `bluenote_relation` | 关注、粉丝、黑名单、关系事件 |
| 7 | `bluenote_counter` | 计数聚合、变更明细、重建任务 |
| 8 | `bluenote_feed` | Feed 收件箱、游标、投递任务 |
| 9 | `bluenote_rank` | 榜单快照、分数明细、排名任务 |
| 10 | `bluenote_push` | 设备、连接、投递日志、用户推送设置 |
| 11 | `bluenote_notification` | 通知、通知聚合、已读状态 |
| 12 | `bluenote_im` | 会话、消息、用户消息链、ACK、未读 |
| 13 | `bluenote_order` | 神券模板、活动、订单、支付、用户券、库存流水 |

DDL 文件建议放在：

```text
backend/sql/
  V001__auth.sql
  V002__user.sql
  V003__file.sql
  V004__note.sql
  V005__comment.sql
  V006__relation.sql
  V007__counter.sql
  V008__feed.sql
  V009__rank.sql
  V010__push.sql
  V011__notification.sql
  V012__im.sql
  V013__order.sql
```

可以使用 Flyway 或手工 SQL 管理。第一阶段如果不引入 Flyway，也必须保证 SQL 文件有版本号、可审查、可回滚。

### 7.2 表设计统一要求

所有业务表默认包含：

| 字段 | 说明 |
|---|---|
| `id` | 主键，使用雪花 ID 或类似全局 ID |
| `created_at` | 创建时间 |
| `updated_at` | 更新时间 |
| `deleted` | 逻辑删除标记，必要表使用 |
| `version` | 乐观锁版本，状态流转或库存扣减表必须使用 |

状态类表必须额外具备：

1. 状态字段。
2. 状态变更时间。
3. 状态变更原因。
4. 状态日志表或关键操作日志。

高并发写表必须明确：

1. 唯一约束。
2. 幂等键。
3. 乐观锁条件。
4. 是否允许重试。
5. 重试失败后的补偿任务。

### 7.3 索引收口要求

每张表必须按查询场景设计索引，而不是只按字段猜索引。

每个索引都应说明：

1. 服务哪个查询。
2. 是否用于唯一约束。
3. 是否支持分页。
4. 是否和状态字段组合。
5. 是否可能造成写放大。

典型索引场景：

| 场景 | 索引方向 |
|---|---|
| 用户登录 | `username` 唯一索引 |
| 用户主页 | `user_id` 唯一索引或主键 |
| 笔记列表 | `author_id + status + publish_time` |
| 评论列表 | `note_id + parent_id + status + created_at` |
| 关注列表 | `user_id + relation_status + created_at` |
| 粉丝列表 | `target_user_id + relation_status + created_at` |
| Feed 拉取 | `user_id + feed_time + note_id` |
| IM 会话列表 | `user_id + updated_at` |
| 订单查询 | `user_id + created_at`、`activity_id + user_id` 唯一 |

### 7.4 数据重建脚本

以下数据不是唯一事实来源，必须具备重建能力：

| 数据 | 事实来源 | 重建方式 |
|---|---|---|
| Redis 用户资料缓存 | MySQL 用户表 | 按用户 ID 重建 |
| Redis 笔记详情缓存 | MySQL 笔记表、媒体表、计数快照 | 按笔记 ID 重建 |
| Redis 计数 Hash | MySQL 互动事实表、关系表、评论表 | 全量聚合或增量修正 |
| Feed 收件箱 | 笔记发布事件、关注关系 | 按用户关注关系重放生成 |
| 排行榜 Redis ZSet | 排行榜快照表、计数聚合表 | 定时任务重建 |
| 秒杀库存缓存 | MySQL 活动库存 | 活动预热任务重建 |
| IM 未读缓存 | IM 用户会话表、ACK 表 | 按用户会话重建 |

## 8. Redis 落地计划

### 8.1 Key 清单产出

Redis Key 应集中维护在：

```text
docs/contracts/redis/
  redis-key.md
```

每个 Key 至少记录：

1. Key 模板。
2. 所属服务。
3. 数据结构。
4. TTL。
5. 写入方。
6. 读取方。
7. 失效策略。
8. 重建方式。

### 8.2 Key 命名基线

统一格式：

```text
bluenote:{env}:{service}:{biz}:{id}
```

示例：

```text
bluenote:prod:user:profile:{userId}
bluenote:prod:note:detail:{noteId}
bluenote:prod:counter:note:{noteId}
bluenote:prod:feed:inbox:{userId}
bluenote:prod:rank:weekly:note:{week}
bluenote:prod:order:seckill:stock:{activityId}
```

### 8.3 Redis 使用边界

Redis 只能用于：

1. 缓存。
2. 限流。
3. 分布式锁的短时保护。
4. 秒杀库存前置扣减。
5. Feed、排行榜、计数等可重建数据。
6. IM 在线状态、连接路由、未读数加速。

Redis 不用于：

1. 保存唯一业务事实。
2. 保存不可恢复订单状态。
3. 保存不可恢复 IM 消息正文。
4. 替代 MySQL 唯一约束。

## 9. RocketMQ 落地计划

### 9.1 事件契约目录

事件契约建议放在：

```text
docs/contracts/mq/
  event-envelope.md
  topic-list.md
  note-event.md
  interaction-event.md
  comment-event.md
  relation-event.md
  counter-event.md
  feed-event.md
  rank-event.md
  notification-event.md
  push-request-event.md
  im-event.md
  order-seckill-task-event.md
  order-timeout-event.md
  order-event.md
```

### 9.2 事件 Envelope

所有事件统一使用基础结构：

```json
{
  "eventId": "uuid",
  "eventType": "NotePublished",
  "eventVersion": 1,
  "occurredAt": "2026-06-04T10:00:00+08:00",
  "traceId": "trace-id",
  "producer": "bluenote-note",
  "bizKey": "noteId",
  "payload": {}
}
```

字段要求：

| 字段 | 要求 |
|---|---|
| `eventId` | 全局唯一，用于消费幂等 |
| `eventType` | 过去式事实事件，或明确的任务事件 |
| `eventVersion` | 从 1 开始，兼容演进 |
| `occurredAt` | 事件发生时间 |
| `traceId` | 链路追踪 |
| `producer` | 生产服务 |
| `bizKey` | 用于顺序、分区或排查 |
| `payload` | 业务字段，不放大对象和文件内容 |

### 9.3 Topic 收口要求

每个 topic 必须明确：

1. 生产者。
2. 消费者。
3. 事件类型。
4. 是否顺序。
5. 是否延迟。
6. 消费幂等方式。
7. 最大重试次数。
8. 死信处理方式。
9. 是否允许重放。

### 9.4 Outbox 与消费幂等

涉及核心业务事实的事件，优先使用本地事务 + Outbox：

1. 本地事务写业务表。
2. 同事务写 `outbox_event`。
3. 后台任务扫描待发送事件。
4. 发送 RocketMQ 成功后标记已发送。
5. 发送失败保留重试。

消费者统一使用消费记录表：

```text
mq_consume_record
  id
  event_id
  topic
  consumer_group
  event_type
  biz_key
  consume_status
  retry_count
  error_message
  consumed_at
  created_at
```

幂等判断优先使用：

1. `event_id + consumer_group` 唯一约束。
2. 业务唯一约束。
3. 状态机 CAS。
4. 幂等结果缓存，作为加速，不作为事实。

## 10. 安全与权限落地计划

### 10.1 必须先冻结的安全参数

编码前需要明确：

| 参数 | 建议初始值 | 说明 |
|---|---|---|
| Access Token 有效期 | 2 小时 | 后续可根据体验调整 |
| Refresh Token 有效期 | 30 天 | 支持轮换和失效 |
| 密码算法 | BCrypt 或 Argon2id | Java 侧优先使用 Spring Security 支持成熟方案 |
| 登录失败限制 | 用户名 + IP 双维度 | 防爆破 |
| 文件上传大小 | 图片单文件 10MB 内 | 具体按移动端体验调整 |
| 秒杀令牌 TTL | 30 到 120 秒 | 防止提前刷接口 |
| 幂等请求 TTL | 5 到 30 分钟 | 按接口类型区分 |

这些值后续可以通过 Nacos 配置化，但第一版代码不能留空。

### 10.2 权限矩阵

需要补充统一权限矩阵：

```text
docs/contracts/security/
  permission-matrix.md
```

权限矩阵至少包含：

1. 接口路径。
2. 请求方法。
3. 是否需要登录。
4. 需要的角色。
5. 是否需要资源归属校验。
6. 是否需要限流。
7. 是否需要幂等键。
8. 是否记录审计日志。

### 10.3 敏感数据处理

必须统一处理：

| 数据 | 处理方式 |
|---|---|
| 密码 | 只存哈希，不打印日志 |
| 手机号 | 第一阶段如不使用手机号登录，则不要强依赖；展示时脱敏 |
| Token | 不打印完整值 |
| 支付回调参数 | 验签前后均要脱敏记录 |
| IM 消息 | 日志不打印完整正文 |
| 推送内容 | 根据用户隐私设置控制预览 |
| 文件 URL | 私有文件使用短期签名或鉴权访问 |

5. MinIO 上传。

这条链路可以验证：

3. Redis 计数聚合。

5. 通知服务事件落地。

## 16. 发布与回滚

### 16.1 发布前检查

发布前必须检查：

1. 当前 Git 版本和镜像版本一致。
2. 数据库变更已审查。
3. Nacos 配置已备份。
4. 环境变量已确认。
5. RocketMQ topic 已存在。
6. MinIO bucket 已存在。
7. Redis maxmemory 已配置。
8. 管理端口未公网暴露。
9. 健康检查接口可用。
10. 回滚版本可用。

### 16.2 发布顺序

推荐发布顺序：

1. 数据库兼容性变更。
2. Nacos 配置。
3. 基础设施配置。
4. 后端应用。
5. 网关路由。
6. 移动端版本。
7. 定时任务和 MQ 消费开关。

如果某个新功能依赖 MQ 消费，建议先部署消费者，再放开生产者流量。

### 16.3 回滚原则

1. 代码可以回滚，数据库破坏性变更不能依赖回滚解决。
2. 数据库字段删除、字段改名、语义变更必须拆成多次发布。
3. MQ 事件 payload 新增字段必须兼容旧消费者。
4. 移动端老版本仍可能调用旧接口，后端接口不能随意删除。
5. 订单和支付相关状态不能通过回滚脚本粗暴修改，必须走状态机或补偿任务。

## 17. 风险控制

| 风险 | 表现 | 应对 |
|---|---|---|
| 文档和代码脱节 | 方案写得很完整，代码实现偏离 | API、DDL、MQ 契约进入仓库，变更必须同步 |
| 服务过多导致资源不够 | 2 核 4G 跑不动多个 JVM | 逻辑拆分、物理合并部署，限制 JVM 内存 |
| MQ 链路难排查 | 消息失败后不知道卡在哪 | eventId、traceId、消费记录表、死信告警 |
| Redis 数据丢失 | 计数、Feed、排行异常 | MySQL 为事实来源，提供重建任务 |
| 秒杀超卖 | Redis 扣减和 DB 不一致 | Redis Lua 前置扣减 + MySQL 乐观扣减 + 唯一约束 + 补偿 |
| IM 消息丢失 | 离线消息不可恢复 | IM 消息先入库，再投递；ACK 和未读独立记录 |
| 推送合规和隐私 | 离线 Push 展示敏感消息 | 用户推送设置、IM 预览开关、内容脱敏 |
| MinIO 正式环境风险 | 单机对象存储磁盘故障 | 备份脚本、目录规范、后续迁移云 OSS |
| 支付回调重复 | 订单状态错乱 | 回调验签、金额校验、唯一约束、状态机 CAS |

```text
09-工程结构与编码规范.md

```

原因是这三类内容会直接影响代码怎么写、接口怎么联调、数据库怎么建表。它们不补齐，直接写代码很容易出现返工。

## 20. 结论

BlueNote 当前已经完成企业级方案设计的主体部分，但还没有完成真实落地所需的工程资产。

后续工作应从“继续扩展设计文档”切换为“把设计转成可执行契约和配置”：

1. 先冻结工程结构、统一规范、接口响应、错误码。
2. 再按主链路产出 DDL、OpenAPI、Redis Key、MQ 事件。
3. 然后搭建本地 Docker Compose 和 Nacos 配置。
4. 最后按 M0 到 M8 的阶段推进开发、联调、部署和验收。

这样做既能保持真实企业项目的设计质量，也不会让第一阶段在资源和复杂度上失控。
