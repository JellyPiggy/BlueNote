# BlueNote 可观测性与运维方案

版本：v0.1  
状态：草案  
更新时间：2026-06-03

## 1. 文档目标

本文档定义 BlueNote 的可观测性与运维方案，包括指标监控、日志、链路追踪、告警、健康检查、发布回滚、备份恢复、容量管理和故障处理流程。

本文不展开具体 Prometheus 配置、Grafana 面板 JSON、日志采集配置和告警规则文件。实际配置会在工程部署阶段维护。

## 2. 设计目标

可观测性目标：

1. 服务是否可用能看到。
2. 请求是否变慢能看到。
3. 错误率是否升高能看到。
4. MQ 是否堆积能看到。
5. MySQL、Redis、RocketMQ 是否异常能看到。
6. 线上问题能通过 traceId 串联日志。
7. 关键故障能告警。
8. 数据能备份，故障后能恢复。

运维目标：

1. 发布流程标准化。
2. 回滚流程可执行。
3. 备份恢复可验证。
4. 配置变更可追踪。
5. 故障处理有基本 runbook。

## 3. 技术栈

| 类型 | 技术 |
|---|---|
| 应用指标 | Spring Boot Actuator + Micrometer |
| 指标采集 | Prometheus |
| 指标展示 | Grafana |
| 日志采集 | Promtail 或 Docker logging driver |
| 日志存储 | Loki |
| 链路追踪 | Micrometer Tracing |
| Trace 后端 | Tempo 或 Zipkin，后续接入 |
| 告警 | Prometheus Alertmanager，后续可接企业微信/飞书/邮件 |
| 健康检查 | Actuator health endpoint |
| 部署 | Docker Compose |

第一阶段可以先实现：

1. Actuator 指标。
2. Prometheus 采集。
3. Grafana 面板。
4. Loki 日志。
5. traceId 日志串联。

完整分布式链路追踪可以后续接入 Tempo 或 Zipkin。

## 4. 指标监控

### 4.1 应用指标

所有 Java 服务必须暴露基础指标。

核心指标：

| 指标 | 说明 |
|---|---|
| HTTP QPS | 请求量 |
| HTTP 4xx | 客户端错误 |
| HTTP 5xx | 服务端错误 |
| HTTP latency | 请求延迟 |
| JVM memory | 堆内存、非堆内存 |
| JVM GC | GC 次数和耗时 |
| Thread pool | 线程数、队列长度、拒绝次数 |
| DB connection pool | 连接池使用率、等待数 |
| Redis latency | Redis 操作延迟 |
| RocketMQ producer | 发送成功、失败、耗时 |
| RocketMQ consumer | 消费成功、失败、耗时 |

### 4.2 业务指标

业务指标用于判断业务是否异常。

| 服务 | 指标 |
|---|---|
| 登录服务 | 登录成功数、登录失败数、刷新 Token 次数 |
| 用户服务 | 用户注册数、资料更新数 |
| 笔记服务 | 笔记发布数、删除数、发布失败数 |
| 文件服务 | 上传凭证申请数、上传确认数、上传失败数 |
| 计数服务 | 计数更新数、计数落库延迟、计数校准差异 |
| 关系服务 | 关注数、取关数、关注失败数 |
| Feed 服务 | Feed 写入数、Feed 拉取延迟、Feed 扩散失败数 |
| 排行榜服务 | 热度分更新数、榜单刷新延迟 |
| 评论服务 | 评论发布数、删除数、发布失败数 |
| IM 服务 | 在线连接数、消息发送数、投递延迟、ACK 延迟 |
| 订单服务 | 创建订单数、支付成功数、关单数、关单延迟 |

### 4.3 中间件指标

MySQL：

1. 连接数。
2. 慢查询数量。
3. QPS。
4. TPS。
5. 锁等待。
6. 磁盘空间。
7. 主从延迟，后续。

Redis：

1. 内存使用率。
2. QPS。
3. 命中率。
4. 大 key。
5. 慢命令。
6. 连接数。
7. evicted keys。

RocketMQ：

1. Topic 堆积。
2. Consumer Group 延迟。
3. 发送失败数。
4. 消费失败数。
5. 死信数量。
6. Broker 磁盘使用率。
7. Broker JVM 内存。

服务器：

1. CPU。
2. 内存。
3. 磁盘。
4. 网络。
5. 容器重启次数。
6. 文件句柄。

## 5. 日志方案

### 5.1 日志目标

日志用于：

1. 排查线上问题。
2. 串联用户请求链路。
3. 排查 MQ 消费失败。
4. 审计关键行为。
5. 分析异常访问。

日志不用于：

1. 保存敏感数据。
2. 替代业务数据库。
3. 长期保存所有明细数据。

### 5.2 日志格式

建议使用 JSON 日志。

字段：

```json
{
  "time": "2026-06-03T16:00:00+08:00",
  "level": "INFO",
  "service": "bluenote-note",
  "traceId": "trace-id",
  "spanId": "span-id",
  "userId": "123",
  "eventId": "event-id",
  "message": "note published",
  "exception": ""
}
```

必要字段：

1. time
2. level
3. service
4. traceId
5. message

按场景补充：

1. userId。
2. deviceId。
3. requestId。
4. eventId。
5. topic。
6. consumerGroup。
7. bizId。

### 5.3 traceId 规范

traceId 生成规则：

1. 如果请求 Header 已有合法 traceId，网关透传。
2. 如果没有，网关生成。
3. 网关向所有下游服务透传。
4. MQ 消息中必须包含 traceId。
5. 消费者日志继续使用消息中的 traceId。

Header 建议：

```text
X-Trace-Id
```

### 5.4 日志脱敏

日志禁止打印：

1. 密码。
2. 验证码。
3. Access Token。
4. Refresh Token。
5. 完整手机号。
6. 对象存储密钥。
7. 支付密钥。
8. 银行卡或支付敏感信息，后续。

日志可以打印：

1. userId。
2. noteId。
3. commentId。
4. orderId。
5. eventId。
6. traceId。
7. 脱敏手机号。

### 5.5 日志级别

| 级别 | 使用场景 |
|---|---|
| DEBUG | 本地开发调试，生产默认关闭 |
| INFO | 正常关键业务节点 |
| WARN | 可恢复异常、降级、重试 |
| ERROR | 请求失败、消费失败、数据异常 |

要求：

1. 生产环境不能大量 DEBUG。
2. ERROR 必须包含 traceId。
3. MQ 消费失败必须包含 eventId、topic、consumerGroup。
4. 不要在循环中打印大量重复日志。

## 6. 链路追踪

### 6.1 第一阶段

第一阶段先保证 traceId 串联。

覆盖：

1. HTTP 请求。
2. 服务间调用。
3. RocketMQ 事件。
4. 定时任务。

即使暂时不接完整 Trace 后端，也要让日志能通过 traceId 查询完整链路。

### 6.2 后续接入

后续接入：

1. Micrometer Tracing。
2. OpenTelemetry Bridge，按技术兼容性评估。
3. Tempo 或 Zipkin。

目标：

1. 查看服务调用拓扑。
2. 查看慢请求在哪个服务耗时。
3. 查看 MQ 异步链路。
4. 查看数据库和 Redis 慢操作。

## 7. 健康检查

### 7.1 应用健康检查

每个服务暴露：

```text
/actuator/health
/actuator/prometheus
```

健康检查包括：

1. 应用存活。
2. MySQL 连接。
3. Redis 连接。
4. RocketMQ 连接，按服务依赖决定。
5. Nacos 注册状态。

### 7.2 容器健康检查

Docker Compose 中配置 healthcheck。

要求：

1. 服务启动后再接流量。
2. 健康检查失败可自动重启。
3. 避免过短间隔导致误判。
4. 区分 liveness 和 readiness，后续 Kubernetes 阶段完善。

## 8. 告警方案

### 8.1 告警等级

| 等级 | 说明 | 示例 |
|---|---|---|
| P0 | 核心服务不可用 | 网关不可用、MySQL 不可用 |
| P1 | 核心链路严重受影响 | 登录失败率大幅升高、MQ 严重堆积 |
| P2 | 局部功能异常 | 排行榜延迟、通知堆积 |
| P3 | 需要关注 | 磁盘使用率升高、慢查询增加 |

### 8.2 第一阶段告警项

必须配置：

1. 网关不可用。
2. 登录服务不可用。
3. MySQL 不可用。
4. Redis 不可用。
5. RocketMQ 不可用。
6. HTTP 5xx 错误率过高。
7. 请求延迟过高。
8. RocketMQ 消费堆积。
9. RocketMQ 死信数量大于 0。
10. 磁盘使用率过高。
11. 容器频繁重启。

建议配置：

1. 登录失败率异常。
2. 文件上传失败率异常。
3. 评论发布失败率异常。
4. IM 在线连接数异常下降。
5. 订单关单延迟，订单服务上线后。

### 8.3 告警通道

第一阶段可选：

1. 邮件。
2. 企业微信。
3. 飞书。
4. Telegram，视个人使用习惯。

要求：

1. P0/P1 必须主动推送。
2. P2/P3 可以聚合推送。
3. 告警内容包含服务名、环境、指标、当前值、trace 或面板链接。

## 10. 发布运维

### 10.1 发布前检查

发布前检查：

1. 单元测试通过。
2. 核心接口联调通过。
3. 数据库脚本已审查。
4. 配置项已准备。
5. 镜像 tag 明确。
6. 回滚镜像存在。
7. 备份已执行，涉及数据库变更时。

### 10.2 发布流程

```text
构建镜像
  -> 推送镜像仓库
  -> 更新测试环境
  -> 验证核心流程
  -> 备份生产关键数据
  -> 更新生产 Compose 配置
  -> 拉取新镜像
  -> 滚动或分组重启
  -> 健康检查
  -> 观察指标和日志
```

2 核 4G 单机环境不一定能真正滚动发布。第一阶段可以采用短暂停机发布，但要控制窗口。

### 10.3 发布后观察

至少观察：

1. 服务健康状态。
2. HTTP 5xx。
3. 请求延迟。
4. JVM 内存。
5. MySQL 连接数。
6. Redis 错误。
7. MQ 消费堆积。
8. 核心业务成功率。

观察时间：

1. 小变更：10 到 30 分钟。
2. 涉及数据库或 MQ：至少 30 分钟。
3. 涉及登录、发布、支付：更长时间观察。

### 10.4 回滚流程

回滚要求：

1. 镜像 tag 可回退。
2. 配置可回退。
3. 数据库变更向前兼容。
4. 发布脚本支持指定版本。
5. 回滚后继续观察指标。

数据库已经执行破坏性变更时，不能简单回滚应用，需要按数据库恢复方案处理。因此数据库变更必须尽量向前兼容。

## 11. 备份与恢复运维

### 11.1 备份对象

必须备份：

1. MySQL。
2. 对象存储重要文件。
3. Nacos 配置。
4. Docker Compose 配置。
5. 环境变量模板，不含真实密钥。

建议备份：

1. Grafana 面板。
2. Prometheus 告警规则。
3. Loki 配置。
4. 运维脚本。

### 11.2 备份频率

| 对象 | 频率 |
|---|---|
| MySQL 全量 | 每日 |
| MySQL binlog | 持续，按磁盘空间保留 |
| 对象存储 | 每日或每周，视文件量 |
| Nacos 配置 | 每次变更后 |
| 运维配置 | 每次变更后提交 Git |

### 11.3 恢复演练

恢复演练要求：

1. 至少每月一次。
2. 在 test 环境恢复。
3. 记录恢复耗时。
4. 校验核心数据。
5. 修正不可用脚本。

只备份不演练，等同于没有可靠备份。

## 12. 故障处理 Runbook

### 12.1 服务不可用

处理步骤：

1. 查看 Grafana 服务状态。
2. 查看容器是否运行。
3. 查看最近发布记录。
4. 查看服务日志 ERROR。
5. 查看依赖 MySQL、Redis、RocketMQ。
6. 尝试重启单个服务。
7. 如果是发布导致，回滚到上一版本。

### 12.2 MySQL 不可用

处理步骤：

1. 查看容器状态。
2. 查看磁盘空间。
3. 查看 MySQL 错误日志。
4. 查看连接数是否打满。
5. 临时降低非核心服务流量。
6. 必要时重启 MySQL。
7. 如果数据损坏，按恢复流程处理。

### 12.3 Redis 不可用

处理步骤：

1. 查看容器状态。
2. 查看内存是否达到 maxmemory。
3. 查看慢命令。
4. 查看是否存在大 key。
5. 重启后执行必要数据重建。
6. 确认登录、计数、Feed、排行榜是否恢复。

### 12.4 RocketMQ 堆积

处理步骤：

1. 定位堆积 Topic 和 Consumer Group。
2. 查看消费者是否存活。
3. 查看消费失败日志。
4. 判断是否单条毒消息。
5. 临时扩容消费者，资源允许时。
6. 修复业务异常。
7. 处理死信消息。

### 12.5 磁盘空间不足

处理步骤：

1. 查看 Docker 日志占用。
2. 查看 MySQL binlog。
3. 查看 RocketMQ commitlog。
4. 查看 Loki 日志。
5. 清理过期日志和备份。
6. 扩容磁盘。
7. 调整保留周期。

### 12.6 IM 连接异常

处理步骤：

1. 查看 IM 服务是否存活。
2. 查看 WebSocket 握手错误。
3. 查看在线连接数变化。
4. 查看网关或反向代理超时配置。
5. 查看 Redis 在线状态。
6. 查看消息投递堆积。

## 13. 容量管理

### 13.1 资源基线

需要定期记录：

1. 每个容器 CPU。
2. 每个容器内存。
3. MySQL 数据大小。
4. Redis 内存大小。
5. RocketMQ 磁盘大小。
6. 对象存储容量。
7. 日志增长速度。

### 13.2 扩容触发

需要扩容或拆分的信号：

1. CPU 长时间超过 70%。
2. 内存长时间超过 80%。
3. 磁盘超过 80%。
4. MySQL 慢查询持续增加。
5. Redis 内存频繁淘汰。
6. MQ 堆积经常发生。
7. IM 连接数接近单机上限。
8. 发布时资源明显不足。

### 13.3 降级策略

资源紧张时可降级：

1. 暂停排行榜实时更新。
2. 降低 Feed 扩散批次。
3. 延迟通知生成。
4. 限制图片上传大小。
5. 暂停非核心离线任务。
6. 降低日志级别。

不能轻易降级：

1. 登录。
2. 笔记发布。
3. 订单支付，后续。
4. 订单关单，后续。

## 14. 配置变更管理

配置变更要求：

1. 记录变更人。
2. 记录变更时间。
3. 记录变更内容。
4. 记录回滚方式。
5. 生产配置变更前先在 test 验证。

高风险配置：

1. 数据库连接。
2. Redis 连接。
3. RocketMQ topic。
4. JWT 密钥。
5. 对象存储 bucket。
6. 网关路由。
7. 限流阈值。

## 15. 日志与指标保留

第一阶段建议：

| 数据 | 保留时间 |
|---|---|
| 应用日志 | 7 到 14 天 |
| 错误日志 | 14 到 30 天 |
| 指标数据 | 15 到 30 天 |
| 审计日志 | 90 天以上，视存储成本 |
| MySQL 备份 | 每日 7 到 14 天，每周 4 到 8 周 |

保留周期需要结合磁盘成本调整。

## 16. 本地旧电脑运维边界

本地旧电脑可以运行：

1. Grafana。
2. Prometheus。
3. Loki。
4. test 环境。
5. 备份副本。
6. 离线任务。

要求：

1. 本地旧电脑不可用时，prod 核心服务继续可用。
2. 本地旧电脑上的监控不可用时，至少云服务器仍有基础日志。
3. 内网穿透入口必须有认证。
4. 不暴露数据库、Redis、RocketMQ 管理端口。
5. 备份传输需要校验完整性。

## 17. 运维脚本要求

建议脚本：

```text
deploy.sh
rollback.sh
health-check.sh
backup-mysql.sh
restore-mysql.sh
backup-nacos.sh
clean-docker-log.sh
rebuild-counter.sh
rebuild-feed.sh
replay-mq-event.sh
```

要求：

1. 脚本可重复执行。
2. 危险操作需要显式参数。
3. 输出清晰日志。
4. 失败时返回非零状态码。
5. 不在脚本中硬编码密钥。
6. prod 操作有确认机制。

## 18. 故障复盘

P0/P1 故障需要复盘。

复盘内容：

1. 故障时间线。
2. 影响范围。
3. 根因。
4. 发现方式。
5. 恢复方式。
6. 为什么监控或告警有效/无效。
7. 后续改进项。
8. 责任不是重点，系统改进是重点。

复盘产物：

1. 文档记录。
2. 告警规则调整。
3. Runbook 更新。
4. 自动化脚本补充。
5. 测试用例补充。

## 19. 风险与应对

| 风险 | 说明 | 应对 |
|---|---|---|
| 监控系统部署在本地旧电脑 | 本地断网时监控不可用 | 云服务器保留基础日志和健康检查 |
| 日志过多打满磁盘 | Loki、Docker 日志增长 | 限制保留周期，日志轮转 |
| 告警太多被忽略 | 无效告警过多 | 分级告警，定期清理规则 |
| 只有备份没有演练 | 真故障时恢复失败 | 每月恢复演练 |
| traceId 不完整 | 异步链路无法串联 | MQ 消息必须携带 traceId |
| 单机资源不足 | 2 核 4G 承载能力有限 | 合并服务、限制 JVM、降级非核心任务 |
| 发布不可回滚 | 数据库破坏性变更 | 向前兼容 DDL，保留旧镜像 |
