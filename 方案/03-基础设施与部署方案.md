# BlueNote 基础设施与部署方案

版本：v0.1  
状态：草案  
更新时间：2026-06-03

## 1. 文档目标

本文档定义 BlueNote 第一阶段的基础设施组成、部署拓扑、环境划分、网络入口、容器化策略、资源规划、备份策略和后续演进方向。

本文不展开具体 Docker Compose 文件内容，也不展开各服务配置项明细。实际部署脚本和环境变量模板会在工程落地阶段单独维护。

## 2. 部署约束

当前项目的部署条件：

1. 一部分服务部署在 2 核 4G 云服务器。
2. 一部分服务部署在本地旧电脑。
3. 本地旧电脑通过内网穿透暴露部分入口。
4. 暂不引入复杂集群。
5. 最多采用单机、主从或冷备方案。
6. 第一阶段优先保证可运行、可维护、可恢复。

核心判断：

1. 用户主链路不能强依赖本地旧电脑。
2. 内网穿透不能作为核心业务链路的唯一入口。
3. 关键数据必须在云服务器或可靠云服务中保存。
4. 本地旧电脑更适合承载监控、备份、开发测试、异步低优先级任务。

## 3. 环境划分

建议至少划分三个环境：

| 环境 | 用途 | 部署位置 |
|---|---|---|
| local | 本机开发、单服务调试 | 开发电脑 |
| test | 联调、功能测试、压测小样本 | 本地旧电脑或云服务器低配环境 |
| prod | 正式对外服务 | 云服务器为主 |

### 3.1 local 环境

local 环境用于开发调试。

建议：

1. 使用 Docker Compose 启动 MySQL、Redis、RocketMQ、Nacos、MinIO。
2. 后端服务可以直接在 IDE 中启动。
3. 移动端连接本地网关或测试网关。
4. local 数据允许随时重置。

### 3.2 test 环境

test 环境用于联调和验证。

建议：

1. 可以部署在本地旧电脑。
2. 允许通过内网穿透提供临时访问。
3. 使用独立数据库和对象存储桶。
4. 不复用 prod 数据库。
5. 可用于验证部署脚本、MQ 消费、备份恢复。

### 3.3 prod 环境

prod 环境用于正式访问。

要求：

1. 主入口在云服务器。
2. 数据库、Redis、RocketMQ 等核心组件不能依赖不稳定穿透链路。
3. 必须有备份。
4. 必须有基础监控和日志。
5. 必须支持快速重启和回滚。

## 4. 基础设施组件

### 4.1 组件清单

| 组件 | 推荐技术 | 第一阶段部署方式 |
|---|---|---|
| 反向代理 | Nginx 或 Caddy | 云服务器 |
| API 网关 | Spring Cloud Gateway | 云服务器 |
| 服务注册配置 | Nacos | 云服务器，单机 |
| 数据库 | MySQL 8.x | 云服务器，单机，后续主从 |
| 缓存 | Redis 7.x | 云服务器，单机，开启持久化 |
| 消息队列 | RocketMQ 5.x | 云服务器或资源允许时独立机器 |
| 对象存储 | MinIO | 第一阶段全环境使用；prod 部署在云服务器，旧电脑做备份；云 OSS/CDN 作为后续演进 |
| 日志系统 | Loki | 本地旧电脑或云服务器 |
| 指标系统 | Prometheus | 本地旧电脑或云服务器 |
| 面板 | Grafana | 本地旧电脑 |
| 镜像仓库 | Docker Hub / GHCR / 私有仓库 | 云端 |
| 内网穿透 | frp / Cloudflare Tunnel / 其他方案 | 本地旧电脑 |

### 4.2 核心链路组件

核心链路组件是用户访问主流程必须依赖的组件：

1. Nginx 或 Caddy
2. API 网关
3. 登录服务
4. 用户服务
5. 笔记服务
6. 计数服务
7. 关系服务
8. Feed 服务
9. 评论服务
10. MySQL
11. Redis
12. RocketMQ

这些组件原则上部署在云服务器，或者使用可靠云服务。

### 4.3 非核心链路组件

非核心链路组件可以部署在本地旧电脑：

1. Grafana
2. Prometheus
3. Loki
4. 测试环境
5. 离线备份任务
6. 低优先级异步任务
7. 管理后台预览环境，后续扩展
8. 文档站点，后续扩展

本地旧电脑不可用时，正式用户核心流程应继续可用。

## 5. 第一阶段部署拓扑

### 5.1 推荐拓扑

```text
用户移动端
  |
  | HTTPS
  v
云服务器：Nginx/Caddy
  |
  v
云服务器：bluenote-gateway
  |
  +--> bluenote-member-app
  +--> bluenote-content-app
  +--> bluenote-social-app
  +--> bluenote-im-app
  +--> bluenote-order-app，后续启用
  |
  +--> MySQL
  +--> Redis
  +--> RocketMQ
  +--> Nacos
  +--> 对象存储

本地旧电脑
  +--> Prometheus
  +--> Grafana
  +--> Loki
  +--> test 环境
  +--> 备份副本
  +--> 内网穿透入口
```

### 5.2 云服务器部署内容

2 核 4G 云服务器建议部署：

| 类型 | 组件 |
|---|---|
| 入口 | Nginx 或 Caddy |
| 网关 | bluenote-gateway-app |
| 应用 | bluenote-member-app、bluenote-content-app、bluenote-social-app |
| 实时 | bluenote-im-app，资源紧张时延后部署 |
| 中间件 | MySQL、Redis、Nacos、RocketMQ |
| 存储 | MinIO 单机部署在云服务器；后续对接云 OSS/CDN |

如果内存压力较大，优先保证：

1. Nginx/Caddy
2. 网关
3. member
4. content
5. social
6. MySQL
7. Redis

RocketMQ、IM、监控可以根据资源情况调整部署位置，但正式链路要避免依赖不稳定节点。

### 5.3 本地旧电脑部署内容

本地旧电脑建议部署：

| 类型 | 组件 |
|---|---|
| 监控 | Prometheus、Grafana |
| 日志 | Loki |
| 测试 | test 环境应用和中间件 |
| 备份 | MySQL 备份副本、对象存储备份 |
| 辅助任务 | 离线统计、低优先级补偿任务 |

本地旧电脑适合处理：

1. 非实时任务。
2. 可失败重试的任务。
3. 有云端数据兜底的任务。
4. 开发测试用途。

不适合处理：

1. 登录。
2. 笔记发布。
3. 支付回调。
4. 订单关单唯一任务。
5. IM 唯一长连接入口。
6. MySQL 主库。

## 6. 物理应用部署分组

第一阶段可以按照逻辑服务边界设计，物理部署合并为较少 Java 进程。

| 物理应用 | 包含逻辑服务 | 是否第一阶段部署 |
|---|---|---|
| bluenote-gateway-app | gateway | 是 |
| bluenote-member-app | auth、user | 是 |
| bluenote-content-app | file、note、comment | 是 |
| bluenote-social-app | counter、relation、feed、rank、notification | 是 |
| bluenote-im-app | im | 可延后 |
| bluenote-order-app | order | 后续启用 |

合并部署要求：

1. 包结构仍按逻辑服务隔离。
2. 数据 schema 仍按逻辑服务隔离。
3. RocketMQ topic 和 consumer group 仍按业务语义隔离。
4. 对外 API 仍按服务资源命名。
5. 后续拆分时尽量不改业务代码，只调整启动模块和部署单元。

## 7. Docker Compose 策略

### 7.1 使用原则

第一阶段使用 Docker Compose 编排。

原因：

1. 资源占用低。
2. 学习和运维成本低。
3. 适合单机或少量机器。
4. 配置清晰，便于迁移。
5. 当前不需要 Kubernetes 的复杂调度能力。

### 7.2 Compose 文件建议

建议拆分：

```text
deploy/
  compose.base.yml
  compose.local.yml
  compose.test.yml
  compose.prod.yml
  env/
    local.env
    test.env
    prod.env.example
```

规则：

1. `compose.base.yml` 放通用服务定义。
2. `compose.local.yml` 放本地开发覆盖项。
3. `compose.test.yml` 放测试环境覆盖项。
4. `compose.prod.yml` 放正式环境覆盖项。
5. `.env` 不提交真实密钥。
6. `prod.env.example` 只放变量名和示例。

### 7.3 容器资源限制

2 核 4G 环境必须限制 JVM 和中间件内存。

建议初始方向：

| 组件 | 内存策略 |
|---|---|
| gateway | JVM 最大堆 256m 到 384m |
| member | JVM 最大堆 384m 到 512m |
| content | JVM 最大堆 512m |
| social | JVM 最大堆 512m |
| im | JVM 最大堆 384m 到 512m |
| MySQL | 限制 buffer pool，避免吃满内存 |
| Redis | 设置 maxmemory 和淘汰策略 |
| RocketMQ | 限制 JVM 参数，控制 topic 和队列数量 |

具体数值需要压测和实际运行后调整。

## 8. 网络与入口

### 8.1 域名规划

建议域名：

| 域名 | 用途 |
|---|---|
| api.bluenote.example.com | 移动端 API |
| im.bluenote.example.com | IM WebSocket |
| oss.bluenote.example.com | 文件访问，若自建 MinIO |
| grafana.bluenote.example.com | 监控面板，仅限管理访问 |
| test-api.bluenote.example.com | 测试环境 API |

实际域名以后替换。

### 8.2 TLS

正式环境必须使用 HTTPS。

可选方案：

1. Caddy 自动申请和续期证书。
2. Nginx + Certbot 管理证书。
3. 使用云厂商证书服务。

移动端只访问 HTTPS API，WebSocket 使用 WSS。

### 8.3 端口规划

示例：

| 组件 | 内部端口 | 是否公网暴露 |
|---|---|---|
| Nginx/Caddy | 80/443 | 是 |
| Gateway | 8080 | 否 |
| IM WebSocket | 8090 | 通过反向代理暴露 |
| Nacos | 8848 | 否 |
| MySQL | 3306 | 否 |
| Redis | 6379 | 否 |
| RocketMQ | 按组件配置 | 否 |
| Grafana | 3000 | 限制访问 |
| Prometheus | 9090 | 否 |
| Loki | 3100 | 否 |

规则：

1. MySQL、Redis、RocketMQ、Nacos 不直接暴露公网。
2. 管理端口必须通过防火墙、VPN 或内网穿透鉴权保护。
3. 对外入口只保留 80/443。
4. IM 通过 HTTPS 入口升级为 WebSocket。

## 9. 配置管理

### 9.1 配置中心

使用 Nacos 作为配置中心。

适合放入：

1. 服务配置。
2. 限流参数。
3. 业务开关。
4. MQ topic 配置。
5. Redis key 前缀。
6. 对象存储 bucket 配置。

不适合直接明文放入：

1. 数据库密码。
2. Redis 密码。
3. JWT 私钥。
4. 对象存储密钥。
5. 第三方支付密钥。

敏感配置可以通过环境变量、Docker secret、宿主机文件权限或后续密钥管理服务处理。

### 9.2 环境变量规范

环境变量命名建议：

```text
BLUENOTE_ENV=prod
BLUENOTE_MYSQL_HOST=
BLUENOTE_MYSQL_PORT=
BLUENOTE_REDIS_HOST=
BLUENOTE_REDIS_PORT=
BLUENOTE_ROCKETMQ_ENDPOINT=
BLUENOTE_NACOS_ADDR=
BLUENOTE_JWT_PRIVATE_KEY_PATH=
BLUENOTE_OSS_ENDPOINT=
```

规则：

1. 不提交真实 `.env`。
2. 提交 `.env.example`。
3. prod 和 test 配置完全隔离。
4. 启动时打印环境名，但不打印密钥值。

## 10. 数据备份与恢复

### 10.1 MySQL 备份

最低要求：

1. 每日全量备份。
2. 关键业务上线后开启 binlog。
3. 备份文件同步到本地旧电脑和另一个云端位置。
4. 定期做恢复演练。

备份保留建议：

| 类型 | 保留时间 |
|---|---|
| 每日备份 | 7 到 14 天 |
| 每周备份 | 4 到 8 周 |
| 每月备份 | 6 到 12 个月，视成本 |

### 10.2 Redis 备份

Redis 不作为唯一事实来源，但仍建议开启持久化。

建议：

1. 开启 AOF 或 RDB，具体按性能评估。
2. 关键计数可从 MySQL 明细和快照重建。
3. Feed、排行榜需要提供重建脚本。
4. 在线状态类数据丢失可接受。

### 10.3 对象存储备份

第一阶段使用 MinIO：

1. 定期同步到本地旧电脑或云端备份。
2. 定期校验对象数量和大小。
3. 图片元数据以 MySQL 为准。

后续迁移云 OSS 时：

1. 开启版本控制或生命周期规则，视成本决定。
2. 重要文件开启跨区域复制，后续再评估。

## 11. 发布与回滚

### 11.1 发布流程

建议流程：

```text
代码合并
  -> 自动测试
  -> 构建 Docker 镜像
  -> 推送镜像仓库
  -> 更新测试环境
  -> 验证核心接口
  -> 更新正式环境
  -> 健康检查
  -> 观察日志和指标
```

### 11.2 回滚策略

要求：

1. 每次发布保留上一个可用镜像。
2. Compose 文件使用明确镜像 tag，不使用 `latest`。
3. 数据库变更必须向前兼容。
4. 先发兼容代码，再执行破坏性数据清理。
5. 回滚脚本要能快速恢复上一版本容器。

### 11.3 数据库变更策略

数据库变更必须谨慎。

原则：

1. 新增字段优先允许为空或有默认值。
2. 删除字段必须经过废弃期。
3. 字段改名采用新增字段、双写、迁移、切读、删除旧字段的流程。
4. 大表变更需要评估锁表风险。
5. 所有 DDL 脚本必须纳入版本管理。

## 12. 高可用演进

第一阶段不做复杂集群，但需要保留演进路径。

| 组件 | 第一阶段 | 后续演进 |
|---|---|---|
| 应用服务 | 单实例或少量实例 | 多实例 + 网关负载均衡 |
| MySQL | 单机 | 主从复制、备库、云 RDS |
| Redis | 单机 | 哨兵或托管 Redis |
| RocketMQ | 单机 | 主从或 DLedger |
| Nacos | 单机 | 集群 |
| 对象存储 | MinIO 单机 | 云 OSS/COS/R2 + CDN |
| 部署 | Docker Compose | Kubernetes |

演进触发条件：

1. 用户量明显增长。
2. 单机资源瓶颈明显。
3. 服务不可用影响用户。
4. 运维成本超过 Compose 可控范围。
5. 需要自动扩缩容或更细粒度隔离。

## 13. 本地旧电脑使用边界

本地旧电脑的定位是辅助节点，不是主站核心节点。

适合：

1. 监控面板。
2. 日志存储。
3. 备份副本。
4. 测试环境。
5. 离线任务。
6. 开发辅助服务。

不适合：

1. MySQL 主库。
2. Redis 主缓存。
3. RocketMQ 唯一 Broker。
4. 支付回调入口。
5. 用户上传文件唯一存储。
6. IM 唯一入口。

如果必须使用内网穿透访问本地服务，要求：

1. 必须有访问认证。
2. 不暴露数据库和缓存端口。
3. 只暴露 HTTP/HTTPS 管理入口。
4. 记录访问日志。
5. 穿透服务不可用时不影响正式用户主流程。

## 14. 运维目录建议

后续工程中建议维护：

```text
deploy/
  compose/
  nginx/
  caddy/
  nacos/
  rocketmq/
  mysql/
  redis/
  scripts/
    deploy.sh
    rollback.sh
    backup-mysql.sh
    restore-mysql.sh
    health-check.sh
  env/
    local.env.example
    test.env.example
    prod.env.example
```

脚本要求：

1. 操作幂等。
2. 有清晰日志。
3. 失败时返回非零状态码。
4. 危险操作需要二次确认或显式参数。
5. 不在脚本中硬编码密钥。

## 15. 风险与应对

| 风险 | 说明 | 应对 |
|---|---|---|
| 云服务器内存不足 | 多个 Java 服务和中间件占用较高 | 合并部署、限制 JVM、必要时拆分中间件 |
| RocketMQ 较重 | 单机资源有限时压力明显 | 控制 topic、队列数和消息量，必要时先部署到资源更充足机器 |
| 本地旧电脑不稳定 | 断电、网络、穿透异常 | 只承载非核心链路 |
| 数据备份不可恢复 | 只备份不演练没有意义 | 定期恢复演练 |
| 配置泄露 | `.env` 或 Nacos 配置泄露 | 密钥不入库、不提交、限制管理端访问 |
| 单点故障 | MySQL、Redis、MQ 第一阶段都是单点 | 保留主从演进路线，先做备份和快速恢复 |
