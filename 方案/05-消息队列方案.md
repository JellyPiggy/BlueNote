# BlueNote 消息队列方案

版本：v0.1  
状态：草案  
更新时间：2026-06-03

## 1. 文档目标

本文档定义 BlueNote 的消息队列总体方案，包括 RocketMQ 选型、Topic 规划、事件规范、生产者策略、消费者策略、幂等、重试、死信、延时消息、顺序消息和后续演进方向。

本文不展开每个事件的完整 payload 字段。具体字段会在各服务详细设计中补充。

## 2. 选型结论

BlueNote 第一阶段选择 RocketMQ 5.x 作为主消息队列。

Kafka 暂不引入。后续如果需要用户行为流、推荐系统、实时分析、数据湖或 Flink 实时计算，再单独评估 Kafka。

## 3. 使用目标

RocketMQ 主要用于：

1. 服务间异步解耦。
2. 跨服务最终一致。
3. 高频行为削峰。
4. Feed 扩散。
5. 计数异步更新。
6. 排行榜准实时更新。
7. 通知异步生成。
8. IM 内部投递。
9. 神券秒杀异步下单。
10. 订单超时关单。
11. 后续补偿和重试。

不用于：

1. 直接传输大文件。
2. 替代数据库保存核心状态。
3. 同步查询。
4. 强实时 RPC。
5. 无限期保存完整业务历史。

## 4. 设计原则

### 4.1 事件表示已经发生的事实

事件命名使用过去式。

正确示例：

```text
NotePublished
CommentCreated
UserFollowed
OrderPaid
ImMessageSent
```

不建议：

```text
PublishNote
CreateComment
FollowUser
PayOrder
```

### 4.2 消息不是数据库

核心状态必须落 MySQL。

RocketMQ 只负责通知其他服务：

1. 某个事实已经发生。
2. 某个异步任务需要处理。
3. 某个延时检查需要触发。

### 4.3 消费者必须幂等

所有消费者都必须假设：

1. 消息可能重复投递。
2. 消息可能乱序到达，除非明确使用顺序消息。
3. 消息可能延迟。
4. 消息处理可能失败后重试。

### 4.4 失败必须可恢复

消息失败不能只依赖人工观察日志。

要求：

1. 自动重试。
2. 死信记录。
3. 幂等表。
4. 补偿任务。
5. 告警。

## 5. RocketMQ 组件部署

第一阶段可以采用单机部署。

组件：

1. NameServer
2. Broker
3. Proxy，视客户端接入方式决定
4. Dashboard，非必须，建议测试环境使用

资源有限时：

1. 控制 topic 数量。
2. 控制队列数量。
3. 控制消息保留时间。
4. 限制 Broker JVM 内存。
5. 禁止将日志、埋点大流量全部打入 RocketMQ。

后续演进：

1. Broker 主从。
2. DLedger 高可用。
3. 独立 MQ 服务器。
4. 托管消息队列服务。

## 6. Topic 规划

### 6.1 Topic 命名规范

建议格式：

```text
bluenote-{env}-{domain}-event
```

示例：

```text
bluenote-prod-note-event
bluenote-prod-relation-event
bluenote-prod-interaction-event
bluenote-prod-order-timeout-event
```

如果 RocketMQ 实例已按环境隔离，topic 中可以省略环境前缀，但文档和配置中必须清晰区分。

### 6.2 Topic 清单

| Topic | 事件类型 | 主要生产者 | 主要消费者 |
|---|---|---|---|
| note-event | NotePublished、NoteUpdated、NoteDeleted、NoteVisibilityChanged、NoteStatusChanged | 笔记服务 | Feed、排行榜、通知、搜索后续、计数 |
| interaction-event | NoteLiked、NoteUnliked、NoteCollected、NoteUncollected | 笔记服务或互动模块 | 计数、通知 |
| relation-event | UserFollowed、UserUnfollowed、UserBlocked | 关系服务 | 计数、Feed、通知、IM |
| comment-event | CommentCreated、CommentDeleted、CommentLiked、CommentUnliked、CommentStatusChanged | 评论服务 | 计数、通知 |
| counter-delta-event | CounterDeltaCreated | 计数服务 Event Adapter | 计数服务 Counter Updater |
| counter-event | CounterChanged、CounterRebuilt | 计数服务 | 排行榜、展示读模型 |
| feed-fanout-task-event | FeedFanoutSubTaskCreated | Feed 服务 Fanout Dispatcher | Feed 服务 Fanout Executor |
| feed-event | FeedDelivered、FeedRebuilt | Feed 服务 | 监控、后续推荐 |
| rank-event | RankChanged | 排行榜服务 | 通知或运营后续 |
| notification-event | NotificationCreated、NotificationAggregated、NotificationRead、NotificationReadBatch、NotificationDeleted | 通知服务 | 通知生命周期订阅方 |
| push-request-event | PushSendRequested | 通知、IM、订单等服务 | 推送与实时投递服务 |
| push-event | PushRequestAccepted、PushDeliverySucceeded、PushDeliveryFailed、PushDeliveryFiltered、PushNotificationClicked | 推送与实时投递服务 | 通知、IM、订单、运营后续 |
| im-message-event | ImMessageSent、ImMessageAcked、ImMessageRead | IM 服务 | IM 内部消费者 |
| order-seckill-task-event | CouponSeckillAccepted | 订单服务 | 订单服务异步下单消费者 |
| order-event | OrderCreated、OrderPaid、OrderClosed、OrderCancelled、CouponIssued | 订单服务 | 通知、运营后续、后续交易相关服务 |
| order-timeout-event | OrderTimeoutCheck | 订单服务 | 订单服务关单消费者 |

### 6.3 Topic 粒度原则

Topic 不宜过细，也不宜过粗。

建议：

1. 按业务域划分 topic。
2. 同一业务域内用 eventType 区分事件。
3. 高优先级和低优先级事件不要混在一个 topic。
4. 延时任务单独 topic。
5. IM 高吞吐消息单独 topic。
6. `counter-delta-event` 是计数服务内部 Topic，不作为跨业务服务协议。
7. `feed-fanout-task-event` 是 Feed 服务内部 Topic，不作为跨业务服务协议。
8. `order-seckill-task-event` 是订单服务内部异步下单 Topic，不作为跨业务服务协议。

## 7. Consumer Group 规划

| Consumer Group | 订阅 Topic | 归属服务 | 用途 |
|---|---|---|---|
| counter-consumer-group | note-event、interaction-event、relation-event、comment-event | 计数服务 | 消费业务事件并生成计数 delta |
| counter-delta-updater-group | counter-delta-event | 计数服务 | 聚合写入 Redis 计数 |
| feed-consumer-group | note-event、relation-event | Feed 服务 | Feed 扩散和清理 |
| feed-fanout-executor-group | feed-fanout-task-event | Feed 服务 | 执行 Feed 扩散子任务 |
| rank-consumer-group | note-event、counter-event | 排行榜服务 | 维护候选笔记并更新热度分 |
| notification-consumer-group | interaction-event、relation-event、comment-event | 通知服务 | 生成通知 |
| push-consumer-group | push-request-event | 推送与实时投递服务 | 在线或离线投递 |
| im-message-consumer-group | im-message-event | IM 服务 | IM 内部消息状态处理 |
| order-seckill-consumer-group | order-seckill-task-event | 订单服务 | 神券秒杀异步下单 |
| order-timeout-consumer-group | order-timeout-event | 订单服务 | 超时关单 |
| search-index-consumer-group | note-event | 搜索服务后续 | 搜索索引更新 |

规则：

1. 不同服务使用不同 consumer group。
2. 同一服务不同语义可以使用不同 consumer group。
3. 消费组名称必须稳定，不能随版本变化。
4. 生产环境和测试环境消费组隔离。

## 8. 消息基础结构

所有领域事件建议使用统一信封：

```json
{
  "eventId": "uuid-or-snowflake-id",
  "eventType": "NotePublished",
  "eventVersion": 1,
  "occurredAt": "2026-06-03T16:00:00+08:00",
  "producer": "bluenote-note",
  "traceId": "trace-id",
  "bizKey": "noteId",
  "payload": {}
}
```

字段说明：

| 字段 | 必填 | 说明 |
|---|---|---|
| eventId | 是 | 全局唯一事件 ID |
| eventType | 是 | 事件类型 |
| eventVersion | 是 | 事件版本 |
| occurredAt | 是 | 事件发生时间 |
| producer | 是 | 生产服务 |
| traceId | 是 | 链路追踪 ID |
| bizKey | 建议 | 业务主键，用于顺序、排查、幂等辅助 |
| payload | 是 | 业务数据 |

要求：

1. `eventId` 全局唯一。
2. `eventType` 稳定。
3. `eventVersion` 用于兼容演进。
4. `payload` 只放消费者需要的信息。
5. 不放大文件和过大的 JSON。

## 9. 生产者策略

### 9.1 普通消息

适合：

1. 点赞收藏事件。
2. 评论事件。
3. 关注事件。
4. 排行榜更新触发。
5. 通知生成。

要求：

1. 发送失败要记录。
2. 关键事件不能只打印日志。
3. 生产者要设置合理超时。
4. 不在用户请求中等待非必要消费者完成。

### 9.2 事务消息

适合：

1. 神券订单创建。
2. 订单支付成功。
3. 订单关闭。
4. 用户券发放成功。
5. 关键笔记发布事件，后续视复杂度决定。

事务消息用于保证：

1. 本地事务成功后消息最终可见。
2. 本地事务失败后消息不投递。
3. RocketMQ 回查时可以根据本地事务状态决定提交或回滚。

注意：

1. 事务消息不是跨服务强事务。
2. 消费端仍然必须幂等。
3. 本地事务状态表必须可靠。

### 9.3 Outbox 模式

第一阶段推荐广泛使用 Outbox 模式，尤其在业务事件较多时更容易排查。

流程：

```text
业务写库事务
  -> 写业务表
  -> 写 outbox_event 表
  -> 提交事务
  -> 后台任务扫描 outbox_event
  -> 发送 RocketMQ
  -> 发送成功后标记 sent
```

优点：

1. 本地事务和事件记录一致。
2. 易于补偿。
3. 易于排查。
4. 不强绑定 MQ 事务机制。

缺点：

1. 事件投递有轻微延迟。
2. 需要后台投递任务。
3. outbox 表需要清理和归档。

### 9.4 延时消息

适合：

1. 付费神券订单超时关单。
2. 文件上传超时清理。
3. 未支付提醒，后续扩展。
4. 审核超时检查，后续扩展。

付费神券订单关单流程：

```text
创建 WAIT_PAY 订单
  -> 订单状态 WAIT_PAY
  -> expire_at 写入 MySQL
  -> 发送 OrderTimeoutCheck 延时消息
  -> 到期消费
  -> 查询订单状态
  -> 若仍是 WAIT_PAY 且已过期，CAS 关闭
```

要求：

1. 延时消息只是触发器。
2. 订单状态以 MySQL 为准。
3. 必须有数据库定时扫描兜底。
4. 支付回调和关单消费必须幂等。
5. 免费神券直接发券，不需要发送关单延时消息。

### 9.5 顺序消息

适合：

1. 同一 IM 会话内消息顺序。
2. 同一订单状态流转顺序。
3. 同一业务对象的少量状态变更。

顺序键建议：

| 场景 | 顺序键 |
|---|---|
| IM 会话消息 | conversationId |
| 订单状态事件 | orderId |
| 笔记状态变更 | noteId |

注意：

1. 只要求局部有序，不要求全局有序。
2. 顺序消息会降低并发度。
3. 顺序键热点会造成单队列压力。

## 10. 消费者策略

### 10.1 幂等处理

消费者必须实现幂等。

常用方式：

1. 数据库唯一约束。
2. `mq_consume_record` 表。
3. Redis 幂等 key。
4. 业务状态机判断。

建议：

1. 核心业务使用数据库幂等记录。
2. 高频非核心事件可以使用 Redis 幂等，配合补偿。
3. 订单、支付、关单必须使用数据库幂等。

### 10.2 消费记录表

建议每个消费服务维护：

```text
mq_consume_record
```

字段：

| 字段 | 说明 |
|---|---|
| id | 主键 |
| consumer_group | 消费组 |
| event_id | 事件 ID |
| event_type | 事件类型 |
| biz_key | 业务键 |
| status | 处理状态 |
| retry_count | 本地处理重试次数 |
| error_message | 最近错误 |
| consumed_at | 消费完成时间 |
| created_at | 创建时间 |
| updated_at | 更新时间 |

唯一索引：

```text
consumer_group + event_id
```

### 10.3 重试策略

可重试错误：

1. 数据库临时不可用。
2. Redis 临时不可用。
3. 下游接口短暂失败。
4. 网络超时。
5. 锁冲突。

不可重试错误：

1. 消息体格式错误。
2. 必填字段缺失。
3. 业务对象不存在且确认不可恢复。
4. 版本不兼容且无法解析。

处理方式：

1. 可重试错误抛出异常，让 MQ 重试。
2. 不可重试错误记录失败并告警，不无限重试。
3. 多次失败进入死信或本地失败表。

### 10.4 死信处理

死信消息要求：

1. 可查询。
2. 可人工重放。
3. 可标记忽略。
4. 可查看失败原因。

建议建立内部运维能力：

1. 查看失败消息。
2. 根据 eventId 重试。
3. 根据时间范围批量重放。
4. 标记不可恢复消息。

第一阶段可以先通过数据库失败表 + 日志 + RocketMQ 控制台处理。

## 11. 典型事件流

### 11.1 笔记发布

```text
笔记服务
  -> 保存笔记
  -> 发布 NotePublished
  -> Feed 服务扩散
  -> 排行榜服务加入候选
  -> 通知服务后续按场景生成通知
  -> 如需提醒则发布 PushSendRequested
```

关键要求：

1. 笔记保存成功后才能发布事件。
2. Feed 扩散失败不回滚笔记发布。
3. Feed 服务可通过补偿任务重建。

### 11.2 点赞笔记

```text
笔记服务或互动模块
  -> 写点赞明细
  -> 发布 NoteLiked
  -> 计数服务增加点赞数
  -> 计数服务发布 CounterChanged
  -> 排行榜服务根据 CounterChanged 更新热度分
  -> 通知服务生成点赞通知
  -> 如需提醒则发布 PushSendRequested
```

关键要求：

1. 点赞明细有唯一约束。
2. 重复点赞必须幂等。
3. 计数可以最终一致。

### 11.3 关注用户

```text
关系服务
  -> 写关注关系
  -> 发布 UserFollowed
  -> 计数服务更新关注数和粉丝数
  -> Feed 服务可补充作者近期笔记
  -> 通知服务生成关注通知
  -> 通知服务发布 PushSendRequested
```

关键要求：

1. 关注关系是事实来源。
2. 计数错误可从关注关系重建。
3. Feed 补充失败可重试。

### 11.4 评论笔记

```text
评论服务
  -> 保存评论
  -> 发布 CommentCreated
  -> 计数服务增加评论数
  -> 计数服务发布 CounterChanged
  -> 排行榜服务根据 CounterChanged 更新热度分
  -> 通知服务生成评论通知
  -> 通知服务发布 PushSendRequested
```

关键要求：

1. 评论可见性和状态以评论服务为准。
2. 评论数最终可由评论表重建。
3. 通知失败不影响评论发布。

### 11.5 IM 消息投递

```text
IM 服务
  -> 接收移动端发消息请求
  -> 消息入库
  -> 发布 ImMessageSent
  -> 发布 PushSendRequested
  -> 推送服务在线 WebSocket 或离线系统 Push 提醒接收方
  -> 接收方拉取 IM 消息详情
  -> 客户端 ACK 后更新状态
```

关键要求：

1. 消息入库先于投递。
2. 同一会话内使用 `conversationSeq` 保证展示顺序。
3. 推送失败不影响 IM 消息事实。
4. 客户端 ACK 幂等。

### 11.6 神券秒杀订单

```text
订单服务
  -> 秒杀接口通过 Redis Lua 预扣库存和用户限购
  -> 发布 CouponSeckillAccepted 到 order-seckill-task-event
  -> 订单服务异步下单消费者消费
  -> MySQL 扣减活动库存并创建订单
  -> 免费神券直接发券并发布 CouponIssued
  -> 付费神券创建 WAIT_PAY 订单并发送 OrderTimeoutCheck 延时消息
  -> 支付成功后发布 OrderPaid 和 CouponIssued
  -> 超时未支付则关闭订单并发布 OrderClosed
```

关键要求：

1. Redis 只做前置拦截，MySQL 才是库存、订单和用户券事实来源。
2. `CouponSeckillAccepted` 是订单服务内部异步下单任务，消费者必须幂等。
3. 延时消息不能作为订单状态依据。
4. 支付成功和超时关单要用状态机控制。
5. 使用 CAS 避免并发状态覆盖。
6. 数据库扫描任务兜底。

## 12. 消息版本管理

事件结构必须支持演进。

原则：

1. 新增字段必须向后兼容。
2. 不直接删除消费者正在使用的字段。
3. 重大变更提升 `eventVersion`。
4. 消费者根据 eventType + eventVersion 解析。
5. 保留一段时间双版本兼容。

不建议：

1. 频繁改字段名。
2. 在 payload 中塞服务内部 Entity。
3. 让消费者依赖生产者数据库结构。

## 13. 消息大小与内容限制

要求：

1. 消息只传必要字段。
2. 不传图片、视频、文件二进制。
3. 不传过大的正文快照，除非消费者确实需要。
4. 大对象用 ID 引用。
5. 敏感字段不进入消息，除非经过明确评估。

建议：

1. 单条消息控制在较小 JSON。
2. payload 中使用业务 ID + 少量展示字段。
3. 消费者需要完整数据时调用源服务批量接口。

## 14. 监控指标

必须监控：

1. Topic 消息堆积。
2. Consumer Group 消费延迟。
3. 消费失败次数。
4. 重试次数。
5. 死信数量。
6. 生产者发送失败次数。
7. Broker 磁盘使用率。
8. Broker JVM 内存。
9. 订单延时消息消费延迟。
10. IM 消息投递延迟。

告警建议：

| 指标 | 告警条件 |
|---|---|
| 消费堆积 | 持续增长超过阈值 |
| 死信数量 | 大于 0 且持续存在 |
| 发送失败 | 连续失败或错误率升高 |
| Broker 磁盘 | 超过 80% |
| 订单关单延迟 | 超过业务允许窗口 |

## 15. 环境隔离

要求：

1. local、test、prod 使用不同 topic 或不同 MQ 实例。
2. 不允许测试消费者订阅生产 topic。
3. 不允许本地开发误连生产 MQ。
4. 生产 topic 权限需要收紧。
5. 生产消息重放必须有明确操作记录。

## 16. 风险与应对

| 风险 | 说明 | 应对 |
|---|---|---|
| 重复消费 | MQ 可能重复投递 | 消费幂等、唯一约束 |
| 消息丢失 | 发送失败或事务不一致 | Outbox、事务消息、补偿 |
| 消息堆积 | 消费能力不足或消费者故障 | 监控告警、扩容、降级 |
| 顺序消息热点 | 单个 conversationId 或 orderId 流量过大 | 局部有序、热点隔离 |
| 延时消息不准时 | 延迟受 Broker 和堆积影响 | 数据库状态为准、定时扫描兜底 |
| 事件版本不兼容 | 生产者升级破坏消费者 | eventVersion、兼容期 |
| 消息体过大 | Broker 压力和网络开销 | 只传 ID 和必要字段 |
