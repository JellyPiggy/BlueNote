# BlueNote 笔记发布服务设计

## 1. 背景与目标

笔记发布服务负责 BlueNote 的核心内容资产，包括笔记创建、草稿、发布、编辑、删除、详情查询、作者笔记列表、媒体顺序、可见性控制，以及笔记点赞、收藏这类直接围绕笔记发生的互动行为事实。

这里需要特别区分两类数据：

1. 点赞、收藏的行为明细，例如“用户 A 是否点赞了笔记 B”，第一阶段归笔记服务维护。
2. 点赞数、收藏数、评论数等聚合计数，归计数服务维护，笔记服务不通过 `COUNT(*)` 实时统计展示计数。

这样设计是为了同时满足两个目标：

1. 笔记服务掌握笔记状态、可见性和互动事实，可以正确判断重复点赞、取消点赞、收藏列表等业务语义。
2. 计数服务专注高频读写的累计数字，使用 Redis Hash 和异步事件承载展示计数，避免 MySQL 明细表承担高频计数压力。

设计目标：

1. 支持图文笔记发布、编辑、删除和详情查询。
2. 支持后端草稿，避免用户跨设备或异常退出时内容丢失。
3. 支持笔记图片、封面、展示顺序和后续视频扩展。
4. 支持公开和私密两种第一阶段可见性。
5. 支持笔记点赞、取消点赞、收藏、取消收藏和收藏列表。
6. 支持笔记详情、作者笔记列表、批量笔记摘要查询。
7. 支持内容版本，为后续审核通过后再生效打基础。
8. 支持内容状态变更事件，解耦 Feed、排行榜、计数、通知等下游服务。
9. 支持缓存、幂等、补偿、日志、指标和告警。
10. 为审核、搜索、推荐、话题、视频处理、互动服务拆分预留扩展空间。

关键约束：

1. MySQL 是笔记、媒体绑定、点赞收藏明细的事实来源。
2. Redis 只作为笔记详情缓存、互动状态缓存和限流辅助，不作为笔记事实来源。
3. 文件本体不进入笔记服务数据库，只保存 `fileId` 和媒体业务顺序。
4. 笔记服务不直接访问 MinIO，只调用文件服务校验、绑定和获取访问 URL。
5. 计数展示值归计数服务，笔记服务不维护最终点赞数、收藏数、评论数。
6. Feed、排行榜、通知不能直接修改笔记服务数据。

## 2. 功能范围

### 2.1 第一阶段支持

| 功能 | 说明 |
|---|---|
| 创建草稿 | 保存标题、正文、媒体 fileId、可见性等草稿内容 |
| 发布笔记 | 校验作者状态、媒体文件、内容规则后发布 |
| 编辑笔记 | 作者可编辑自己的笔记，生成新版本 |
| 删除笔记 | 作者逻辑删除自己的笔记，下游通过事件清理 |
| 查询笔记详情 | 返回笔记内容、媒体、作者摘要、计数、当前用户互动状态 |
| 作者笔记列表 | 查询某个用户已发布且当前访问者可见的笔记列表 |
| 我的笔记列表 | 查询当前用户自己的草稿、已发布、私密、审核失败等笔记 |
| 笔记点赞 | 记录点赞明细并发布 `NoteLiked` 事件 |
| 取消点赞 | 更新点赞明细并发布 `NoteUnliked` 事件 |
| 笔记收藏 | 记录收藏明细并发布 `NoteCollected` 事件 |
| 取消收藏 | 更新收藏明细并发布 `NoteUncollected` 事件 |
| 我的收藏列表 | 查询当前用户收藏过且仍可见的笔记 |
| 媒体绑定 | 保存笔记图片顺序、封面 fileId，并调用文件服务绑定 |
| 内容基础校验 | 标题、正文长度、图片数量、敏感词、发布频率 |
| 详情缓存 | Redis 缓存笔记详情或摘要，状态变更时失效 |
| 事件发布 | 通过 outbox 发布笔记和互动事件 |
| 补偿任务 | 文件绑定、outbox 投递、缓存失效等补偿 |

### 2.2 第一阶段不支持

| 功能 | 暂不支持原因 |
|---|---|
| 独立互动服务 | 点赞收藏规模未验证，先由笔记服务维护行为明细 |
| 完整审核中心 | 暂无审核团队和第三方审核服务，先预留状态和事件 |
| 推荐页 Feed | 推荐系统不是第一阶段目标 |
| 全站搜索 | 暂不引入 Elasticsearch |
| 复杂话题系统 | 第一阶段只保存简单话题文本或标签 |
| 视频正式开放 | MinIO 单机、2 核 4G 服务器资源有限，先预留结构 |
| 长文章对象存储 | BlueNote 第一阶段正文较短，正文直接存 MySQL 版本表 |
| 多人协作编辑 | 当前没有协作内容场景 |
| 付费内容 | 当前订单能力未展开 |
| 复杂可见性 | `FOLLOWER_ONLY` 等关系型可见性后续依赖用户关系服务再开放 |

### 2.3 后续扩展

后续可以扩展：

1. 拆分 `bluenote-interaction` 服务，承接点赞、收藏、浏览、分享等行为明细。
2. 接入审核服务，支持先审后发、发后审、热门内容追加审核。
3. 接入搜索服务，支持标题、正文、话题、用户的搜索。
4. 接入推荐系统，消费笔记和互动事件。
5. 完善话题服务，支持话题页、话题热榜、话题订阅。
6. 开放视频笔记，接入视频转码、封面抽帧和多码率播放。
7. 增加内容举报、申诉、管理端下架和恢复。
8. 增加内容版权、水印、防搬运等能力。
9. 增加浏览记录、分享记录和转发。

这些扩展不应改变“笔记事实归笔记服务、聚合计数归计数服务、文件本体归文件服务”的基本边界。

## 3. 服务边界

### 3.1 笔记服务负责

笔记服务负责：

1. 笔记主体元数据。
2. 笔记正文和版本。
3. 笔记媒体 fileId 列表、顺序和封面。
4. 笔记生命周期状态。
5. 笔记可见性。
6. 作者笔记列表。
7. 笔记详情查询。
8. 点赞和收藏行为明细。
9. 当前用户是否点赞、是否收藏的判断。
10. 发布笔记相关事件和互动事件。

典型数据包括：

| 数据 | 说明 |
|---|---|
| `noteId` | 笔记 ID |
| `authorId` | 作者用户 ID |
| `title` | 标题 |
| `content` | 正文，第一阶段限制长度后存 MySQL |
| `noteType` | `IMAGE_TEXT` / `VIDEO` |
| `visibility` | `PUBLIC` / `PRIVATE` / 后续 `FOLLOWER_ONLY` |
| `noteStatus` | 草稿、发布中、已发布、审核中、已下架、已删除等 |
| `currentVersion` | 当前对外生效版本 |
| `latestVersion` | 最近一次编辑版本 |
| `coverFileId` | 封面文件 ID |
| `mediaFileIds` | 图片或视频文件 ID 列表 |
| `topicNames` | 第一阶段简单话题文本 |
| `commentEnabled` | 是否允许评论 |

### 3.2 文件服务负责

文件服务负责：

1. 上传凭证。
2. 文件元数据。
3. MinIO 对象访问。
4. 文件归属、场景、大小、类型校验。
5. 文件绑定和解绑。
6. 文件 URL 生成。
7. 文件清理和删除。

笔记服务不保存 bucket、objectKey 作为业务事实，也不直接操作 MinIO。

### 3.3 用户服务负责

用户服务负责：

1. 用户资料。
2. 用户状态。
3. 用户公开摘要。
4. 用户是否允许发布、编辑、点赞、收藏等业务操作的状态校验。

笔记服务在关键写操作前调用用户服务校验用户状态，不能只依赖异步事件快照。

### 3.4 计数服务负责

计数服务负责：

1. 笔记点赞数。
2. 笔记收藏数。
3. 笔记评论数。
4. 用户作品数。
5. 用户获赞数。
6. Redis Hash 计数读写。
7. 计数校准和重建。

笔记服务发布 `NoteLiked`、`NoteUnliked`、`NoteCollected`、`NoteUncollected`、`NotePublished`、`NoteDeleted` 等事件，计数服务消费后更新聚合计数。

### 3.5 Feed 服务负责

Feed 服务负责关注页收件箱、Feed 扩散和 Feed 读模型。

笔记服务只发布笔记状态事件，不直接写 Feed 收件箱。

### 3.6 排行榜服务负责

排行榜服务负责热度分计算、榜单 ZSet 和榜单快照。

笔记服务只提供笔记摘要查询和事件，不负责热度分。

### 3.7 评论服务负责

评论服务负责评论内容、二级评论、评论删除、评论点赞。

评论服务发布 `CommentCreated`、`CommentDeleted` 事件后，计数服务更新评论数。笔记服务不直接维护评论数。

## 4. 核心概念

### 4.1 笔记 ID

`noteId` 是笔记全局唯一 ID。

原则：

1. 由笔记服务生成。
2. 对外 API 返回字符串，避免前端数字精度问题。
3. 所有服务之间引用笔记时使用 `noteId`。
4. 不在 `noteId` 中暴露业务含义。

### 4.2 笔记类型

第一阶段主要支持图文笔记。

| 类型 | 说明 |
|---|---|
| `IMAGE_TEXT` | 图文笔记，第一阶段主类型 |
| `VIDEO` | 视频笔记，第一阶段预留 |

图文笔记规则建议：

1. 图片数量 1 到 9 张。
2. 正文最多 2000 字，标题最多 60 字。
3. 必须有封面，默认使用第一张图片。
4. 视频字段保留，但第一阶段可以通过配置关闭。

### 4.3 笔记状态

`noteStatus` 表示笔记生命周期。

| 状态 | 说明 |
|---|---|
| `DRAFT` | 草稿，只作者可见，不进入 Feed 和排行榜 |
| `PUBLISHING` | 发布中，文件绑定或状态切换处理中，不对外展示 |
| `AUDIT_PENDING` | 待审核，后续审核服务使用 |
| `AUDIT_REJECTED` | 审核拒绝，只作者可见拒绝原因 |
| `PUBLISHED` | 已发布，可按可见性规则展示 |
| `OFFLINE` | 管理端或系统下架，不对普通用户展示 |
| `DELETED` | 作者删除或系统删除，不对外展示 |

状态流转：

```text
DRAFT
  -> PUBLISHING
  -> PUBLISHED

PUBLISHED
  -> PUBLISHING
  -> PUBLISHED

PUBLISHED
  -> OFFLINE
  -> PUBLISHED

PUBLISHED
  -> DELETED

AUDIT_PENDING
  -> PUBLISHED
  -> AUDIT_REJECTED
```

第一阶段如果不接审核服务，发布和编辑可以走：

```text
DRAFT/PUBLISHED -> PUBLISHING -> PUBLISHED
```

但表结构和事件保留审核状态，后续接入审核时不用推翻笔记版本模型。

### 4.4 可见性

`visibility` 表示用户设置的可见范围。

| 可见性 | 第一阶段 | 说明 |
|---|---|---|
| `PUBLIC` | 支持 | 公开可见，可进入 Feed 和排行榜 |
| `PRIVATE` | 支持 | 仅作者可见，不进入 Feed 和排行榜 |
| `FOLLOWER_ONLY` | 预留 | 仅粉丝可见，依赖用户关系服务 |

可见性和状态必须同时判断：

```text
可展示 = noteStatus = PUBLISHED 且 visibility 规则允许当前访问者查看
```

私密笔记可以保存在服务端，但不会被 Feed、排行榜、通知等下游扩散。

### 4.5 笔记版本

笔记内容支持版本。

核心字段：

| 字段 | 说明 |
|---|---|
| `currentVersion` | 当前对外生效版本 |
| `latestVersion` | 作者最近一次提交版本 |
| `versionStatus` | 某个版本的状态 |

版本设计原因：

1. 后续审核服务需要“新版本审核通过后再覆盖线上版本”。
2. 编辑失败或审核拒绝不能破坏当前已发布版本。
3. 可以追溯内容修改历史。
4. 可以避免旧事件覆盖新内容。

第一阶段不接审核时，编辑成功后可以直接让 `currentVersion = latestVersion`。

### 4.6 笔记媒体

笔记媒体指笔记引用的图片或视频文件。

原则：

1. 笔记服务保存 `fileId`、媒体类型、排序、封面标记。
2. 文件服务保存文件元数据和对象存储信息。
3. 文件服务不保存笔记图片顺序。
4. 业务服务和移动端不依赖 MinIO objectKey。
5. 删除或替换媒体时，笔记服务调用文件服务解绑旧文件。

### 4.7 点赞和收藏行为事实

点赞和收藏行为事实用于回答：

1. 当前用户是否点赞了这篇笔记。
2. 当前用户是否收藏了这篇笔记。
3. 用户收藏列表里有哪些笔记。
4. 某次点赞或收藏是否重复请求。
5. 取消点赞或取消收藏是否合法。

这些事实第一阶段归笔记服务维护。

计数服务只维护展示数值，例如：

```text
counter:note:{noteId}
  like_count
  collect_count
  comment_count
```

笔记服务不能为了展示点赞数而频繁执行：

```sql
SELECT COUNT(*) FROM note_like WHERE note_id = ?
```

## 5. 核心业务流程

### 5.1 创建草稿

流程：

```text
移动端提交草稿
  -> 网关校验 Access Token
  -> 笔记服务获取 authorId
  -> 校验标题、正文、媒体数量
  -> 调用用户服务校验作者状态
  -> 如有 fileId，调用文件服务批量校验
  -> 保存 note、note_version、note_media，状态 DRAFT
  -> 如有媒体，调用文件服务 batch-bind
  -> 返回 noteId 和版本号
```

草稿特点：

1. 只作者可见。
2. 不发布 `NotePublished`。
3. 不进入 Feed、排行榜、计数。
4. 可以绑定媒体，避免文件服务把草稿图片当作未绑定文件清理。
5. 如果文件绑定失败，需要记录补偿任务。

### 5.2 发布新笔记

发布前移动端应先通过文件服务上传图片并拿到 `fileId`。

流程：

```text
移动端提交发布请求
  -> 网关校验登录态
  -> 笔记服务校验 clientRequestId 幂等
  -> 调用用户服务校验作者状态
  -> 校验标题、正文、图片数量、可见性
  -> 调用文件服务 batch-validate 校验 fileId
  -> 本地事务创建 note、note_version、note_media，状态 PUBLISHING
  -> 调用文件服务 batch-bind
  -> 本地事务更新 note 状态为 PUBLISHED
  -> 写 note_outbox_event(NotePublished)
  -> 删除或预热笔记详情缓存
  -> 返回发布成功
```

关键点：

1. 文件绑定成功前，笔记保持 `PUBLISHING`，不对普通用户展示。
2. `NotePublished` 只能在笔记已经可对外展示后发布。
3. 私密笔记也可以发布，但事件中必须携带 `visibility = PRIVATE`，下游不能扩散到公开 Feed 或榜单。
4. 发布请求必须携带 `clientRequestId`，避免移动端重试导致重复创建。

### 5.3 从草稿发布

流程：

```text
移动端发布草稿
  -> 校验 noteId 属于当前用户
  -> 校验当前状态为 DRAFT 或 AUDIT_REJECTED
  -> 校验最新版本内容和媒体
  -> 状态改为 PUBLISHING
  -> 文件服务补齐绑定
  -> 状态改为 PUBLISHED
  -> 写 outbox NotePublished
```

如果草稿文件已经被清理：

```text
返回 NOTE_MEDIA_INVALID
```

移动端需要提示用户重新选择图片。

### 5.4 编辑已发布笔记

流程：

```text
移动端提交编辑
  -> 校验登录态
  -> 校验作者权限
  -> 校验笔记状态允许编辑
  -> 校验内容和媒体
  -> 创建新的 note_version
  -> 创建新版本 note_media
  -> 调用文件服务绑定新媒体
  -> 第一阶段直接切换 currentVersion
  -> 删除笔记详情缓存
  -> 写 outbox NoteUpdated
  -> 异步解绑旧版本不再使用的媒体
```

后续接入审核后，编辑流程变为：

```text
创建新版本
  -> AUDIT_PENDING
  -> 审核通过后 currentVersion 切到新版本
  -> 发布 NoteUpdated
```

审核拒绝时，当前线上版本不变。

### 5.5 删除笔记

流程：

```text
作者删除笔记
  -> 校验登录态
  -> 校验作者权限
  -> 校验状态允许删除
  -> 本地事务更新 noteStatus = DELETED
  -> 写 note_operation_log
  -> 写 outbox NoteDeleted
  -> 删除 Redis 缓存
  -> 异步调用文件服务 unbind
```

删除影响：

1. 笔记详情普通用户不可见。
2. 作者自己的删除记录是否展示，后续按产品设计。
3. Feed 服务消费 `NoteDeleted` 后移除或标记不可见。
4. 排行榜服务消费后移除候选。
5. 计数服务可以保留历史计数，也可以按策略减少作品数。
6. 文件不立即物理删除，由文件服务生命周期策略处理。

### 5.6 查询笔记详情

流程：

```text
移动端请求详情
  -> 网关透传 viewerId，可允许未登录后续再评估
  -> 笔记服务读取详情缓存
  -> 未命中则查 note、note_version、note_media
  -> 判断 noteStatus 和 visibility
  -> 调用用户服务获取作者摘要
  -> 调用文件服务批量生成媒体访问 URL
  -> 调用计数服务获取聚合计数
  -> 查询当前用户点赞收藏状态
  -> 组装响应并回填缓存
```

降级策略：

1. 用户服务失败时，作者摘要可以返回基础 `authorId` 和降级标记。
2. 计数服务失败时，计数返回 0 或缓存值，并标记 `degraded = true`。
3. 文件服务访问 URL 失败时，详情可以返回媒体占位状态，但要记录告警。
4. 笔记状态为 `DELETED`、`OFFLINE` 时，普通用户返回 `NOTE_NOT_FOUND` 或 `NOTE_UNAVAILABLE`。

### 5.7 作者笔记列表

流程：

```text
移动端查询用户主页笔记列表
  -> 笔记服务按 authorId + status + visibility 游标分页
  -> 只返回当前访问者可见的笔记
  -> 批量查询媒体封面 URL
  -> 批量查询计数
  -> 返回笔记卡片列表
```

分页建议：

1. 使用游标分页，不使用深分页。
2. 游标由 `publishedAt + noteId` 组成。
3. 每页最大 20 条。

### 5.8 我的笔记列表

当前用户可以查询自己的：

1. 草稿。
2. 已发布。
3. 私密。
4. 审核失败。
5. 已下架。

作者查询自己的笔记时，可以看到非公开状态；其他用户只能看到可见笔记。

### 5.9 点赞笔记

流程：

```text
用户点击点赞
  -> 网关校验登录态
  -> 笔记服务校验用户状态
  -> 校验笔记存在且当前用户可见
  -> 查询 note_like 是否已存在
  -> 不存在则插入 ACTIVE
  -> 已取消则更新为 ACTIVE
  -> 已点赞则幂等返回成功
  -> 写 outbox NoteLiked，仅状态发生变化时写
  -> 删除当前用户互动状态缓存
  -> 返回 liked = true
```

幂等要求：

1. `note_like` 使用唯一约束 `note_id + user_id`。
2. 重复点赞不重复发送 `NoteLiked`。
3. 并发点赞通过唯一约束和状态判断兜底。

### 5.10 取消点赞

流程：

```text
用户取消点赞
  -> 校验登录态和用户状态
  -> 校验笔记是否存在
  -> 查询 note_like
  -> 已点赞则更新为 CANCELED
  -> 未点赞或已取消则幂等返回成功
  -> 写 outbox NoteUnliked，仅状态发生变化时写
  -> 删除互动状态缓存
```

取消点赞不要求笔记仍然公开可见。用户对已删除或已下架笔记取消点赞时，可以允许清理自己的行为状态。

### 5.11 收藏笔记

收藏流程和点赞类似，但收藏还需要支持“我的收藏列表”。

流程：

```text
用户收藏
  -> 校验用户状态
  -> 校验笔记存在且可见
  -> 写 note_collection
  -> 发布 NoteCollected
  -> 返回 collected = true
```

收藏列表查询以 `note_collection` 为入口，再批量查询笔记摘要。已删除、下架或当前用户不可见的笔记不应展示。

## 6. 异常流程

| 异常 | 处理方式 |
|---|---|
| 重复发布请求 | 根据 `clientRequestId` 幂等返回首次结果 |
| 作者不存在 | 返回 `NOTE_AUTHOR_INVALID` |
| 作者被禁用 | 返回 `USER_DISABLED` 或 `NOTE_AUTHOR_FORBIDDEN` |
| 标题过长 | 返回 `NOTE_TITLE_TOO_LONG` |
| 正文过长 | 返回 `NOTE_CONTENT_TOO_LONG` |
| 图片数量不合法 | 返回 `NOTE_MEDIA_COUNT_INVALID` |
| fileId 不存在 | 返回 `NOTE_MEDIA_INVALID` |
| fileId 不属于当前用户 | 返回 `NOTE_MEDIA_OWNER_MISMATCH` |
| 文件服务不可用 | 发布失败或进入 `PUBLISHING` 补偿，视阶段策略 |
| 文件绑定失败 | 笔记不切到 `PUBLISHED`，记录补偿任务 |
| 编辑旧版本冲突 | 返回 `NOTE_VERSION_CONFLICT` |
| 笔记不存在 | 返回 `NOTE_NOT_FOUND` |
| 笔记已删除 | 返回 `NOTE_NOT_FOUND` 或 `NOTE_DELETED` |
| 笔记已下架 | 普通用户不可见，作者可见下架状态 |
| 无权编辑或删除 | 返回 `NOTE_PERMISSION_DENIED` |
| 私密笔记被他人访问 | 返回 `NOTE_NOT_FOUND`，避免暴露存在性 |
| 重复点赞 | 幂等成功，不重复发事件 |
| 重复取消点赞 | 幂等成功，不重复发事件 |
| Redis 不可用 | 降级查 MySQL，不影响核心写流程 |
| 计数服务不可用 | 详情计数降级，不影响笔记详情主数据 |
| RocketMQ 不可用 | outbox 保留待投递，不影响本地事务 |
| outbox 堆积 | 告警并重试投递 |

补偿重点：

1. `PUBLISHING` 状态超时。
2. 文件绑定失败。
3. 文件解绑失败。
4. outbox 事件投递失败。
5. 缓存删除失败。
6. 计数事件漏消费或重复消费。
7. 用户状态变化后笔记状态未及时处理。

### 7.12 索引设计

| 表名 | 索引 | 说明 |
|---|---|---|
| `note` | `PRIMARY KEY(note_id)` | 按笔记 ID 查询 |
| `note` | `idx_note_author_status_time(author_id, note_status, published_at, note_id)` | 作者笔记列表 |
| `note` | `idx_note_status_visibility_time(note_status, visibility, published_at, note_id)` | 公开笔记补偿、榜单候选 |
| `note` | `idx_note_updated(updated_at)` | 补偿和管理查询 |
| `note_version` | `uk_note_version(note_id, version_no)` | 版本唯一 |
| `note_media` | `idx_note_media_version(note_id, version_no, sort_order)` | 查询版本媒体 |
| `note_media` | `idx_note_media_file(file_id)` | 文件反查绑定 |
| `note_topic` | `idx_note_topic_name(topic_name, note_id)` | 后续话题查询 |
| `note_like` | `uk_note_like_user(note_id, user_id)` | 点赞幂等 |
| `note_like` | `idx_note_like_user_time(user_id, like_status, liked_at)` | 用户点赞记录，后续扩展 |
| `note_like` | `idx_note_like_note_time(note_id, like_status, liked_at)` | 点赞列表和校准 |
| `note_collection` | `uk_note_collection_user(note_id, user_id)` | 收藏幂等 |
| `note_collection` | `idx_note_collection_user_time(user_id, collection_status, collected_at, note_id)` | 我的收藏列表 |
| `note_operation_log` | `idx_note_op_note_time(note_id, created_at)` | 操作审计 |
| `note_idempotent_request` | `PRIMARY KEY(idempotent_key)` | 请求幂等 |
| `note_idempotent_request` | `idx_note_idem_expire(expire_at)` | 清理过期幂等记录 |
| `note_event_consume_log` | `PRIMARY KEY(event_id)` | MQ 消费幂等 |
| `note_outbox_event` | `idx_note_outbox_status_retry(send_status, next_retry_at)` | 扫描待投递事件 |
| `note_outbox_event` | `idx_note_outbox_aggregate(aggregate_id, event_type)` | 排查某笔记事件 |

索引原则：

1. 列表查询按 `author_id + status + published_at + note_id` 游标分页。
2. 点赞、收藏必须用唯一约束保护幂等。
3. 不在正文 `content` 上做普通索引。
4. 第一阶段不做复杂搜索，避免 MySQL 模糊查询拖慢主库。

## 8. Redis Key 设计

Redis 只作为缓存、限流和临时状态，不作为笔记事实来源。

### 8.1 笔记详情缓存

```text
bluenote:{env}:note:detail:{noteId}
```

数据结构：

```text
String(JSON)
```

内容：

1. noteId
2. authorId
3. title
4. content
5. coverFileId
6. mediaFileIds
7. visibility
8. noteStatus
9. currentVersion
10. publishedAt

TTL：

```text
5 分钟到 30 分钟，加随机抖动
```

失效时机：

1. 笔记编辑成功。
2. 笔记删除。
3. 笔记下架。
4. 可见性变化。
5. 文件被封禁。

### 8.2 笔记摘要缓存

```text
bluenote:{env}:note:summary:{noteId}
```

用于 Feed、排行榜、收藏列表批量补全时降低 MySQL 压力。

TTL：

```text
10 分钟到 1 小时
```

### 8.3 用户互动状态缓存

```text
bluenote:{env}:note:viewer-action:{userId}:{noteId}
```

数据结构：

```text
Hash
  liked
  collected
```

TTL：

```text
5 分钟到 30 分钟
```

说明：

1. 只用于减少详情页重复查点赞收藏明细。
2. 写点赞、收藏状态变化后删除该缓存。
3. Redis 不可用时直接查 MySQL。

### 8.4 发布频率限制

```text
bluenote:{env}:note:rate:publish:{userId}
bluenote:{env}:note:rate:edit:{userId}
bluenote:{env}:note:rate:interaction:{userId}
```

说明：

1. 发布、编辑、点赞收藏都需要限流。
2. 限流失败不影响已保存数据。
3. Redis 不可用时可降级为更严格的网关限流或记录风险日志。

### 8.5 热点笔记本地缓存

第一阶段可以暂不引入本地缓存。后续热点笔记详情 QPS 升高后，可在应用内使用短 TTL 本地缓存。

本地缓存必须监听笔记事件或依赖短 TTL，避免下架和删除后长时间可见。

## 10. 安全与风控设计

### 10.1 身份和权限

要求：

1. 发布、编辑、删除、点赞、收藏必须登录。
2. `authorId` 从网关注入的用户上下文获取，不能信任客户端传入。
3. 只有作者本人可以编辑和删除笔记。
4. 管理端操作必须使用独立权限体系，不能复用普通移动端权限。
5. 内部接口必须服务间鉴权，不能暴露公网。

### 10.2 用户状态校验

以下操作前必须调用用户服务或使用强实时状态校验：

1. 发布笔记。
2. 编辑笔记。
3. 点赞笔记。
4. 收藏笔记。

用户状态为 `DISABLED`、`CANCELLED` 时禁止写操作。`LIMITED` 状态按后续策略可以禁止发布和评论，但允许浏览。

### 10.3 内容安全

第一阶段内容安全策略：

1. 标题和正文做长度限制。
2. 标题和正文做基础敏感词校验。
3. 图片和视频通过文件服务限制类型、大小、场景。
4. 预留 `audit_status` 和管理下架能力。
5. 对发布频率异常的用户限流并记录风险日志。

第一阶段不接第三方审核时，不能宣称具备完整内容安全能力，只能做到基础预防和后续治理。

### 10.4 防刷与限流

建议限制：

| 操作 | 限制 |
|---|---|
| 发布笔记 | 每分钟最多 3 次，每天最多可配置 |
| 保存草稿 | 每分钟最多 20 次 |
| 编辑笔记 | 每分钟最多 10 次 |
| 点赞收藏 | 每秒和每分钟双维度限流 |
| 查询详情 | 网关按 IP 和用户限流 |
| 批量摘要内部接口 | 单次最多 100 个 noteId |

限流使用网关 + Redis。Redis 不可用时，至少保留网关级粗粒度限流。

### 10.5 可见性安全

规则：

1. 私密笔记非作者访问时返回 `NOTE_NOT_FOUND`，不暴露存在性。
2. 下架和删除笔记普通用户不可见。
3. Feed、排行榜只处理公开笔记。
4. 内部批量摘要接口必须明确 `includeInvisible`，默认不返回不可见笔记。

## 11. 前后端实现要点

### 11.1 移动端实现

发布页流程：

1. 用户选择图片。
2. 调用文件服务申请上传凭证。
3. 移动端直传 MinIO。
4. 调用文件服务确认上传完成，拿到 `fileId`。
5. 移动端维护图片顺序、封面和话题。
6. 发布时提交 `fileId` 列表和 `clientRequestId`。
7. 发布成功后跳转笔记详情或个人主页。

实现要点：

1. 不保存 objectKey 作为业务字段。
2. 发布按钮要防重复点击，并依赖 `clientRequestId` 兜底。
3. 本地草稿可以和后端草稿结合，网络差时先保存在本地。
4. 上传成功但发布失败时，不需要移动端主动删除文件，后端清理任务处理。
5. 点赞收藏可以做乐观 UI，但必须以后端返回结果为准。
6. 详情页如果计数降级，可以展示 0 或隐藏部分计数，避免阻塞正文展示。

### 11.2 后端实现

后端模块建议：

```text
bluenote-note
  ├── api
  ├── application
  ├── domain
  ├── infrastructure
  │   ├── mysql
  │   ├── redis
  │   ├── client
  │   └── mq
  ├── mq
  └── job
```

实现要点：

1. Controller 只处理参数校验和用户上下文。
2. Application Service 编排用户校验、文件校验、事务、缓存、事件。
3. Domain 层封装笔记状态机、可见性规则、版本规则、互动幂等规则。
4. Infrastructure 层封装 MySQL、Redis、用户服务客户端、文件服务客户端、计数服务客户端和 RocketMQ。
5. 内部接口和外部接口分包，避免误暴露。
6. outbox 投递任务独立运行，失败可重试。
7. 文件绑定补偿任务独立运行。
8. 所有列表接口必须限制 page size。

### 11.3 文件服务集成

发布和编辑时调用：

```text
POST /internal/files/batch-validate
POST /internal/files/batch-bind
POST /internal/files/unbind
```

校验内容：

1. 文件存在。
2. 文件属于当前用户。
3. 文件场景为 `NOTE_IMAGE` 或后续 `NOTE_VIDEO`。
4. 文件状态为 `UPLOADED` 或允许复用的 `BOUND`。
5. 文件数量不超过限制。
6. 文件类型和大小符合笔记规则。

### 11.4 用户服务集成

写操作前调用：

```text
POST /internal/users/status-check
```

场景：

```text

```

详情和列表需要作者摘要时，调用：

```text

```

### 11.5 计数服务集成

详情和列表查询时调用计数服务批量查询：


## 12. 数据一致性与事件

### 12.1 本地事务边界

发布笔记时，本地事务负责：

1. 写 `note`。
2. 写 `note_version`。
3. 写 `note_media`。
4. 写 `note_topic`。
5. 写 `note_outbox_event`，仅在笔记真正可见时写。

文件服务调用不进入本地事务。通过 `PUBLISHING` 状态和补偿任务保证最终一致。

### 12.2 文件绑定一致性

推荐流程：

```text
本地事务创建 PUBLISHING 笔记
  -> 调用文件服务 batch-bind
  -> 成功后本地事务改为 PUBLISHED 并写 NotePublished
```

如果 `batch-bind` 失败：

1. 笔记保持 `PUBLISHING`，普通用户不可见。
2. 写补偿任务重试绑定。
3. 超过阈值后改为 `DRAFT` 或 `AUDIT_REJECTED` 类似的失败状态，具体按产品提示。
4. 不发布 `NotePublished`。

### 12.3 互动和计数一致性

点赞流程：

```text
本地事务：
  -> 插入或更新 note_like
  -> 写 outbox NoteLiked

异步：
  -> 计数服务消费 NoteLiked
  -> Redis Hash like_count +1
```

取消点赞同理。

原则：

1. 点赞事实以 `note_like` 为准。
2. 点赞数以计数服务为展示来源。
3. 计数服务可定期用 `note_like` 校准。
4. 重复点赞不重复增加计数。

### 12.4 缓存一致性

笔记更新流程：

```text
更新 MySQL
  -> 删除 note detail 缓存
  -> 删除 note summary 缓存
  -> 写 outbox NoteUpdated
```

如果缓存删除失败：

1. 记录失败日志。
2. 写缓存重删任务。
3. 依赖 TTL 最终修复。

### 12.5 发布事件

#### note-event

| 事件 | 触发时机 | 消费方 |
|---|---|---|
| `NotePublished` | 笔记对外发布成功 | Feed、排行榜、计数、通知、搜索后续 |
| `NoteUpdated` | 当前生效版本更新 | Feed、排行榜、搜索后续 |
| `NoteDeleted` | 笔记删除 | Feed、排行榜、计数、通知 |
| `NoteVisibilityChanged` | 可见性变化 | Feed、排行榜 |
| `NoteStatusChanged` | 下架、恢复、审核状态变化 | Feed、排行榜、搜索后续 |

#### interaction-event

| 事件 | 触发时机 | 消费方 |
|---|---|---|
| `NoteLiked` | 点赞状态从未点赞或取消变为已点赞 | 计数、排行榜、通知 |
| `NoteUnliked` | 点赞状态从已点赞变为取消 | 计数、排行榜 |
| `NoteCollected` | 收藏状态从未收藏或取消变为已收藏 | 计数、排行榜、通知 |
| `NoteUncollected` | 收藏状态从已收藏变为取消 | 计数、排行榜 |

```json
  "eventVersion": 1,
  "occurredAt": "2026-06-03T22:01:00+08:00",
  "producer": "bluenote-note",
  "traceId": "trace-id",
  "bizKey": "800001",
  "payload": {
    "noteId": "800001",
    "authorId": "10001",
    "noteType": "IMAGE_TEXT",
    "visibility": "PUBLIC",
    "noteStatus": "PUBLISHED",
    "currentVersion": 1,
    "title": "杭州周末咖啡店记录",
```

说明：

1. 事件只传摘要，不传完整正文和文件二进制。
2. `visibility` 必填，下游据此决定是否扩散。

### 12.7 NoteLiked 事件示例

```json
{
  "eventId": "evt_note_liked_800001_10002",
  "eventType": "NoteLiked",
  "eventVersion": 1,
  "occurredAt": "2026-06-03T22:05:00+08:00",
  "producer": "bluenote-note",
  "traceId": "trace-id",
  "bizKey": "800001",
  "payload": {
    "noteId": "800001",
    "authorId": "10001",
    "actorId": "10002",
    "likedAt": "2026-06-03T22:05:00+08:00"
  }
}
```

消费者幂等：

1. 计数服务按 `eventId` 或业务键记录消费。
2. 通知服务按 `noteId + actorId + eventType` 可做通知去重。
3. 排行榜服务重复消费时不能重复加分。

### 12.8 订阅事件

| 事件 | 来源 | 用途 |
|---|---|---|
| `UserStatusChanged` | 用户服务 | 用户被禁用或注销时，限制发布和必要时隐藏内容 |
| `UserProfileUpdated` | 用户服务 | 可选，用于更新笔记侧作者展示快照，第一阶段可不落库 |
| `FileBlocked` | 文件服务 | 文件违规时下架或隐藏引用该文件的笔记 |
| `MediaProcessCompleted` | 文件服务 | 后续视频处理完成后更新媒体状态 |

第一阶段关键写操作仍以同步接口校验为准，不能只依赖事件快照。

不记录：

1. 过长正文全文。
2. MinIO 预签名 URL 完整签名参数。
3. 敏感内部 Header。

### 13.3 告警

建议告警：

1. 发布失败率异常升高。
2. 文件绑定失败持续增加。
3. `PUBLISHING` 状态超时数量超过阈值。
4. outbox 堆积超过阈值。
5. Redis 缓存异常不可用。
6. 计数服务调用降级率异常升高。
7. 用户服务状态校验失败率异常升高。
8. MQ 消费失败或死信增加。
9. 笔记详情 P95 延迟异常。
10. 点赞收藏写入错误率异常。

## 14. 测试重点

### 14.1 单元测试

1. 笔记状态机流转。
2. 可见性判断。
3. 标题和正文长度校验。
4. 图片数量和封面规则。

10. 内容摘要生成。

### 14.2 集成测试

1. 发布笔记成功写入 `note`、`note_version`、`note_media`。
2. 文件服务校验失败时不发布笔记。
3. 文件绑定失败时笔记不进入 `PUBLISHED`。
4. 发布成功后写 outbox `NotePublished`。
5. 编辑成功后切换 `currentVersion`。
6. 删除笔记后缓存失效并发布 `NoteDeleted`。
7. 点赞成功后写 `note_like` 和 outbox `NoteLiked`。
8. 重复点赞不重复发送事件。
9. 收藏列表过滤已删除笔记。
10. 计数服务不可用时详情降级。

### 14.3 接口测试

1. 未登录不能发布、编辑、删除、点赞、收藏。
2. 用户不能编辑或删除他人笔记。
3. 私密笔记非作者不能访问。
4. 已删除笔记普通用户不可访问。
5. 图片数量为 0 时发布失败。
6. 超过 9 张图片发布失败。
7. 重复发布请求幂等返回同一 noteId。
8. `baseVersion` 过旧时编辑失败。
9. 批量摘要超过数量限制失败。
10. 取消未点赞笔记幂等成功。

### 14.4 性能测试

第一阶段重点压测：

1. 笔记详情查询。
2. 作者笔记列表。
3. 批量笔记摘要。
4. 点赞和取消点赞。
5. 收藏和取消收藏。
6. 发布笔记时文件服务延迟对接口的影响。

建议目标：

1. 笔记详情 Redis 命中场景 P95 小于 100ms。
2. 笔记详情 Redis 未命中且下游正常 P95 小于 300ms。
3. 作者笔记列表 P95 小于 200ms。
4. 点赞收藏接口 P95 小于 100ms，不等待计数服务消费。
5. 发布接口 P95 受文件服务影响，第一阶段目标小于 500ms。

## 15. 风险与后续演进

| 风险 | 说明 | 应对 |
|---|---|---|
| 点赞收藏放在笔记服务增加写压力 | 第一阶段服务少，但互动写会进入笔记库 | 唯一索引、异步计数、后续拆互动服务 |
| 文件绑定失败导致发布卡住 | 文件服务和笔记服务跨服务一致性 | `PUBLISHING` 状态 + 补偿任务 |
| 计数短暂不准 | 计数异步消费存在延迟或失败 | 计数服务幂等、重试、校准 |
| 私密笔记泄露 | Feed、排行榜误处理私密事件 | 事件携带 visibility，下游过滤，接口二次校验 |
| 删除后缓存未失效 | 用户仍可能短时间看到旧内容 | 删除缓存 + TTL + 重删任务 |
| 内容审核不足 | 第一阶段未接审核服务 | 敏感词、限流、举报和管理下架预留 |
| 视频资源消耗 | MinIO 单机和云服务器带宽有限 | 第一阶段关闭或严格限制视频 |
| 作者状态扩散延迟 | 用户被禁用后旧事件未处理 | 写操作实时调用用户服务状态校验 |
| MySQL 列表慢查询 | 作者笔记、收藏列表持续增长 | 游标分页、组合索引、限制 page size |

后续演进：

1. 当点赞、收藏、浏览、分享写入明显增多时，拆分 `bluenote-interaction`。
2. 接入审核服务后，把编辑版本改为审核通过后再生效。
3. 引入搜索服务后，消费 `NotePublished`、`NoteUpdated`、`NoteDeleted` 建索引。
4. 引入推荐系统后，消费笔记和互动事件作为推荐特征。
5. 开放视频笔记后，接入媒体处理任务和视频播放策略。
6. 增加举报、申诉、管理端内容治理。
7. 对冷门笔记和历史互动明细做归档。
8. 根据访问量增加本地缓存和热点保护。
