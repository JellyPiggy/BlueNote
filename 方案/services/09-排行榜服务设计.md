# BlueNote 排行榜服务设计

## 1. 背景与目标

排行榜服务负责 BlueNote 的榜单读模型和排名能力。它不保存笔记、评论、点赞、收藏等业务事实，而是消费笔记状态事件和计数变化事件，构建面向移动端展示和创作者激励的榜单。

第一阶段设计两个排行榜：

| 榜单 | rankCode | 成员 | 定位 |
|---|---|---|---|
| 本周热门笔记榜 | `WEEKLY_HOT_NOTE` | 笔记 `noteId` | 给普通用户发现热门内容 |
| 本年创作者成长榜 | `YEARLY_CREATOR_GROWTH` | 创作者 `userId` | 给创作者荣誉展示和成长激励 |

两个榜单采用不同的排名策略：

1. 本周热门笔记榜只展示 Top 100，使用 Redis ZSET 做精确排名，不做粗估排名。
2. 本年创作者成长榜展示 Top 100，同时支持创作者查询自己的大致排名，因此采用“Top 100 精确排名 + 尾部粗估排名”。

整体链路：

```text
笔记服务维护笔记状态、作者、发布时间、可见性
  -> 计数服务维护点赞、评论、收藏展示计数
  -> 排行榜服务消费 note-event 和 counter-event
  -> 维护榜单分数、Redis 排名、MySQL 快照和重建任务
```

设计目标：

1. 支持本周热门笔记榜 Top 100 精确展示。
2. 支持本年创作者成长榜 Top 100 精确展示。
3. 支持创作者查询本人本年成长榜排名，Top 100 内返回精确排名，Top 100 外返回粗估排名。
4. 热度分和成长分基于点赞、评论、收藏的周期内净增量计算。
5. 排行榜通过 `CounterChanged` 更新分数，不直接消费点赞、评论、收藏原始事件算分。
6. 通过 `note-event` 维护笔记是否可参与榜单，以及 `noteId -> authorId -> publishedAt` 的轻量索引。
7. 使用 MySQL 保存分数、来源贡献、榜单快照、消费幂等和重建任务。
8. 使用 Redis ZSET 保存在线精确榜，使用 Redis Hash 保存创作者成长榜的分段统计。
9. 支持榜单周期滚动、榜单快照、Redis 重建、分数校准和异常补偿。

关键约束：

1. 笔记状态以笔记服务为事实来源。
2. 点赞、评论、收藏计数以计数服务为展示来源。
3. 排行榜服务不判断用户是否点赞、是否收藏、是否评论。
4. 排行榜服务不直接访问其他服务数据库。
5. Redis 只是在线排名读模型，必须可以从 MySQL 和来源服务重建。
6. 私密、删除、下架、审核不通过的笔记不能贡献公开榜单分数。
7. 榜单允许最终一致，可以接受秒级到分钟级延迟。

## 2. 功能范围

### 2.1 第一阶段支持

| 功能 | 说明 |
|---|---|
| 本周热门笔记榜 | 按自然周统计笔记本周互动热度，展示 Top 100 |
| 本年创作者成长榜 | 按自然年统计创作者今年公开笔记获得的互动成长分，展示 Top 100 |
| 创作者排名查询 | 创作者查询本人今年成长榜排名，支持精确和粗估 |
| 精确排名 | Redis ZSET 维护两个榜单的 Top 100 |
| 粗估排名 | 仅用于本年创作者成长榜 Top 100 之外的排名估算 |
| 热度分计算 | 点赞、评论、收藏按权重计算 |
| 笔记轻量索引 | 保存 `noteId`、`authorId`、发布时间、状态、可见性 |
| 来源贡献记录 | 保存某篇笔记对某个榜单成员的分数贡献，便于状态变化时扣回 |
| 榜单快照 | 定时保存 Top 100 到 MySQL |
| 事件幂等 | 对 `note-event`、`counter-event` 做消费幂等 |
| Redis 重建 | 从 MySQL 分数和来源贡献重建 ZSET 与分段统计 |
| 分数校准 | 低频调用计数服务或重放分数流水修正异常 |
| 榜单裁剪 | Redis ZSET 对外 Top 100，内部保留少量缓冲 |

### 2.2 第一阶段不支持

| 功能 | 暂不支持原因 |
|---|---|
| 日榜、月榜 | 第一阶段先做好周榜和年榜两个典型场景 |
| 话题榜、城市榜 | 依赖话题体系、地理位置和隐私策略 |
| 热门大 V 粉丝榜 | 粉丝数更适合用户关系和计数体系后续单独设计 |
| 分享数、浏览数参与排名 | 当前还没有分享和浏览行为事实设计 |
| 运营置顶、人工加权 | 需要管理后台和运营权限体系 |
| 完整反作弊系统 | 第一阶段只做基础异常监控，刷榜治理由来源服务先兜底 |
| 精确全量长榜 | 创作者成长榜 Top 100 外用粗估排名即可 |
| 实时流式计算框架 | 当前不引入 Flink、Kafka 实时数仓 |

### 2.3 后续扩展

后续可以扩展：

1. 增加日榜、月榜、历史年度榜。
2. 增加话题热门笔记榜、城市热门笔记榜。
3. 接入分享数、浏览数、举报数、完播率等更多信号。
4. 引入时间衰减，避免榜单早期高分内容长期占位。
5. 引入反作弊服务，对异常互动降权。
6. 接入运营后台，支持人工下榜、置顶、冻结分数。
7. 将排行榜作为推荐系统召回源。
8. 数据量增大后按榜单或周期拆分 Redis 实例。

## 3. 服务边界

### 3.1 排行榜服务负责

排行榜服务负责：

1. 榜单定义和周期管理。
2. 热度分、成长分的计算规则。
3. 笔记轻量索引和候选资格快照。
4. 榜单成员当前分数。
5. 笔记对榜单成员的来源贡献记录。
6. Redis ZSET 精确榜维护。
7. 创作者成长榜分段统计维护。
8. 移动端榜单查询接口。
9. 内部排名查询、快照、重建、校准接口。
10. 消费幂等、失败重试、补偿任务。
11. 榜单日志、指标和告警。

### 3.2 非排行榜服务职责

排行榜服务不负责：

1. 发布、编辑、删除笔记。
2. 保存笔记正文和媒体最终事实。
3. 保存点赞、收藏、评论行为明细。
4. 维护点赞数、评论数、收藏数最终展示计数。
5. 判断用户是否刷赞、刷评论。
6. 维护 Feed 收件箱。
7. 维护粉丝、关注关系。
8. 承担推荐系统个性化排序。

### 3.3 依赖关系

| 依赖 | 方式 | 用途 |
|---|---|---|
| 笔记服务 | MQ + 同步接口 | 消费笔记状态事件，批量查询笔记摘要 |
| 计数服务 | MQ + 同步接口 | 消费计数变化事件，批量查询展示计数 |
| 用户服务 | 同步接口 | 查询创作者摘要和用户状态 |
| Redis | 存储 | ZSET 精确榜、分段统计、短缓存 |
| MySQL | 存储 | 分数、贡献、快照、幂等、任务 |
| RocketMQ | 消息 | 订阅 `note-event`、`counter-event`，发布 `rank-event` |

## 4. 核心概念

### 4.1 榜单定义

第一阶段固定两个榜单。

| rankCode | 名称 | memberType | periodType | 展示规模 | 粗估排名 |
|---|---|---|---|---|---|
| `WEEKLY_HOT_NOTE` | 本周热门笔记榜 | `NOTE` | `WEEKLY` | Top 100 | 不支持 |
| `YEARLY_CREATOR_GROWTH` | 本年创作者成长榜 | `USER` | `YEARLY` | Top 100 | 支持 |

榜单定义需要配置化，至少包括：

1. 榜单编码。
2. 周期类型。
3. 成员类型。
4. 分数字段权重。
5. 精确榜展示数量。
6. Redis 保留数量。
7. 是否启用粗估排名。
8. 是否对移动端公开。

### 4.2 榜单周期

榜单由 `rankCode + periodId` 唯一确定。

本周热门笔记榜：

```text
periodType = WEEKLY
periodId = 2026W23
周期 = 北京时间周一 00:00:00 到下周一 00:00:00
```

本年创作者成长榜：

```text
periodType = YEARLY
periodId = 2026
周期 = 北京时间 2026-01-01 00:00:00 到 2027-01-01 00:00:00
```

事件归属周期按事件 `occurredAt` 计算，不按消费时间计算。

### 4.3 本周热门笔记榜

成员：

```text
memberType = NOTE
memberId = noteId
```

统计口径：

```text
某篇公开笔记在本周内获得的点赞、评论、收藏净增量加权分。
```

特点：

1. 面向普通用户内容发现。
2. 只展示 Top 100。
3. 不提供 Top 100 外笔记的粗估排名。
4. 老笔记如果本周重新获得大量互动，可以重新进入本周榜。
5. 私密、删除、下架笔记不能展示在榜单。

展示计数和排序分要区分：

```text
排序分：本周净互动热度分
展示计数：笔记当前总点赞数、总评论数、总收藏数
```

这样既能体现“本周热门”，也能让移动端展示用户熟悉的总计数。

### 4.4 本年创作者成长榜

成员：

```text
memberType = USER
memberId = creatorUserId
```

统计口径：

```text
创作者在本年发布的公开笔记，在本年内获得的点赞、评论、收藏净增量加权分。
```

这个口径包含两个时间条件：

1. 笔记发布时间属于当前自然年。
2. 互动发生时间也属于当前自然年。

示例：

```text
2026 年发布的笔记，在 2026 年获得互动 -> 计入 2026 年榜
2025 年发布的笔记，在 2026 年获得互动 -> 不计入 2026 年创作者成长榜
2026 年发布的笔记，在 2027 年获得互动 -> 不计入 2026 年榜
```

特点：

1. 面向创作者激励。
2. Top 100 做精确排名。
3. Top 100 外支持本人粗估排名。
4. 更适合使用“精确排名 + 粗估排名”组合方案。
5. 年度结束后榜单冻结，后续只保留历史快照。

### 4.5 分数公式

第一阶段两个榜单使用同一组基础互动权重：

| 字段 | 权重 | 原因 |
|---|---|---|
| `like_count` | 1 | 点赞成本最低，权重最低 |
| `comment_count` | 2 | 评论互动意愿更强，权重中等 |
| `collect_count` | 3 | 收藏代表内容有价值或值得回看，权重最高 |

公式：

```text
scoreDelta = likeDelta * 1
           + commentDelta * 2
           + collectDelta * 3
```

分数要求：

1. 分数按净增量计算，取消点赞、删除评论、取消收藏会扣分。
2. 单个成员当前周期分数最低为 0。
3. 权重必须配置化，并记录 `scoreVersion`。
4. 公式调整后不直接重写历史榜单，必要时创建新版本规则。

第一阶段不额外给发布笔记数量加分。

原因：

1. 发布数量容易诱导低质量内容。
2. 成长榜更应该奖励内容获得的真实互动。
3. 用户作品数已经由计数服务维护，不需要混入成长分。

### 4.6 候选资格

笔记参与榜单贡献必须满足：

1. `noteStatus = PUBLISHED`。
2. `visibility = PUBLIC`。
3. 未删除。
4. 未下架。
5. 未处于审核失败或不可展示状态。

对两个榜单的影响：

| 状态变化 | 本周热门笔记榜 | 本年创作者成长榜 |
|---|---|---|
| 公开发布 | 可开始参与本周互动分 | 如果发布时间在本年，可开始贡献作者成长分 |
| 改为私密 | 从周榜移出 | 扣回该笔记对作者年榜的当前周期贡献 |
| 删除 | 从周榜移出 | 扣回该笔记对作者年榜的当前周期贡献 |
| 下架 | 从周榜移出 | 扣回该笔记对作者年榜的当前周期贡献 |
| 恢复公开 | 可恢复当前周期贡献 | 可恢复当前年该笔记贡献 |

为了支持扣回和恢复，排行榜服务需要维护“来源贡献”。

### 4.7 来源贡献

来源贡献用于记录某个来源对象对某个榜单成员的分数贡献。

本周热门笔记榜：

```text
榜单成员 = noteId
来源对象 = noteId
```

本年创作者成长榜：

```text
榜单成员 = authorId
来源对象 = noteId
```

来源贡献的用途：

1. 笔记变为私密、删除、下架时，可以扣回它已经贡献的分数。
2. 笔记恢复公开时，可以恢复它在当前周期内的贡献。
3. 排查某个创作者成长分由哪些笔记贡献。
4. Redis 丢失时可以从 MySQL 重建成员分数。

### 4.8 精确排名与粗估排名

精确排名使用 Redis ZSET。

```text
ZSET member = memberId
ZSET score = rankScore
```

Top N 设计：

| 榜单 | 对外展示 | Redis 内部保留 |
|---|---|---|
| 本周热门笔记榜 | Top 100 | Top 120 |
| 本年创作者成长榜 | Top 100 | Top 120 |

保留 120 是为了减少第 100 名附近频繁进出造成的抖动。对移动端仍只展示 Top 100。

粗估排名只对本年创作者成长榜启用：

```text
Top 100 内：ZSET 返回精确排名 EXACT
Top 100 外：根据分段统计返回粗估排名 ESTIMATED
无分数：返回 NOT_RANKED
```

粗估排名使用固定分段或线段树思想：

1. 服务本地保存固定分段结构。
2. Redis Hash 保存每个分段当前人数。
3. 更新创作者分数时，同步维护旧分段和新分段人数。
4. 查询尾部排名时，根据分数所在分段估算大致名次。

### 4.9 同分排序

Redis ZSET 同分时会按 member 字典序排序，不符合用户直觉。

本服务采用：

```text
rankScore = realScore + tieBreaker
```

原则：

1. 整数部分是真实分数。
2. 小数部分是同分排序因子。
3. 同分时，先达到当前分数的成员排在前面。
4. API 返回真实分数，不返回小数排序分。

实现注意：

1. 不能直接用 `ZINCRBY` 简单累加。
2. 更新时需要重新计算真实分和 tieBreaker，然后 `ZADD`。
3. Redis 更新应使用 Lua 脚本原子处理旧分、去重、ZSET、分段统计。

## 5. 核心流程

### 5.1 笔记事件维护轻量索引

排行榜服务消费：

```text
NotePublished
NoteUpdated
NoteDeleted
NoteVisibilityChanged
NoteStatusChanged
```

流程：

```text
消费 note-event
  -> 根据 consumerGroup + eventId 幂等
  -> 更新 rank_note_index
  -> 判断笔记当前是否可贡献榜单分数
  -> 如果变为不可贡献，扣回相关来源贡献
  -> 如果恢复可贡献，恢复当前周期内来源贡献
  -> 更新 Redis 精确榜和年榜分段统计
```

`rank_note_index` 至少保存：

```text
noteId
authorId
publishedAt
noteStatus
visibility
eligibleStatus
```

当计数变化先于笔记索引到达时：

1. 排行榜服务调用笔记服务 `batch-summary` 或内部校验接口补齐索引。
2. 如果笔记服务不可用，当前事件进入可重试状态。
3. 不允许在未知可见性时直接给榜单加分。

### 5.2 CounterChanged 更新本周热门笔记榜

排行榜服务消费 `CounterChanged`。

处理条件：

```text
targetType = NOTE
changedFields 包含 like_count/comment_count/collect_count
note 当前可公开展示
```

流程：

```text
CounterChanged
  -> 计算事件 occurredAt 对应的 week periodId
  -> 查询或补齐 rank_note_index
  -> 判断笔记是否可贡献
  -> 根据 changedFields.delta 计算 scoreDelta
  -> 更新 rank_member_score(memberType=NOTE, memberId=noteId)
  -> 更新 rank_score_contribution(sourceId=noteId)
  -> 写 rank_score_change_log
  -> Redis Lua 更新 WEEKLY_HOT_NOTE ZSET
  -> 裁剪到内部保留 Top 120
  -> 标记榜单 dirty
```

示例：

```json
{
  "targetType": "NOTE",
  "targetId": "800001",
  "changedFields": {
    "like_count": {
      "delta": 3,
      "currentValue": 128
    },
    "comment_count": {
      "delta": 1,
      "currentValue": 20
    }
  }
}
```

分数变化：

```text
scoreDelta = 3 * 1 + 1 * 2 = 5
```

注意：

1. 周榜按互动发生时间归属周期。
2. 老笔记本周获得互动也可以进入本周榜。
3. 分数降到 0 时可以从 ZSET 移除。
4. 对外查询只展示 Top 100。

### 5.3 CounterChanged 更新本年创作者成长榜

处理条件：

```text
targetType = NOTE
changedFields 包含 like_count/comment_count/collect_count
note 当前可公开展示
note.publishedAt 所在年份 = event.occurredAt 所在年份
```

流程：

```text
CounterChanged
  -> 计算事件 occurredAt 对应的 year periodId
  -> 查询 rank_note_index 得到 authorId 和 publishedAt
  -> 判断笔记发布时间是否属于该年
  -> 判断笔记当前是否可贡献
  -> 根据 changedFields.delta 计算 scoreDelta
  -> 更新 rank_member_score(memberType=USER, memberId=authorId)
  -> 更新 rank_score_contribution(sourceId=noteId)
  -> 写 rank_score_change_log
  -> Redis Lua 更新 YEARLY_CREATOR_GROWTH ZSET
  -> Redis Lua 更新 YEARLY_CREATOR_GROWTH 分段统计
  -> 裁剪精确 ZSET 到内部保留 Top 120
  -> 标记榜单 dirty
```

如果一篇 2026 年发布的公开笔记在 2026 年获得互动：

```text
给该笔记作者的 2026 年成长分加分
```

如果一篇 2025 年发布的笔记在 2026 年获得互动：

```text
不计入 2026 年创作者成长榜
```

### 5.4 笔记不可见时扣回贡献

触发事件：

```text
NoteDeleted
NoteVisibilityChanged(PUBLIC -> PRIVATE)
NoteStatusChanged(PUBLISHED -> OFFLINE)
```

流程：

```text
笔记变为不可贡献
  -> 更新 rank_note_index.eligibleStatus = INELIGIBLE
  -> 查询该 noteId 在当前周榜的来源贡献
  -> 扣回 WEEKLY_HOT_NOTE 中 noteId 的分数
  -> 查询该 noteId 在当前年榜的来源贡献
  -> 扣回 YEARLY_CREATOR_GROWTH 中 authorId 的分数
  -> 更新 Redis ZSET 和年榜分段统计
  -> 记录操作日志
```

说明：

1. 周榜扣回的是该笔记本周贡献。
2. 年榜扣回的是该笔记本年对作者的贡献。
3. 历史已关闭周期不默认回写，除非有管理端重算任务。

### 5.5 笔记恢复公开时恢复贡献

触发事件：

```text
NoteVisibilityChanged(PRIVATE -> PUBLIC)
NoteStatusChanged(OFFLINE -> PUBLISHED)
```

流程：

```text
笔记恢复可贡献
  -> 更新 rank_note_index.eligibleStatus = ELIGIBLE
  -> 查询当前周期内该 noteId 的来源贡献
  -> 恢复 WEEKLY_HOT_NOTE 分数，若属于当前周
  -> 恢复 YEARLY_CREATOR_GROWTH 分数，若发布时间和当前年匹配
  -> 更新 Redis ZSET 和年榜分段统计
```

如果贡献记录不存在，不主动扫描业务明细表，只等待后续计数变化或校准任务处理。

### 5.6 查询本周热门笔记榜

移动端接口：

```text
GET /api/ranks/weekly-hot-notes?cursor=&size=20
```

流程：

```text
移动端请求
  -> 计算当前 week periodId
  -> 读取 WEEKLY_HOT_NOTE ZSET Top 100 中的一页
  -> 批量查询 rank_note_index 过滤不可见笔记
  -> 调用笔记服务 batch-summary 获取笔记摘要
  -> 调用计数服务 batch 获取展示计数
  -> 调用用户服务 batch-summary 获取作者摘要
  -> 组装榜单卡片并返回
```

查询时必须二次校验：

1. Redis 可能短时间残留不可见笔记。
2. 笔记服务摘要接口是展示前的权威校验。
3. 被过滤的笔记需要向后补齐，尽量保证页面数量。

### 5.7 查询本年创作者成长榜

流程：

```text
  -> 计算当前 year periodId
```

榜单卡片建议包含：

1. 排名。
2. 创作者昵称、头像、简介。
3. 成长分。
4. 本年上榜作品数或公开作品数，后续可选。
5. 获赞数、粉丝数等展示字段，后续按产品决定。

### 5.8 查询创作者本人年榜排名

接口：

```text
GET /api/ranks/creators/me/yearly-growth-rank
```

流程：

```text
查询当前登录用户 creatorId
  -> 计算当前 year periodId
  -> 先查 YEARLY_CREATOR_GROWTH ZSET
  -> 如果命中，返回 EXACT 精确排名
  -> 如果未命中，查询 rank_member_score
  -> 如果分数为 0 或不存在，返回 NOT_RANKED
  -> 根据分段统计估算排名，返回 ESTIMATED
```

| rankMode | 说明 |
|---|---|
| `EXACT` | Top 100 内精确排名 |
| `ESTIMATED` | Top 100 外粗估排名 |
| `NOT_RANKED` | 当前年无有效成长分 |

### 5.9 周期滚动

周榜：

```text
每周一 00:00:00 创建新 WEEKLY_HOT_NOTE 周期
上一周榜单进入 CLOSED 状态
Redis 历史 Key 保留 14 到 30 天
MySQL 快照长期保留或归档
```

年榜：

```text
每年 1 月 1 日 00:00:00 创建新 YEARLY_CREATOR_GROWTH 周期
上一年榜单冻结
冻结后不再因新互动改变上一年分数
如需修正，只能通过管理端重算任务
```

事件延迟处理：

1. 按 `occurredAt` 归属周期。
2. 已关闭但未冻结的周期可以接受迟到事件。
3. 已冻结的年度榜默认不接收迟到事件，只记录异常或进入人工修正。

### 5.10 榜单快照

快照任务：

```text
定时读取 Redis ZSET Top 100
  -> 批量补充笔记或用户摘要
  -> 写 rank_snapshot
  -> 写 rank_snapshot_item
  -> 标记 snapshotVersion
```

用途：

1. Redis 故障时降级展示。
2. 历史榜单回看。
3. 排查榜单变化。
4. 年榜冻结留档。

建议第一阶段：

1. 周榜每 1 到 5 分钟快照一次。
2. 年榜每 5 到 15 分钟快照一次。
3. 年底冻结时生成最终快照。

### 5.11 Redis 重建

流程：

```text
创建 rank_rebuild_task
  -> 读取 rank_member_score
  -> 过滤 eligibleStatus 或 memberStatus
  -> 重建对应 rankCode + periodId 的 ZSET
  -> 如果是 YEARLY_CREATOR_GROWTH，同时重建分段统计 Hash
  -> 裁剪 ZSET 到内部保留 Top 120
  -> 生成快照
  -> 标记任务完成
```

如果怀疑 MySQL 分数不准：

1. 重放 `rank_score_change_log`。
2. 根据 `rank_score_contribution` 汇总成员分数。
3. 必要时调用计数服务和笔记服务做低频校准。

## 6. 异常流程

| 异常 | 处理方式 |
|---|---|
| 重复消费事件 | 根据 `consumerGroup + eventId` 幂等跳过 |
| 事件缺少必要字段 | 记录失败，不更新榜单，必要时告警 |
| 事件版本不兼容 | 进入失败表，等待兼容处理 |
| 计数事件先于笔记索引到达 | 调用笔记服务补齐索引，失败则重试 |
| 笔记不可见 | 不加分；如果已有贡献，扣回当前周期贡献 |
| 负向 delta | 按权重扣分，成员分数最低为 0 |
| Redis 写入失败 | MQ 重试，超过阈值进入补偿任务 |
| MySQL 成功但 Redis 失败 | 通过重建任务从 MySQL 修复 Redis |
| Redis 成功但 MySQL 状态更新失败 | 依赖消费幂等和 Redis 幂等防止重复加分 |
| 榜单查询 Redis 不可用 | 降级读取最近 MySQL 快照 |
| 笔记服务摘要失败 | 周榜不展示无法确认可见性的笔记 |
| 用户服务摘要失败 | 年榜可返回 creatorId 和降级标记 |
| 年榜分段统计异常 | 暂停粗估排名，触发 `REBUILD_SEGMENT` |
| 周期边界迟到事件 | 按 `occurredAt` 归属周期 |
| 冻结年度榜收到迟到事件 | 记录异常，不自动改榜 |

```text
bluenote:prod:rank:WEEKLY_HOT_NOTE:2026W23:exact
bluenote:prod:rank:YEARLY_CREATOR_GROWTH:2026:exact
```

结构：

```text
ZSET
Member = memberId
Score = rankScore
```

用途：

1. Top 100 精确榜。
2. 精确排名查询。
3. 榜单快照来源。

保留策略：

1. 当前周期不设置短 TTL。
2. 周榜历史 Redis Key 保留 14 到 30 天。
3. 年榜历史 Redis Key 可保留更久，或只保留 MySQL 最终快照。

#### 7.3.2 年榜分段统计 Hash

```text
bluenote:{env}:rank:YEARLY_CREATOR_GROWTH:{periodId}:segment
```

结构：

```text
HASH
Field = scoreRange
Value = creatorCount
```

用途：

1. 年度创作者成长榜尾部粗估排名。
2. 不支持周榜。

#### 7.3.3 成员分数缓存

```text
bluenote:{env}:rank:{rankCode}:{periodId}:score:{memberType}:{memberId}
```

结构：

```text
HASH
score
rank_score
score_version
member_status
last_score_at
```

用途：

1. Redis Lua 快速读取旧分。
2. 降低 MySQL 热点读取。

#### 7.3.4 事件短期去重

```text
bluenote:{env}:rank:dedupe:{yyyyMMdd}
```

结构：

```text
SET
Member = eventId
```

TTL：

```text
30 天
```

MySQL `rank_event_consume_log` 是长期幂等依据，Redis 只是短期加速。

#### 7.3.5 榜单 dirty 集合

```text

```

结构：

```text

Member = rankCode:periodId
Score = dirtyAtMillis
```

用途：

1. 快照任务只处理有变化的榜单。
2. 降低无效快照写入。

#### 7.3.6 榜单页短缓存

```text
bluenote:{env}:rank:{rankCode}:{periodId}:page:{cursorHash}
```

TTL：

```text
5 到 30 秒
```

说明：

1. 只缓存非强个性化字段。
2. 如果后续展示“当前用户是否关注作者”等个性化字段，需要单独补齐。

## 9. 安全与风控设计

### 9.1 接口权限

要求：

1. 移动端只能通过网关访问 `/api/ranks/**`。
2. `/internal/ranks/**` 只能服务间调用。
3. 重建、校准、快照查询等接口需要内部权限。
4. `GET /api/ranks/creators/me/yearly-growth-rank` 必须登录。

### 9.2 可见性安全

本周热门笔记榜展示前必须校验：

1. 笔记仍为公开。
2. 笔记仍为已发布。
3. 笔记未删除、未下架。
4. 笔记服务摘要接口没有返回不可见。

本年创作者成长榜不展示具体贡献笔记列表时，可以只展示创作者；但分数贡献仍必须来自可公开展示的笔记。

### 9.3 防刷与限流

排行榜服务不替代来源服务风控，但要识别异常。

第一阶段措施：

1. 单篇笔记短时间分数暴涨告警。
2. 单个创作者短时间成长分暴涨告警。
3. 负向 delta、大额 delta、异常字段进入失败表。
4. 榜单查询按 IP、用户、设备限流。
5. 内部重建接口限制频率。

来源服务职责：

1. 点赞、收藏防刷在笔记服务或后续互动服务。
2. 评论防刷在评论服务。
3. 计数异常校准在计数服务。

### 9.4 数据安全

排行榜服务不保存：

1. 手机号、邮箱、密码。
2. 笔记完整正文。
3. Access Token。
4. MinIO 预签名 URL 完整签名参数。

日志中避免记录大批量完整成员列表，只记录数量、榜单编码、周期和必要 ID。

## 10. 前后端实现要点

### 10.1 移动端实现

移动端建议有两个入口：

1. 发现页或热门页展示“本周热门笔记”。
2. 创作者中心或榜单页展示“本年创作者成长榜”。

实现要点：

1. 移动端不计算榜单分数。
2. 移动端不直接请求计数服务、笔记服务内部接口。
3. 榜单列表使用游标分页。
4. `degraded = true` 时仍可展示已有榜单，不弹复杂错误。
5. 点击周榜笔记进入笔记详情，详情仍以笔记服务为准。
6. 点击年榜创作者进入用户主页。

### 10.3 Redis Lua 实现要点

Lua 脚本应处理：

1. 短期事件去重。
2. 读取旧分。
3. 计算新分。
4. 分数低于 0 时钳制为 0。
5. 写 ZSET。
6. 更新成员分数缓存。
7. 年榜更新旧分段和新分段人数。
8. 标记榜单 dirty。

MySQL 仍是恢复依据。Redis 写成功不等于整个消费完成，消费记录和分数流水必须可靠落库。

## 11. 数据一致性与事件

### 11.1 总体一致性

排行榜链路：

```text
来源服务本地事务
  -> outbox 发布业务事件
  -> 计数服务消费并更新计数
  -> 计数服务发布 CounterChanged
  -> 排行榜服务消费 CounterChanged
  -> 更新 MySQL 成员分数和来源贡献
  -> 更新 Redis 榜单读模型
  -> 定时生成榜单快照
```

一致性级别：

1. 笔记状态以笔记服务为准。
2. 计数以计数服务为准。
3. 榜单成员分数以排行榜 MySQL 为恢复依据。
4. Redis 排名是可重建读模型。
5. 移动端榜单最终一致。

### 11.2 订阅事件

排行榜服务订阅：

| Topic | 事件 | 用途 |
|---|---|---|
| `note-event` | `NotePublished` | 建立笔记索引，判断候选资格 |
| `note-event` | `NoteUpdated` | 更新标题、封面等摘要快照，必要时 |
| `note-event` | `NoteDeleted` | 扣回贡献，移出周榜 |
| `note-event` | `NoteVisibilityChanged` | 公开性变化导致贡献启停 |
| `note-event` | `NoteStatusChanged` | 下架、恢复、审核状态影响贡献 |
| `counter-event` | `CounterChanged` | 根据计数变化更新分数 |
| `counter-event` | `CounterRebuilt` | 计数重建后触发榜单校准 |

第一阶段排行榜服务不直接订阅 `interaction-event` 或 `comment-event` 更新分数。

原因：

1. 互动和评论已经统一进入计数服务。
2. 计数服务处理了幂等、聚合、负数保护。
3. 排行榜直接消费原始事件容易双算。
4. 后续拆互动服务时，排行榜事件口径不需要跟着变化。

### 11.3 发布事件

Topic：

```text
rank-event
```

事件：

| 事件 | 触发时机 | 消费方 |
|---|---|---|
| `RankChanged` | 榜单 Top 100 明显变化 | 通知、运营、监控后续 |
| `RankFrozen` | 年榜冻结完成 | 运营、监控后续 |

第一阶段可以先实现 outbox 和事件定义，不强依赖下游消费。

### 11.4 幂等策略

| 层级 | 幂等方式 |
|---|---|
| MQ 消费 | `rank_event_consume_log` 唯一约束 |
| 分数流水 | `source_event_id + rank_code + member_type + member_id + field_name` 唯一 |
| 成员分数 | `rank_code + period_id + member_type + member_id` 唯一 |
| 来源贡献 | `rank_code + period_id + member_type + member_id + source_type + source_id` 唯一 |
| Redis 更新 | `eventId` 短期去重 + Lua 原子执行 |
| outbox | `event_id` 唯一 |

### 11.5 乱序处理

处理原则：

1. 计数事件到达但笔记索引不存在时，先补齐索引。
2. 笔记删除、私密、下架事件优先级高于加分事件。
3. 状态类事件必须带前后状态或对象版本；缺失时调用笔记服务校验。
4. 旧状态事件不能覆盖新状态。
5. 周期按 `occurredAt` 判断，不能按消费时间判断。

## 14. 风险与后续演进

| 风险 | 说明 | 应对 |
|---|---|---|
| 榜单延迟 | 计数和排行榜都是异步链路 | 明确最终一致，监控消费延迟 |
| 周榜和年榜口径混淆 | 周榜按互动周，年榜还要求笔记发布年 | 文档和代码中封装周期规则 |
| 老笔记影响周榜 | 老笔记本周再次热门可以上榜 | 这是周榜内容发现的合理行为 |
| 老笔记影响年榜 | 年榜只统计本年发布笔记 | 通过 `publishedAt` 判断过滤 |
| 私密笔记贡献未扣回 | 状态变化异步处理可能失败 | 来源贡献表 + 补偿任务 |
| 事件重复导致分数错误 | MQ 至少一次投递 | 消费表、流水唯一约束、Redis dedupe |
| 原始事件和计数事件双算 | 同时消费容易重复加分 | 第一阶段只用 `counter-event` 算分 |
| 年榜粗估排名不准 | 分段统计只能估算 | 明确 `ESTIMATED`，只用于本人参考 |
| Redis 数据丢失 | Redis 是在线读模型 | MySQL 分数和贡献重建 |
| 刷榜行为 | 点赞评论收藏成本较低 | 来源服务限流，排行榜异常监控 |
| 2 核 4G 资源有限 | 排行榜、计数、Feed 可能合并部署 | Top 100、保留 120、控制快照频率 |
