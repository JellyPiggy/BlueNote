# BlueNote 评论服务设计

## 1. 背景与目标

评论服务负责 BlueNote 笔记下的评论、回复、删除、评论点赞和评论列表查询。它是笔记详情页互动体验的核心能力，也是通知、计数、排行榜和后续内容审核的重要事件来源。

本方案参考 `参考资料/12.评论服务.md` 中“二级模式服务设计”的主体思路：采用一级评论 + 二级评论的常见评论区模型，使用面向内容查询和面向用户查询的冗余表解决查询入口不同带来的索引冲突，使用 Redis ZSET 维护热门评论和时间序评论 ID 列表。

结合 BlueNote 第一阶段的资源和技术栈，本方案做以下收敛：

1. 不使用图数据库。
2. 不引入独立分布式 KV 存储评论正文，第一阶段用 MySQL 垂直拆分保存评论元信息和正文。
3. 不引入 Elasticsearch 做评论搜索和审核后台，先做敏感词校验和状态预留。
4. 不把发布评论做成 MQ 异步落库，第一阶段同步写 MySQL，保证用户发完即可读。
5. 评论点赞行为事实归评论服务，评论点赞数、笔记评论数等聚合计数归计数服务。

设计目标：

1. 支持发布一级评论。
2. 支持发布二级评论，二级评论可以回复一级评论或同一一级评论下的另一条二级评论。
3. 支持二级评论区，不支持无限楼中楼展示。
4. 支持删除自己的评论，笔记作者可按规则删除自己笔记下的评论。
5. 支持笔记评论区一级评论分页，支持按热度和时间排序。
6. 支持一级评论下的二级评论分页，按时间从早到晚展示。
7. 支持评论点赞、取消点赞和当前用户点赞状态查询。
8. 支持用户评论历史查询。
9. 支持评论状态、基础敏感词、限流、缓存、事件和补偿。
10. 为评论审核、举报、置顶、评论图片、异步写评论、搜索和图数据库预留扩展空间。

关键约束：

1. MySQL 是评论事实、评论正文、评论点赞明细的事实来源。
2. Redis 只保存评论列表缓存、热门评论、评论摘要和限流数据。
3. 评论服务不维护笔记评论数、评论点赞数的最终展示计数。
4. 评论服务不保存笔记正文和笔记媒体。
5. 评论服务不保存用户昵称、头像等用户资料事实，只可保存展示快照。
6. 评论服务发布事件，下游计数、排行榜、通知异步消费。

## 2. 功能范围

### 2.1 第一阶段支持

| 功能 | 说明 |
|---|---|
| 发布一级评论 | 用户对笔记发布评论 |
| 发布二级评论 | 用户回复一级评论或同一一级评论下的二级评论 |
| 删除评论 | 评论作者删除自己的评论，笔记作者删除自己笔记下的评论 |
| 查询一级评论列表 | 查询某笔记下的一级评论，支持热度序和时间序 |
| 查询二级评论列表 | 查询某条一级评论下的二级评论，按时间正序 |
| 查询我的评论 | 查询当前用户发布过的评论 |
| 评论点赞 | 对评论点赞 |
| 取消评论点赞 | 取消已点赞评论 |
| 评论状态过滤 | 正常、审核中、自见、已删除等状态按业务层过滤 |
| 基础内容安全 | 长度、敏感词、发布频率校验 |
| 评论缓存 | Redis 缓存热门评论、时间序评论 ID、评论摘要 |
| 发布事件 | 发布 `CommentCreated`、`CommentDeleted`、`CommentLiked`、`CommentUnliked` |
| 补偿任务 | outbox 投递、缓存失效、计数校准来源等补偿 |

### 2.2 第一阶段不支持

| 功能 | 暂不支持原因 |
|---|---|
| 图数据库 | 当前只采用二级评论，不需要复杂回复链路查询 |
| 无限楼中楼 | 展示复杂、查询复杂，BlueNote 第一阶段采用二级模式 |
| 评论正文独立 KV | 当前没有独立 KV 存储，MySQL 垂直拆分足够 |
| 评论异步写入 | 第一阶段评论量小，同步写库体验更稳定 |
| 写聚合 | 只有热点评论高并发时才需要 |
| Elasticsearch 评论搜索 | 第一阶段没有完整审核后台和搜索需求 |
| 评论图片 | 文件服务可支持，但评论区先只做文本 |
| 点踩评论 | 当前产品不需要 |
| 评论置顶 | 依赖运营和审核，后续扩展 |
| 评论举报 | 后续配合审核流程设计 |
| 好友可见评论 | 依赖关系服务隐私策略，后续扩展 |
| 本地缓存 | 第一阶段先使用 Redis，热点明显后再做本地缓存 |

### 2.3 后续扩展

后续可以扩展：

1. 评论图片和表情。
2. 评论举报和审核后台。
3. Elasticsearch 评论搜索和关键词召回。
4. 评论置顶、精选评论、作者赞过。
5. 评论异步写入和写聚合。
6. 热点评论本地缓存。
7. 评论隐私状态，例如好友可见。
8. 黑名单限制评论。
9. 评论楼中楼或图数据库。
10. 评论风控模型和反垃圾策略。

## 3. 服务边界

### 3.1 评论服务负责

评论服务负责：

1. 评论元信息。
2. 评论正文。
3. 评论层级关系。
4. 评论状态。
5. 评论点赞明细。
6. 评论列表缓存。
7. 评论事件发布。
8. 评论删除和可见性过滤。
9. 给计数服务提供评论数、评论点赞数校准来源。

典型数据包括：

| 数据 | 说明 |
|---|---|
| `commentId` | 评论 ID |
| `noteId` | 被评论笔记 ID |
| `authorId` | 笔记作者 ID 快照 |
| `userId` | 评论发布者 ID |
| `rootId` | 一级评论 ID，一级评论自身为 0 或自身 ID，具体见表设计 |
| `level` | `1` 一级评论，`2` 二级评论 |
| `replyCommentId` | 实际回复的评论 ID |
| `replyUserId` | 实际回复的用户 ID |
| `commentStatus` | 评论状态 |
| `content` | 评论正文 |

### 3.2 笔记服务负责

笔记服务负责：

1. 笔记是否存在。
2. 笔记状态是否允许评论。
3. 评论开关 `commentEnabled`。
4. 笔记作者 ID。
5. 笔记可见性。

发布评论前，评论服务调用笔记服务内部接口：

```text
POST /internal/notes/comment-check
```

评论服务不直接访问笔记服务数据库，也不维护笔记正文。

### 3.3 用户服务负责

用户服务负责：

1. 评论用户是否存在。
2. 评论用户状态是否允许评论。
3. 用户昵称、头像等展示资料。

评论服务可以在评论表中保存用户展示快照，用于列表降级展示，但用户资料事实仍以用户服务为准。

### 3.4 计数服务负责

计数服务负责：

1. 笔记评论数。
2. 一级评论回复数。
3. 评论点赞数。

评论服务发布 `CommentCreated`、`CommentDeleted`、`CommentLiked`、`CommentUnliked` 后，计数服务异步更新聚合计数。

评论服务可以在查询评论列表时调用计数服务批量获取评论点赞数、回复数，也可以使用评论表中的低频快照字段作为降级值。

### 3.5 通知服务负责

通知服务消费评论事件生成通知：

1. 评论笔记通知笔记作者。
2. 回复评论通知被回复用户。
3. 评论点赞通知评论作者，第一阶段可选择不做点赞通知。

评论服务不保存通知内容，也不处理通知已读。

### 3.6 排行榜服务负责

排行榜服务可以消费评论事件，把评论数、评论活跃度纳入笔记热度计算。评论服务不负责榜单排序。

### 3.7 关系服务负责

第一阶段评论可见性不依赖关系服务。后续如果支持好友可见、黑名单限制评论、互相关注评论权限，再调用关系服务。

## 4. 核心概念

### 4.1 一级评论

一级评论是用户直接对笔记发表的评论。

特点：

1. `level = 1`。
2. 归属某个 `noteId`。
3. 可以出现在笔记评论区列表。
4. 可以拥有自己的二级评论区。
5. 可以参与热度排序。

### 4.2 二级评论

二级评论是一级评论下的回复。

二级评论可以回复：

1. 一级评论。
2. 同一一级评论下的另一条二级评论。

但展示上仍归属于同一个一级评论的二级评论区，不再展开第三层楼层。

### 4.3 rootId

`rootId` 表示二级评论区归属。

建议规则：

| 评论类型 | rootId |
|---|---|
| 一级评论 | `0`，或保存自身 `commentId` |
| 二级评论 | 所属一级评论的 `commentId` |

本方案建议一级评论 `root_id = comment_id`。这样列表查询可以统一使用：

```text
note_id = 笔记 noteId and level = 1
root_id = 一级评论 commentId and level = 2
```

但由于一级评论按笔记查询更常用，表中仍单独保留 `note_id`，实际 SQL 以 `note_id + level` 查询一级评论，以 `root_id + level` 查询二级评论。

### 4.4 replyCommentId 和 replyUserId

`replyCommentId` 表示当前评论实际回复哪条评论。

`replyUserId` 表示当前评论实际回复哪个用户。

示例：

| 场景 | level | rootId | replyCommentId | replyUserId |
|---|---|---|---|---|
| 用户 A 评论笔记 N | 1 | 自身 commentId | 0 | 0 |
| 用户 B 回复一级评论 C1 | 2 | C1 | C1 | A |
| 用户 C 回复二级评论 C2 | 2 | C1 | C2 | B |

这样既可以保持二级展示，又可以在文案上展示“回复 @某用户”。

### 4.5 评论状态

建议状态：

| 状态 | 说明 |
|---|---|
| `VISIBLE` | 正常可见 |
| `SELF_VISIBLE` | 仅评论发布者自己可见 |
| `REVIEWING` | 审核中，对外表现为仅自己可见 |
| `REJECTED` | 审核不通过，对外不可见或仅自己可见 |
| `DELETED` | 用户或作者删除 |
| `HIDDEN` | 管理端或系统隐藏，后续扩展 |

第一阶段默认评论经过基础敏感词校验后进入 `VISIBLE`。命中可疑词时可以进入 `REVIEWING` 或 `SELF_VISIBLE`。

### 4.6 热门评论

热门评论只针对一级评论。

第一阶段热度公式可以简单设为：

```text
hotScore = likeCount * 0.4 + replyCount * 0.6
```

说明：

1. 点赞数越多，热度越高。
2. 回复数越多，讨论度越高。
3. 热门评论只保留 Top 1000，避免 Redis 大 key。
4. 热门评论列表只保存评论 ID，完整评论仍从缓存或 MySQL 获取。

## 5. 核心业务流程

### 5.1 发布一级评论

流程：

```text
移动端提交评论
  -> 网关校验 Access Token
  -> 评论服务获取当前 userId
  -> 校验评论内容长度、敏感词、频率
  -> 调用用户服务校验用户状态
  -> 调用笔记服务校验笔记可评论
  -> 生成 commentId
  -> 写 content_comment
  -> 写 user_comment
  -> 写 comment_content
  -> 写 comment_outbox_event(CommentCreated)
  -> 删除或更新评论列表缓存
  -> 返回评论结果
```

本地事务包括：

1. 评论元信息。
2. 用户评论冗余表。
3. 评论正文。
4. outbox 事件。

第一阶段同步写库，不使用 MQ 异步落库。原因是当前评论量预期不大，同步写能保证发布后立即可见，业务语义更清楚。

### 5.2 发布二级评论

流程：

```text
移动端提交回复
  -> 校验登录态
  -> 校验内容、频率、用户状态
  -> 查询被回复评论
  -> 校验被回复评论存在且未删除
  -> 如果被回复的是一级评论，rootId = 被回复 commentId
  -> 如果被回复的是二级评论，rootId = 被回复评论 rootId
  -> 校验 root 一级评论仍可见
  -> 写 content_comment(level=2)
  -> 写 user_comment
  -> 写 comment_content
  -> 写 CommentCreated outbox
  -> 删除或更新二级评论列表缓存
  -> 返回回复结果
```

二级评论发布后：

1. 计数服务消费事件后增加一级评论回复数。
2. 通知服务通知被回复用户。
3. 热门评论消费者根据回复数变化更新一级评论热度。

### 5.3 删除评论

删除权限：

| 操作人 | 权限 |
|---|---|
| 评论作者 | 可以删除自己的评论 |
| 笔记作者 | 可以删除自己笔记下的评论 |
| 管理端 | 后续可删除违规评论 |

流程：

```text
用户请求删除评论
  -> 校验登录态
  -> 查询评论元信息
  -> 校验评论未删除
  -> 校验操作权限
  -> 更新 content_comment.comment_status = DELETED
  -> 更新 user_comment.comment_status = DELETED
  -> 写 comment_operation_log
  -> 计算 affectedCommentCount 和 affectedReplyCount
  -> 写 CommentDeleted outbox
  -> 删除评论相关 Redis 缓存
  -> 返回删除成功
```

删除一级评论时，第一阶段建议：

1. 一级评论本身变为 `DELETED`。
2. 二级评论不物理删除。
3. 前端展示“该评论已删除”或直接隐藏整组二级评论，具体展示策略由产品决定。

为了降低复杂度，第一阶段可以隐藏已删除一级评论及其二级评论。后续如果要保留二级回复，需要专门设计折叠展示。

计数影响要求：

1. 删除二级评论时，`affectedCommentCount = 1`，`affectedReplyCount = 1`。
2. 删除一级评论且隐藏整组二级评论时，`affectedCommentCount = 1 + 当前可见二级评论数`，`affectedReplyCount = 当前可见二级评论数`。
3. `CommentDeleted` 事件必须携带这两个字段，计数服务据此减少笔记评论数和一级评论回复数。
4. 如果删除时无法精确得到可见二级评论数，可以先用快照值并触发计数校准任务修正。

### 5.4 查询一级评论列表

支持两种排序：

| 排序 | 说明 |
|---|---|
| `HOT` | 先读热门评论，不足时补时间序 |
| `TIME` | 直接按发布时间倒序或正序读取 |

第一阶段建议：

1. 默认排序使用 `HOT`。
2. 热门评论刷完后进入时间序。
3. 移动端分页游标中携带当前来源：`hot` 或 `time`。

流程：

```text
查询笔记评论区
  -> 校验笔记是否可见，可由笔记服务或详情页前置保证
  -> 获取评论 ID 列表
     -> HOT: 先查 comment:hot:{noteId}
     -> 不足则查 comment:time:{noteId}:level1 或 MySQL
  -> 批量获取评论摘要
     -> Redis MGET
     -> 未命中查 MySQL + comment_content
  -> 调用计数服务批量获取评论点赞数、回复数
  -> 查询当前用户是否点赞这些评论
  -> 按评论状态做业务层过滤
  -> 返回评论列表和 nextCursor
```

业务层过滤原因：

1. `SELF_VISIBLE` 和 `REVIEWING` 对不同用户可见性不同。
2. 后续好友可见需要关系服务判断。
3. SQL 只负责取候选集，最终可见性由评论服务判断。

### 5.5 查询二级评论列表

二级评论区通常按发布时间从早到晚展示。

流程：

```text
查询一级评论下回复
  -> 校验一级评论存在
  -> 查询 comment:replies:{rootCommentId}
  -> Redis 未命中查 content_comment
  -> 批量获取评论摘要和正文
  -> 查询当前用户点赞状态
  -> 业务层过滤状态
  -> 返回列表
```

二级评论列表不参与热度排序。

### 5.6 查询我的评论

用户查看自己发布过的评论。

流程：

```text
查询当前 userId
  -> 查询 user_comment
  -> 按 created_at 倒序分页
  -> 批量读取 comment_content
  -> 过滤 DELETED 或按产品要求展示已删除标记
  -> 返回评论历史
```

`user_comment` 的存在是为了让用户历史查询不依赖 `note_id` 分片或扫描 `content_comment`。

### 5.7 评论点赞

评论点赞行为事实归评论服务维护。

流程：

```text
用户点赞评论
  -> 校验登录态
  -> 查询评论存在且可见
  -> 查询 comment_like
  -> 不存在则插入 LIKED
  -> 已 CANCELLED 则更新为 LIKED
  -> 已 LIKED 则幂等成功
  -> 写 CommentLiked outbox
  -> 删除当前用户评论点赞状态缓存
  -> 返回成功
```

评论点赞数由计数服务消费 `CommentLiked` 后更新。

### 5.8 取消评论点赞

流程：

```text
用户取消点赞
  -> 校验登录态
  -> 查询 comment_like
  -> 不存在或已取消则幂等成功
  -> 已 LIKED 则更新为 CANCELLED
  -> 写 CommentUnliked outbox
  -> 删除点赞状态缓存
  -> 返回成功
```

### 5.9 热门评论更新

评论服务发布以下事件：

1. `CommentCreated`
2. `CommentDeleted`
3. `CommentLiked`
4. `CommentUnliked`

评论服务内部或后续独立消费者可以消费这些事件更新热门评论 Redis ZSET：

```text
收到影响热度的事件
  -> 找到一级评论 rootCommentId
  -> 查询计数服务或本地快照获取 likeCount、replyCount
  -> 计算 hotScore
  -> ZADD comment:hot:{noteId} hotScore rootCommentId
  -> 如果评论删除，则 ZREM
  -> 定期保留 Top 1000
```

第一阶段可以把热门评论更新作为评论服务内部消费者实现，不单独拆服务。

## 6. 异常流程

| 异常 | 处理方式 |
|---|---|
| 用户未登录 | 禁止发布、删除、点赞 |
| 用户被禁用或禁言 | 禁止发布评论和点赞，允许浏览 |
| 笔记不存在 | 返回 `COMMENT_NOTE_NOT_FOUND` |
| 笔记已删除或不可见 | 返回不可评论 |
| 笔记关闭评论 | 返回 `COMMENT_NOTE_COMMENT_DISABLED` |
| 评论内容为空 | 返回参数错误 |
| 评论内容过长 | 返回 `COMMENT_CONTENT_TOO_LONG` |
| 评论命中敏感词 | 拒绝或进入 `SELF_VISIBLE` / `REVIEWING` |
| 被回复评论不存在 | 返回 `COMMENT_REPLY_TARGET_NOT_FOUND` |
| 被回复评论已删除 | 返回 `COMMENT_REPLY_TARGET_DELETED` |
| root 一级评论已删除 | 禁止继续回复 |
| 重复点赞 | 幂等成功 |
| 重复取消点赞 | 幂等成功 |
| 删除无权限 | 返回 `COMMENT_DELETE_FORBIDDEN` |
| Redis 不可用 | 降级 MySQL，不影响写主流程 |
| 计数服务不可用 | 评论列表点赞数、回复数使用降级值 |
| RocketMQ 不可用 | outbox 保留待投递 |
| 缓存删除失败 | 记录补偿任务或等待 TTL |

补偿重点：

1. outbox 投递失败。
2. 评论列表缓存失效失败。
3. 热门评论 ZSET 更新失败。
4. `content_comment` 和 `user_comment` 不一致。
5. 评论计数与评论事实不一致。

## 8. Redis Key 设计

### 8.1 热门一级评论列表

```text
comment:hot:{noteId}
```

| 项 | 说明 |
|---|---|
| 数据结构 | ZSET |
| member | `commentId` |
| score | `hotScore` |
| 保留长度 | Top 1000 |
| TTL | 24 小时，可加随机抖动 |
| 写入时机 | 评论点赞、取消点赞、回复、删除后异步更新 |
| 失效策略 | 评论删除时 ZREM，定期裁剪 |
| 重建方式 | 从 `content_comment` + 计数服务或快照重算 |

### 8.2 一级评论时间序列表

```text
comment:time:{noteId}:level1
```

| 项 | 说明 |
|---|---|
| 数据结构 | ZSET |
| member | `commentId` |
| score | `createdAt` 毫秒时间戳 |
| 保留长度 | 最近或最前 N 条，第一阶段建议 1000 |
| TTL | 6 到 24 小时 |
| 写入时机 | 查询回源后回填；发布评论后增量写或删除缓存 |
| 重建方式 | 从 `content_comment` 按 `note_id + level` 重建 |

### 8.3 二级评论时间序列表

```text
comment:replies:{rootCommentId}
```

| 项 | 说明 |
|---|---|
| 数据结构 | ZSET |
| member | `commentId` |
| score | `createdAt` 毫秒时间戳 |
| 保留长度 | 前 N 条，第一阶段建议 500 |
| TTL | 6 到 24 小时 |
| 写入时机 | 查询回源后回填；发布回复后增量写或删除缓存 |
| 重建方式 | 从 `content_comment` 按 `root_id + level` 重建 |

### 8.4 评论摘要缓存

```text
comment:meta:{commentId}
```

| 项 | 说明 |
|---|---|
| 数据结构 | String |
| value | 评论元信息 + 正文摘要 JSON |
| TTL | 1 到 6 小时 |
| 写入时机 | 评论查询回源后写入 |
| 失效策略 | 评论删除、状态变化、计数快照变化后删除 |
| 重建方式 | 从 `content_comment` + `comment_content` 查询 |

摘要内容建议包含：

```json
{
  "commentId": "30001",
  "noteId": "90001",
  "userId": "10001",
  "rootId": "30001",
  "level": 1,
  "replyCommentId": "0",
  "replyUserId": "0",
  "content": "写得真好",
  "commentStatus": "VISIBLE",
  "createdAt": "2026-06-03T20:30:00+08:00"
}
```

### 8.5 当前用户评论点赞状态

```text
comment:liked:{userId}:{commentId}
```

| 项 | 说明 |
|---|---|
| 数据结构 | String |
| value | `LIKED` / `NOT_LIKED` |
| TTL | 1 小时 |
| 写入时机 | 查询评论列表或点赞操作后 |
| 失效策略 | 点赞、取消点赞后删除或更新 |
| 重建方式 | 从 `comment_like` 查询 |

批量列表场景也可以使用 Hash：

```text
comment:liked:batch:{userId}
```

但要控制 TTL 和容量，避免大 key。

### 8.6 限流 Key

```text
rate:comment:create:{userId}
rate:comment:like:{userId}
rate:comment:delete:{userId}
rate:comment:note:{noteId}
```

用途：

1. 限制单用户高频评论。
2. 限制单用户高频点赞。
3. 限制热点笔记评论写入压力。
4. 限制删除操作频率。

## 10. 安全与风控设计

### 10.1 身份和权限

要求：

1. 发布评论、删除评论、点赞评论必须登录。
2. 当前用户 ID 只能来自网关注入的用户上下文。
3. 移动端不能传 `userId` 作为评论作者。
4. 评论作者可以删除自己的评论。
5. 笔记作者可以删除自己笔记下的评论。
6. 内部接口不能暴露公网。

### 10.2 用户状态校验

发布评论前调用用户服务校验：

```text
POST /internal/users/status-check
scene = CREATE_COMMENT
```

禁用、注销、禁言用户不允许发布评论。是否允许点赞可以按用户状态策略确定，第一阶段建议禁用用户不能点赞。

### 10.3 笔记状态校验

发布评论前调用笔记服务：

```text
POST /internal/notes/comment-check
```

需要校验：

1. 笔记存在。
2. 笔记已发布。
3. 笔记未删除、未下架。
4. 当前用户有权限查看该笔记。
5. 笔记允许评论。

### 10.4 内容安全

第一阶段内容安全：

1. 评论长度限制，例如 1 到 500 字。
2. 去除前后空白。
3. 禁止纯空白、纯重复字符。
4. 本地敏感词校验。
5. 命中严重敏感词直接拒绝。
6. 命中可疑词进入 `SELF_VISIBLE` 或 `REVIEWING`。

后续可以接入审核服务、举报和 ES 召回。

### 10.5 防刷策略

建议限制：

| 操作 | 限制 |
|---|---|
| 发布评论 | 单用户每分钟、每天限制 |
| 同一笔记评论 | 单用户对同一笔记短时间限制 |
| 回复同一评论 | 单用户短时间限制 |
| 评论点赞 | 单用户每分钟限制 |
| 删除评论 | 单用户每分钟限制 |

热点笔记评论写入压力较大时，可以增加 `rate:comment:note:{noteId}` 限流。

## 11. 前后端实现要点

### 11.1 移动端实现

移动端页面：

1. 笔记详情评论区。
2. 二级评论展开页或弹层。
3. 评论输入框。
4. 我的评论列表。

实现要点：

1. 评论发布成功后可先插入本地列表，但要以服务端返回 `commentId` 为准。
2. 一级评论默认按 `HOT` 查询。
3. 二级评论按时间正序分页。
4. 评论点赞可乐观更新，失败时回滚。
5. 计数存在最终一致延迟，前端不应依赖本地计数作为事实。
6. 删除评论后从列表移除或展示“已删除”，按后端状态处理。
7. `REVIEWING`、`SELF_VISIBLE` 评论只对作者展示。

### 11.2 后端实现

后端模块建议：

```text
bluenote-comment
  ├── api
  ├── application
  ├── domain
  ├── infrastructure
  ├── mq
  └── job
```

实现要点：

1. Controller 只做参数校验和用户上下文获取。
2. Application Service 编排用户校验、笔记校验、数据库写入、缓存失效和 outbox。
3. Domain 层封装评论层级、状态流转、删除权限和点赞幂等。
4. Infrastructure 层封装 MySQL、Redis、RocketMQ、用户服务、笔记服务、计数服务客户端。
5. MQ 生产使用 outbox，失败可重试。
6. Job 负责 outbox 投递、缓存重建、冗余表校准。

### 11.3 笔记服务集成

发布评论时，评论服务只依赖笔记服务内部接口，不访问笔记表。

建议 `comment-check` 返回：

1. `noteId`
2. `noteStatus`
3. `authorId`
4. `commentEnabled`
5. `visibleToCurrentUser`

评论服务把 `authorId` 保存为 `note_author_id` 快照，用于删除权限和通知事件。

### 11.4 用户服务集成

评论列表需要用户展示信息。

两种方式：

1. 评论服务查询时调用用户服务批量摘要。
2. 评论表保存用户快照，用户服务事件更新或列表展示允许短暂旧头像旧昵称。

第一阶段建议：

1. 评论创建时保存用户快照。
2. 列表查询时优先使用快照。
3. 如果需要强一致展示，再调用用户服务批量补全。

### 11.5 计数服务集成

评论列表需要展示：

1. 评论点赞数。
2. 一级评论回复数。

可以调用计数服务批量查询：

```text
POST /internal/counters/batch
```

计数服务失败时：

1. 使用 `content_comment.like_count_snapshot`。
2. 使用 `content_comment.reply_count_snapshot`。
3. 响应中标记 `degraded = true`。

## 12. 数据一致性与事件

### 12.1 本地事务边界

发布评论本地事务：

```text
写 content_comment
写 user_comment
写 comment_content
写 comment_operation_log
写 comment_outbox_event(CommentCreated)
```

点赞评论本地事务：

```text
写或更新 comment_like
写 comment_outbox_event(CommentLiked / CommentUnliked)
```

删除评论本地事务：

```text
更新 content_comment
更新 user_comment
写 comment_operation_log
写 comment_outbox_event(CommentDeleted)
```

缓存删除可以在事务提交后执行，失败后由 TTL 或补偿修复。

### 12.2 发布事件

Topic：

```text
comment-event
```

事件：

| 事件 | 触发时机 | 消费方 |
|---|---|---|
| `CommentCreated` | 评论发布成功 | 计数、通知、排行榜 |
| `CommentDeleted` | 评论删除成功 | 计数、通知、排行榜 |
| `CommentLiked` | 评论点赞成功 | 计数、通知、排行榜、评论热度消费者 |
| `CommentUnliked` | 取消评论点赞成功 | 计数、排行榜、评论热度消费者 |
| `CommentStatusChanged` | 评论状态变化，后续 | 审核、通知、计数 |

### 12.3 CommentCreated 事件示例

```json
{
  "eventId": "evt_comment_created_30001",
  "eventType": "CommentCreated",
  "eventVersion": 1,
  "occurredAt": "2026-06-03T20:30:00+08:00",
  "producer": "bluenote-comment",
  "traceId": "trace-id",
  "payload": {
    "commentId": "30001",
    "noteId": "90001",
    "noteAuthorId": "20001",
    "userId": "10001",
    "level": 1,
    "rootId": "30001",
    "replyCommentId": "0",
    "replyUserId": "0",
    "commentStatus": "VISIBLE",
    "createdAt": "2026-06-03T20:30:00+08:00"
  }
}
```

消费者建议：

| 消费方 | 处理 |
|---|---|
| 计数服务 | 笔记评论数 +1；如果是二级评论，一级评论回复数 +1 |
| 通知服务 | 通知笔记作者或被回复用户 |
| 排行榜服务 | 增加笔记互动热度 |
| 评论热度消费者 | 二级评论创建后更新一级评论热度 |

### 12.4 CommentDeleted 事件示例

```json
{
  "eventId": "evt_comment_deleted_30001",
  "eventType": "CommentDeleted",
  "eventVersion": 1,
  "occurredAt": "2026-06-03T20:40:00+08:00",
  "producer": "bluenote-comment",
  "traceId": "trace-id",
  "payload": {
    "commentId": "30001",
    "noteId": "90001",
    "userId": "10001",
    "level": 1,
    "rootId": "30001",
    "affectedCommentCount": 9,
    "affectedReplyCount": 8,
    "operatorId": "10001",
    "operatorType": "USER",
    "deletedAt": "2026-06-03T20:40:00+08:00"
  }
}
```

说明：

1. 删除一级评论并隐藏整组回复时，`affectedCommentCount` 包含一级评论本身和受影响的可见二级评论。
2. 删除二级评论时，`affectedCommentCount = 1`，`affectedReplyCount = 1`。
3. 计数服务消费该事件后减少笔记评论数，并按需减少一级评论回复数。

### 12.5 CommentLiked 事件示例

```json
{
  "eventId": "evt_comment_liked_30001_10002",
  "eventType": "CommentLiked",
  "eventVersion": 1,
  "occurredAt": "2026-06-03T20:41:00+08:00",
  "producer": "bluenote-comment",
  "traceId": "trace-id",
  "payload": {
    "commentId": "30001",
    "noteId": "90001",
    "commentUserId": "10001",
    "userId": "10002",
    "rootId": "30001",
    "likedAt": "2026-06-03T20:41:00+08:00"
  }
}
```

### 12.6 订阅事件

第一阶段评论服务可订阅：

| 事件 | 来源 | 用途 |
|---|---|---|
| `NoteDeleted` | 笔记服务 | 将笔记下评论隐藏或标记不可见 |
| `NoteStatusChanged` | 笔记服务 | 笔记下架时隐藏评论区 |
| `UserStatusChanged` | 用户服务 | 禁止异常用户继续评论，必要时隐藏历史评论 |
| `UserProfileUpdated` | 用户服务 | 后续更新评论用户展示快照 |

消费要求：

1. 按 `eventId` 幂等。
2. 不直接物理删除历史评论。
3. 状态变更后删除相关 Redis 缓存。

### 12.7 冗余表一致性

第一阶段同步写 `content_comment` 和 `user_comment`，两者在同一事务中提交。

如果后续评论量上升，可以演进为：

```text
content_comment 作为主表
  -> outbox 或 CDC
  -> user_comment 作为派生表
```

当前阶段不引入 Canal 到评论服务，避免 CDC 复杂度继续扩散。

### 12.8 缓存一致性

缓存处理：

1. 评论发布后，删除或增量更新 `comment:time:{noteId}:level1` 或 `comment:replies:{rootId}`。
2. 评论删除后，删除 `comment:meta:{commentId}`，并从列表 ZSET 移除。
3. 评论点赞后，删除点赞状态缓存，并异步更新热度 ZSET。
4. 缓存删除失败记录日志，交给 TTL 或重建任务修复。

## 15. 风险与后续演进

| 风险 | 说明 | 应对 |
|---|---|---|
| 热门评论和时间序混合分页复杂 | HOT 列表不足时要切换 TIME 列表 | 游标中携带来源和偏移，服务端统一处理 |
| 评论状态导致列表不足 | 业务层过滤后可能不足一页 | 多取候选评论，最多补拉一定次数 |
| 双表写入不一致 | `content_comment` 和 `user_comment` 需要保持一致 | 第一阶段同事务写，后续补偿校准 |
| 评论正文在 MySQL 中增长 | 长期评论量上升会增加存储压力 | 垂直拆分正文表，后续迁移 KV |
| 热点笔记评论写压力 | 同一笔记短时间大量评论 | 单笔记限流，后续异步写和写聚合 |
| 计数短暂不准 | 评论数、点赞数异步更新 | 计数服务幂等、重试、校准，列表可降级 |
| 审核能力不足 | 第一阶段只有敏感词 | 预留状态，后续接入审核和举报 |
