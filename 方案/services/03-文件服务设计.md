# BlueNote 文件服务设计

## 1. 背景与目标

文件服务负责 BlueNote 中图片、视频、头像、主页背景图等文件资源的上传凭证、元数据、访问地址、业务绑定、生命周期和清理任务。文件本体不进入 MySQL，也不通过普通业务接口上传到后端 JVM，而是通过对象存储承载。

第一阶段的实际存储实现统一使用部署在 2 核 4G 云服务器上的单机 MinIO。开发、测试、正式环境都按 MinIO 落地，以降低成本和部署复杂度。但文件服务的接口、对象模型和存储适配层按企业级对象存储方式设计，业务服务只依赖 `fileId` 和文件服务接口，不直接依赖 MinIO 的 bucket、objectKey 或访问 URL。

这样可以同时满足两个目标：

1. 当前阶段真实落地，文件存在自己的服务器中，不额外购买云 OSS/COS/R2。
2. 后续如果图片数量、访问量、带宽或可用性要求提高，可以迁移到云 OSS/COS/R2 + CDN，而不推翻移动端、用户服务、笔记服务和 IM 服务接口。

设计目标：

1. 支持移动端申请上传凭证。
2. 支持移动端直传 MinIO。
3. 支持上传完成确认和对象校验。
4. 支持文件元数据管理。
5. 支持头像、主页背景图、笔记图片、笔记视频、IM 图片等业务场景。
6. 支持业务服务校验文件归属、类型、大小和状态。
7. 支持业务服务绑定和解绑文件。
8. 支持文件访问 URL 生成。
9. 支持未绑定文件清理和延迟物理删除。
10. 为图片压缩、视频转码、内容审核、云对象存储、CDN 预留扩展点。

关键约束：

1. 文件流量不经过后端业务服务。
2. 文件本体不存 MySQL。
3. RocketMQ 不传输文件本体，只传 `fileId`、`objectKey`、处理状态等轻量信息。
4. MinIO 控制台不暴露公网。
5. 正式环境 MinIO 数据目录必须持久化，并定期备份到本地旧电脑。
6. 文件 URL 不是长期事实来源，长期事实来源是 `fileId`、`bucket`、`objectKey` 和文件元数据。
7. 业务服务保存自己的业务事实，例如用户服务保存当前头像 `fileId`，笔记服务保存笔记媒体列表。

## 2. 功能范围

### 2.1 第一阶段支持

| 功能 | 说明 |
|---|---|
| 申请上传凭证 | 根据场景、文件类型、大小申请预签名上传 URL |
| 移动端直传 MinIO | 移动端使用预签名 URL 上传文件本体 |
| 上传完成确认 | 客户端通知文件服务确认上传完成 |
| 对象存在性校验 | 文件服务通过 MinIO `HEAD Object` 校验对象是否存在 |
| 文件元数据保存 | 保存 `fileId`、bucket、objectKey、大小、MIME、ownerId、状态等 |
| 文件归属校验 | 用户服务、笔记服务、IM 服务绑定前校验 ownerId |
| 文件场景校验 | 按头像、背景图、笔记图片、笔记视频等限制类型和大小 |
| 文件绑定 | 业务服务绑定 `fileId`，文件状态进入 `BOUND` |
| 文件解绑 | 业务删除或替换时解除绑定 |
| 文件访问 URL | 生成可访问 URL 或签名 URL |
| 未绑定文件清理 | 定时清理上传后长期未绑定的临时文件 |
| 延迟物理删除 | 业务删除后先逻辑删除，再异步删除 MinIO 对象 |
| 文件事件 | 发布 `FileUploaded`、`FileBound`、`FileDeleted` 等事件 |

### 2.2 第一阶段不支持

| 功能 | 暂不支持原因 |
|---|---|
| 云 OSS/COS/R2 | 第一阶段控制成本，统一使用自建 MinIO |
| CDN | 当前用户量和图片量很小，暂不额外接入 |
| 完整视频转码 | 资源消耗较高，第一阶段只预留状态和任务结构 |
| 图片多规格压缩 | 第一阶段可先使用原图或简单限制图片尺寸 |
| 第三方图片审核 | 需要付费服务或额外平台，第一阶段只预留审核字段 |
| 秒传 | 需要全局 hash 去重和复杂权限判断，后续再评估 |
| 分片上传 | 第一阶段文件大小受控，后续视频变大时再补 |
| 文件跨区域复制 | 单机 MinIO 阶段通过备份任务保障 |
| 私有云多节点 MinIO 集群 | 当前服务器资源有限，先单机部署 |

### 2.3 后续扩展

后续可以扩展：

1. 云 OSS/COS/R2 存储实现。
2. CDN 加速域名。
3. 图片压缩和多尺寸缩略图。
4. 视频转码、封面抽帧、时长识别。
5. 内容审核服务。
6. 分片上传和断点续传。
7. 文件 hash 去重。
8. 冷热存储和归档。
9. 多 bucket 生命周期策略。
10. 管理端文件查询和违规处理。

这些扩展不能改变业务服务只依赖 `fileId` 的基本约定。

## 3. 服务边界

### 3.1 文件服务负责

文件服务负责：

1. 生成上传凭证。
2. 生成服务端 objectKey。
3. 管理文件元数据。
4. 校验文件所有权。
5. 校验文件类型、大小、场景限制。
6. 生成访问 URL。
7. 记录文件业务绑定提示。
8. 清理未绑定文件。
9. 删除对象存储中的文件本体。
10. 发布文件生命周期事件。

文件服务的数据事实：

| 数据 | 说明 |
|---|---|
| `fileId` | 文件全局 ID |
| `ownerId` | 上传用户 |
| `bucket` | 对象存储 bucket |
| `objectKey` | 对象存储 key |
| `storageType` | 第一阶段为 `MINIO` |
| `fileSize` | 文件大小 |
| `mimeType` | MIME 类型 |
| `fileExt` | 文件扩展名 |
| `fileHash` | 文件 hash，第一阶段可选 |
| `fileStatus` | 文件生命周期状态 |
| `scene` | 文件业务场景 |

### 3.2 用户服务负责

用户服务负责：

1. 当前用户头像使用哪个 `fileId`。
2. 当前用户主页背景图使用哪个 `fileId`。
3. 修改资料时调用文件服务校验文件。
4. 用户资料中保存头像 URL 快照时，仍以 `fileId` 为事实来源。

用户服务不直接访问 MinIO，也不直接读取文件服务数据库。

### 3.3 笔记发布服务负责

笔记发布服务负责：

1. 笔记包含哪些图片。
2. 笔记图片展示顺序。
3. 笔记使用哪个视频。
4. 笔记封面图选择。
5. 笔记发布、编辑、删除时的媒体绑定关系。

文件服务只校验文件是否可被当前用户、当前业务场景使用，不判断笔记是否发布成功。

### 3.4 IM 服务负责

IM 服务负责：

1. IM 消息引用哪个 `fileId`。
2. 图片消息和后续视频消息的消息状态。
3. 消息撤回、删除等业务语义。

文件服务不负责 IM 消息投递和已读状态。

### 3.5 对象存储负责

第一阶段对象存储为 MinIO，负责文件本体保存和对象读写能力。

文件服务通过存储适配层访问 MinIO：

```text
ObjectStorageClient
  -> MinioObjectStorageClient
  -> 后续 AliyunOssObjectStorageClient / TencentCosObjectStorageClient / R2ObjectStorageClient
```

业务服务不直接依赖 `MinioClient`。

## 4. 核心概念

### 4.1 fileId

`fileId` 是 BlueNote 内部文件 ID。业务服务只保存和传递 `fileId`，不保存 MinIO 的内部访问细节。

原则：

1. `fileId` 全局唯一。
2. `fileId` 由文件服务生成。
3. 文件服务按 `fileId` 查询文件元数据。
4. 业务服务通过 `fileId` 调用文件服务校验和获取访问 URL。
5. 后续迁移存储实现时，`fileId` 不变化。

### 4.2 objectKey

`objectKey` 是对象存储中的文件路径，由文件服务生成，客户端不能指定。

建议格式：

```text
{env}/{scene}/{yyyy}/{MM}/{dd}/{ownerId}/{fileId}.{ext}
```

示例：

```text
prod/note-image/2026/06/03/10001/900001.jpg
```

设计原因：

1. 便于按环境隔离。
2. 便于按场景做生命周期管理。
3. 便于排查用户上传问题。
4. 防止客户端构造路径覆盖其他文件。

### 4.3 bucket

第一阶段可以使用一个或少量 bucket。

建议：

| bucket | 用途 |
|---|---|
| `bluenote-public` | 公开可访问资源，例如发布后的笔记图片、头像 |
| `bluenote-private` | 私有或临时资源，例如上传中、审核中、未发布文件 |

如果为了简化部署，也可以第一阶段只使用：

```text
bluenote-files
```

并通过 objectKey 前缀区分场景。文档和代码仍保留 bucket 字段，后续可以平滑拆分。

### 4.4 文件场景

`scene` 表示文件用途。不同场景有不同限制。

| scene | 说明 |
|---|---|
| `USER_AVATAR` | 用户头像 |
| `USER_HOME_COVER` | 用户主页背景图 |
| `NOTE_IMAGE` | 笔记图片 |
| `NOTE_VIDEO` | 笔记视频 |
| `IM_IMAGE` | IM 图片 |
| `IM_VIDEO` | IM 视频，后续扩展 |

### 4.5 文件状态

文件状态建议：

| 状态 | 说明 |
|---|---|
| `UPLOADING` | 已申请上传凭证，等待客户端上传 |
| `UPLOADED` | 对象存储中已存在，尚未被业务绑定 |
| `BOUND` | 已被业务对象绑定 |
| `ABANDONED` | 长时间未绑定，等待清理 |
| `DELETE_PENDING` | 已逻辑删除，等待物理删除 |
| `DELETED` | 已删除 |
| `BLOCKED` | 违规或审核失败，禁止访问 |

状态流转：

```text
UPLOADING -> UPLOADED
UPLOADING -> ABANDONED
UPLOADED -> BOUND
UPLOADED -> ABANDONED
BOUND -> DELETE_PENDING
BOUND -> BLOCKED
ABANDONED -> DELETE_PENDING
DELETE_PENDING -> DELETED
```

上传成功不等于业务使用成功。只有用户服务、笔记服务、IM 服务等业务方完成绑定后，文件才进入 `BOUND`。

### 4.6 绑定关系

文件绑定关系表示文件已被某个业务对象引用。

示例：

| bindType | bindId | 说明 |
|---|---|---|
| `USER_AVATAR` | userId | 用户头像 |
| `USER_HOME_COVER` | userId | 用户主页背景图 |
| `NOTE_MEDIA` | noteId | 笔记媒体 |
| `IM_MESSAGE` | messageId | IM 图片消息 |

文件服务的绑定表是生命周期管理依据，不是业务事实来源。比如“某篇笔记到底有哪些图片”仍然以笔记服务自己的媒体表为准。

### 4.7 访问 URL

访问 URL 是派生结果，不是事实来源。

文件事实来源：

```text
fileId + bucket + objectKey + storageType
```

访问 URL 可以根据部署环境生成：

```text
https://oss.bluenote.example.com/bluenote-files/prod/note-image/...
```

或生成签名 URL：

```text
https://oss.bluenote.example.com/...?X-Amz-Signature=...
```

后续迁移 CDN 或云 OSS 时，只需要调整文件服务的 URL 生成逻辑。

## 5. 核心业务流程

### 5.1 申请上传凭证

移动端上传文件前，先向文件服务申请上传凭证。

流程：

```text
移动端选择图片或视频
  -> 调用 POST /api/files/upload-token
  -> 网关校验 Access Token
  -> 文件服务校验 scene、mimeType、size
  -> 生成 fileId
  -> 生成 bucket 和 objectKey
  -> 写 file_object，状态为 UPLOADING
  -> 调用 MinIO 生成预签名 PUT URL
  -> 返回 fileId、uploadUrl、headers、expireAt
```

服务端必须校验：

1. 用户是否登录。
2. 文件场景是否合法。
3. 文件大小是否超过场景限制。
4. MIME 类型是否允许。
5. 用户上传频率是否超过限制。
6. 单次申请数量是否超过限制。

### 5.2 移动端直传 MinIO

移动端拿到预签名 URL 后直接上传 MinIO。

流程：

```text
移动端
  -> 使用 uploadUrl 执行 PUT
  -> 上传文件本体到 MinIO
  -> MinIO 返回成功
  -> 移动端调用文件服务确认上传
```

注意：

1. 上传文件流不进入网关和后端业务服务。
2. 预签名 URL 有短有效期，例如 10 分钟。
3. 客户端必须按文件服务返回的 Header 上传。
4. 客户端文件名只做展示参考，不参与 objectKey 生成。

### 5.3 上传完成确认

移动端上传成功后，调用文件服务确认。

流程：

```text
移动端调用 POST /api/files/{fileId}/confirm
  -> 文件服务查询 file_object
  -> 校验 ownerId 是当前用户
  -> 校验状态为 UPLOADING
  -> 调用 MinIO HEAD Object
  -> 校验对象存在
  -> 校验 Content-Length
  -> 校验 Content-Type 或重新识别 MIME
  -> 更新状态为 UPLOADED
  -> 写 outbox FileUploaded
  -> 返回 fileId、fileStatus、accessUrl
```

如果确认时对象不存在：

```text
返回 FILE_OBJECT_NOT_FOUND
```

如果对象大小和申请时不一致：

```text
返回 FILE_SIZE_MISMATCH
```

这种情况下文件状态保持 `UPLOADING` 或进入异常清理流程。

### 5.4 用户头像绑定

用户更新头像时，用户服务调用文件服务校验并绑定。

流程：

```text
移动端上传头像
  -> 文件服务返回 fileId
  -> 移动端提交 PUT /api/users/me/profile
  -> 用户服务调用 /internal/files/validate
  -> 文件服务校验 ownerId、scene、mimeType、size、status
  -> 用户服务更新 user_profile.avatar_file_id
  -> 用户服务调用 /internal/files/bind
  -> 文件服务记录 USER_AVATAR 绑定
  -> 文件状态进入 BOUND
```

用户更换头像时：

```text
用户服务绑定新头像 fileId
  -> 解绑旧头像 fileId
  -> 旧头像如果无其他绑定，进入延迟删除或保留策略
```

第一阶段为了避免误删，可以对旧头像采用延迟删除，例如 7 天后清理。

### 5.5 笔记图片绑定

笔记发布时，笔记服务需要校验并绑定图片文件。

流程：

```text
移动端上传笔记图片
  -> 得到多个 fileId
  -> 发布笔记时提交 fileId 列表和顺序
  -> 笔记服务调用 /internal/files/batch-validate
  -> 校验所有文件 ownerId、scene、状态、类型、大小
  -> 笔记服务保存 note_media
  -> 笔记服务调用 /internal/files/batch-bind
  -> 文件服务记录 NOTE_MEDIA 绑定
```

原则：

1. 笔记图片顺序归笔记服务维护。
2. 文件服务不保存笔记图片排序。
3. 单篇笔记图片数量限制建议为 1 到 9 张。
4. 发布失败时，文件仍保持 `UPLOADED`，后续未绑定清理任务处理。

### 5.6 笔记视频绑定

第一阶段可以预留视频上传能力，但严格控制大小和时长。

建议：

1. 第一阶段笔记视频最大 100MB 或 200MB。
2. 暂不做完整转码。
3. 只允许常见移动端格式，例如 `video/mp4`、`video/quicktime`。
4. 视频封面可以由客户端上传图片作为 `NOTE_IMAGE`，后续再做抽帧。

如果后续引入转码，流程扩展为：

```text
视频上传确认
  -> FileUploaded
  -> MediaProcessTask 创建
  -> 转码/抽帧
  -> MediaProcessCompleted
  -> 笔记服务允许发布或更新视频状态
```

### 5.7 文件访问

业务服务或移动端需要展示文件时，可以通过文件服务获取访问 URL。

流程：

```text
业务服务持有 fileId
  -> 调用文件服务获取访问 URL
  -> 文件服务校验文件状态
  -> 根据 storageType、bucket、objectKey 生成 URL
  -> 返回访问 URL
```

第一阶段建议：

1. 已发布的头像、笔记图片可返回公网可访问 URL。
2. 上传中、未绑定、被封禁文件不返回公开访问 URL。
3. 私有或临时文件返回短期签名 URL。

### 5.8 文件删除

业务删除文件引用时，先解绑，再由文件服务延迟删除。

流程：

```text
业务服务删除或替换文件
  -> 调用 /internal/files/unbind
  -> 文件服务判断是否还有其他绑定
  -> 无绑定则更新为 DELETE_PENDING 或保留一段时间
  -> 删除任务异步删除 MinIO 对象
  -> 删除成功后状态变为 DELETED
  -> 发布 FileDeleted
```

不建议业务服务直接物理删除对象。文件删除需要异步、可重试、可审计。

## 6. 异常流程

| 异常 | 处理方式 |
|---|---|
| 上传凭证过期 | 移动端重新申请 |
| 上传到 MinIO 失败 | 移动端重试，超过次数重新申请凭证 |
| 确认上传时对象不存在 | 返回 `FILE_OBJECT_NOT_FOUND` |
| 对象大小不匹配 | 返回 `FILE_SIZE_MISMATCH`，文件进入异常清理 |
| MIME 类型不匹配 | 返回 `FILE_TYPE_NOT_ALLOWED` |
| 用户绑定他人文件 | 返回 `FILE_OWNER_MISMATCH` |
| 文件场景不匹配 | 返回 `FILE_SCENE_MISMATCH` |
| 文件已删除 | 返回 `FILE_DELETED` |
| 文件被封禁 | 返回 `FILE_BLOCKED` |
| 重复确认上传 | 如果已是 `UPLOADED` 或 `BOUND`，幂等返回成功 |
| 重复绑定 | 根据 `bindType + bindId + fileId` 幂等成功 |
| 业务发布失败 | 文件保持 `UPLOADED`，等待未绑定清理 |
| MinIO 临时不可用 | 上传凭证申请、确认、访问 URL 生成失败，记录告警 |
| 删除 MinIO 对象失败 | 删除任务重试，超过阈值告警 |
| RocketMQ 不可用 | outbox 保留待投递，不影响核心数据库事务 |

异常补偿重点：

1. `UPLOADING` 超时未确认。
2. `UPLOADED` 长时间未绑定。
3. `DELETE_PENDING` 删除失败。
4. outbox 事件投递失败。
5. MinIO 对象存在但 MySQL 元数据缺失。
6. MySQL 元数据存在但 MinIO 对象缺失。

## 7. 数据库设计

### 7.1 表清单

| 表名 | 说明 |
|---|---|
| `file_object` | 文件元数据主表 |
| `file_upload_session` | 上传会话表 |
| `file_binding` | 文件业务绑定表 |
| `file_delete_task` | 文件删除任务表 |
| `file_process_task` | 媒体处理任务表，后续扩展 |
| `file_event_consume_log` | 文件服务消费事件幂等表，后续扩展 |
| `file_outbox_event` | 文件服务 outbox 事件表 |
| `file_audit_log` | 文件操作审计表 |

### 7.2 file_object

文件元数据主表。

| 字段 | 类型 | 说明 |

## 8. 对象存储设计

### 8.1 第一阶段 MinIO 部署口径

第一阶段所有环境统一使用 MinIO。

| 环境 | 存储实现 | 说明 |
|---|---|---|
| dev | 本机 Docker Compose MinIO | 可重置数据 |
| test | 测试服务器或旧电脑 MinIO | 验证上传流程和备份 |
| prod | 2 核 4G 云服务器 MinIO | 小规模真实使用 |

正式环境 MinIO 要求：

1. 数据目录挂载到持久化磁盘。
2. 不暴露 MinIO 控制台公网入口。
3. 对象访问通过 HTTPS 域名或反向代理。
4. 定期备份到本地旧电脑。
5. 配置磁盘容量告警。
6. 控制图片和视频大小。
7. 保留迁移云 OSS 的适配层。

### 8.2 存储适配层

后端代码不应在业务逻辑中直接使用 MinIO API。

建议抽象：

```text
ObjectStorageClient
  generatePresignedPutUrl(request)
  generatePresignedGetUrl(request)
  headObject(bucket, objectKey)
  deleteObject(bucket, objectKey)
  copyObject(source, target)
```

第一阶段实现：

```text
MinioObjectStorageClient
```

后续扩展：

```text
AliyunOssObjectStorageClient
TencentCosObjectStorageClient
CloudflareR2ObjectStorageClient
```

### 8.3 bucket 和路径规范

第一阶段推荐：

```text
bucket: bluenote-files
```

objectKey：

```text
{env}/{scene}/{yyyy}/{MM}/{dd}/{ownerId}/{fileId}.{ext}
```

示例：

```text
prod/user-avatar/2026/06/03/10001/900001.jpg
prod/note-image/2026/06/03/10001/900002.webp
prod/note-video/2026/06/03/10001/900003.mp4
```

后续如果需要拆分 bucket，可以按访问级别和生命周期拆：

```text
bluenote-public
bluenote-private
bluenote-temp
```

### 8.4 访问域名

建议域名：

```text
https://oss.bluenote.example.com
```

正式环境可以通过 Nginx 或 Caddy 反向代理 MinIO 对象访问 API。

不建议暴露：

```text
https://minio-console.bluenote.example.com
```

如果必须访问控制台，只允许：

1. 内网访问。
2. VPN。
3. 带访问控制的内网穿透管理入口。
4. 强密码和最小权限账号。

### 8.5 URL 生成策略

访问 URL 由文件服务生成。

公开资源：

```text
https://oss.bluenote.example.com/{bucket}/{objectKey}
```

私有资源：

```text
预签名 GET URL，短有效期
```

第一阶段为了简化，可以将头像、已发布笔记图片作为公开资源访问，但上传中、未绑定、审核中、封禁文件不能公开访问。

### 8.6 备份策略

MinIO 单机不是高可用存储。第一阶段用户量小可以接受，但必须做备份。

建议：

1. 每天增量同步 MinIO 数据到本地旧电脑。
2. 每周做一次对象数量和大小校验。
3. 每月至少做一次恢复演练。
4. MySQL 文件元数据和 MinIO 对象备份要保持时间接近。
5. 备份脚本记录日志并纳入告警。

可以使用 MinIO `mc mirror` 或对象同步脚本实现。

## 9. 接口设计

### 9.1 外部接口

外部接口通过网关暴露给移动端。

#### 9.1.1 申请上传凭证

```text
POST /api/files/upload-token
```

请求：

```json
{
  "scene": "NOTE_IMAGE",
  "filename": "demo.jpg",
  "mimeType": "image/jpeg",
  "fileSize": 2048000
}
```

响应：

```json
{
  "code": "OK",
  "message": "success",
  "data": {
    "fileId": "900001",
    "uploadMethod": "PRESIGNED_PUT",
    "uploadUrl": "https://oss.bluenote.example.com/...",
    "headers": {
      "Content-Type": "image/jpeg"
    },
    "expireAt": "2026-06-03T21:10:00+08:00",
    "objectKey": "prod/note-image/2026/06/03/10001/900001.jpg"
  }
}
```

说明：

1. `objectKey` 返回给客户端仅用于排查，不作为业务事实。

```json
  "message": "success",

  }
}
```

#### 9.1.3 获取文件访问 URL



响应：

```json
{
  "code": "OK",
```

公开资源可以返回长期 URL；私有资源返回签名 URL。

用于排查或后续草稿箱，当前移动端可以先不做入口。

要求：

1. 申请上传凭证必须登录。
2. `ownerId` 从网关注入的用户上下文获取，不能信任客户端传入。
3. 客户端不能指定 bucket 和 objectKey。
4. 上传 URL 有短有效期。
5. 上传 URL 只能用于指定 objectKey。
6. 用户上传频率必须限制。

### 11.2 文件类型限制

第一阶段建议限制：

| scene | MIME 类型 | 大小限制 | 数量限制 |
|---|---|---:|---:|
| `USER_AVATAR` | `image/jpeg`, `image/png`, `image/webp` | 5MB | 1 |
| `USER_HOME_COVER` | `image/jpeg`, `image/png`, `image/webp` | 10MB | 1 |
| `NOTE_IMAGE` | `image/jpeg`, `image/png`, `image/webp` | 10MB/张 | 单笔记最多 9 张 |
| `NOTE_VIDEO` | `video/mp4`, `video/quicktime` | 100MB 到 200MB | 单笔记最多 1 个 |
| `IM_IMAGE` | `image/jpeg`, `image/png`, `image/webp` | 10MB | 单消息 1 个 |

视频能力第一阶段可以先预留接口，实际启用时再根据服务器带宽和磁盘容量调小限制。

### 11.3 文件内容安全

文件属于用户生成内容，必须预留审核能力。

字段：

```text
audit_status: PENDING / PASSED / REJECTED / SKIPPED
```

第一阶段建议：

1. 头像、笔记图片默认 `SKIPPED` 或 `PENDING`，根据是否接入审核决定。
2. 不接第三方审核时，先依靠用户举报和管理端处理。
3. 发现违规后将文件状态改为 `BLOCKED`。
4. `BLOCKED` 文件不生成访问 URL。
5. 发布 `FileBlocked` 事件通知业务服务处理引用。

### 11.4 访问安全

要求：

1. MinIO 管理控制台不暴露公网。
2. 上传使用预签名 PUT URL。
3. 私有文件使用预签名 GET URL。
4. 公开文件只允许读，不允许匿名写。
5. MinIO AccessKey 和 SecretKey 不进入代码仓库。
6. 移动端永远不持有 MinIO 永久密钥。

### 11.5 防刷与配额

建议限制：

| 维度 | 限制 |
|---|---|
| 单用户申请上传凭证 | 每分钟 30 次 |
| 单用户图片上传总量 | 第一阶段可配置，例如每天 200MB |
| 单用户视频上传总量 | 第一阶段可配置，例如每天 500MB |
| 单次批量校验 | 最多 20 个 fileId |
| 单篇笔记图片 | 最多 9 张 |

Redis Key 示例：

```text
rate:file:upload-token:{userId}
quota:file:daily-size:{userId}:{yyyyMMdd}
```

Redis 不可用时，可以降级为只依赖单文件大小限制，并记录告警。

## 12. 前后端实现要点

### 12.1 移动端实现

移动端上传流程：

1. 用户选择图片或视频。
2. 移动端读取文件大小、后缀、MIME。
3. 调用文件服务申请上传凭证。
4. 使用 `uni.uploadFile` 或平台能力上传到预签名 URL。
5. 上传成功后调用确认接口。
6. 得到 `fileId`。
7. 在用户资料或笔记发布接口中提交 `fileId`。

实现注意：

1. 移动端不要直接保存 objectKey 作为业务字段。
2. 上传失败要支持重试。
3. 上传凭证过期后重新申请。
4. 笔记发布前要确认所有图片都已 `UPLOADED`。
5. 取消发布后不需要移动端主动删除文件，后端清理任务处理。

### 12.2 后端实现

后端模块建议：

```text
bluenote-file
  ├── api
  ├── application
  ├── domain
  ├── infrastructure
  │   ├── mysql
  │   ├── redis
  │   └── storage
  ├── mq
  └── job
```

实现要点：

1. Controller 只处理参数校验和用户上下文。
2. Application Service 编排上传凭证、确认、绑定、删除。
3. Domain 层封装文件状态机和场景限制。
4. Storage 层通过 `ObjectStorageClient` 访问 MinIO。
5. 内部接口和外部接口分包，防止误暴露。
6. 删除任务和 outbox 投递任务独立执行。
7. 所有绑定、确认、删除操作都要幂等。

### 12.3 MinIO 客户端实现

MinIO 客户端负责：

1. 创建预签名 PUT URL。
2. 创建预签名 GET URL。
3. `HEAD Object`。
4. 删除对象。
5. 读取 ETag、Content-Length、Content-Type。

连接配置通过 Nacos 和环境变量管理：

```text
BLUENOTE_OSS_ENDPOINT=
BLUENOTE_OSS_ACCESS_KEY=
BLUENOTE_OSS_SECRET_KEY=
BLUENOTE_OSS_BUCKET=
BLUENOTE_OSS_PUBLIC_ENDPOINT=
```

密钥不能写入 Git。

### 12.4 用户服务集成

用户服务修改头像或主页背景图时：

1. 调用文件服务 `validate`。
2. 更新用户资料表。
3. 调用文件服务 `bind`。
4. 解绑旧文件。

如果 `bind` 调用失败：

1. 用户资料更新已经成功时，需要补偿重试绑定。
2. 可以在用户服务记录待绑定任务。
3. 文件服务的未绑定清理任务需要避免过早删除刚被业务引用但绑定延迟的文件。

### 12.5 笔记服务集成

笔记发布时：

1. 调用文件服务批量校验。
2. 保存笔记和媒体绑定表。
3. 调用文件服务批量绑定。
4. 发布 `NotePublished`。

如果笔记保存失败，不绑定文件，后续由未绑定清理任务处理。

## 13. 数据一致性与补偿

### 13.1 上传一致性

上传涉及文件服务 MySQL 和 MinIO。

流程：

```text
申请上传凭证
  -> 写 file_object(UPLOADING)
  -> 写 file_upload_session
  -> 返回预签名 URL

客户端上传 MinIO
  -> 调用 confirm
  -> HEAD Object 校验
  -> 更新 file_object(UPLOADED)
```

可能不一致：

1. MySQL 有 `UPLOADING`，MinIO 无对象。
2. MinIO 有对象，客户端未确认。
3. MySQL 更新失败，但 MinIO 已有对象。

补偿：

1. 扫描超时 `UPLOADING`。
2. 对超时对象执行 `HEAD Object`。
3. 如果对象存在，补确认或标记异常。
4. 如果对象不存在，标记 `ABANDONED` 并删除任务。

### 13.2 绑定一致性

绑定涉及业务服务和文件服务。

建议业务流程：

```text
业务服务本地事务保存业务事实
  -> 调用文件服务 bind
  -> bind 失败则记录补偿任务
```

原因：

1. 文件绑定是生命周期提示。
2. 业务事实必须先由业务服务保存。
3. 文件绑定失败可以通过补偿修复。

为避免误删，未绑定文件清理任务应设置较长宽限期，例如：

```text
UPLOADED 超过 24 小时仍未绑定才清理
```

### 13.3 删除一致性

删除涉及 MySQL 状态和 MinIO 对象。

流程：

```text
本地事务：
  -> file_object 状态改为 DELETE_PENDING
  -> 写 file_delete_task
  -> 写 file_outbox_event

异步任务：
  -> 删除 MinIO 对象
  -> 成功后 file_object 状态改为 DELETED
```

如果删除 MinIO 对象失败：

1. 删除任务按退避策略重试。
2. 超过阈值告警。
3. 文件继续保持 `DELETE_PENDING`。

### 13.4 事件一致性

文件事件通过 outbox 保证最终投递。

要求：

1. 文件状态变更和 outbox 写入在同一个本地事务中。
2. outbox 投递失败可重试。
3. 消费方必须幂等。
4. 事件只传轻量元数据，不传文件本体。

## 14. 清理与生命周期任务

### 14.1 上传超时清理

扫描条件：

```text
file_status = UPLOADING
created_at < now - 30 minutes
```

处理：

1. 查询 MinIO 对象是否存在。
2. 不存在则标记 `ABANDONED`。
3. 存在但未确认，可以尝试补确认或标记待人工排查。

### 14.2 未绑定文件清理

扫描条件：

```text
file_status = UPLOADED
uploaded_at < now - 24 hours
没有有效 file_binding
```

处理：

1. 标记 `ABANDONED`。
2. 创建 `file_delete_task`。
3. 异步删除 MinIO 对象。

头像、笔记发布等业务可能存在短暂延迟，所以不建议上传后很快清理。

### 14.3 逻辑删除后的物理删除

扫描条件：

```text
file_status = DELETE_PENDING
```

处理：

1. 执行 MinIO 删除。
2. 成功后状态变为 `DELETED`。
3. 失败则更新 `retry_count` 和 `next_retry_at`。

### 14.4 备份任务

正式环境 MinIO 需要备份到本地旧电脑。

建议：

```text
每天凌晨执行增量同步
每周执行对象数量校验
每月执行恢复演练
```

备份日志需要记录：

1. 同步开始和结束时间。
2. 新增对象数量。
3. 删除对象数量。
4. 同步失败对象。
5. 总文件大小。

4. 文件校验失败率异常升高。

6. `UPLOADING` 超时数量异常。
7. `UPLOADED` 未绑定数量异常。
8. outbox 堆积超过阈值。
9. 备份任务失败。

### 16.1 单元测试

3. 文件大小限制。
4. objectKey 生成规则。
5. 文件状态流转。
6. 绑定幂等。
7. 删除任务重试时间计算。
8. URL 生成策略。

### 16.2 集成测试

1. 申请上传凭证写入 `file_object` 和 `file_upload_session`。
2. 使用 MinIO 预签名 URL 上传成功。
3. 确认上传时 `HEAD Object` 成功。
4. 文件大小不匹配时确认失败。
5. 用户服务校验头像文件成功。
6. 笔记服务批量校验图片成功。
7. 重复确认上传幂等成功。
8. 重复绑定幂等成功。
9. 删除任务成功删除 MinIO 对象。
10. outbox 投递失败后重试。

1. 未登录不能申请上传凭证。
2. 非法 MIME 类型申请失败。
3. 超过大小限制申请失败。
4. 用户不能绑定他人文件。
5. 场景不匹配时校验失败。

7. 被封禁文件不能生成访问 URL。

### 16.4 性能测试

1. 上传凭证接口 P95。
2. 上传确认接口 P95。

## 17. 风险与后续演进

| 风险 | 说明 | 应对 |
|---|---|---|
| MinIO 单机故障 | 正式环境文件访问依赖单机 MinIO | 数据持久化、备份、恢复演练 |
| 云服务器磁盘不足 | 图片、视频持续增长会占用磁盘 | 上传限制、清理任务、磁盘告警 |
| 带宽不足 | 图片访问走云服务器带宽 | 控制图片大小，后续迁移 OSS/CDN |
| 未绑定文件堆积 | 用户上传后取消发布 | 定时清理 `UPLOADED` 未绑定文件 |
| 误删文件 | 业务绑定延迟或绑定表不完整 | 较长清理宽限期、绑定补偿、逻辑删除 |
| URL 绑定 MinIO | 业务服务保存 MinIO URL 会影响迁移 | 业务服务保存 `fileId`，URL 由文件服务生成 |
| 内容违规 | 第一阶段未接图片审核 | 预留审核状态、举报和管理端处理 |
| 视频资源消耗 | 视频上传占磁盘和带宽 | 第一阶段限制视频大小或暂缓启用 |
| MinIO 密钥泄露 | 对象存储可被非法访问 | 密钥不入库、不进 Git、定期轮换 |

后续演进：

1. 将 `MinioObjectStorageClient` 替换或扩展为云 OSS/COS/R2 实现。
2. 接入 CDN，文件服务统一生成 CDN URL。
3. 图片上传后生成缩略图和多尺寸图。
4. 视频上传后转码、抽帧、生成封面。
5. 接入内容审核服务。
6. 支持分片上传和断点续传。
7. 按 hash 做文件去重和秒传。
8. 增加管理端文件检索、封禁、恢复能力。
9. 增加对象存储备份校验和自动恢复工具。
