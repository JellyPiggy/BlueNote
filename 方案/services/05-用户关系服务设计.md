# BlueNote 用户关系服务设计

## 1. 背景与目标

用户关系服务负责 BlueNote 用户之间的关注关系，是用户主页、关注页 Feed、通知、IM 私信策略、后续好友关系和社交推荐能力的基础服务。

这里需要先区分三类数据：

1. 关注行为事实，例如“用户 A 当前是否关注用户 B”，归用户关系服务维护。
2. 关注数、粉丝数等聚合计数，归计数服务维护，用户关系服务不通过 `COUNT(*)` 实时统计展示计数。
3. Feed 收件箱、通知、IM 权限等下游读模型，由对应服务基于关系事件维护，用户关系服务不直接写入。

本方案参考 `参考资料/10.用户关系服务.md` 的主线设计：以关注方向表作为数据中心，通过 binlog 订阅链路派生粉丝方向表、Redis 缓存和业务事件。第一阶段暂不引入图数据库，但会引入 Canal 监听 MySQL binlog，作为用户关系服务的一个技术亮点。

设计目标：

1. 支持关注、取消关注、关注列表、粉丝列表和关注关系查询。
2. 使用 MySQL 保存关注关系事实，Redis 只作为查询缓存。
3. 使用 `relation_following` 作为关注事实主表，`relation_follower` 作为粉丝方向查询表。
4. 使用 Canal 监听 MySQL binlog，通过 RocketMQ 驱动粉丝表、Redis 缓存和业务事件最终一致。
5. 对外发布稳定的业务事件 `UserFollowed`、`UserUnfollowed`，不让计数、Feed、通知直接消费原始 CDC。
6. 支持幂等、乱序保护、补偿校准和 Redis 重建。
7. 为黑名单、互相关注好友、共同关注、图数据库等后续能力预留扩展空间。

关键约束：

1. MySQL 是关注关系事实来源。
2. Redis 不能作为关注关系事实来源，Redis 数据丢失后必须可从 MySQL 重建。
3. 关注数和粉丝数归计数服务，关系服务只提供校准来源。
4. Canal CDC 链路属于关系服务内部同步机制，跨服务仍以业务事件作为边界。
5. 第一阶段不使用图数据库，不做复杂关系链路查询。
6. 关系服务不保存用户昵称、头像、笔记正文、评论内容和 IM 消息。

## 2. 功能范围

### 2.1 第一阶段支持

| 功能 | 说明 |
|---|---|
| 关注用户 | 当前登录用户关注目标用户 |
| 取消关注 | 当前登录用户取消关注目标用户 |
| 查询关注列表 | 查询某个用户关注了哪些用户，最新关注在前 |
| 查询粉丝列表 | 查询某个用户的粉丝，最新关注在前 |
| 查询关注状态 | 查询当前用户是否关注目标用户 |
| 批量查询关注状态 | 查询当前用户对一批用户的关注关系 |
| 批量查询粉丝关系 | 查询一批用户是否关注了目标用户，供 IM、主页、后续场景使用 |
| 内部获取关注列表 | Feed 服务重建或拉模式读取时使用 |
| 内部获取粉丝列表 | Feed 扩散时使用 |
| 关系 CDC 同步 | Canal 监听关注变更，派生粉丝表、Redis 缓存和业务事件 |
| 关系缓存 | Redis 缓存关注列表、最近粉丝列表和热点粉丝关系 |
| 事件发布 | 发布 `UserFollowed`、`UserUnfollowed` |
| 补偿校准 | 粉丝表、Redis、业务事件投递失败时支持重试和校准 |

### 2.2 第一阶段不支持

| 功能 | 暂不支持原因 |
|---|---|
| 图数据库 | 当前规模和部署资源不需要，后续复杂关系查询再引入 |
| 黑名单 | 会影响笔记可见性、评论、IM，需要单独设计权限策略 |
| 好友系统 | 可由互相关注推导，第一阶段先不单独产品化 |
| 共同关注 | 需要较重的集合计算或图存储，后续扩展 |
| 推荐关系 | 推荐系统不是第一阶段目标 |
| 分库分表 | 当前用户量小，先用单库单表并按分片字段设计索引 |
| 本地缓存 | 可作为大 V 关注列表优化，第一阶段先使用 Redis |
| 关注分组 | 第一阶段不做分组管理 |
| 关注申请审核 | 第一阶段关注是即时生效 |
| 隐私关注列表 | 后续配合用户隐私设置设计 |

### 2.3 后续扩展

后续可以扩展：

1. 黑名单和取消拉黑。
2. 互相关注好友关系。
3. 共同关注、共同粉丝、你关注的人也关注了谁。
4. 图数据库作为复杂关系查询和热备存储。
5. 关注分组、特别关注。
6. 隐私设置，例如关注列表不可见、粉丝列表不可见。
7. 大 V 关注列表本地缓存。
8. 关注关系分库分表。
9. 基于关注关系的推荐召回。
10. 关系变更审计和运营管理。

## 3. 服务边界

### 3.1 用户关系服务负责

用户关系服务负责：

1. 关注关系事实。
2. 关注方向查询。
3. 粉丝方向查询。
4. 关注关系状态查询。
5. 关注关系缓存。
6. 关系 CDC 消费和派生读模型同步。
7. 关系业务事件发布。
8. 关系数据校准和重建。

典型数据包括：

| 数据 | 说明 |
|---|---|
| `followerId` | 关注者用户 ID |
| `followeeId` | 被关注者用户 ID |
| `relationStatus` | `FOLLOWING` / `CANCELLED` |
| `followedAt` | 最近一次关注时间 |
| `cancelledAt` | 最近一次取消关注时间 |
| `relationVersion` | 关系版本，用于 CDC 幂等和乱序保护 |
| `changeId` | 关系变更流水 ID |

### 3.2 用户服务负责

用户服务负责用户资料和用户业务状态。关系服务在关注、取消关注前调用用户服务校验：

1. 当前用户是否存在。
2. 目标用户是否存在。
3. 当前用户是否允许关注。
4. 目标用户是否允许被关注。

关系服务不保存昵称、头像、简介等资料。如果列表需要展示用户信息，移动端或 BFF 通过用户服务批量补全。

### 3.3 计数服务负责

计数服务负责：

1. 用户关注数。
2. 用户粉丝数。
3. 关注数和粉丝数缓存。
4. 计数校准和重建。

关系服务发布 `UserFollowed`、`UserUnfollowed`，计数服务消费后更新计数。计数错误时可调用关系服务内部接口或直接按关系事实表做离线校准。

### 3.4 Feed 服务负责

Feed 服务负责关注页 Feed 收件箱和推拉结合策略。

关系服务只提供：

1. 粉丝列表查询。
2. 关注列表查询。
3. `UserFollowed`、`UserUnfollowed` 事件。

Feed 服务可以在用户关注作者后补充作者最近笔记，也可以在用户取消关注后清理该作者的 Feed 项，但这些逻辑不进入关系服务。

### 3.5 通知服务负责

通知服务消费 `UserFollowed` 生成关注通知。关系服务不保存通知内容，也不关心通知是否已读。

### 3.6 IM 服务负责

IM 服务可以调用关系服务判断私信权限，例如后续只允许互相关注用户发起会话。

第一阶段如果 IM 不限制陌生人私信，关系服务只提供接口能力，不主动参与 IM 消息链路。

## 4. 核心概念

### 4.1 关注者和被关注者

当用户 A 关注用户 B 时：

| 角色 | 含义 |
|---|---|
| `followerId` | 用户 A，关注者 |
| `followeeId` | 用户 B，被关注者 |

从用户 A 视角看，用户 B 出现在 A 的关注列表中。

从用户 B 视角看，用户 A 出现在 B 的粉丝列表中。

### 4.2 关注关系状态

关注关系不建议物理删除，而是使用状态字段记录当前状态。

| 状态 | 说明 |
|---|---|
| `FOLLOWING` | 正在关注 |
| `CANCELLED` | 已取消关注 |

这样做的原因：

1. 方便重复关注、重复取关做幂等判断。
2. 方便生成关系变更流水。
3. 方便 CDC 消费端按版本判断事件新旧。
4. 后续可支持关系历史审计。

### 4.3 Following 表

`relation_following` 是关注事实主表，以 `follower_id` 为核心查询字段。

它负责支持：

1. 查询用户是否关注某个目标用户。
2. 查询用户关注列表。
3. 批量查询用户对若干目标用户的关注状态。
4. 作为 CDC 数据中心驱动下游派生数据。

第一阶段关注、取关接口只直接写 `relation_following` 和 `relation_change_log`。`relation_follower`、Redis 缓存、业务事件由 Canal CDC 链路异步派生。

### 4.4 Follower 表

`relation_follower` 是粉丝方向查询表，以 `followee_id` 为核心查询字段。

它负责支持：

1. 查询用户粉丝列表。
2. 批量查询若干用户是否关注了目标用户。
3. Feed 扩散时获取作者粉丝。

`relation_follower` 不作为关注事实入口，不允许移动端或业务写接口直接修改。

### 4.5 关系变更流水

`relation_change_log` 记录每一次关注和取关动作，是 Canal 监听的推荐对象。

相比直接监听 `relation_following`，监听变更流水有几个好处：

1. 每一次业务动作都有独立 `changeId`，适合做 MQ 消费幂等。
2. 即使 `relation_following` 被同一行反复更新，也不会丢失中间业务动作。
3. 下游可以清楚区分 `FOLLOW` 和 `UNFOLLOW`。
4. 更适合排查、重放和补偿。

### 4.6 CDC 事件和业务事件

关系服务内部有两类事件：

| 类型 | 说明 | 消费者 |
|---|---|---|
| CDC 事件 | Canal 从 binlog 解析出的数据库变更事件 | 关系服务内部 CDC 消费器 |
| 业务事件 | `UserFollowed`、`UserUnfollowed` | 计数、Feed、通知、IM 等业务服务 |

跨服务只暴露业务事件，不暴露原始 CDC。这样可以避免其他服务和关系服务表结构强耦合。

## 5. 核心业务流程

### 5.1 关注用户

流程：

```text
移动端点击关注
  -> 网关校验 Access Token
  -> 关系服务获取当前 userId
  -> 校验 followerId != followeeId
  -> 调用用户服务校验双方状态
  -> 查询 relation_following
  -> 不存在则插入 FOLLOWING
  -> 已 CANCELLED 则更新为 FOLLOWING 并 relation_version + 1
  -> 已 FOLLOWING 则幂等成功
  -> 写 relation_change_log(FOLLOW)
  -> 提交本地事务
  -> 返回关注成功
  -> MySQL binlog 被 Canal 监听
  -> relation-cdc 消费器派生粉丝表、缓存和业务事件
```

本地事务内只保证关注事实和变更流水一致。粉丝表、Redis、计数、Feed、通知都是最终一致。

重复关注处理：

1. 如果当前已经 `FOLLOWING`，直接返回成功，不重复写 `FOLLOW` 变更流水。
2. 如果当前是 `CANCELLED`，更新为 `FOLLOWING`，递增 `relation_version`，写一条新的 `FOLLOW` 流水。

### 5.2 取消关注

流程：

```text
移动端点击取消关注
  -> 网关校验 Access Token
  -> 关系服务获取当前 userId
  -> 校验 followerId != followeeId
  -> 查询 relation_following
  -> 不存在则幂等成功
  -> 已 CANCELLED 则幂等成功
  -> 已 FOLLOWING 则更新为 CANCELLED 并 relation_version + 1
  -> 写 relation_change_log(UNFOLLOW)
  -> 提交本地事务
  -> 返回取消关注成功
  -> Canal CDC 派生粉丝表、缓存和业务事件
```

取消关注不强依赖用户服务状态。即使目标用户已禁用或注销，用户也应该可以取消关注。

### 5.3 Canal CDC 同步

Canal 监听 `relation_change_log` 的 binlog，将变更投递到 RocketMQ 内部 Topic：

```text
Topic: relation-cdc
Tag: RelationChangeLogInserted
```

关系服务内部 CDC 消费器处理：

```text
消费 relation-cdc
  -> 按 changeId 做幂等判断
  -> 解析 followerId、followeeId、action、relationVersion
  -> 查询 relation_following 当前版本，做乱序保护
  -> 同步 relation_follower
  -> 更新或删除 Redis 关系缓存
  -> 写 relation_outbox_event(UserFollowed/UserUnfollowed)
  -> 标记 CDC 消费成功
```

为什么不让 Canal 直接投递 `relation-event`：

1. Canal 事件是数据库行变更，不是稳定业务协议。
2. 业务事件需要脱离表结构，字段应按业务语义组织。
3. 关系服务需要先完成粉丝表和缓存派生，再发布对外事件。
4. 后续表结构调整时，不应该影响计数、Feed、通知消费者。

### 5.4 查询关注列表

查询某个用户关注了哪些人，默认最新关注在前。

流程：

```text
查询 following:list:{userId}
  -> Redis 命中则返回 userId 列表
  -> Redis 未命中则查询 relation_following
  -> 回填 Redis ZSET
  -> 返回列表
  -> 调用用户服务批量补全用户摘要，可由 BFF 或移动端聚合
```

关注列表可以全量缓存，原因是产品上会限制单个用户最多关注人数。第一阶段建议先设置上限，例如 5000，具体数值可在实现前按产品体验调整。

### 5.5 查询粉丝列表

查询某个用户的粉丝，默认最新关注在前。

流程：

```text
如果查询最近粉丝页
  -> 查询 follower:recent:{userId}
  -> Redis 命中则返回
  -> Redis 未命中则查询 relation_follower 最近 N 条
  -> 回填 Redis ZSET
  -> 返回列表

如果查询深分页
  -> 做限流
  -> 查询 relation_follower
  -> 不强制回填完整 Redis
```

粉丝列表不能无限全量缓存。参考资料中建议只缓存最近 10000 个粉丝，BlueNote 第一阶段也可以采用该阈值作为设计值。

### 5.6 查询关注状态

查询当前用户是否关注目标用户。

流程：

```text
查询 following:list:{currentUserId}
  -> 如果关注列表缓存存在，用 ZSCORE 判断目标用户是否存在
  -> 如果缓存不存在，查询 relation_following
  -> 返回 FOLLOWING / NOT_FOLLOWING
```

如果只是单个关系查询，不强制拉取完整关注列表。实现时可以先查 Redis ZSET，未命中后直接查 MySQL 单行，并视情况回填关系短缓存。

### 5.7 批量查询关注状态

用于用户主页、Feed、推荐列表、评论列表等场景批量判断“我是否关注了这些用户”。

流程：

```text
入参 currentUserId + targetUserIds
  -> 限制 targetUserIds 数量
  -> 优先读取 following:list:{currentUserId}
  -> 缓存存在则批量判断
  -> 缓存不存在则查询 relation_following where follower_id = ? and followee_id in (...)
  -> 返回每个 targetUserId 的状态
```

接口必须按入参顺序返回，方便调用方组装列表。

### 5.8 批量查询粉丝关系

用于判断一批用户是否关注了目标用户。例如 IM 后续判断“这些用户是否是我的粉丝”。

处理策略：

1. 如果目标用户粉丝数不高，可以读取最近粉丝列表缓存或 MySQL 批量查 `relation_follower`。
2. 如果目标用户是大 V，最近粉丝列表不一定覆盖全部粉丝，使用 Redis Hash 缓存最近查询过的粉丝关系。
3. Hash 未命中的用户再回源 `relation_follower`，并把结果写回 Hash。

示例：

```text
follower:rel:{followeeId}
  field: followerId
  value: FOLLOWING / NOT_FOLLOWING
```

该 Hash 必须设置 TTL，避免热点用户的关系缓存无限增长。

### 5.9 Feed 服务获取粉丝

Feed 服务在作者发布笔记后需要获取粉丝列表做推送。

第一阶段内部接口支持：

1. 按 `followeeId` 分页查询粉丝。
2. 按游标继续拉取。
3. 返回 `followerId`、`followedAt`。

Feed 服务不应该直接访问关系服务数据库。大 V 推拉结合阈值在 Feed 服务设计中确定，关系服务只提供稳定分页能力。

### 5.10 关系数据校准

CDC 链路出现延迟或失败时，需要校准派生数据。

校准对象：

1. `relation_follower` 是否和 `relation_following` 一致。
2. Redis 关注列表是否和 `relation_following` 一致。
3. Redis 最近粉丝列表是否和 `relation_follower` 一致。
4. `UserFollowed`、`UserUnfollowed` 事件是否存在漏发。

第一阶段可以提供后台 job：

```text
按 relation_change_log 时间范围扫描
  -> 对比 relation_cdc_consume_log
  -> 重放失败或缺失 changeId
  -> 修复 relation_follower
  -> 删除或重建对应 Redis key
  -> 补发缺失业务事件
```

## 6. 异常流程

| 异常 | 处理方式 |
|---|---|
| 用户关注自己 | 返回 `RELATION_FOLLOW_SELF_NOT_ALLOWED` |
| 目标用户不存在 | 返回 `RELATION_TARGET_USER_NOT_FOUND` |
| 当前用户被禁用 | 禁止关注，返回 `USER_DISABLED` |
| 目标用户被禁用或注销 | 禁止关注，返回目标用户不可关注 |
| 重复关注 | 幂等成功，不重复发事件 |
| 重复取消关注 | 幂等成功，不重复发事件 |
| 用户服务不可用 | 关注操作失败；取消关注可按现有关系幂等处理 |
| MySQL 唯一键冲突 | 重新查询当前关系状态，按幂等结果返回 |
| Canal 延迟 | 主写接口不受影响，粉丝表、计数、Feed 最终一致 |
| Canal 停止消费 | 触发告警，恢复后从位点继续消费 |
| RocketMQ 不可用 | Canal 投递失败或 CDC 消费失败，依赖重试和告警 |
| CDC 重复消费 | 通过 `changeId` 幂等 |
| CDC 乱序 | 通过 `relationVersion` 判断，只应用新版本 |
| Redis 不可用 | 查询降级 MySQL，写缓存失败记录降级日志 |
| 粉丝深分页过多 | 限流或返回稍后重试 |
| 业务事件投递失败 | outbox 保留待重试，超过阈值告警 |

补偿重点：

1. `relation_follower` 同步失败。
2. Redis 缓存更新失败。
3. `relation-event` 投递失败。
4. Canal 消费位点异常。
5. CDC 消费重复、乱序、漏消费。

## 7. 存储设计

### 7.1 MySQL 表清单

| 表名 | 说明 |
|---|---|
| `relation_following` | 关注事实主表，按关注者查询 |
| `relation_follower` | 粉丝方向查询表，按被关注者查询 |
| `relation_change_log` | 关系变更流水，Canal 监听对象 |
| `relation_cdc_consume_log` | CDC 消费幂等表 |
| `relation_event_consume_log` | 消费其他服务事件的幂等表 |
| `relation_outbox_event` | 关系服务对外业务事件 outbox |
| `relation_rebuild_task` | 关系派生数据重建任务表 |

### 7.2 relation_following

关注事实主表。

| 字段 | 类型 | 说明 |
|---|---|---|
| `id` | BIGINT | 主键 |
| `follower_id` | BIGINT | 关注者用户 ID |
| `followee_id` | BIGINT | 被关注者用户 ID |

### 7.9 索引设计

| 表名 | 索引 | 说明 |
|---|---|---|
| `relation_following` | `PRIMARY KEY(id)` | 主键 |
| `relation_following` | `uk_following_pair(follower_id, followee_id)` | 保证一对用户只有一条关系 |
| `relation_following` | `idx_following_list(follower_id, relation_status, followed_at, followee_id)` | 查询关注列表，覆盖 `followee_id` |
| `relation_following` | `idx_following_updated(updated_at)` | 补偿扫描 |
| `relation_follower` | `PRIMARY KEY(id)` | 主键 |
| `relation_follower` | `uk_follower_pair(followee_id, follower_id)` | 粉丝方向唯一 |
| `relation_follower` | `idx_follower_list(followee_id, relation_status, followed_at, follower_id)` | 查询粉丝列表，覆盖 `follower_id` |
| `relation_follower` | `idx_follower_source_change(source_change_id)` | 按变更流水排查 |
| `relation_change_log` | `PRIMARY KEY(change_id)` | Canal 和补偿基准 |
| `relation_change_log` | `idx_change_pair(follower_id, followee_id, change_id)` | 查询一对用户的变更历史 |
| `relation_change_log` | `idx_change_occurred(occurred_at, change_id)` | 按时间范围重放 |
| `relation_cdc_consume_log` | `PRIMARY KEY(change_id)` | CDC 幂等 |
| `relation_cdc_consume_log` | `idx_cdc_status_retry(consume_status, updated_at)` | 扫描失败消费 |
| `relation_outbox_event` | `PRIMARY KEY(event_id)` | 事件唯一 |
| `relation_outbox_event` | `uk_outbox_change(source_change_id, event_type)` | 防止同一变更重复生成业务事件 |
| `relation_outbox_event` | `idx_outbox_status_retry(send_status, next_retry_at)` | 扫描待投递事件 |
| `relation_rebuild_task` | `idx_rebuild_status(task_status, updated_at)` | 扫描待执行任务 |

索引原则：

1. `relation_following` 后续如分表，应按 `follower_id` 分片。
2. `relation_follower` 后续如分表，应按 `followee_id` 分片。
3. 列表查询索引把排序字段 `followed_at` 和返回字段放入联合索引，减少额外排序和回表。
4. 深分页应优先使用游标，不使用大 offset。

## 8. Redis Key 设计

### 8.1 关注列表缓存

```text
relation:following:{userId}
```

| 项 | 说明 |
|---|---|
| 数据结构 | ZSET |
| member | `followeeId` |
| score | `followedAt` 毫秒时间戳 |
| TTL | 24 小时，可加随机抖动 |
| 写入时机 | 查询回源后回填；CDC 消费后增量更新或删除 |
| 失效策略 | 关注或取消关注后由 CDC 消费器更新 |
| 重建方式 | 从 `relation_following` 按 `follower_id` 查询重建 |

关注列表有产品上限，可以全量缓存。

### 8.2 最近粉丝列表缓存

```text
relation:follower:recent:{userId}
```

| 项 | 说明 |
|---|---|
| 数据结构 | ZSET |
| member | `followerId` |
| score | `followedAt` 毫秒时间戳 |
| 保留长度 | 最近 10000 个粉丝 |
| TTL | 12 小时，可加随机抖动 |
| 写入时机 | 查询最近粉丝回源后回填；CDC 消费后增量更新 |
| 失效策略 | CDC 消费器维护 ZADD / ZREM，并执行 ZREMRANGEBYRANK 控制长度 |
| 重建方式 | 从 `relation_follower` 按 `followee_id` 查询最近 10000 个重建 |

粉丝列表不做无限全量缓存，避免大 V 粉丝占用过多 Redis 内存。

### 8.3 粉丝关系 Hash 缓存

```text
relation:follower:rel:{followeeId}
```

| 项 | 说明 |
|---|---|
| 数据结构 | HASH |
| field | `followerId` |
| value | `FOLLOWING` / `NOT_FOLLOWING` |
| TTL | 1 到 6 小时，按访问热度设置 |
| 写入时机 | 批量粉丝关系查询回源后写入 |
| 失效策略 | CDC 消费器按关注或取关更新对应 field |
| 重建方式 | 不做全量重建，缺失时回源 MySQL |

该缓存主要解决大 V 场景下“最近 10000 粉丝列表无法覆盖全部关系判断”的问题。

### 8.4 关系操作限流

```text
rate:relation:follow:{userId}
rate:relation:unfollow:{userId}
rate:relation:follower:deep-page:{userId}
```

用途：

1. 限制短时间大量关注、取关。
2. 限制粉丝列表深分页请求。
3. 降低刷粉、刷接口和数据库击穿风险。

### 8.5 CDC 延迟标记

```text
relation:cdc:lag
```

可选，用于记录 Canal 或 CDC 消费器最新处理到的 `changeId`、延迟毫秒数和最后消费时间。该数据只用于监控，不作为业务事实。

## 9. Canal 与 RocketMQ 设计

### 9.1 Canal 监听范围

Canal 第一阶段只监听关系服务库中的关系变更流水表：

```text
database: bluenote_relation
table: relation_change_log
event: INSERT
```

不建议第一阶段监听所有业务表。范围越小，排查和稳定性越容易控制。

### 9.2 MySQL binlog 要求

MySQL 需要开启：

```text
log-bin=mysql-bin
binlog-format=ROW
server-id=唯一值
```

Canal 连接 MySQL 的账号需要具备复制相关权限。

### 9.3 Canal 到 RocketMQ

内部 CDC Topic：

```text
Topic: relation-cdc
Tag: RelationChangeLogInserted
```

消息 Key 建议：

```text
changeId
```

分区顺序建议：

```text
hash(followerId + ":" + followeeId)
```

这样同一对用户的关注、取关变更尽量进入同一队列，降低乱序概率。即使出现乱序，消费端仍需使用 `relationVersion` 防护。

### 9.4 CDC 消费器职责

CDC 消费器属于关系服务内部模块，职责：

1. 消费 `relation-cdc`。
2. 按 `changeId` 幂等。
3. 同步 `relation_follower`。
4. 更新 Redis 关注列表、粉丝列表、粉丝关系 Hash。
5. 生成 `relation_outbox_event`。
6. 记录消费日志和指标。

CDC 消费器不直接调用计数、Feed、通知服务。

### 9.5 CDC 消费失败处理

失败处理：

1. 短暂异常交给 RocketMQ 重试。
2. 多次失败后进入死信或记录 `relation_cdc_consume_log.FAIL`。
3. 补偿任务扫描失败记录并重试。
4. Canal 位点异常时，按 `relation_change_log.change_id` 范围做业务重放。

注意：CDC 消费失败不能影响用户关注接口返回，因为关注事实已经写入 MySQL。

## 11. 安全与风控设计

### 11.1 身份和权限

要求：

1. 关注和取消关注必须登录。
2. 当前用户 ID 只能来自网关注入的用户上下文。
3. 移动端不能传 `followerId` 作为操作人。
4. 查询公开关注列表和粉丝列表第一阶段允许登录用户访问，后续受隐私设置控制。
5. 内部接口只能服务间访问，不能暴露给公网。

### 11.2 用户状态校验

关注前调用用户服务校验：

1. 当前用户存在且状态允许关注。
2. 目标用户存在且状态允许被关注。

取消关注原则上不依赖目标用户状态，避免目标用户被封禁后无法解除关系。

### 11.3 防刷策略

建议限制：

| 操作 | 限制 |
|---|---|
| 关注 | 单用户每分钟、每天关注次数限制 |
| 取消关注 | 单用户每分钟取消次数限制 |
| 关注人数上限 | 单用户最多关注人数限制 |
| 批量关系查询 | 单次最多 100 个目标用户 |
| 粉丝深分页 | 对大 offset 或深游标请求限流 |

第一阶段可以使用 Redis + 网关限流实现。风控规则命中时记录审计日志。

### 11.4 数据安全

关系服务不保存手机号、邮箱、密码等敏感信息。

日志中不记录：

1. Access Token。
2. 内部服务签名。
3. 大批量 userId 全量列表，必要时记录数量和摘要。

### 11.5 CDC 链路安全

Canal 账号要求：

1. 只授予必要复制权限。
2. 不使用 MySQL root 账号。
3. Canal 配置中的账号密码通过环境变量或配置中心管理。
4. RocketMQ Topic 权限按环境隔离。

## 12. 前后端实现要点

### 12.1 移动端实现

移动端页面：

1. 用户主页关注按钮。
2. 关注列表页。
3. 粉丝列表页。
4. 我的页面关注数、粉丝数展示。
5. Feed 或评论列表中的关注状态标记。

实现要点：

1. 点击关注后可乐观更新按钮状态，但失败必须回滚。
2. 关注成功后，关注数和粉丝数可能短暂延迟，由计数服务最终更新。
3. 关注列表和粉丝列表使用游标分页。
4. 批量列表展示时，优先使用批量关注状态接口，避免逐条请求。
5. 如果后端返回 CDC 延迟或计数未更新，前端不需要特殊提示。
6. 粉丝深分页被限流时，展示稍后重试。

### 12.2 后端实现

后端模块建议：

```text
bluenote-relation
  ├── api
  ├── application
  ├── domain
  ├── infrastructure
  ├── cdc
  ├── mq
  └── job
```

实现要点：

1. Controller 只做参数校验和用户上下文获取。
2. Application Service 编排用户状态校验、关系写入和流水写入。
3. Domain 层封装关注、取关、状态流转和幂等规则。
4. Infrastructure 层封装 MySQL、Redis、用户服务客户端、RocketMQ。
5. CDC 模块只处理 `relation-cdc`，不暴露给外部业务。
6. MQ 模块负责发布 `relation-event`。
7. Job 模块负责 CDC 失败重试、Redis 重建和粉丝表校准。

### 12.3 Canal 部署实现

第一阶段 Canal 可以作为独立容器部署在云服务器，与 MySQL、RocketMQ 处于同一内网。

部署建议：

1. local/test/prod 都使用 Canal，但 local 可简化为单实例。
2. prod 如果资源紧张，Canal JVM 内存需要严格限制。
3. Canal 只监听 `bluenote_relation.relation_change_log`。
4. Canal 投递到当前环境对应 RocketMQ Topic，例如 `bluenote-prod-relation-cdc`。
5. 记录 Canal 位点和消费延迟指标。

### 12.4 用户服务集成

关注前调用：

```text
POST /internal/users/status-check
```

场景：

```text
FOLLOW_USER
```

关系服务不缓存用户完整资料。可以短时间缓存用户状态校验结果，但禁用、注销等状态变更必须能通过事件或 TTL 及时失效。

### 12.5 计数服务集成

关系服务不调用计数服务更新数字，而是发布事件：

1. `UserFollowed`
2. `UserUnfollowed`

计数服务消费后维护：

1. `FOLLOWING_COUNT:{followerId}`
2. `FOLLOWER_COUNT:{followeeId}`

如果计数服务异常，不影响关注关系事实。

## 13. 数据一致性与事件

### 13.1 本地事务边界

关注和取关本地事务包括：

```text
更新 relation_following
写 relation_change_log
提交事务
```

不包括：

1. 写 `relation_follower`。
2. 写 Redis。
3. 写计数服务。
4. 写 Feed 服务。
5. 写通知服务。

这样可以保证主链路短小，降低关注接口延迟和失败面。

### 13.2 CDC 派生一致性

CDC 派生流程：

```text
relation_change_log INSERT
  -> MySQL ROW binlog
  -> Canal
  -> RocketMQ relation-cdc
  -> 关系服务 CDC 消费器
  -> relation_follower
  -> Redis 缓存
  -> relation_outbox_event
```

一致性级别：

1. `relation_following` 强一致。
2. `relation_follower` 最终一致。
3. Redis 最终一致，可丢失可重建。
4. 计数、Feed、通知最终一致。

### 13.3 业务事件发布

业务事件 Topic：

```text
Topic: relation-event
```

Tag：

```text
UserFollowed
UserUnfollowed
```

事件由 outbox 投递任务发送。发送成功后更新 `relation_outbox_event.send_status = SENT`。

### 13.4 UserFollowed 事件示例

```json
{
  "eventId": "evt_relation_followed_10001_20001_23",
  "eventType": "UserFollowed",
  "eventVersion": 1,
  "occurredAt": "2026-06-03T20:30:00+08:00",
  "producer": "bluenote-relation",
  "traceId": "trace-id",
  "payload": {
    "changeId": "900001",
    "followerId": "10001",
    "followeeId": "20001",
    "relationVersion": 23,
    "followedAt": "2026-06-03T20:30:00+08:00"
  }
}
```

消费者建议：

| 消费方 | 处理 |
|---|---|
| 计数服务 | `followerId` 关注数 +1，`followeeId` 粉丝数 +1 |
| Feed 服务 | 可补充作者最近笔记到关注者 Feed |
| 通知服务 | 给被关注者生成关注通知 |
| IM 服务 | 后续更新会话权限或好友状态 |

### 13.5 UserUnfollowed 事件示例

```json
{
  "eventId": "evt_relation_unfollowed_10001_20001_24",
  "eventType": "UserUnfollowed",
  "eventVersion": 1,
  "occurredAt": "2026-06-03T20:35:00+08:00",
  "producer": "bluenote-relation",
  "traceId": "trace-id",
  "payload": {
    "changeId": "900002",
    "followerId": "10001",
    "followeeId": "20001",
    "relationVersion": 24,
    "cancelledAt": "2026-06-03T20:35:00+08:00"
  }
}
```

消费者建议：

| 消费方 | 处理 |
|---|---|
| 计数服务 | `followerId` 关注数 -1，`followeeId` 粉丝数 -1 |
| Feed 服务 | 可清理该作者在关注者 Feed 中的笔记 |
| 通知服务 | 一般不生成取消关注通知 |
| IM 服务 | 后续重新判断私信权限 |

### 13.6 消费事件

第一阶段关系服务可消费：

| 事件 | 来源 | 用途 |
|---|---|---|
| `UserStatusChanged` | 用户服务 | 清理用户状态缓存，后续限制异常用户关系操作 |

消费要求：

1. 按 `eventId` 幂等。
2. 只更新本服务缓存或派生状态。
3. 不直接删除关注关系，除非后续有明确的注销清理流程。

### 13.7 幂等和乱序

幂等策略：

1. 关注写入依赖 `uk_following_pair(follower_id, followee_id)`。
2. CDC 消费依赖 `relation_cdc_consume_log.change_id`。
3. 对外事件依赖 `relation_outbox_event.event_id` 和 `source_change_id + event_type`。
4. 下游消费者按 `eventId` 幂等。

乱序策略：

1. 同一对用户尽量按 RocketMQ 分区顺序消费。
2. 消费端仍以 `relationVersion` 为准。
3. 当 CDC 事件版本小于当前 `relation_follower.relation_version` 时，跳过该事件。

## 14. 日志、指标与告警

### 14.1 日志

必须记录：

1. 关注成功、失败、幂等成功。
2. 取消关注成功、失败、幂等成功。
3. 用户状态校验失败。
4. MySQL 唯一键冲突。
5. Canal CDC 消费成功和失败。
6. `relation_follower` 派生失败。
7. Redis 更新失败和降级。
8. outbox 投递失败。
9. 补偿任务执行结果。

日志字段：

| 字段 | 说明 |
|---|---|
| `traceId` | 链路 ID |
| `followerId` | 关注者 |
| `followeeId` | 被关注者 |
| `changeId` | 关系变更流水 ID |
| `relationVersion` | 关系版本 |
| `operation` | `FOLLOW` / `UNFOLLOW` / `CDC_SYNC` |
| `result` | 成功或失败 |

### 14.3 告警

建议告警：

1. 关注接口失败率异常升高。
2. 用户服务状态校验超时率升高。
3. Canal 连接 MySQL 失败。
4. Canal 到 RocketMQ 投递失败。
5. CDC 延迟超过阈值。
6. `relation_cdc_consume_log.FAIL` 持续增加。
7. `relation_outbox_event` 堆积超过阈值。
8. Redis 缓存更新失败持续增加。
9. 粉丝表和关注表校准差异超过阈值。
10. 粉丝深分页限流次数异常升高。

## 15. 测试重点

### 15.1 单元测试

1. 不允许关注自己。
2. 关注状态流转：不存在 -> `FOLLOWING`。
3. 取消关注状态流转：`FOLLOWING` -> `CANCELLED`。
4. 重复关注幂等。
5. 重复取消关注幂等。
6. `relationVersion` 递增。
7. 游标分页参数解析。
8. 批量关系查询结果按入参顺序返回。

### 15.2 集成测试

1. 关注后写 `relation_following` 和 `relation_change_log`。
2. 取消关注后写变更流水。
3. Canal 模拟 CDC 后同步 `relation_follower`。
4. CDC 重复消费不重复生成业务事件。
5. CDC 乱序消费不覆盖新版本。
6. Redis 关注列表 CDC 增量更新。
7. Redis 最近粉丝列表只保留限定长度。
8. outbox 投递失败后可重试。
9. 用户服务不可用时关注失败。
10. Redis 不可用时查询降级 MySQL。

### 15.3 接口测试

1. 未登录不能关注和取关。
2. 不能关注自己。
3. 关注不存在用户返回错误。
4. 查询关注列表按最新关注倒序返回。
5. 查询粉丝列表按最新关注倒序返回。
6. 批量查询数量超限返回错误。
7. 粉丝深分页触发限流。
8. 内部接口不能从公网访问。

### 15.4 CDC 链路测试

重点验证：

1. MySQL binlog 为 ROW 模式时 Canal 能正确解析 `relation_change_log`。
2. Canal 能投递 `relation-cdc` 到 RocketMQ。
3. 同一对用户连续关注、取关、再关注时最终状态正确。
4. Canal 停止一段时间后恢复，可以继续消费。
5. RocketMQ 重复投递时 CDC 消费幂等。
6. CDC 消费失败进入补偿任务后可修复。

### 15.5 性能测试

重点压测：

1. 关注和取消关注接口。
2. 批量关注状态查询。
3. 关注列表 Redis 命中和未命中。
4. 粉丝列表最近页查询。
5. Feed 内部拉取粉丝分页接口。
6. CDC 消费吞吐和延迟。

第一阶段目标建议：

1. 关注/取关接口 P95 小于 150ms，不包含用户服务严重抖动。
2. 批量关系查询单次最多 100 个用户，Redis 命中时 P95 小于 100ms。
3. CDC 正常情况下延迟小于 3 秒。
4. Redis 不可用时核心写操作不受影响。

## 16. 风险与后续演进

| 风险 | 说明 | 应对 |
|---|---|---|
| Canal 增加部署复杂度 | 第一阶段多一个中间件 | 只在关系服务使用，监听范围收敛到 `relation_change_log` |
| CDC 延迟 | 粉丝表、计数、Feed 可能短暂落后 | 前端接受最终一致，监控 `relation_cdc_lag_ms` |
| CDC 重复或乱序 | RocketMQ 重试和分区可能带来重复、乱序 | `changeId` 幂等，`relationVersion` 防乱序 |
| 粉丝表派生失败 | 粉丝列表查询可能不准确 | 失败重试 + 校准任务 + 必要时回源 following |
| Redis 缓存不一致 | CDC 更新失败或 Redis 丢失 | TTL + 删除重建 + MySQL 重建 |
| 大 V 粉丝列表过大 | Redis 无法全量缓存 | 只缓存最近 10000 个粉丝，深分页限流 |
| 关注刷量 | 恶意关注取关影响计数和通知 | 限流、风控、后续黑名单 |
| 事件边界混乱 | 下游直接消费 CDC 会强耦合表结构 | CDC 只在关系服务内部消费，对外发布业务事件 |
| 2 核 4G 资源紧张 | Canal、RocketMQ、MySQL 都占资源 | 控制 JVM 内存，按需合并业务服务，观察资源再扩容 |

后续演进：

1. 补充 Canal 选型 ADR，并同步总体技术方案、基础设施方案和消息队列方案。
2. 对 `relation_following` 按 `follower_id` 分库分表，对 `relation_follower` 按 `followee_id` 分库分表。
3. 引入大 V 关注列表本地缓存。
4. 引入黑名单和 IM 私信权限策略。
5. 引入互相关注好友关系。
6. 引入共同关注等复杂关系接口。
7. 评估图数据库作为复杂关系查询和热备存储。
8. 将关注关系变更纳入数据分析和推荐系统。
