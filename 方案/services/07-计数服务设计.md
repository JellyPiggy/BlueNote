# BlueNote 计数服务设计

## 1. 背景与目标

计数服务负责 BlueNote 中高频展示、高频变更的聚合数字，例如笔记点赞数、收藏数、评论数，用户关注数、粉丝数、作品数、获赞数，以及评论点赞数、回复数。

这类数据有几个特点：

1. 读取频率高，笔记详情、用户主页、评论区、Feed 卡片都会展示。
2. 更新频率高，点赞、收藏、评论、关注等动作都会触发计数变化。
3. 不是强资产类数据，可以接受短暂最终一致。
4. 可以从业务事实数据重建，例如点赞明细、收藏明细、关注关系、评论表。

因此计数服务不能依赖业务明细表实时 `COUNT(*)`，也不能把点赞、关注、评论等行为事实抢到计数服务里。正确边界是：

```text
业务服务维护行为事实
  -> 发布业务事件
  -> 计数服务维护展示计数
  -> 计数异常时从业务事实校准和重建
```

本方案参考 `参考资料/08.通用计数系统.md` 的核心思路：使用 Redis Hash 保存同一对象的多维计数，通过 RocketMQ 异步写入，并由 Counter Updater 做批量消费和写聚合。

结合 BlueNote 第一阶段资源约束，本方案采用以下取舍：

1. 采用 Redis Hash 作为在线计数读写存储。
2. 采用 RocketMQ 实现计数异步写。
3. 第一阶段正式实现计数写聚合，不只做预留。
4. MySQL 保存计数快照、计数变更流水、消费幂等和重建任务。
5. 暂不定制 Redis 对象，不引入 RocksDB、Pika 等冷热分离存储。
6. 计数服务内部使用 `counter-delta-event` Topic，上游业务服务不直接生产计数 delta。

设计目标：

1. 支持笔记、用户、评论三类对象的常用计数。
2. 支持批量计数查询，满足详情页、列表页、Feed、排行榜等聚合展示。
3. 消费笔记、评论、关系等业务事件，转换为标准计数变更指令。
4. 通过内部 `counter-delta-event` 实现计数异步写和写聚合。
5. Redis Hash 使用 Lua 脚本批量增减，保证单个对象计数更新原子性。
6. 支持事件幂等、delta 幂等、重复消费保护和负数保护。
7. 支持 MySQL 快照刷盘、Redis 预热、计数校准和重建。
8. 支持发布 `CounterChanged`、`CounterRebuilt` 事件，供排行榜等服务使用。
9. 支持降级、补偿、日志、指标和告警。

关键约束：

1. 点赞、收藏行为事实归笔记服务。
2. 评论和评论点赞行为事实归评论服务。
3. 关注关系事实归用户关系服务。
4. 计数服务不判断用户是否点赞、是否收藏、是否关注。
5. Redis 是在线计数读写层，但不是不可替代的事实来源；计数必须可以从业务事实重建。
6. MySQL 计数快照用于启动恢复、降级查询和运维排查，不替代业务事实。
7. 钱包余额、订单金额、库存这类强一致数字不能进入通用计数服务。

## 2. 功能范围

### 2.1 第一阶段支持

| 功能 | 说明 |
|---|---|
| 批量查询计数 | 按对象批量查询指定计数字段 |
| 笔记计数 | 点赞数、收藏数、评论数 |
| 用户计数 | 关注数、粉丝数、公开作品数、获赞数 |
| 评论计数 | 评论点赞数、一级评论回复数 |
| 业务事件适配 | 消费业务事件并转换为标准计数 delta |
| 计数内部 Topic | 使用 `counter-delta-event` 承载标准计数变更 |
| 异步写计数 | Counter Updater 消费 delta 后异步写 Redis |
| 写聚合 | 按 `targetType + targetId + counterField` 聚合同批 delta |
| Redis Hash 存储 | 同一对象的多个计数字段保存到一个 Hash |
| Lua 原子更新 | 批量去重、增减、负数保护和脏标记 |
| 计数快照 | 定期把 Redis 计数刷入 MySQL 快照表 |
| 计数校准 | 调用来源服务低频校准计数 |
| 计数重建 | Redis 丢失或计数异常时从来源服务重建 |
| 事件发布 | 发布 `CounterChanged`、`CounterRebuilt` |
| 幂等处理 | 外部事件幂等、delta 幂等、Redis 应用幂等 |
| 降级策略 | Redis 或 MQ 异常时有明确降级和恢复方案 |

### 2.2 第一阶段不支持

| 功能 | 暂不支持原因 |
|---|---|
| 钱包余额、订单金额 | 属于强一致资产数据，必须由交易服务用事务保证 |
| 库存扣减 | 库存有超卖风险，需要单独库存模型和事务策略 |
| 业务服务直接更新计数 | 会让计数规则散落在上游服务，破坏边界 |
| 业务服务直接生产 `counter-delta-event` | 内部 Topic 只归计数服务使用 |
| 定制化 Redis `CountInt` | 开发和维护成本高，当前数据量不需要 |
| RocksDB / Pika 冷热分离 | 当前预计数据量小，Redis Hash + MySQL 快照足够 |
| 大数据实时统计 | 计数服务只做在线展示计数，不替代数仓分析 |
| 精确到每一秒的全站统计 | 这类报表后续由离线或流式分析系统承担 |
| 任意字段动态计数 | 只支持注册过的对象类型和计数字段 |

### 2.3 后续扩展

后续可以扩展：

1. 接入独立互动服务后，把笔记点赞、收藏行为事实从笔记服务拆出。
2. 对热点 Key 增加本地聚合缓冲，进一步减少 Redis 写入。
3. 对冷门历史计数做冷热分离，迁移到磁盘 KV 或归档表。
4. 对超大规模计数评估定制 Redis 对象或 Redis Module。
5. 按对象类型拆分计数 Redis 实例，例如 note-counter、user-counter。
6. 根据排行榜需要增加更多衍生计数，例如近 1 小时点赞数、近 24 小时评论数。
7. 接入流式计算或数仓，用于运营报表和反作弊分析。
8. 增加管理端计数修正、计数冻结、异常回滚等运维能力。

## 3. 服务边界

### 3.1 计数服务负责

计数服务负责：

1. 在线展示计数。
2. 标准计数类型和字段定义。
3. 批量计数查询接口。
4. 外部业务事件消费。
5. 业务事件到计数 delta 的转换。
6. 内部 `counter-delta-event` 投递。
7. Counter Updater 批量消费和写聚合。
8. Redis Hash 计数更新。
9. MySQL 计数快照。
10. 计数校准和重建。
11. `CounterChanged`、`CounterRebuilt` 事件发布。

典型数据包括：

| 数据 | 说明 |
|---|---|
| `targetType` | 计数对象类型，例如 `NOTE`、`USER`、`COMMENT` |
| `targetId` | 计数对象 ID |
| `counterField` | 计数字段，例如 `like_count` |
| `counterValue` | 当前计数值 |
| `deltaId` | 标准计数变更 ID |
| `deltaValue` | 本次计数增量，可以为正数或负数 |
| `sourceEventId` | 来源业务事件 ID |
| `snapshotVersion` | 计数快照版本 |

### 3.2 非计数服务职责

计数服务不负责：

1. 判断当前用户是否点赞笔记。
2. 判断当前用户是否收藏笔记。
3. 判断当前用户是否关注另一个用户。
4. 保存评论内容。
5. 保存关注关系。
6. 保存点赞、收藏、评论点赞明细。
7. 判断笔记、评论、用户是否可见。
8. 处理钱包余额、支付金额、订单状态、库存等强一致数据。
9. 负责 Feed 收件箱和排行榜排序结果。

### 3.3 来源服务职责

| 来源服务 | 行为事实 | 计数服务消费 |
|---|---|---|
| 笔记服务 | 笔记发布、删除、点赞、收藏事实 | 笔记计数、用户作品数、用户获赞数 |
| 评论服务 | 评论、回复、评论点赞事实 | 笔记评论数、评论回复数、评论点赞数 |
| 用户关系服务 | 关注关系事实 | 用户关注数、用户粉丝数 |
| 用户服务 | 用户状态和资料 | 不直接提供计数事实，后续可影响展示策略 |

### 3.4 下游服务职责

| 下游服务 | 使用方式 |
|---|---|
| 笔记服务 | 查询笔记详情和列表时批量获取笔记计数 |
| 用户服务 | 用户主页聚合时获取关注数、粉丝数、作品数、获赞数 |
| 评论服务 | 评论列表查询时获取评论点赞数、回复数 |
| Feed 服务 | 组装 Feed 卡片时批量获取笔记计数 |
| 排行榜服务 | 消费 `CounterChanged` 或查询计数后计算热度分 |
| 通知服务 | 一般不依赖计数，后续可展示聚合通知数量 |

## 4. 核心概念

### 4.1 计数对象

计数对象由 `targetType + targetId` 唯一定位。

第一阶段支持：

| targetType | targetId 示例 | 说明 |
|---|---|---|
| `NOTE` | `noteId` | 笔记维度计数 |
| `USER` | `userId` | 用户维度计数 |
| `COMMENT` | `commentId` | 评论维度计数 |

### 4.2 计数字段

不同对象类型支持不同字段。

#### NOTE

| 字段 | 说明 | 来源 |
|---|---|---|
| `like_count` | 笔记点赞数 | `NoteLiked` / `NoteUnliked` |
| `collect_count` | 笔记收藏数 | `NoteCollected` / `NoteUncollected` |
| `comment_count` | 笔记可见评论数 | `CommentCreated` / `CommentDeleted` / `CommentStatusChanged` |

#### USER

| 字段 | 说明 | 来源 |
|---|---|---|
| `following_count` | 用户关注数 | `UserFollowed` / `UserUnfollowed` |
| `follower_count` | 用户粉丝数 | `UserFollowed` / `UserUnfollowed` |
| `note_count` | 用户公开作品数 | `NotePublished` / `NoteDeleted` / `NoteVisibilityChanged` |
| `liked_count` | 用户作品累计获赞数 | `NoteLiked` / `NoteUnliked` |

#### COMMENT

| 字段 | 说明 | 来源 |
|---|---|---|
| `like_count` | 评论点赞数 | `CommentLiked` / `CommentUnliked` |
| `reply_count` | 一级评论可见回复数 | 二级 `CommentCreated` / `CommentDeleted` / `CommentStatusChanged` |

### 4.3 计数 Delta

计数 delta 是计数服务内部标准变更指令。

示例：

```json
{
  "deltaId": "evt_note_liked_90001_10002:NOTE:90001:like_count",
  "sourceEventId": "evt_note_liked_90001_10002",
  "sourceEventType": "NoteLiked",
  "targetType": "NOTE",
  "targetId": "90001",
  "counterField": "like_count",
  "deltaValue": 1,
  "occurredAt": "2026-06-03T20:30:00+08:00"
}
```

`deltaId` 生成规则：

```text
sourceEventId + ":" + targetType + ":" + targetId + ":" + counterField
```

同一个业务事件可能生成多条 delta。例如 `UserFollowed` 会生成：

```text
USER:{followerId}:following_count +1
USER:{followeeId}:follower_count +1
```

### 4.4 Event Adapter

Event Adapter 是计数服务内部模块，负责消费外部业务事件，并转换成标准计数 delta。

职责：

1. 消费 `note-event`、`interaction-event`、`comment-event`、`relation-event`。
2. 校验事件版本和必要字段。
3. 按事件类型生成一条或多条 `counter_delta_log`。
4. 写 `counter_delta_outbox_event`。
5. 投递内部 `counter-delta-event`。

上游业务服务不需要知道计数 Redis Key、计数字段、聚合窗口，也不直接写计数 delta。

### 4.5 Counter Updater

Counter Updater 是计数服务内部异步写入模块。

职责：

1. 消费 `counter-delta-event`。
2. 批量拉取 delta。
3. 按 `targetType + targetId + counterField` 聚合同批增量。
4. 使用 Lua 脚本批量写 Redis Hash。
5. 做 delta 应用幂等。
6. 标记 `counter_delta_log` 状态。
7. 把变更过的对象加入 dirty 集合，等待快照刷盘。
8. 必要时发布 `CounterChanged`。

### 4.6 写聚合

写聚合指 Counter Updater 在一个短窗口内把多条同对象、同字段的 delta 合并成一次 Redis 更新。

示例：

```text
NOTE:90001:like_count +1
NOTE:90001:like_count +1
NOTE:90001:like_count +1
NOTE:90001:comment_count +1
```

聚合后：

```text
HINCRBY counter:note:90001 like_count 3
HINCRBY counter:note:90001 comment_count 1
```

建议第一阶段参数：

| 参数 | 建议值 | 说明 |
|---|---|---|
| 单次拉取数量 | 100 到 500 | 根据 2 核 4G 资源压测调整 |
| 聚合窗口 | 50ms 到 200ms | 热点明显时收益更高 |
| 单目标单次脚本 delta 数 | 不超过 500 | 避免 Lua 参数过大 |
| Counter Updater 实例数 | 1 到 2 | 第一阶段控制复杂度 |

### 4.7 计数快照

计数快照是 Redis 中当前计数的 MySQL 副本。

用途：

1. Redis 重启后预热。
2. 查询 Redis 失败时低频降级。
3. 运维排查计数变化。
4. 计数重建前后对比。

计数快照不是行为事实。最终校准仍以来源服务的业务明细为准。

### 4.8 计数校准

计数校准指计数服务按目标对象调用来源服务的内部 `counter-source` 接口，重新计算某些计数字段，并修正 Redis 和 MySQL 快照。

校准适用于：

1. MQ 重复、漏消费排查。
2. Redis 数据丢失。
3. 计数出现负数或明显异常。
4. 定时低频抽样检查。
5. 运维手动触发修复。

### 4.9 计数展示一致性

计数采用最终一致。

用户点赞后：

```text
点赞按钮状态以笔记服务返回为准
点赞数以计数服务异步更新为准
```

因此短时间内可能出现：

```text
用户已点赞，但点赞数还没 +1
```

这是可接受的。移动端可以做乐观展示，但刷新后以后端计数为准。

## 5. 核心流程

### 5.1 批量查询计数

调用方传入多个计数对象和需要的字段。

流程：

```text
业务服务请求批量计数
  -> 计数服务校验 targetType 和 counterField 白名单
  -> 按对象类型分组
  -> 批量 HMGET Redis Hash
  -> Redis 命中则组装返回
  -> Redis 未命中则查 counter_snapshot
  -> 快照存在则回填 Redis 并返回
  -> 快照不存在则按 0 初始化返回
```

降级策略：

1. Redis 短暂不可用时，低频查 MySQL 快照。
2. MySQL 快照也不可用时，返回 0 或调用方传入的降级值，并标记 `degraded = true`。
3. 查询接口不直接回源业务明细表，避免在线查询被拖慢。

### 5.2 笔记点赞计数流程

笔记服务处理点赞事实：

```text
用户点赞笔记
  -> 笔记服务写 note_like
  -> 笔记服务发布 NoteLiked
```

计数服务处理：

```text
Event Adapter 消费 NoteLiked
  -> 按 sourceEventId 幂等
  -> 生成 NOTE:{noteId}:like_count +1
  -> 生成 USER:{authorId}:liked_count +1
  -> 写 counter_delta_log
  -> 投递 counter-delta-event
  -> Counter Updater 聚合并写 Redis
```

取消点赞同理：

```text
NoteUnliked
  -> NOTE:{noteId}:like_count -1
  -> USER:{authorId}:liked_count -1
```

注意：

1. 是否重复点赞由笔记服务判断。
2. 计数服务只处理已经发生的 `NoteLiked` 或 `NoteUnliked`。
3. 如果笔记服务幂等成功但没有真实状态变化，不应该发布事件。

### 5.3 笔记收藏计数流程

收藏事件映射：

| 事件 | delta |
|---|---|
| `NoteCollected` | `NOTE:{noteId}:collect_count +1` |
| `NoteUncollected` | `NOTE:{noteId}:collect_count -1` |

收藏数不影响用户获赞数。

### 5.4 评论计数流程

一级评论创建：

```text
CommentCreated(level=1, commentStatus=VISIBLE)
  -> NOTE:{noteId}:comment_count +1
```

二级评论创建：

```text
CommentCreated(level=2, commentStatus=VISIBLE)
  -> NOTE:{noteId}:comment_count +1
  -> COMMENT:{rootId}:reply_count +1
```

评论删除：

```text
CommentDeleted
  -> NOTE:{noteId}:comment_count -affectedCommentCount
  -> 如果删除的是二级评论，COMMENT:{rootId}:reply_count -1
  -> 如果删除一级评论并隐藏整组回复，按 affectedReplyCount 修正 root 回复数
```

这里要求评论服务的 `CommentDeleted` 事件携带以下字段之一：

1. `affectedCommentCount`：本次删除对笔记可见评论数的影响。
2. `affectedReplyCount`：本次删除对一级评论回复数的影响。

如果第一阶段评论服务暂时只传单条删除事件，计数服务默认只减 1；一级评论整组隐藏导致的差异由校准任务修正。更推荐在评论服务事件中补齐影响数量，避免计数长期偏差。

### 5.5 评论点赞计数流程

评论点赞事件映射：

| 事件 | delta |
|---|---|
| `CommentLiked` | `COMMENT:{commentId}:like_count +1` |
| `CommentUnliked` | `COMMENT:{commentId}:like_count -1` |

评论点赞数只归评论对象，不增加用户获赞数。后续如果产品需要“评论获赞数”，可以新增 `USER:comment_liked_count` 字段。

### 5.6 用户关注和粉丝计数流程

关注事件映射：

```text
UserFollowed
  -> USER:{followerId}:following_count +1
  -> USER:{followeeId}:follower_count +1
```

取消关注事件映射：

```text
UserUnfollowed
  -> USER:{followerId}:following_count -1
  -> USER:{followeeId}:follower_count -1
```

注意：

1. 关注事实以用户关系服务为准。
2. Canal CDC 链路只在关系服务内部使用。
3. 计数服务只消费关系服务对外发布的业务事件 `relation-event`。

### 5.7 用户作品数流程

第一阶段用户作品数统计公开已发布笔记。

事件映射：

| 事件 | 条件 | delta |
|---|---|---|
| `NotePublished` | `visibility = PUBLIC` 且 `noteStatus = PUBLISHED` | `USER:{authorId}:note_count +1` |
| `NoteDeleted` | 删除前为公开已发布笔记 | `USER:{authorId}:note_count -1` |
| `NoteVisibilityChanged` | `PRIVATE -> PUBLIC` | `USER:{authorId}:note_count +1` |
| `NoteVisibilityChanged` | `PUBLIC -> PRIVATE` | `USER:{authorId}:note_count -1` |
| `NoteStatusChanged` | 下架公开笔记 | `USER:{authorId}:note_count -1` |
| `NoteStatusChanged` | 恢复公开笔记 | `USER:{authorId}:note_count +1` |

如果第一阶段只支持公开和私密，不做复杂审核，则主要处理 `NotePublished`、`NoteDeleted` 和 `NoteVisibilityChanged`。

### 5.8 Counter Updater 写 Redis

Counter Updater 消费 `counter-delta-event` 后执行：

```text
批量拉取 delta
  -> 按 targetType + targetId 分组
  -> 组内按 counterField 聚合 deltaValue
  -> 调用 Lua 脚本
      -> 对每个 deltaId 做 Redis 去重
      -> 未应用过的 delta 才参与 HINCRBY
      -> 计数减到 0 以下时钳制为 0
      -> 把 targetKey 写入 dirty ZSET
  -> 更新 counter_delta_log 状态为 APPLIED
  -> 按策略写 counter_outbox_event(CounterChanged)
```

Lua 脚本职责：

1. `SADD` 或 `HSETNX` 记录已应用 `deltaId`。
2. 对未应用 delta 执行 `HINCRBY`。
3. 如果结果小于 0，修正为 0 并记录异常标记。
4. 更新 dirty 集合，用于后续快照刷盘。

Redis 去重 Key 建议按天或按月分桶：

```text
bluenote:{env}:counter:dedupe:{yyyyMMdd}
```

TTL 建议 30 天。MySQL `counter_delta_log` 长期保留更完整的审计和重放信息。

### 5.9 快照刷盘流程

快照刷盘任务处理 Redis dirty 集合：

```text
定时任务扫描 counter:dirty
  -> 批量读取 Redis Hash
  -> upsert counter_snapshot
  -> 更新 flushedAt 和 snapshotVersion
  -> 从 dirty 集合移除已刷盘对象
```

建议第一阶段：

1. 每 30 秒到 1 分钟执行一次。
2. 每批处理 100 到 1000 个对象。
3. 刷盘失败不影响在线计数读写。
4. dirty 对象保留时间过长需要告警。

### 5.10 计数校准流程

计数校准按对象和字段触发。

流程：

```text
创建 counter_rebuild_task
  -> 根据 targetType 和 counterField 找到来源服务
  -> 调用来源服务 counter-source 内部接口
  -> 获取事实计数
  -> 覆盖 Redis Hash 对应字段
  -> upsert counter_snapshot
  -> 写 CounterRebuilt outbox
  -> 标记任务成功
```

来源服务：

| 计数 | 来源接口 |
|---|---|
| 笔记点赞数、收藏数 | `POST /internal/notes/counter-source` |
| 笔记评论数、评论回复数、评论点赞数 | `POST /internal/comments/counter-source` |
| 用户关注数、粉丝数 | `POST /internal/relations/counter-source` |
| 用户作品数 | `POST /internal/notes/counter-source` |
| 用户获赞数 | 笔记服务按作者聚合点赞明细，或由计数 delta 重放后校准 |

校准不用于在线列表展示，只用于低频修复。

### 5.11 Redis 重建流程

Redis 数据丢失或新环境初始化时：

```text
优先从 counter_snapshot 加载
  -> 批量写入 Redis Hash
  -> 标记预热完成
  -> 对关键对象启动来源服务校准
```

如果快照不存在或明显过旧：

```text
按目标范围创建 rebuild task
  -> 分批调用来源服务 counter-source
  -> 重写 Redis 和 MySQL 快照
```

重建期间查询接口可以：

1. 返回 MySQL 快照值。
2. 对没有快照的对象返回 0。
3. 标记 `degraded = true`。

## 6. 异常流程

| 异常 | 处理方式 |
|---|---|
| 重复业务事件 | `counter_event_consume_log` 或 `counter_delta_log.delta_id` 幂等 |
| 重复 delta 投递 | Redis dedupe + MySQL delta 状态双重保护 |
| MQ 短暂不可用 | outbox 保留待投递，告警但不丢 delta |
| Counter Updater 消费失败 | RocketMQ 重试，多次失败进入死信或失败状态 |
| Redis 不可用 | Updater 暂停或失败重试，查询降级 MySQL 快照 |
| Redis 写成功但 MySQL 状态更新失败 | Redis dedupe 防止重试重复增加，补偿任务修复 delta 状态 |
| MySQL 快照刷盘失败 | dirty 集合保留，后续重试 |
| 计数减成负数 | Lua 钳制为 0，记录 `counter_negative_guard_total` 指标 |
| 业务事件字段缺失 | 记录失败并进入异常队列，不生成 delta |
| 事件乱序 | 大多数计数增减可交换；可见性和状态类事件需要事件版本或来源校准 |
| `CounterChanged` 投递失败 | outbox 重试，不影响在线计数 |
| 来源服务校准接口失败 | rebuild task 标记失败并重试，超过阈值告警 |
| 批量查询目标过多 | 返回 `COUNTER_BATCH_SIZE_EXCEEDED` |
| 未注册计数字段 | 返回 `COUNTER_FIELD_NOT_SUPPORTED` |

补偿重点：

1. `counter_delta_outbox_event` 投递失败。
2. `counter_delta_log` 长时间未应用。
3. Redis dirty 对象长时间未刷盘。
4. Redis 计数和来源服务事实计数差异过大。
5. `CounterChanged` 事件漏发。

```text

```

### 7.10 Redis 持久化建议

第一阶段 Redis 建议开启：

```text
appendonly yes
appendfsync everysec
aof-use-rdb-preamble yes
```

理由：

1. 计数在线存储依赖 Redis。
2. AOF everysec 最多丢失约 1 秒写入，结合 MQ 和 delta log 可以修复。
3. 混合持久化恢复速度更快。

Redis 内存策略建议避免随意淘汰计数 Key。资源紧张时应优先扩容或做冷数据策略，而不是让关键计数被不可控淘汰。

## 9. 安全与风控设计

### 9.1 接口权限

要求：

1. 计数服务不暴露移动端公网接口。
2. 内部接口必须经过服务间鉴权。
3. 管理类接口只能允许运维或管理端服务调用。
4. 网关不能直接转发公网请求到 `/internal/counters/**`。

### 9.2 字段白名单

计数对象和字段必须注册。

禁止调用方传入任意字段，例如：

```text
balance
amount
stock
```

这样可以避免把强一致数字误接入通用计数服务。

### 9.3 批量限制

建议限制：

| 接口 | 限制 |
|---|---|
| `/internal/counters/batch` | 单次最多 100 个 target |
| `/internal/counters/reconcile` | 同一 target 短时间只能触发一次 |
| `/internal/counters/warmup` | 单次最多 500 个 target |

超过限制直接返回错误，避免计数服务被聚合接口拖垮。

### 9.4 数据安全

计数服务不保存手机号、邮箱、密码、正文、图片 URL 等敏感内容。

日志中不记录：

1. Access Token。
2. 内部服务签名。
3. 大批量完整 target 列表。

必要时只记录数量、摘要和前几个对象 ID。

### 9.5 防刷关系

点赞、收藏、评论、关注的防刷主要在来源服务处理。

计数服务只做补充保护：

1. 对异常高频 delta 来源打指标。
2. 对单对象计数突增触发告警。
3. 对负数保护触发告警。
4. 对来源事件字段异常进入失败队列。

计数服务不能代替业务服务判断行为是否合法。

## 10. 前后端实现要点

### 10.1 移动端实现

移动端不直接对接计数服务，但需要理解计数最终一致：

1. 点赞、收藏、关注按钮状态以对应业务接口返回为准。
2. 数字可以乐观更新，但刷新后以后端返回为准。
3. 如果详情或列表接口返回 `degraded = true`，可以展示已有值或 0，不阻断主内容展示。
4. 不要为了刷新一个数字单独轮询计数接口。

### 10.2 后端模块

后端模块建议：

```text
bluenote-counter
  ├── api
  ├── application
  ├── domain
  ├── adapter
  │   ├── note
  │   ├── interaction
  │   ├── comment
  │   └── relation
  ├── updater
  ├── infrastructure
  │   ├── mysql
  │   ├── redis
  │   ├── mq
  │   └── client
  ├── mq
  └── job
```

职责划分：

| 模块 | 职责 |
|---|---|
| `api` | 内部查询、校准、预热接口 |
| `application` | 编排查询、校准、快照、重建 |
| `domain` | 计数对象、字段、delta、映射规则 |
| `adapter` | 外部业务事件到 delta 的转换 |
| `updater` | delta 聚合和 Redis 写入 |
| `infrastructure` | MySQL、Redis、RocketMQ、来源服务客户端 |
| `job` | outbox、快照刷盘、重建、异常补偿 |

### 10.3 事件适配实现

事件适配规则建议集中管理，不散落在 Consumer 代码里。

示例：

```text
NoteLiked
  -> NOTE.like_count +1 by noteId
  -> USER.liked_count +1 by authorId

UserFollowed
  -> USER.following_count +1 by followerId
  -> USER.follower_count +1 by followeeId
```

实现要求：

1. 每个事件类型必须有明确版本。
2. 事件缺字段时不生成 delta。
3. 同一业务事件生成多条 delta 时，要么全部写入成功，要么整体失败重试。
4. 生成 delta 后通过 outbox 投递内部 Topic。

### 10.4 Counter Updater 实现

实现要点：

1. 消费 `counter-delta-event` 时使用 `targetType:targetId` 作为消息 Key。
2. 同一对象尽量进入同一队列，提高聚合效果。
3. 单批次先按对象分组，再按字段聚合。
4. Lua 脚本必须同时处理去重、增减、负数保护和 dirty 标记。
5. 单次 Lua 参数不能无限增长，需要设置批量上限。
6. Redis 写成功后再更新 `counter_delta_log` 状态。
7. 状态更新失败时依赖 Redis dedupe 防止重复加减。

### 10.5 快照和重建实现

快照刷盘：

1. 只处理 dirty 对象。
2. 批量读取 Redis Hash。
3. upsert MySQL `counter_snapshot`。
4. 失败保留 dirty 标记。

重建任务：

1. 根据字段路由到来源服务。
2. 来源服务接口必须限制批量大小。
3. 重建结果覆盖 Redis 和 MySQL 快照。
4. 重建完成发布 `CounterRebuilt`。

### 10.6 调用方集成

笔记服务查询详情时：

```text
查笔记主数据
  -> 批量查计数服务
  -> 查当前用户点赞收藏状态
  -> 组装响应
```

评论服务查询评论列表时：

```text
查评论列表
  -> 批量查 COMMENT.like_count 和 COMMENT.reply_count
  -> 查当前用户点赞状态
  -> 组装响应
```

用户服务查询主页时：

```text
查用户资料
  -> 批量查 USER.following_count、follower_count、note_count、liked_count
  -> 组装主页头部
```

## 11. 数据一致性与事件

### 11.1 总体一致性

计数链路：

```text
业务服务本地事务
  -> outbox 发布业务事件
  -> 计数服务 Event Adapter 消费
  -> 写 counter_delta_log
  -> 投递 counter-delta-event
  -> Counter Updater 聚合写 Redis
  -> 快照任务写 MySQL
```

一致性级别：

1. 业务事实强一致，归来源服务本地事务保证。
2. 计数 delta 最终一致。
3. Redis 在线计数最终一致。
4. MySQL 快照最终一致。
5. 排行榜等下游基于 `CounterChanged` 最终一致。

### 11.2 订阅外部事件

| Topic | 事件 | 用途 |
|---|---|---|
| `note-event` | `NotePublished` | 用户公开作品数 +1 |
| `note-event` | `NoteDeleted` | 用户公开作品数 -1 |
| `note-event` | `NoteVisibilityChanged` | 用户公开作品数增减 |
| `note-event` | `NoteStatusChanged` | 下架、恢复影响公开作品数 |
| `interaction-event` | `NoteLiked` | 笔记点赞数 +1，作者获赞数 +1 |
| `interaction-event` | `NoteUnliked` | 笔记点赞数 -1，作者获赞数 -1 |
| `interaction-event` | `NoteCollected` | 笔记收藏数 +1 |
| `interaction-event` | `NoteUncollected` | 笔记收藏数 -1 |
| `comment-event` | `CommentCreated` | 笔记评论数、一级评论回复数增加 |
| `comment-event` | `CommentDeleted` | 笔记评论数、一级评论回复数减少 |
| `comment-event` | `CommentLiked` | 评论点赞数 +1 |
| `comment-event` | `CommentUnliked` | 评论点赞数 -1 |
| `comment-event` | `CommentStatusChanged` | 可见性变化导致评论计数修正 |
| `relation-event` | `UserFollowed` | 关注数、粉丝数 +1 |
| `relation-event` | `UserUnfollowed` | 关注数、粉丝数 -1 |

### 11.3 内部事件

内部 Topic：

```text
Topic: counter-delta-event
Tag: CounterDeltaCreated
Producer: bluenote-counter Event Adapter
Consumer: bluenote-counter Counter Updater
```

该 Topic 只允许计数服务内部使用。

### 11.4 CounterDeltaCreated 示例

```json
{
  "eventId": "evt_counter_delta_evt_note_liked_90001_10002_NOTE_90001_like_count",
  "eventType": "CounterDeltaCreated",
  "eventVersion": 1,
  "occurredAt": "2026-06-03T20:30:00+08:00",
  "producer": "bluenote-counter",
  "traceId": "trace-id",
  "payload": {
    "deltaId": "evt_note_liked_90001_10002:NOTE:90001:like_count",
    "sourceEventId": "evt_note_liked_90001_10002",
    "sourceEventType": "NoteLiked",
    "targetType": "NOTE",
    "targetId": "90001",
    "counterField": "like_count",
    "deltaValue": 1
  }
}
```

### 11.5 发布对外事件

Topic：

```text
counter-event
```

事件：

| 事件 | 触发时机 | 消费方 |
|---|---|---|
| `CounterChanged` | 重要计数变化或聚合窗口结束 | 排行榜、展示读模型后续 |
| `CounterRebuilt` | 计数重建完成 | 排行榜、运维监控、后续读模型 |

第一阶段不建议每个 delta 都发布 `CounterChanged`，否则事件量会很大。建议按以下策略发布：

1. 对排行榜关注的字段发布，例如 `NOTE.like_count`、`NOTE.comment_count`、`NOTE.collect_count`。
2. 同一对象短时间内合并发布。
3. 小计数变化可以只更新 Redis，不发布对外事件。

### 11.6 CounterChanged 示例

```json
{
  "eventId": "evt_counter_changed_NOTE_90001_1748953800000",
  "eventType": "CounterChanged",
  "eventVersion": 1,
  "occurredAt": "2026-06-03T20:30:00+08:00",
  "producer": "bluenote-counter",
  "traceId": "trace-id",
  "payload": {
    "targetType": "NOTE",
    "targetId": "90001",
    "changedFields": {
      "like_count": {
        "delta": 3,
        "currentValue": 128
      },
      "comment_count": {
        "delta": 1,
        "currentValue": 20
      }
    }
  }
}
```

### 11.7 幂等策略

| 层级 | 幂等方式 |
|---|---|
| 来源服务 | 业务明细唯一约束，避免重复发布真实事件 |
| Event Adapter | `consumerGroup + eventId` 幂等 |
| Delta 生成 | `deltaId` 唯一 |
| Delta 投递 | `counter_delta_outbox_event.event_id` 唯一 |
| Updater 消费 | `deltaId` + Redis dedupe |
| Redis 应用 | Lua 脚本只应用未处理 delta |
| 对外事件 | `counter_outbox_event.event_id` 唯一 |

### 11.8 乱序处理

普通增减计数具备交换性：

```text
+1 后 -1
-1 后 +1
```

最终结果一致，因此点赞、收藏、关注、评论点赞这类计数主要依赖幂等，不强依赖顺序。

但状态类事件需要注意：

1. `NoteVisibilityChanged`
2. `NoteStatusChanged`
3. `CommentStatusChanged`

这类事件必须带来源对象版本或状态前后值。计数服务按 `fromStatus -> toStatus`、`fromVisibility -> toVisibility` 判断 delta。如果事件缺少前后状态，则只触发校准任务，不直接增减计数。

## 14. 风险与后续演进

| 风险 | 说明 | 应对 |
|---|---|---|
| 计数短暂延迟 | 异步写链路导致展示数字稍慢 | 移动端乐观展示，监控 delta 延迟 |
| 重复消费导致多加 | MQ 至少一次投递 | deltaId、Redis dedupe、MySQL 幂等 |
| Redis 数据丢失 | Redis 是在线计数层 | AOF everysec、MySQL 快照、来源服务重建 |
| delta log 持续增长 | 所有计数变化都会写流水 | 分区归档、保留近期热数据、冷数据压缩 |
| 内部 Topic 增加复杂度 | 多一段 MQ 链路 | 统一封装，监控 pending 和 lag |
| CounterChanged 事件风暴 | 每次计数变化都发事件会放大流量 | 合并发布，只对关键字段发布 |
| 评论删除影响数量不清 | 删除一级评论可能影响多个回复 | 评论事件携带 affected count，缺失时校准 |
| 状态类事件乱序 | 可见性、下架恢复不具备简单交换性 | 事件携带前后状态和版本，异常触发校准 |
| 负数计数 | 取消事件重复或来源异常 | Lua 钳制为 0，告警并校准 |
| 2 核 4G 资源有限 | Redis、MQ、MySQL 都在小资源环境 | 控制批量大小、Updater 实例数和快照频率 |

后续演进：

1. 点赞、收藏规模增大后，拆分独立互动服务，计数服务继续消费互动事件。
2. 对排行榜需要的近实时窗口计数增加独立字段，例如 `like_count_1h`。
3. 对超级热点 Key 增加本地内存聚合，再批量写 Redis。
4. 对历史冷门计数做冷热分离，降低 Redis 内存成本。
5. 评估 Redis Cluster 或主从读写分离。
6. 计数 delta log 做归档和重放工具。
7. 接入管理端，支持人工校准、计数冻结和异常对比。
8. 在数据量极大时评估定制 Redis 对象或 Redis Module。
