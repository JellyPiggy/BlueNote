# services 目录协作规则

本目录用于编写 BlueNote 各业务服务的详细设计文档。

这些规则用于保证后续服务设计文档结构一致、粒度适中、边界清晰。这里不记录某个具体服务的业务结论；具体业务流程、表结构、接口、状态和事件仍然写在对应服务设计文档中。

## 文档定位

每个服务设计文档应该回答：

1. 这个服务为什么存在。
2. 第一阶段做什么、不做什么。
3. 它和其他服务的边界在哪里。
4. 核心业务对象是什么。
5. 主要流程和异常流程如何处理。
6. 数据怎么存、怎么查、怎么保证一致性。
7. 对外和对内接口如何设计。
8. 安全、风控、日志、指标、测试和后续演进如何考虑。

不要只写接口和表结构。服务设计文档必须能帮助后续开发、联调、排查和演进。

## 推荐目录

后续服务设计文档优先采用以下结构：

```markdown
# BlueNote Xxx 服务设计

## 1. 背景与目标

## 2. 功能范围

### 2.1 第一阶段支持
### 2.2 第一阶段不支持
### 2.3 后续扩展

## 3. 服务边界

### 3.1 本服务负责
### 3.2 非本服务职责
### 3.3 依赖关系

## 4. 核心概念

## 5. 核心流程

## 6. 异常流程

## 7. 存储设计

### 7.1 MySQL 表设计
### 7.2 索引设计
### 7.3 Redis Key 设计

## 8. 接口设计

### 8.2 内部接口

```

## 裁剪规则

目录可以按服务复杂度裁剪，但不能随意省略关键内容。

可以裁剪或合并：

1. 不使用 Redis 的服务，可以不写 Redis Key，或在存储设计中说明“不使用 Redis”。
2. 不发布、不订阅 MQ 的服务，可以简写“数据一致性与事件”。
3. 没有移动端直接交互的内部服务，可以弱化“移动端实现要点”。
4. 没有复杂状态流转的服务，可以不单独写状态机，只在表字段和流程中说明状态枚举。
5. 没有明显风控场景的服务，可以把风控合并到安全设计中。

不建议省略：

1. 背景与目标。
2. 功能范围。
3. 服务边界。
4. 核心概念。
5. 核心流程。
6. 异常流程。
7. 存储设计。
8. 索引设计。
9. 接口设计。
10. 安全设计。
11. 数据一致性说明。
12. 日志、指标与告警。
13. 测试重点。
14. 风险与后续演进。

## 写作规则

### 先范围，后细节

每个服务必须先写清楚：

1. 第一阶段支持什么。
2. 第一阶段不支持什么。
3. 后续可能扩展什么。

不要一开始就写表结构或接口，否则容易把暂缓能力误设计成当前能力。

### 先边界，后依赖

每个服务必须明确：

1. 本服务负责什么。
2. 本服务不负责什么。
3. 哪些能力属于其他服务。
4. 依赖哪些服务和基础设施。

跨服务职责不能含糊。一个业务对象只能有一个主写服务。

### 核心概念要先定义

涉及业务对象时，先定义概念再写流程。

例如：

1. 账号、凭证、会话、Token。
2. 笔记、草稿、媒体、状态。
3. 关注关系、粉丝、黑名单。
4. 评论、一级评论、二级评论。
5. 会话、消息、ACK、未读数。

概念不清楚时，不要直接进入接口和数据库设计。

### 流程必须包含异常

核心流程不能只写 happy path。

每个关键流程至少补充：

1. 输入校验失败。
2. 业务对象不存在。
3. 状态不允许。
4. 重复请求。
5. 下游服务失败。
6. MQ 投递失败。
7. Redis 或缓存不可用。

### 存储设计要包含索引

MySQL 表设计至少包含：

1. 表用途。
2. 关键字段。
3. 状态枚举。
4. 唯一约束。
5. 常用查询索引。

不能只写字段，不写索引。高频查询、唯一业务约束和分页查询必须说明索引依据。

### Redis 不能当事实来源

如果服务使用 Redis，必须说明：

1. Key 格式。
2. 数据结构。
3. TTL。
4. 写入时机。
5. 失效策略。
6. Redis 丢失后的重建方式。

关键业务状态以 MySQL 或对应事实库为准。

### 接口要区分外部和内部

接口设计分为：

1. 移动端接口：通过网关访问。
2. 内部接口：服务间调用，不暴露给移动端。

内部接口需要说明调用方、用途和是否支持批量查询。

### 事件要写清楚生产和消费

如果服务使用 RocketMQ，必须说明：

1. 生产哪些事件。
2. 订阅哪些事件。
3. 事件触发时机。
4. 消费幂等方式。
5. 失败重试和补偿方式。

事件表示已经发生的事实，命名优先使用过去式。

### 安全与风控按服务场景写

每个服务都要写自己的安全点。

示例：

1. 登录服务关注密码、Token、登录失败限制。
2. 文件服务关注上传大小、类型、访问权限。
3. 笔记服务关注作者权限、内容状态、媒体归属。
4. 评论服务关注删除权限、频率限制、内容审核。
5. IM 服务关注连接鉴权、会话权限、消息频率。
6. 订单服务关注支付回调验签、金额校验、幂等。

不要只写一句“按全局安全方案执行”。

### 可观测性要服务化

每个服务至少写：

1. 关键日志。
2. 核心指标。
3. 需要告警的异常。

日志必须包含 traceId。涉及 MQ 的服务还要记录 eventId、topic、consumerGroup。

### 测试重点要覆盖风险

测试重点至少覆盖：

1. 正常流程。
2. 异常流程。
3. 权限和安全。
4. 幂等。
5. 数据一致性。
6. 事件消费，若使用 MQ。
7. 缓存失效或重建，若使用 Redis。

不要只写接口成功用例。

## 不应写入本文件的内容

以下内容不应抽到 `services/AGENTS.md`：

1. 某个服务的具体业务决策。
2. 某个服务的表结构。
3. 某个服务的接口路径和请求响应。
4. 某个服务的状态枚举。
5. 某个服务的 Redis Key。
6. 某个服务的 RocketMQ 事件清单。
7. 某个服务的具体阈值，例如限流次数、锁定时间。
8. 某个服务的具体风控规则。
9. 某个服务的具体前端页面设计。

判断标准：

```text
能指导所有服务如何写方案的，放在 services/AGENTS.md。
只描述某个服务自身业务事实的，放在具体服务设计文档。
```

## 文档粒度

服务设计文档采用“小公司可维护版”粒度：

1. 该严谨的地方必须严谨，例如边界、状态、索引、安全、一致性。
2. 不为了形式写过重内容，例如大规模容量模型、复杂集群方案、完整运维手册。
3. 当前阶段不实现的能力要写清楚“不支持”和“后续扩展”，不要混进第一阶段方案。
4. 如果某个服务天然复杂，可以适度增加章节，例如状态机、容量估算、补偿任务。

目标是让文档能支撑真实开发，而不是堆模板。
