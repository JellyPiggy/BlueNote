# BlueNote 登录服务设计

版本：v0.1  
状态：草案  
更新时间：2026-06-03

## 1. 背景与目标

登录服务负责 BlueNote 用户身份认证、密码凭证、登录会话、Token 签发和登录安全控制，是所有需要用户身份的业务服务的基础。

第一阶段不接入短信、邮箱、微信、Apple 等外部服务，避免引入额外成本和审核流程。当前只支持用户名 + 密码登录。手机号、邮箱和第三方登录只做架构预留，后续在具备验证码、邮箱验证或 OAuth 能力后再启用。

设计目标：

1. 支持用户名 + 密码注册和登录。
2. 密码不可明文存储。
3. 支持 Access Token + Refresh Token。
4. 支持设备会话。
5. 支持当前设备登出。
6. 支持修改密码。
7. 支持登录失败限制和基础风控。
8. 为手机号、邮箱、第三方登录保留扩展结构。
9. 登录态校验主要在网关完成，避免下游服务频繁访问登录服务。

关键约束：

1. 第一阶段不使用手机号作为登录账号。
2. 未验证手机号不能用于登录、找回密码或身份认证。
3. 用户名只作为登录标识，不等同于用户昵称。
4. 用户资料由用户服务维护，不由登录服务维护。

## 2. 功能范围

### 2.1 第一阶段支持

| 功能 | 说明 |
|---|---|
| 用户名密码注册 | 用户使用唯一用户名和密码注册账号 |
| 用户名密码登录 | 用户使用用户名和密码登录 |
| Access Token 签发 | 登录成功后签发短期访问令牌 |
| Refresh Token 签发 | 登录成功后签发长期刷新令牌 |
| Token 刷新 | 使用 Refresh Token 获取新的 Access Token 和 Refresh Token |
| 当前设备登出 | 当前设备会话失效 |
| 修改密码 | 已登录用户输入旧密码后修改密码 |
| 设备会话记录 | 记录登录设备、IP、App 版本等信息 |
| 登录审计日志 | 记录登录、登出、刷新、失败等行为 |
| 基础风控 | 登录失败次数、IP 限流、设备限流 |

### 2.2 第一阶段不支持

| 功能 | 原因 |
|---|---|
| 手机号注册 | 没有短信验证码，无法确认手机号归属 |
| 手机号密码登录 | 手机号必须先验证绑定后才能作为登录身份 |
| 手机验证码登录 | 短信服务有成本和审核流程 |
| 邮箱验证码登录 | 邮件服务和送达率后续再评估 |
| 找回密码 | 需要已验证手机号或邮箱 |
| 第三方登录 | 需要接入 OAuth 平台，第一阶段暂缓 |
| 扫码登录 | 当前只做移动端，不设计 PC 扫码登录 |
| 多因素认证 | 后续安全增强时再设计 |

### 2.3 后续扩展

后续可扩展：

1. 绑定手机号。
2. 绑定邮箱。
3. 手机号验证码登录。
4. 邮箱验证码登录。
5. 手机号密码登录，前提是手机号已验证。
6. 微信登录。
7. Apple 登录。
8. 找回密码。
9. 登出全部设备。
10. 管理端封禁和解封账号。

扩展原则：

1. 只有 `verified = true` 的手机号或邮箱才能用于登录和找回密码。
2. 第三方登录必须保存第三方身份和 BlueNote 用户 ID 的绑定关系。
3. 扩展登录方式不应推翻当前账号、密码、会话、Token 模型。

## 3. 服务边界

### 3.1 登录服务负责

登录服务负责：

1. 用户名注册。
2. 用户名唯一性校验。
3. 密码哈希存储。
4. 密码校验。
5. Access Token 生成。
6. Refresh Token 生成、存储和轮换。
7. 设备会话管理。
8. 当前设备登出。
9. 修改密码。
10. 登录审计。
11. 登录失败限制。
12. 发布用户注册、登录、登出等事件。

### 3.2 用户服务负责

用户服务负责：

1. 用户昵称。
2. 用户头像。
3. 用户简介。
4. 用户主页信息。
5. 用户展示状态。
6. 用户资料编辑。

登录服务不保存昵称、头像、简介等资料字段。

### 3.3 依赖关系

| 依赖 | 用途 |
|---|---|
| MySQL | 账号、密码、会话、审计日志 |
| Redis | 登录失败计数、限流、Token 黑名单、会话缓存 |
| RocketMQ | 发布注册、登录、登出、密码修改事件 |
| 用户服务 | 注册后创建默认用户资料，或通过事件异步创建 |
| 网关 | 统一校验 Access Token 并注入用户上下文 |

### 3.4 与网关的关系

移动端请求登录服务的公开接口：

1. 注册。
2. 登录。
3. 刷新 Token。

其他需要登录的接口由网关校验 Access Token 后转发。

网关校验通过后向下游注入：

```text
X-User-Id
X-Device-Id
X-Session-Id
X-Trace-Id
```

下游服务不直接信任客户端传入的用户身份 Header。反向代理和网关必须清理客户端伪造的内部 Header。

## 4. 核心概念

### 4.1 账号

账号表示 BlueNote 内部用户认证主体。

第一阶段账号由用户名创建：

```text
username + password
```

账号不包含用户昵称、头像、简介。用户名用于登录，昵称用于展示，二者由不同服务维护。

### 4.2 密码凭证

密码凭证用于保存用户密码的安全哈希。

要求：

1. 不保存明文密码。
2. 不保存可逆加密后的密码。
3. 使用 BCrypt 或 Argon2id。
4. 保存密码算法和版本，便于后续升级。
5. 登录日志不得打印密码。

### 4.3 登录身份

登录身份表示用户可以用什么外部标识登录。

第一阶段只使用 `USERNAME`，并直接存在账号表中。

后续扩展身份：

| identity_type | 说明 | 是否第一阶段启用 |
|---|---|---|
| PHONE | 手机号 | 否 |
| EMAIL | 邮箱 | 否 |
| WECHAT | 微信 openId/unionId | 否 |
| APPLE | Apple subject | 否 |

手机号和邮箱必须验证后才能启用登录：

```text
verified = true
```

### 4.4 设备会话

设备会话表示一次登录设备上的登录状态。

一个用户可以有多个设备会话，例如：

1. Android 手机。
2. iOS 手机。
3. H5 浏览器。

每个设备会话有独立的 Refresh Token。登出当前设备只影响当前 session。

### 4.5 Token

Token 分为：

| Token | 类型 | 用途 |
|---|---|---|
| Access Token | JWT | 访问业务接口 |
| Refresh Token | 随机字符串 | 刷新 Access Token |

Access Token 短有效期，用于高频请求本地校验。Refresh Token 长有效期，服务端存哈希，用于控制会话和刷新登录态。

## 5. 核心流程

### 5.1 用户名密码注册

流程：

```text
移动端
  -> 输入 username/password/deviceInfo
  -> 网关
  -> 登录服务
  -> 校验 username 格式
  -> 校验 password 强度
  -> 检查 username 是否已存在
  -> 生成 userId
  -> 哈希密码
  -> 写 auth_account
  -> 写 auth_password
  -> 写 outbox UserRegistered
  -> 创建设备会话
  -> 签发 Access Token 和 Refresh Token
  -> 返回登录结果
```

注册成功后默认自动登录。

用户名规则建议：

1. 长度 4 到 20。
2. 只能包含小写字母、数字、下划线。
3. 不能以下划线开头或结尾。
4. 不允许连续下划线。
5. 注册时统一转小写。
6. 保留系统关键字，例如 `admin`、`root`、`system`。

密码规则建议：

1. 长度 8 到 64。
2. 不能全是空白字符。
3. 不能和用户名相同。
4. 不允许常见弱密码，例如 `12345678`。
5. 第一阶段不强制复杂字符组合，避免过度影响体验。

### 5.2 用户名密码登录

流程：

```text
移动端
  -> 输入 username/password/deviceInfo
  -> 网关
  -> 登录服务
  -> 归一化 username
  -> 检查登录失败限制
  -> 查询 auth_account
  -> 检查账号状态
  -> 查询 auth_password
  -> 校验密码哈希
  -> 密码正确则清理失败计数
  -> 创建或更新设备会话
  -> 签发 Access Token 和 Refresh Token
  -> 写登录审计日志
  -> 返回登录结果
```

错误提示统一：

```text
用户名或密码错误
```

不要区分用户名不存在和密码错误，避免账号枚举。

### 5.3 刷新 Token

流程：

```text
移动端
  -> 提交 Refresh Token
  -> 登录服务计算 Refresh Token hash
  -> 查询 auth_session
  -> 校验 session 状态
  -> 校验 refresh token 是否过期
  -> 校验账号状态
  -> 生成新的 Access Token
  -> 生成新的 Refresh Token
  -> 更新 session.refresh_token_hash
  -> 更新 session.last_active_at
  -> 返回新 Token
```

要求：

1. Refresh Token 每次刷新都轮换。
2. 旧 Refresh Token 立即失效。
3. 刷新接口需要防并发，多次并发刷新只能有一次成功。
4. Refresh Token 不使用 JWT，使用高熵随机字符串。
5. 数据库存储 Refresh Token 哈希，不存明文。

并发刷新处理：

1. 根据 `refresh_token_hash` 查询 session。
2. 使用数据库行锁或乐观锁更新。
3. 更新成功后返回新 token。
4. 更新失败要求客户端重新登录或重新使用最新 token。

### 5.4 当前设备登出

流程：

```text
移动端
  -> 携带 Access Token 请求登出
  -> 网关校验 Access Token
  -> 登录服务读取 userId/sessionId
  -> 标记 auth_session 为 LOGGED_OUT
  -> 清空 refresh_token_hash 或标记失效
  -> 将当前 Access Token jti 加入 Redis 黑名单
  -> 写登出审计日志
  -> 返回成功
```

说明：

1. 登出后 Refresh Token 不能再使用。
2. Access Token 如果未进入黑名单，理论上会在过期前继续有效。
3. 第一阶段建议对登出 token 写黑名单，TTL 设置为 Access Token 剩余有效期。

### 5.5 修改密码

流程：

```text
移动端
  -> 已登录状态提交 oldPassword/newPassword
  -> 网关校验 Access Token
  -> 登录服务查询当前密码哈希
  -> 校验 oldPassword
  -> 校验 newPassword 强度
  -> 写入新 password_hash
  -> 更新 password_version
  -> 使其他设备 session 失效
  -> 当前设备重新签发 Token
  -> 发布 PasswordChanged 事件
```

策略：

1. 修改密码必须输入旧密码。
2. 修改成功后建议踢出其他设备。
3. 当前设备可以继续登录，但需要重新签发 Token。
4. 后续支持找回密码后，找回密码成功应使所有设备失效。

## 6. 异常流程

| 场景 | 处理 |
|---|---|
| 用户名已存在 | 注册失败，提示用户名已被使用 |
| 用户名格式错误 | 注册失败，提示格式要求 |
| 密码过弱 | 注册或修改密码失败 |
| 用户名不存在 | 登录失败，提示用户名或密码错误 |
| 密码错误 | 登录失败，增加失败计数 |
| 密码错误次数过多 | 临时锁定登录，提示稍后再试 |
| 账号禁用 | 拒绝登录，提示账号状态异常 |
| 账号注销 | 拒绝登录 |
| Refresh Token 过期 | 要求重新登录 |
| Refresh Token 已轮换 | 要求重新登录 |
| Session 已登出 | 要求重新登录 |
| Access Token 过期 | 移动端尝试刷新 Token |
| Access Token 非法 | 清理本地登录态并跳转登录 |
| Redis 不可用 | 登录失败限制降级，核心登录流程仍以 MySQL 为准 |
| RocketMQ 不可用 | outbox 记录待投递，不影响注册登录主流程 |

登录失败计数建议：

1. 同用户名连续失败 5 次，锁定 15 分钟。
2. 同 IP 高频失败，网关限流。
3. 同设备 高频失败，短期限制。
4. 成功登录后清理对应失败计数。

具体阈值放配置中心，后续按运行情况调整。

## 7. 存储设计

### 7.1 MySQL 表设计

登录服务使用 schema：

```text
bluenote_auth
```

#### 7.1.1 auth_account

账号表。

| 字段 | 类型 | 说明 |
|---|---|---|
| user_id | BIGINT | 用户 ID，主键 |
| username | VARCHAR(32) | 登录用户名，唯一 |

| created_at | DATETIME(3) | 创建时间 |
| updated_at | DATETIME(3) | 更新时间 |
| deleted | TINYINT | 逻辑删除 |

账号状态：

| 状态 | 说明 |
|---|---|
| NORMAL | 正常 |
| DISABLED | 禁用，不能登录 |
| BANNED | 封禁，不能登录 |
| DELETED | 已注销 |

#### 7.1.2 auth_password

密码凭证表。

| 字段 | 类型 | 说明 |
|---|---|---|
| id | BIGINT | 主键 |
| user_id | BIGINT | 用户 ID，唯一 |
| password_hash | VARCHAR(255) | 密码哈希 |
| password_algo | VARCHAR(32) | 哈希算法，例如 BCrypt |
| password_version | INT | 密码版本 |
| password_updated_at | DATETIME(3) | 密码更新时间 |
| need_reset | TINYINT | 是否要求重置密码 |
| created_at | DATETIME(3) | 创建时间 |
| updated_at | DATETIME(3) | 更新时间 |

说明：

1. `password_hash` 使用成熟 PasswordEncoder 生成。
2. 不单独存 salt，BCrypt/Argon2id 结果中已包含必要参数。
3. `password_version` 用于让旧 Token 失效或后续做密码策略升级。

#### 7.1.3 auth_identity

登录身份表，第一阶段预留。

| status | VARCHAR(32) | 绑定状态 |
| bound_at | DATETIME(3) | 绑定时间 |
| created_at | DATETIME(3) | 创建时间 |
| updated_at | DATETIME(3) | 更新时间 |

绑定状态：

| 状态 | 说明 |
|---|---|
| BOUND | 已绑定 |
| UNBOUND | 已解绑 |
| REVOKED | 被撤销 |

第一阶段不写入手机号和邮箱。后续只有 `verified = true` 的身份才能用于登录和找回密码。

### 7.2 索引设计

登录路径关键查询：

1. 根据 username 查询 account。
2. 根据 user_id 查询 password。
3. 根据 refresh_token_hash 查询 session。

这些查询必须命中唯一索引或高选择性索引。

### 7.3 Redis Key 设计

Key 前缀：

```text
bluenote:{env}:auth
```

| Key | 类型 | TTL | 用途 |
|---|---|---|---|
| login_fail:username:{username} | String | 15 分钟 | 用户名登录失败次数 |
| login_fail:ip:{ip} | String | 15 分钟 | IP 登录失败次数 |
| login_fail:device:{deviceId} | String | 15 分钟 | 设备登录失败次数 |
| rate:register:ip:{ip} | String | 1 小时 | IP 注册限流 |
| rate:login:ip:{ip} | String | 15 分钟 | IP 登录限流 |
| token:blacklist:{jti} | String | Access Token 剩余有效期 | Access Token 黑名单 |
| session:{sessionId} | Hash | 30 分钟 | 会话缓存，可选 |

说明：

1. Redis 用于风控、限流和短期黑名单。
2. 核心会话事实以 MySQL 的 `auth_session` 为准。
3. Redis 不可用时，登录失败限制可降级，但注册和登录主流程应尽量可用。

## 8. Token 与会话设计

### 8.1 Access Token

Access Token 使用 JWT。

建议：

1. 签名算法使用 RS256。
2. 登录服务持有私钥。
3. 网关持有公钥。
4. 下游服务不直接依赖客户端 Token。

JWT Claims：

```json
{
  "iss": "bluenote-auth",
  "sub": "userId",
  "jti": "tokenId",
  "sessionId": "sessionId",
  "deviceId": "deviceId",
  "passwordVersion": 1,
  "iat": 1780000000,
  "exp": 1780003600
}
```

有效期建议：

```text
30 分钟到 1 小时
```

第一阶段建议使用 1 小时。后续如果需要更及时的封禁和踢下线，可以缩短到 30 分钟。

### 8.2 Refresh Token

Refresh Token 使用随机高熵字符串。

要求：

1. 至少 256 bit 随机。
2. 只在登录和刷新接口返回。
3. 服务端只保存 hash。
4. 每次刷新都轮换。
5. 过期时间建议 30 天。

Refresh Token 用于控制服务端会话。只要 session 被登出、撤销或过期，Refresh Token 就不能继续使用。

### 8.3 会话有效期

建议：

| 类型 | 有效期 |
|---|---|
| Access Token | 1 小时 |
| Refresh Token | 30 天 |
| 设备会话 | 与 Refresh Token 一致 |
| Token 黑名单 | Access Token 剩余有效期 |

### 8.4 强制下线时效

如果用户被封禁或设备被踢出：

1. Refresh Token 立即失效。
2. 新的 Access Token 无法刷新。
3. 已签发 Access Token 在过期前可能仍可用。
4. 如果需要立即失效，将 Access Token jti 加入 Redis 黑名单。

第一阶段对当前设备登出写入黑名单。批量封禁、踢全设备后续再做更完整的 token version 或 session version 方案。

## 10. 安全与风控设计

### 10.1 传输安全

要求：

1. 正式环境必须 HTTPS。
2. 移动端只请求 HTTPS API。
3. WebSocket 使用 WSS。
4. H5 场景禁止混合内容。

第一阶段不强制客户端 RSA 加密密码。原因是 HTTPS、密码哈希、日志脱敏、限流这些基础措施优先级更高。后续如果需要增强登录链路，可以增加登录公钥接口和客户端加密。

### 10.2 密码安全

要求：

1. 使用 BCrypt 或 Argon2id。
2. 不使用 MD5、SHA1 保存密码。
3. 不保存明文密码。
4. 不打印密码。
5. 不在异常信息中返回密码相关内部细节。
6. 修改密码后更新 `password_version`。

### 10.3 Token 安全

要求：

1. Access Token 短有效期。
2. Refresh Token 高熵随机。
3. Refresh Token 只存 hash。
4. Refresh Token 每次刷新轮换。
5. 登出时 Refresh Token 失效。
6. Access Token 使用 `jti` 支持黑名单。

### 10.4 账号枚举防护

登录失败统一提示：

```text
用户名或密码错误
```

不返回：

1. 用户不存在。
2. 密码错误。
3. 用户名存在但未设置密码。

注册接口可以提示用户名已被使用，因为注册场景必须让用户知道用户名不可用。

### 10.5 限流与锁定

第一阶段基础规则：

| 维度 | 策略 |
|---|---|
| username | 连续失败 5 次锁定 15 分钟 |
| IP | 15 分钟内失败超过阈值限流 |
| deviceId | 高频失败限流 |
| 注册 IP | 每小时限制注册次数 |
| 刷新 Token | 同 session 高频刷新限制 |

这些阈值放配置中心，后续根据真实流量调整。

### 10.6 审计

必须记录：

1. 注册成功。
2. 登录成功。
3. 登录失败。
4. 刷新 Token 成功。
5. 刷新 Token 失败。
6. 登出。
7. 修改密码成功。
8. 修改密码失败。

日志中不得记录明文密码、Access Token、Refresh Token。

## 11. 前后端实现要点

### 11.1 移动端

移动端需要实现：

1. 注册页。
2. 登录页。
3. 修改密码页。
4. Token 本地保存。
5. 请求自动携带 Access Token。
6. Access Token 过期后自动刷新。
7. Refresh Token 失效后跳转登录。
8. 登出时清理本地 Token。

Token 保存建议：

1. App 端优先使用安全存储能力。
2. H5 场景避免把敏感 token 暴露给第三方脚本。
3. 不在前端日志打印 token。
4. 不保存用户密码。

刷新 Token 要点：

1. 遇到 401 且原因是 Access Token 过期时发起刷新。
2. 多个请求同时 401 时，只发起一次刷新。
3. 刷新成功后重放等待中的请求。
4. 刷新失败后清理登录态并跳转登录页。

设备 ID：

1. 首次安装生成 UUID。
2. 保存在本地。
3. 卸载重装后变化可接受。
4. 服务端不完全信任 deviceId。

### 11.2 后端

后端建议分层：

```text
controller
application
domain
infrastructure
```

核心组件：

| 组件 | 职责 |
|---|---|
| PasswordService | 密码哈希和校验 |
| TokenService | Access Token 和 Refresh Token 生成 |
| SessionService | 设备会话管理 |
| LoginRiskService | 登录失败、限流、锁定 |
| AuthAuditService | 登录审计 |
| AuthEventService | outbox 事件写入和投递 |

后端实现要求：

1. 注册、登录、刷新接口参数必须校验。
2. 密码字段禁止进入普通日志。
3. 核心写操作使用本地事务。
4. Refresh Token 更新要防并发。
5. 事件投递通过 outbox 或 RocketMQ 事务消息保证最终投递。

## 12. 数据一致性与事件

### 12.1 注册一致性

注册涉及登录服务和用户服务。

建议流程：

```text
登录服务本地事务
  -> 写 auth_account
  -> 写 auth_password
  -> 写 auth_session
  -> 写 outbox UserRegistered
  -> 返回注册成功

outbox 投递
  -> RocketMQ UserRegistered
  -> 用户服务消费事件
  -> 创建默认用户资料
```

要求：

1. `UserRegistered` 事件必须幂等。
2. 用户服务按 `userId` 做唯一约束。
3. 用户资料创建失败要重试。
4. 登录账号创建成功是注册事实来源。
5. 用户资料可通过补偿任务修复。

### 12.2 事件清单

| 事件 | 触发时机 | 消费方 |
|---|---|---|
| UserRegistered | 用户注册成功 | 用户服务、通知服务后续 |
| UserLoggedIn | 用户登录成功 | 审计、风控后续 |
| UserLoggedOut | 用户登出 | IM 服务后续 |
| PasswordChanged | 密码修改成功 | 风控、通知后续 |
| UserSessionRevoked | 会话被撤销 | IM 服务后续 |

事件基础结构沿用全局消息规范：

```json
{
  "eventId": "uuid",
  "eventType": "UserRegistered",
  "eventVersion": 1,
  "occurredAt": "2026-06-03T18:00:00+08:00",
  "producer": "bluenote-auth",
  "traceId": "trace-id",
  "bizKey": "userId",
  "payload": {}
}
```

### 12.3 补偿任务

需要补偿：

1. outbox 事件发送失败。
2. 用户服务未创建默认资料。
3. 会话过期状态未及时更新。
4. 审计日志异步写入失败，后续如果改为异步。

## 13. 日志、指标与告警

### 13.1 日志

必须包含：

1. traceId。
2. userId，登录失败时可为空。
3. username，必要时脱敏或按安全策略处理。
4. ip。
5. deviceId。
6. action。
7. result。
8. failReason。

禁止打印：

1. password。
2. password_hash。
3. accessToken。
4. refreshToken。
5. refresh_token_hash。

### 13.2 指标

| 指标 | 说明 |
|---|---|
| auth_register_total | 注册次数 |
| auth_login_success_total | 登录成功次数 |
| auth_login_fail_total | 登录失败次数 |
| auth_token_refresh_total | Token 刷新次数 |
| auth_token_refresh_fail_total | Token 刷新失败次数 |
| auth_logout_total | 登出次数 |
| auth_password_change_total | 修改密码次数 |
| auth_active_session_total | 活跃会话数 |
| auth_risk_block_total | 风控拦截次数 |

### 13.3 告警

建议告警：

1. 登录失败率异常升高。
2. Token 刷新失败率异常升高。
3. 注册失败率异常升高。
4. 同 IP 登录失败次数异常。
5. Redis 不可用。
6. MySQL 不可用。
7. outbox 堆积。

## 14. 测试重点

### 14.1 正常流程

1. 用户名密码注册成功。
2. 注册后自动登录。
3. 用户名密码登录成功。
4. Access Token 访问受保护接口成功。
5. Refresh Token 刷新成功。
6. 当前设备登出成功。
7. 修改密码成功。

### 14.2 异常流程

1. 重复用户名注册失败。
2. 非法用户名注册失败。
3. 弱密码注册失败。
4. 密码错误登录失败。
5. 用户名不存在登录失败。
6. 账号禁用后登录失败。
7. Access Token 过期后刷新。
8. Refresh Token 过期后要求重新登录。
9. Refresh Token 轮换后旧 token 不能再次使用。
10. 登出后 Refresh Token 不能使用。

### 14.3 安全测试

1. 登录失败次数达到阈值后被限制。
2. 日志中不出现明文密码。
3. 日志中不出现 token。
4. 修改密码后旧 Refresh Token 失效。
5. 并发刷新 Token 只有一次成功。
6. 伪造 Access Token 被拒绝。
7. 被加入黑名单的 Access Token 被拒绝。

### 14.4 数据一致性测试

1. 注册成功后 outbox 事件生成。
2. 重复消费 UserRegistered 不重复创建用户资料。
3. outbox 投递失败后可重试。
4. 用户服务暂时不可用时，注册事件可补偿。

## 15. 风险与后续演进

| 风险 | 说明 | 应对 |
|---|---|---|
| 无找回密码能力 | 第一阶段没有手机号和邮箱验证 | 后续引入邮箱或手机号绑定 |
| 用户名被抢注 | 用户名是唯一登录标识 | 保留用户名规则和注册限流 |
| 密码撞库 | 只用密码登录存在撞库风险 | 登录失败限制、IP 限流、后续验证码 |
| Access Token 登出延迟 | JWT 在过期前可用 | 短有效期 + 黑名单 |
| Redis 不可用 | 风控和黑名单能力受影响 | 核心认证以 MySQL 和 JWT 为准，Redis 恢复后继续 |
| 用户资料异步创建失败 | 注册后用户服务未及时创建资料 | UserRegistered 事件幂等消费 + 补偿任务 |

后续演进：

1. 邮箱绑定和邮箱验证。
2. 手机号绑定和短信验证。
3. 手机号验证码登录。
4. 找回密码。
5. 第三方登录。
6. 登出全部设备。
7. 设备管理列表。
8. 登录二次验证。
9. 管理端账号封禁。
10. 密钥轮换和 Token version 机制。
