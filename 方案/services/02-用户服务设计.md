# BlueNote 用户服务设计

## 1. 背景与目标

用户服务负责维护 BlueNote 用户的业务资料、公开身份、主页基础信息和业务侧用户状态。它是笔记、评论、关系、Feed、排行榜、IM 等服务获取用户展示信息和校验用户业务状态的基础服务。

登录服务已经确定第一阶段只支持用户名 + 密码登录。这里需要明确：登录用户名属于认证标识，不等同于用户昵称，也不应该作为社交展示字段。用户服务维护的是用户在 BlueNote 内对外展示的资料，例如昵称、头像、简介、BlueNote 号、主页背景图等。

设计目标：

1. 支持注册后自动创建默认用户资料。
2. 支持用户查看和编辑个人资料。
3. 支持用户主页头部信息查询。
4. 支持其他服务批量查询用户公开摘要。
5. 支持业务侧用户状态管理。
6. 支持用户资料缓存和缓存失效。
7. 支持用户资料变更事件，方便评论、Feed、IM 等服务维护展示快照。
8. 为后续手机号、邮箱、实名认证、达人认证、隐私设置等能力预留扩展空间。

关键约束：

1. 用户服务不保存密码、Refresh Token、登录失败记录等认证数据。
2. 用户服务不保存关注关系明细。
3. 用户服务不直接维护关注数、粉丝数、作品数、获赞数的最终计数。
4. 用户服务不保存笔记正文、评论内容和 IM 消息。
5. MySQL 是用户资料的事实来源，Redis 只作为缓存。

## 2. 功能范围

### 2.1 第一阶段支持

| 功能 | 说明 |
|---|---|
| 创建默认用户资料 | 消费 `UserRegistered` 事件后创建用户资料 |
| 查询当前用户资料 | App 个人资料页使用 |
| 编辑当前用户资料 | 修改昵称、头像、简介、性别、生日、地区、主页背景图 |
| 查询用户公开资料 | 笔记详情、评论、IM、个人主页等场景使用 |
| 批量查询用户摘要 | Feed、评论列表、IM 会话列表等批量补全用户信息 |
| 查询用户主页头部 | 聚合用户资料、计数、关注状态等轻量信息 |
| 用户状态校验 | 对其他服务提供用户是否存在、是否可用的内部接口 |
| 用户状态变更 | 管理端后续使用，第一阶段先设计服务能力 |
| 用户资料缓存 | Redis 缓存公开资料摘要 |
| 发布用户事件 | 发布 `UserProfileUpdated`、`UserStatusChanged` 等事件 |

### 2.2 第一阶段不支持

| 功能 | 暂不支持原因 |
|---|---|
| 手机号绑定 | 登录服务当前不启用手机号身份，后续引入短信验证后再做 |
| 邮箱绑定 | 后续配合邮箱验证和找回密码一起设计 |
| 实名认证 | 当前业务不涉及强实名场景 |
| 达人认证 | 依赖运营和审核体系，后续设计 |
| 用户黑名单 | 归用户关系服务，不放在用户服务 |
| 用户关注列表 | 归用户关系服务 |
| 用户笔记列表 | 归笔记发布服务 |
| 用户计数最终值 | 归计数服务 |
| 管理后台完整用户运营 | 当前只预留状态字段和内部接口 |

### 2.3 后续扩展

后续可以扩展：

1. 绑定手机号和邮箱。
2. 用户隐私设置。
3. 用户标签和兴趣画像。
4. 达人认证、企业认证。
5. 资料审核。
6. 注销流程。
7. 管理端用户查询和封禁。
8. 用户主页装修。

这些能力不应改变当前用户资料、认证身份、关系、计数的基本边界。

## 3. 服务边界

### 3.1 用户服务负责

用户服务负责：

1. 用户业务资料。
2. 用户公开身份。
3. 用户主页基础配置。
4. 用户业务状态。
5. 用户公开资料缓存。
6. 用户资料变更事件。
7. 用户状态变更事件。
8. 给内部服务提供用户轻量信息查询。

典型数据包括：

| 数据 | 说明 |
|---|---|
| `userId` | 内部用户 ID，与登录服务账号 ID 保持一致 |
| `bluenoteNo` | 对外展示的 BlueNote 号，唯一 |
| `nickname` | 昵称，可重复，可修改 |
| `avatarFileId` | 头像文件 ID |
| `avatarUrl` | 头像访问地址，可由文件服务或 CDN 生成 |
| `bio` | 个人简介 |
| `gender` | 性别，可不填 |
| `birthday` | 生日，可不填 |
| `regionCode` | 地区编码 |
| `homeCoverFileId` | 主页背景图文件 ID |
| `userStatus` | 用户业务状态 |

### 3.2 登录服务负责

登录服务负责：

1. 用户名。
2. 密码哈希。
3. 登录身份绑定。
4. Access Token。
5. Refresh Token。
6. 设备会话。
7. 登录失败限制。
8. 认证侧账号状态。

登录服务不保存昵称、头像、简介等用户资料。用户服务也不读取登录服务数据库。

### 3.3 用户关系服务负责

用户关系服务负责：

1. 关注。
2. 取关。
3. 关注列表。
4. 粉丝列表。
5. 拉黑关系。
6. 好友关系，后续可由互相关注推导。

用户服务在主页头部聚合时可以调用用户关系服务查询当前登录用户是否已关注目标用户，但不保存关注关系事实。

### 3.4 计数服务负责

计数服务负责：

1. 关注数。
2. 粉丝数。
3. 作品数。
4. 获赞数。
5. 收藏数等其他聚合计数。

用户服务可以查询计数服务获取主页头部展示数据，但不把这些数字作为事实来源保存到用户资料表。

### 3.5 笔记服务负责

笔记发布服务负责：

1. 用户发布的笔记内容。
2. 笔记状态。
3. 笔记媒体绑定。
4. 作者笔记列表。

用户主页中的笔记列表应由笔记服务分页查询。用户服务只负责主页头部信息。

## 4. 核心概念

### 4.1 用户 ID

`userId` 是 BlueNote 内部全局用户 ID。登录服务注册成功时生成，用户服务消费 `UserRegistered` 事件后使用同一个 `userId` 创建用户资料。

原则：

1. `userId` 不允许修改。
2. 用户服务按 `userId` 做唯一约束。
3. 所有服务之间传递用户身份时使用 `userId`。
4. 对外页面可以展示 `bluenoteNo`，但接口内部仍以 `userId` 为主。

### 4.2 登录用户名

登录用户名由登录服务维护，用于登录。它不是用户服务字段，也不是用户昵称。

原则：

1. 用户名不作为个人主页展示字段。
2. 用户名不参与用户资料搜索。
3. 用户名不由用户服务修改。
4. 用户名和昵称允许完全不同。

### 4.3 昵称

昵称是用户对外展示名，由用户服务维护。

规则建议：

1. 长度 2 到 20。
2. 可包含中文、英文、数字、下划线。
3. 允许重复。
4. 不允许包含明显违规词。
5. 不允许全空白字符。
6. 修改频率需要限制，例如 30 天最多修改 3 次。

昵称允许重复的原因是：社交产品里昵称是展示属性，不是登录标识。如果强制昵称唯一，会导致用户体验差，也会把登录标识和展示标识混在一起。

### 4.4 BlueNote 号

`bluenoteNo` 是对外展示的公开唯一号，类似小红书号。它用于个人主页展示、搜索和分享，不用于登录。

建议规则：

1. 系统注册时自动生成。
2. 全局唯一。
3. 默认不可频繁修改。
4. 第一阶段可以不提供用户主动修改能力。
5. 后续允许用户修改时，需要限制次数并保留历史记录。

示例：

```text
BN10008392
```

### 4.5 用户状态

用户状态表示业务侧用户是否可正常使用产品功能。

建议状态：

| 状态 | 说明 |
|---|---|
| `NORMAL` | 正常 |
| `DISABLED` | 被禁用，不允许发布、评论、关注、发送 IM |
| `LIMITED` | 部分限制，例如只读或禁言 |
| `CANCELLED` | 已注销 |
| `PENDING_INIT` | 资料初始化中 |

用户状态和登录服务的账号状态需要区分：

| 状态来源 | 含义 |
|---|---|
| 登录服务 `account_status` | 是否允许认证登录 |
| 用户服务 `user_status` | 是否允许使用业务功能 |

当用户服务状态变更为 `DISABLED` 或 `CANCELLED` 时，应发布 `UserStatusChanged` 事件。登录服务可以消费该事件，更新认证侧状态或撤销会话。

## 5. 核心业务流程

### 5.1 注册后创建默认用户资料

登录服务注册成功后发布 `UserRegistered` 事件。用户服务消费事件并创建默认资料。

流程：

```text
用户提交注册
  -> 登录服务创建 auth_account
  -> 登录服务写 UserRegistered outbox
  -> RocketMQ 投递 UserRegistered
  -> 用户服务消费事件
  -> 检查 user_profile 是否已存在
  -> 生成 bluenoteNo
  -> 创建 user_profile
  -> 创建 user_profile_stat_ext 初始扩展记录，可选
  -> 记录事件消费日志
```

默认资料建议：

| 字段 | 默认值 |
|---|---|
| `userId` | 来自 `UserRegistered` |
| `nickname` | `BlueNote用户` + `userId` 后 6 位 |
| `bluenoteNo` | 系统生成唯一号 |
| `avatarFileId` | 默认头像文件 ID，可为空 |
| `bio` | 空字符串 |
| `gender` | `UNKNOWN` |
| `userStatus` | `NORMAL` |

幂等要求：

1. `user_profile.user_id` 唯一。
2. `user_profile.bluenote_no` 唯一。
3. 事件消费记录按 `event_id` 唯一。
4. 重复消费 `UserRegistered` 时，如果资料已存在，直接返回消费成功。

### 5.2 查询当前用户资料

当前登录用户进入资料编辑页时，移动端调用：

```text
GET /api/users/me
```

流程：

```text
移动端请求
  -> 网关校验 Access Token
  -> 注入 userId
  -> 用户服务查询 user_profile
  -> 返回当前用户完整可编辑资料
```

如果用户资料还未初始化完成，返回明确状态：

```text
USER_PROFILE_INITIALIZING
```

前端可以短暂重试，后端补偿任务负责修复。

### 5.3 编辑当前用户资料

用户可以编辑昵称、头像、简介、地区、性别、生日、主页背景图等资料。

流程：

```text
移动端提交资料修改
  -> 网关校验登录态
  -> 用户服务校验字段合法性
  -> 校验用户状态是否允许修改
  -> 如涉及 fileId，调用文件服务校验归属和状态
  -> 更新 user_profile
  -> 删除 Redis 用户资料缓存
  -> 写 outbox UserProfileUpdated
  -> 返回更新后的资料
```

修改规则：

1. 昵称、简介需要做长度和敏感词校验。
2. 头像、主页背景图必须属于当前用户，且文件状态可用。
3. 昵称修改需要频率限制。
4. `bluenoteNo` 第一阶段不提供修改接口。
5. 用户处于 `DISABLED` 或 `CANCELLED` 状态时不允许修改。

### 5.4 查询用户公开资料

笔记详情、评论、IM、用户主页都需要展示用户公开资料。

接口：

```text
GET /api/users/{userId}/public
```

返回内容只包含公开字段：

```json
{
  "userId": "10001",
  "nickname": "小蓝",
  "avatarUrl": "https://cdn.example.com/avatar.png",
  "bluenoteNo": "BN10001",
  "bio": "记录生活",
  "userStatus": "NORMAL"
}
```

不返回：

1. 登录用户名。
2. 手机号。
3. 邮箱。
4. 生日完整值，除非后续隐私策略允许。
5. 内部风控字段。

### 5.5 批量查询用户摘要

Feed、评论列表、IM 会话列表经常需要一次性补全多个用户的昵称和头像。

接口：

```text
POST /internal/users/batch-summary
```

流程：

```text
调用方传入 userId 列表
  -> 用户服务限制批量大小
  -> 优先批量读取 Redis
  -> 未命中部分批量查 MySQL
  -> 回填 Redis
  -> 按请求 userId 顺序返回
```

批量大小建议：

```text
单次最多 100 个 userId
```

返回缺失用户时不要整体失败，应在对应项标记：

```text
NOT_FOUND
```

这样评论、Feed、IM 可以局部降级。

### 5.6 查询用户主页头部

个人主页头部需要聚合用户资料、计数和当前关系状态。

接口：

```text
GET /api/users/{userId}/home
```

聚合内容：

| 字段 | 来源 |
|---|---|
| 用户公开资料 | 用户服务 MySQL/Redis |
| 关注数 | 计数服务 |
| 粉丝数 | 计数服务 |
| 作品数 | 计数服务 |
| 获赞数 | 计数服务 |
| 当前登录用户是否已关注 | 用户关系服务 |

主页头部聚合原则：

1. 用户资料是主数据，查不到用户资料时直接返回用户不存在。
2. 计数服务失败时，计数字段可以降级为 0 或展示上一次缓存值。
3. 用户关系服务失败时，`followStatus` 返回 `UNKNOWN`。
4. 用户笔记列表不由用户服务聚合，移动端单独请求笔记服务。

### 5.7 用户状态变更

用户状态变更通常由管理端、风控系统或后续审核系统触发。

流程：

```text
管理端提交状态变更
  -> 用户服务校验操作权限
  -> 查询用户当前状态
  -> 校验状态流转是否合法
  -> 更新 user_profile.user_status
  -> 写 user_status_history
  -> 删除用户资料缓存
  -> 写 outbox UserStatusChanged
  -> 返回变更结果
```

状态流转建议：

```text
NORMAL -> LIMITED
NORMAL -> DISABLED
NORMAL -> CANCELLED
LIMITED -> NORMAL
LIMITED -> DISABLED
DISABLED -> NORMAL
DISABLED -> CANCELLED
```

`CANCELLED` 原则上不可恢复，除非后续专门设计账号恢复流程。

## 6. 异常流程

| 异常 | 处理方式 |
|---|---|
| `UserRegistered` 重复消费 | 根据 `event_id` 或 `user_id` 幂等成功 |
| 用户资料未初始化 | 返回 `USER_PROFILE_INITIALIZING`，后台补偿 |
| `bluenoteNo` 生成冲突 | 重新生成并重试，超过次数进入补偿 |
| 昵称格式错误 | 返回参数错误 |
| 昵称包含敏感词 | 拒绝修改，提示昵称不可用 |
| 昵称修改过于频繁 | 返回修改频率受限 |
| 头像文件不存在 | 返回文件不可用 |
| 头像文件不属于当前用户 | 返回无权限使用该文件 |
| Redis 不可用 | 降级查 MySQL，不影响主流程 |
| 计数服务不可用 | 主页计数降级 |
| 用户关系服务不可用 | 主页关注状态返回 `UNKNOWN` |
| RocketMQ 不可用 | outbox 保留待投递，不影响资料修改主流程 |
| 用户已禁用 | 禁止资料修改和部分业务操作 |
| 用户已注销 | 查询公开资料时返回用户不存在或已注销状态 |

补偿重点：

1. 注册后用户资料未创建。
2. 用户资料更新事件投递失败。
3. 用户状态变更事件投递失败。
4. 缓存删除失败。
5. 文件绑定状态未及时同步。

## 7. 数据库设计

### 7.1 表清单

| 表名 | 说明 |
|---|---|
| `user_profile` | 用户主资料表 |
| `user_profile_audit` | 用户资料修改审计表 |
| `user_status_history` | 用户状态变更历史表 |
| `user_bluenote_no_history` | BlueNote 号历史表，后续允许修改时使用 |
| `user_event_consume_log` | MQ 消费幂等表 |
| `user_outbox_event` | 用户服务 outbox 事件表 |

### 7.2 user_profile

用户主资料表。

| 字段 | 类型 | 说明 |
|---|---|---|
| `user_id` | BIGINT | 用户 ID，主键 |
| `bluenote_no` | VARCHAR(32) | BlueNote 号，唯一 |
| `nickname` | VARCHAR(64) | 昵称 |
| `avatar_file_id` | BIGINT NULL | 头像文件 ID |
| `avatar_url` | VARCHAR(512) NULL | 头像访问地址或快照 |
| `bio` | VARCHAR(256) | 个人简介 |
| `gender` | VARCHAR(16) | `UNKNOWN` / `MALE` / `FEMALE` |
| `birthday` | DATE NULL | 生日 |
| `region_code` | VARCHAR(32) NULL | 地区编码 |
| `home_cover_file_id` | BIGINT NULL | 主页背景图文件 ID |
| `home_cover_url` | VARCHAR(512) NULL | 主页背景图访问地址或快照 |
| `user_status` | VARCHAR(32) | 用户业务状态 |
| `profile_version` | BIGINT | 资料版本号 |
| `created_at` | DATETIME(3) | 创建时间 |
| `updated_at` | DATETIME(3) | 更新时间 |
| `deleted` | TINYINT | 逻辑删除 |

说明：

1. `user_id` 与登录服务注册账号 ID 一致。
2. `nickname` 允许重复，不建唯一索引。
3. `avatar_url` 可以保存展示快照，正式文件权限和生命周期仍归文件服务。
4. `profile_version` 用于缓存、事件和快照更新判断。
5. `deleted = 1` 用于注销或合规删除后的逻辑状态，具体注销流程后续单独设计。

### 7.3 user_profile_audit

用户资料修改审计表。

| 字段 | 类型 | 说明 |
|---|---|---|
| `id` | BIGINT | 主键 |
| `user_id` | BIGINT | 用户 ID |
| `field_name` | VARCHAR(64) | 修改字段 |
| `old_value_mask` | VARCHAR(512) NULL | 修改前值，必要时脱敏 |
| `new_value_mask` | VARCHAR(512) NULL | 修改后值，必要时脱敏 |
| `operator_id` | BIGINT | 操作人 ID |
| `operator_type` | VARCHAR(32) | `USER` / `ADMIN` / `SYSTEM` |
| `reason` | VARCHAR(256) NULL | 修改原因 |
| `trace_id` | VARCHAR(64) | 链路 ID |
| `created_at` | DATETIME(3) | 创建时间 |

说明：

1. 用户自己修改资料时，`operator_type = USER`。
2. 管理端修改资料时，`operator_type = ADMIN`。
3. 审计表不保存敏感原文，必要字段做脱敏或摘要。

### 7.4 user_status_history

| 表名 | 索引 | 说明 |

## 8. 缓存设计

### 8.1 缓存对象

用户公开摘要是高频读取数据，适合缓存。

Redis Key：

```text
user:profile:{userId}
```

类型：

```text
Hash 或 String(JSON)
```

建议第一阶段使用 String(JSON)，便于版本化和整体失效。

缓存内容：

```json
{
  "userId": "10001",
  "bluenoteNo": "BN10001",
  "nickname": "小蓝",
  "avatarFileId": "90001",
  "avatarUrl": "https://cdn.example.com/avatar.png",
  "bio": "记录生活",
  "userStatus": "NORMAL",
  "profileVersion": 12
}
```

TTL 建议：

```text
6 小时 + 随机抖动
```

### 8.2 读缓存策略

```text
查询用户公开摘要
  -> 读 Redis
  -> 命中则返回
  -> 未命中查 MySQL
  -> 用户存在则写 Redis
  -> 用户不存在则写短 TTL 空值缓存
```

空值缓存：

```text
user:profile:null:{userId}
TTL 60 秒
```

用于防止不存在用户被频繁穿透数据库。

### 8.3 写缓存策略

用户资料更新时采用：

```text
先更新 MySQL
再删除 Redis 缓存
再发布 UserProfileUpdated
```

不建议直接更新缓存作为事实来源。删除缓存后，下次读取从 MySQL 重建。

### 8.4 批量查询优化

批量查询用户摘要时：

1. 使用 Redis pipeline 批量读取。
2. 只对未命中的 userId 查询 MySQL。
3. 查询 MySQL 时按 userId 排序，避免返回顺序依赖数据库。
4. 回填 Redis 时设置随机 TTL。
5. 返回结果按请求顺序组装。

### 8.5 缓存一致性风险

| 风险 | 应对 |
|---|---|
| 删除缓存失败 | 记录日志，补偿任务重删 |
| 缓存重建并发 | 可接受，用户资料读多写少 |
| 旧事件覆盖新快照 | 事件携带 `profileVersion`，消费者只接受更新版本 |
| Redis 不可用 | 降级查 MySQL，必要时对批量查询限流 |

## 11. 安全与权限设计

### 11.1 外部接口权限

| 接口 | 权限要求 |
|---|---|
| `GET /api/users/me` | 必须登录，只能查自己 |
| `PUT /api/users/me/profile` | 必须登录，只能改自己 |
| `GET /api/users/{userId}/public` | 可登录访问，后续可支持公开访问 |
| `GET /api/users/{userId}/home` | 可登录访问，后续根据隐私设置决定 |

用户服务信任网关注入的用户上下文，不信任移动端传入的 `userId` 作为当前用户身份。

### 11.2 内部接口权限

内部接口要求：

1. 只允许网关或服务内网访问。
2. 请求头携带内部调用签名或服务身份。
3. 网关不能把 `/internal/**` 暴露给公网。
4. 内部批量接口必须限制单次数量和调用频率。

### 11.3 字段安全

公开资料不返回敏感字段。即使后续用户服务保存手机号、邮箱绑定状态，也不能在公开接口返回原文。

字段处理原则：

| 字段 | 处理 |
|---|---|
| 手机号 | 不在第一阶段保存；后续只返回脱敏值 |
| 邮箱 | 不在第一阶段保存；后续只返回脱敏值 |
| 生日 | 当前用户可见，公开资料默认不返回完整生日 |
| 地区 | 可公开展示，后续受隐私设置控制 |
| 操作原因 | 管理端可见，普通用户不可见 |

### 11.4 内容安全

用户资料属于用户生成内容，需要做基础内容安全：

1. 昵称敏感词校验。
2. 简介敏感词校验。
3. 昵称长度和字符集限制。
4. 简介长度限制。
5. 头像和主页背景图需要通过文件服务状态校验。

第一阶段可以先使用本地敏感词词表，后续再接入审核服务。

### 11.5 频率限制

建议限制：

| 操作 | 限制 |
|---|---|
| 修改昵称 | 30 天最多 3 次 |
| 修改头像 | 每小时最多 20 次 |
| 修改简介 | 每小时最多 20 次 |
| 查询公开资料 | 网关按 IP 和用户限流 |
| 批量摘要查询 | 内部接口单次最多 100 个用户 |

频率限制可以使用 Redis：

```text
rate:user:nickname:update:{userId}
rate:user:profile:update:{userId}
```

Redis 不可用时，资料修改主流程可以继续，但需要记录降级日志。

## 12. 前后端实现要点

### 12.1 移动端实现

移动端页面：

1. 我的页面。
2. 编辑资料页。
3. 用户主页页。
4. 头像选择和裁剪页，后续可做。

实现要点：

1. 当前用户资料缓存到 Pinia，但不要长期信任本地缓存。

3. 编辑资料成功后更新本地用户资料缓存。
4. 用户主页头部调用 `/api/users/{userId}/home`。
5. 用户主页笔记列表单独调用笔记服务接口。
6. 头像上传先走文件服务，拿到 `fileId` 后再更新用户资料。
7. 当 `followStatus = UNKNOWN` 时，关注按钮可以置为不可操作或点击后重试关系查询。

前端展示原则：

1. 不展示登录用户名。
2. 昵称和 BlueNote 号分开展示。
3. 简介为空时展示空状态，不要写死默认文案到后端。
4. 用户状态异常时按后端返回状态展示受限页面。

### 12.2 后端实现

后端模块建议：

```text
bluenote-user
  ├── api
  ├── application
  ├── domain
  ├── infrastructure
  ├── mq
  └── job
```

实现要点：

1. Controller 只做参数校验和用户上下文获取。
2. Application Service 编排文件校验、数据库更新、缓存删除、outbox 写入。
3. Domain 层封装用户状态流转、资料修改规则。
4. Infrastructure 层封装 MySQL、Redis、RocketMQ、文件服务客户端。
5. MQ 消费必须幂等。
6. outbox 投递任务独立执行，失败可重试。
7. 内部接口和外部接口分包，避免误暴露。

### 12.3 文件服务依赖

头像和主页背景图更新时，用户服务需要校验：

1. 文件存在。
2. 文件归属当前用户。
3. 文件类型为图片。
4. 文件状态为可用。
5. 文件大小符合头像或背景图限制。

第一阶段可以只保存 `fileId` 和 URL 快照。文件删除、迁移、URL 过期等问题由文件服务后续统一处理。

### 12.4 计数和关系依赖

用户主页头部依赖计数服务和关系服务，但这些依赖不能影响用户资料主查询。

降级策略：

1. 计数服务超时 200ms 后降级。
2. 关系服务超时 200ms 后降级。
3. 用户主页头部响应中返回 `degraded = true`。
4. 降级场景记录指标，便于排查。

## 13. 数据一致性与补偿

### 13.1 注册资料一致性

注册账号事实来源在登录服务。用户资料由用户服务异步创建，属于最终一致。

一致性要求：

1. `UserRegistered` 事件必须可靠投递。
2. 用户服务消费必须幂等。
3. 用户资料创建失败必须重试。
4. 注册成功但资料未创建时，不能创建第二个账号。
5. 补偿任务可以根据登录服务事件或对账接口修复缺失资料。

补偿任务：

```text
定时扫描 user_event_consume_log 中失败的 UserRegistered
  -> 重试创建用户资料
```

如后续登录服务提供对账接口：

```text
查询最近注册用户
  -> 对比 user_profile
  -> 补齐缺失资料
```

### 13.2 资料更新一致性

资料更新流程：

```text
本地事务：
  -> 更新 user_profile
  -> 写 user_profile_audit
  -> 写 user_outbox_event

事务提交后：
  -> 删除 Redis 缓存
  -> outbox 异步投递 UserProfileUpdated
```

如果缓存删除失败：

1. 记录失败日志。
2. 写入缓存重删任务。
3. 缓存 TTL 到期后自然修复。

如果事件投递失败：

1. outbox 保留 `FAIL` 状态。
2. 定时任务按退避策略重试。
3. 超过阈值告警。

### 13.3 状态变更一致性

状态变更流程：

```text
本地事务：
  -> 更新 user_profile.user_status
  -> 写 user_status_history
  -> 写 user_outbox_event(UserStatusChanged)
  -> 删除缓存
```

用户服务是业务状态事实来源。其他服务消费状态变更事件后维护自己的限制逻辑。

如果状态事件延迟，其他服务在关键写操作前仍应通过内部接口校验用户状态。例如发布笔记、评论、关注、发送 IM 时，不能只依赖本地旧快照。

### 13.4 快照一致性

评论、Feed、IM 等服务可能保存用户昵称和头像快照，用于提升列表查询性能。

快照一致性原则：

1. 用户服务发布 `UserProfileUpdated`。
2. 下游服务按 `profileVersion` 更新快照。
3. 快照短时间不一致可以接受。
4. 用户主页、资料页应以用户服务实时数据为准。

## 14. 日志、指标与告警

### 14.1 日志

必须记录：

1. 用户资料创建成功和失败。
2. 用户资料修改成功和失败。
3. 用户状态变更。
4. 文件校验失败。
5. Redis 降级。
6. 计数服务和关系服务调用降级。
7. MQ 消费失败。
8. outbox 投递失败。

日志字段：

| 字段 | 说明 |
|---|---|
| `traceId` | 链路 ID |
| `userId` | 用户 ID |
| `operatorId` | 操作人 ID |
| `operation` | 操作类型 |
| `result` | 成功或失败 |
| `errorCode` | 错误码 |
| `costMs` | 耗时 |

不记录：

1. 敏感原文。
2. 过长简介全文，必要时截断。
3. 后续手机号、邮箱等敏感数据原文。

### 14.2 指标

| 指标 | 说明 |
|---|---|
| `user_profile_create_total` | 用户资料创建次数 |
| `user_profile_create_fail_total` | 用户资料创建失败次数 |
| `user_profile_update_total` | 用户资料更新次数 |
| `user_profile_update_fail_total` | 用户资料更新失败次数 |
| `user_public_query_total` | 用户公开资料查询次数 |
| `user_batch_summary_total` | 批量摘要查询次数 |
| `user_cache_hit_ratio` | 用户资料缓存命中率 |
| `user_home_query_total` | 用户主页头部查询次数 |
| `user_home_degraded_total` | 用户主页降级次数 |
| `user_status_changed_total` | 用户状态变更次数 |
| `user_mq_consume_fail_total` | MQ 消费失败次数 |
| `user_outbox_pending_total` | outbox 待投递数量 |

### 14.3 告警

建议告警：

1. `UserRegistered` 消费失败持续增加。
2. 用户资料创建失败率超过阈值。
3. outbox 堆积超过阈值。
4. Redis 不可用或缓存命中率异常下降。
5. 用户主页降级率异常升高。
6. 批量摘要接口耗时异常。
7. 用户状态变更事件投递失败。
8. MySQL 慢查询增加。

## 15. 测试重点

### 15.1 单元测试

1. 昵称校验。
2. 简介校验。
3. 用户状态流转校验。
4. BlueNote 号生成冲突重试。
5. 公开资料字段过滤。
6. 资料版本号递增。
7. 批量查询结果顺序组装。

### 15.2 集成测试

1. 消费 `UserRegistered` 创建默认资料。
2. 重复消费 `UserRegistered` 不重复创建。
3. 修改资料后删除 Redis 缓存。
4. 修改资料后写 outbox。
5. outbox 投递失败后可重试。
6. 文件服务校验失败时资料不更新。
7. 计数服务不可用时主页头部降级。
8. 关系服务不可用时关注状态降级。

### 15.3 接口测试

1. 未登录不能访问 `/api/users/me`。
2. 登录用户只能修改自己的资料。
3. 查询不存在用户返回 `USER_NOT_FOUND`。
4. 用户资料初始化中返回 `USER_PROFILE_INITIALIZING`。
5. 批量摘要超过数量限制返回 `USER_BATCH_SIZE_EXCEEDED`。
6. 禁用用户不能修改资料。
7. 公开资料不包含登录用户名。

### 15.4 性能测试

重点压测：

1. 批量用户摘要查询。
2. 用户主页头部查询。
3. Redis 命中和未命中场景。
4. 大量 Feed/评论列表并发补全用户信息。

第一阶段目标：

1. 批量摘要接口 P95 小于 100ms，Redis 命中场景。
2. 用户主页头部接口 P95 小于 200ms，不含严重下游降级。
3. 单次批量摘要最多 100 个用户。

## 16. 风险与后续演进

| 风险 | 说明 | 应对 |
|---|---|---|
| 注册后资料创建延迟 | 登录成功后用户服务可能还没消费事件 | 前端短暂重试 + 后台补偿 |
| 用户资料缓存不一致 | 更新后缓存删除失败或下游快照延迟 | TTL + 重删 + 版本号 |
| 主页聚合依赖过多 | 计数、关系服务失败会影响主页头部完整性 | 超时降级，不影响用户资料返回 |
| 昵称违规 | 用户资料属于 UGC | 敏感词校验，后续接入审核 |
| 批量摘要接口压力大 | Feed、评论、IM 高频调用 | Redis 缓存、批量限制、调用方本地短缓存 |
| 用户状态扩散延迟 | 状态事件消费需要时间 | 关键写操作实时调用用户状态校验 |
| BlueNote 号冲突 | 随机生成可能冲突 | 唯一索引 + 重试 + 补偿 |

后续演进：

1. 引入邮箱、手机号绑定信息，但敏感身份仍要和登录服务边界保持清楚。
2. 增加用户隐私设置，控制生日、地区、关注列表等展示权限。
3. 增加用户资料审核流程。
4. 增加达人认证、企业认证。
5. 增加用户搜索能力，必要时再评估搜索服务。
6. 增加注销和数据清理流程。
7. 增加管理端用户运营能力。
8. 对高频用户摘要查询增加本地缓存或多级缓存。
