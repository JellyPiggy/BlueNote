# BlueNote 方案设计阶段工作准则

本文用于约束 Codex 在 `方案/` 目录下继续设计 BlueNote 项目方案时的行为。

当前阶段不是写代码阶段，而是方案设计阶段。除非用户明确要求生成代码、脚手架或配置文件，否则只产出设计文档、接口草案、数据结构、事件规范、部署方案和技术决策记录。

## 1. 项目定位

BlueNote 是一个简易版小红书项目，面向移动端用户，核心能力包括：

1. 登录、注册、Token、设备会话。
2. 用户资料、主页信息。
3. 笔记发布、编辑、详情、媒体元数据。
4. 点赞、收藏、评论、关注、粉丝等计数。
5. 用户关注关系。
6. 关注页 Timeline Feed。
7. 笔记热度排行榜。
8. 二级评论。
9. IM 单聊、会话、消息、未读数。
10. 文件上传与对象存储。
11. 后续可扩展订单、支付、通知、审核、搜索、推荐、数据分析等能力。

本项目不是一次性做完全部功能的 MVP 原型，而是按照真实企业项目标准设计。第一阶段可以控制部署复杂度和资源消耗，但代码结构、服务边界、数据一致性、可观测性、安全认证、消息事件规范等必须从一开始按长期演进来设计。

## 2. 当前权威上下文

在继续设计任何方案前，必须优先阅读并遵守：

1. `01-总体技术方案.md`
2. 当前任务相关的已有方案文档
3. 如果任务进入开发、部署、接口、DDL、MQ 契约或验收阶段，必须同时阅读 `08-落地实施方案.md`
4. `decisions/` 目录下的技术决策记录，如存在

`01-总体技术方案.md` 是当前总体方案的基线。不要在后续服务方案中随意推翻已确定的总体技术栈、服务边界、非目标范围和部署原则。

如果发现总体方案存在明显问题，应先输出“变更建议”，说明：

1. 为什么需要修改。
2. 会影响哪些已有文档。
3. 替代方案是什么。
4. 是否需要新增或修改 `decisions/` 中的技术决策记录。

未经用户确认，不要直接大改总体技术路线。

## 3. 已确定的核心技术路线

后续方案默认继承以下技术选择：

3. TypeScript

5. `uni.request` 封装
6. Vite，随 uni-app 体系
7. 目标平台优先 Android / iOS / H5，后续可扩展小程序

移动端最终效果直接参考小红书，如果需要图片，可让用户截图发送。

5. Spring Security

### 3.3 网关与服务治理

1. Spring Cloud Gateway

3. Nacos 配置中心

### 3.4 存储与中间件

1. MySQL 8.x

## 4. 当前阶段非目标

第一阶段不要主动引入以下内容，除非用户明确要求重新评估：

1. Kubernetes
2. Kafka 作为主消息队列
3. Seata 或复杂分布式事务框架
4. Elasticsearch
5. 多机房容灾
6. 完整推荐系统
7. 复杂实时数仓或 Flink 体系

可以为这些能力保留扩展点，但不要把它们设计成第一阶段必须依赖。

## 5. 设计原则

### 5.1 企业级，而不是 MVP

不要用“先简单够用”“MVP 先不考虑”“后面再说”来省略关键设计。

允许第一阶段简化部署，但不能省略以下设计：

1. 服务职责边界
2. 数据表设计、索引设计
3. 接口契约
4. 错误码与响应结构
5. 鉴权与安全边界
6. 幂等设计
7. 缓存一致性策略
8. MQ 事件结构
9. 异常与补偿机制
10. 可观测性
11. 部署约束
12. 后续演进路径

### 5.2 有限资源优先

方案必须考虑：

1. 2 核 4G 云服务器资源有限。
2. 本地旧电脑可以参与部署，但不能承载用户主链路的强依赖。
3. 内网穿透适合测试、管理、备份、低优先级异步任务，不适合作为核心线上依赖。
4. 代码可以按服务边界设计，部署时允许合并低流量服务。
5. JVM 内存、容器数量、中间件资源占用需要被考虑。

### 5.3 数据库是事实来源

1. MySQL 是核心业务事实库。
2. Redis 和 MQ 不作为唯一事实来源。
3. Redis 中的计数、Feed、在线状态等数据必须可恢复或可重建。
4. 所有核心异步消费必须具备幂等能力。
5. 跨服务一致性优先采用本地事务 + RocketMQ 事件 + 幂等消费 + 补偿任务。
6. 关键消息发送可使用 RocketMQ 事务消息或 Outbox 表。

### 5.4 服务间协作

1. 查询类和强即时结果类操作可以使用同步 RPC。
3. 不允许跨服务直接访问对方数据库 schema。
4. 不允许跨库 join。
5. 跨服务数据同步必须通过接口或事件完成。

## 6. 文档风格要求

1. 使用中文输出。
2. 保留清晰的一级、二级、三级标题。
3. 不要只列标题，必须给出解释。
4. 不要写成营销文案，要写成可以指导后续开发的工程文档。
5. 对关键技术决策给出原因、取舍和风险。
6. 表格适合表达结构化信息时可以使用表格。
7. 流程复杂时优先使用 Mermaid。
8. 不要泛泛而谈，必须结合 BlueNote 的业务场景。
9. 不要凭空引入未在总体方案中出现的新技术。
10. 如果必须引入新技术，必须先说明为什么现有技术栈无法满足。

## 8. 技术决策记录要求

遇到以下情况，应建议新增或更新 `decisions/` 下的 ADR 文档：

1. 选择某个核心技术。
2. 修改已确定的总体技术栈。
3. 改变服务边界。
4. 引入新的基础设施。
5. 对一致性、事务、缓存、MQ、部署架构作出重要取舍。

ADR 内容建议包含：

1. 背景
2. 决策
3. 备选方案
4. 选择原因
5. 代价与风险
6. 后续重新评估条件

## 9. 与用户协作方式

1. 如果用户要求“设计方案”，直接产出完整方案，不要只给提纲。
2. 如果信息不足但可以合理假设，应先做合理假设并继续输出，不要频繁追问。
7. 每次新增方案文档后，应检查它是否与 `01-总体技术方案.md` 冲突。

## 10. 禁止行为

1. 不要把 BlueNote 写成普通 CRUD 单体项目。
2. 不要随意降低到 MVP 级别。
3. 不要默认省略鉴权、幂等、缓存一致性、消息可靠性、日志监控。
4. 不要让移动端直接调用内部服务。
5. 不要让一个服务直接访问另一个服务的数据库。
6. 不要把 Redis 当成唯一事实来源。
7. 不要通过 MQ 传输大文件。
8. 不要在第一阶段强依赖本地旧电脑承载用户主链路。
10. 不要输出无法落地的空泛架构描述。
