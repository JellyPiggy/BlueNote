# ADR-003-MQ 选择 RocketMQ

状态：Accepted  
日期：2026-06-03

## 背景

BlueNote 需要消息队列支撑多个业务场景：

1. 点赞、收藏、评论、关注后的计数异步更新。
2. 笔记发布后的 Feed 扩散。
3. 排行榜热度更新。
4. 通知异步生成。
5. IM 内部消息投递。
6. 后续订单超时关单。
7. 跨服务最终一致。

参考资料中 IM 服务可能使用 Kafka，但 BlueNote 的整体业务不只是 IM，还包括大量业务消息和后续订单场景。

## 备选方案

| 方案 | 说明 |
|---|---|
| RabbitMQ | 简单易用，适合普通业务队列，但延时、事务和高吞吐扩展不是当前最优 |
| Kafka | 适合日志流、行为流、实时分析和大吞吐事件流 |
| RocketMQ | 适合业务消息、延时消息、事务消息、顺序消息 |

## 决策

BlueNote 第一阶段主消息队列选择：

```text
RocketMQ 5.x
```

Kafka 暂不引入。

## 决策原因

1. RocketMQ 原生支持延时消息，适合订单超时关单。
2. RocketMQ 支持事务消息，适合本地事务和消息发送最终一致。
3. RocketMQ 支持顺序消息，适合 IM 会话内消息和订单状态流转。
4. 当前项目更偏业务消息系统，而不是大规模数据流平台。
5. 只维护一个 MQ 能降低第一阶段部署和运维复杂度。

## 影响

正向影响：

1. 订单超时关单实现更自然。
2. 计数、Feed、通知、排行榜可以统一异步事件模型。
3. IM 内部投递可以使用局部顺序消息。
4. 不需要同时维护 Kafka 和 RocketMQ。

代价：

1. RocketMQ 运维比 RabbitMQ 更重。
2. 如果后续做大规模埋点、推荐、实时计算，可能仍需要引入 Kafka。
3. 需要从一开始设计好 topic、consumer group、幂等和死信处理。

## 后续重新评估条件

出现以下情况时重新评估：

1. 项目需要大规模用户行为日志流。
2. 需要接入 Flink、推荐系统或数据湖。
3. IM 消息量远超普通业务消息，且需要独立事件流平台。
4. RocketMQ 单机或主从部署无法满足吞吐和运维要求。
