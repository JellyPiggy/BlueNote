# ADR-007-第一阶段对象存储使用 MinIO

状态：Accepted  
日期：2026-06-03

## 背景

BlueNote 需要保存用户头像、主页背景图、笔记图片、笔记视频和后续 IM 图片等文件资源。

总体方案原先保留了 `MinIO 或云 OSS/COS/R2` 的选择，并倾向正式环境使用云对象存储和 CDN。但当前项目目标是低成本真实落地，预期第一阶段用户数量较少，文件数量也很少，短期内购买云 OSS/COS/R2 和 CDN 的收益不明显。

同时，文件服务仍需要按真实企业项目方式设计，不能把文件简单保存成后端本地目录，也不能让业务服务直接依赖具体存储实现。

## 备选方案

| 方案 | 说明 |
|---|---|
| 全环境使用云 OSS/COS/R2 | 生产能力更强，但需要额外费用和云厂商配置 |
| 开发测试使用 MinIO，正式使用云 OSS | 企业常见方案，但当前正式用户量很小，成本优先级不匹配 |
| 全环境使用 MinIO | 成本低，可本地和正式环境一致，但正式环境需要承担单机风险和备份责任 |
| 后端本地磁盘目录 | 实现简单，但不符合对象存储模型，后续迁移成本高 |

## 决策

第一阶段开发、测试、正式环境统一使用 MinIO。

正式环境 MinIO 部署在 2 核 4G 云服务器上，作为小规模真实可用的对象存储实现。本地旧电脑不承载正式用户文件访问主链路，只作为测试环境、备份副本或恢复演练环境。

文件服务仍按对象存储模式设计：

1. 移动端先向文件服务申请上传凭证。
2. 移动端使用预签名 URL 直传 MinIO。
3. 文件服务确认上传结果并保存文件元数据。
4. 用户服务、笔记服务、IM 服务只依赖 `fileId` 和文件服务接口。
5. 业务服务不直接依赖 MinIO bucket、objectKey 或访问 URL。
6. 后续迁移云 OSS/COS/R2 + CDN 时，通过替换或新增 `ObjectStorageClient` 实现完成。

## 决策原因

1. 当前项目预计第一阶段用户量和文件量较小，MinIO 单机可以满足落地需求。
2. MinIO 兼容 S3 对象存储模型，便于保持企业级上传流程和接口边界。
3. 统一使用 MinIO 可以降低开发、测试、正式环境差异。
4. 不购买云 OSS/COS/R2 可以降低早期成本。
5. 文件服务抽象存储适配层后，后续迁移云对象存储不影响移动端和业务服务。

## 影响

正向影响：

1. 第一阶段部署成本更低。
2. 开发、测试、正式环境对象存储行为更一致。
3. 文件服务仍保留向云对象存储演进的接口形态。

代价和风险：

1. 正式环境 MinIO 单机不是高可用存储。
2. 文件访问会占用云服务器磁盘和带宽。
3. 需要自行负责 MinIO 数据目录持久化、备份和恢复演练。
4. 视频上传需要严格限制，否则容易快速消耗磁盘和带宽。

应对措施：

1. MinIO 数据目录必须挂载到持久化磁盘。
2. MinIO 控制台不暴露公网。
3. 文件访问通过 HTTPS 域名或反向代理暴露。
4. 定时备份 MinIO 数据到本地旧电脑。
5. 配置磁盘容量、对象数量、上传失败率告警。
6. 定时清理上传后长期未绑定文件。
7. 第一阶段严格限制单文件大小、用户上传频率和视频能力。

## 后续重新评估条件

出现以下任一情况时，重新评估迁移云 OSS/COS/R2 + CDN：

1. 文件总量或磁盘占用接近云服务器容量上限。
2. 图片访问消耗明显影响 API、数据库或 MQ 的主链路资源。
3. 用户数量或访问量增长，MinIO 单机成为明显瓶颈。
4. 需要更稳定的公网图片访问、CDN 加速或跨区域可用性。
5. 视频能力正式开放且存储、带宽、转码成本上升。
6. MinIO 备份恢复复杂度超过继续自建的收益。
