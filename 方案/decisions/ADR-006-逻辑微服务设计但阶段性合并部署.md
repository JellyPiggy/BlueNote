# ADR-006-逻辑微服务设计但阶段性合并部署

状态：Accepted  
日期：2026-06-03

## 背景

BlueNote 包含登录、用户、文件、笔记、计数、关系、Feed、排行榜、评论、IM、通知、订单等多个业务域。

按照真实企业项目标准，这些业务域需要有清晰服务边界、数据归属和事件契约。但第一阶段部署资源有限，如果每个逻辑服务都独立成 Java 进程，会给 2 核 4G 云服务器带来明显内存压力。

## 备选方案

| 方案 | 说明 |
|---|---|
| 完全单体应用 | 部署最简单，但边界容易混乱，后续拆分成本高 |
| 所有服务独立部署 | 微服务边界清晰，但第一阶段资源压力大 |
| 逻辑微服务设计，物理应用阶段性合并部署 | 保留边界，同时控制部署成本 |

## 决策

BlueNote 采用：

```text
逻辑微服务设计，第一阶段物理应用合并部署
```

第一阶段物理分组：

| 物理应用 | 包含逻辑服务 |
|---|---|
| bluenote-gateway-app | gateway |
| bluenote-member-app | auth、user |
| bluenote-content-app | file、note、comment |
| bluenote-social-app | relation、counter、feed、rank、notification |
| bluenote-im-app | im |
| bluenote-order-app | order，后续启用 |

## 决策原因

1. 逻辑边界清晰，有利于后续服务独立拆分。
2. 阶段性合并部署能降低 Java 进程数量和内存占用。
3. 业务域的数据 schema、事件、接口仍按服务边界设计。
4. 当前更需要先把业务链路、数据一致性和运维体系建立起来。
5. 合并部署不影响未来按压力和发布频率逐步拆分。

## 影响

正向影响：

1. 兼顾企业项目边界和现实资源限制。
2. 第一阶段部署复杂度可控。
3. 后续拆分路径清晰。
4. 避免过早微服务化导致运维负担过高。

代价：

1. 代码结构必须严格约束，否则容易退化为单体混乱依赖。
2. 合并应用内模块可能绕过接口边界，需要工程规范限制。
3. 某个模块故障可能影响同一物理应用内其他模块。
4. 后续拆分时仍需要处理配置、部署、监控和调用链调整。

## 约束

即使物理合并部署，也必须遵守：

1. 数据 schema 按逻辑服务拆分。
2. 包结构按逻辑服务拆分。
3. 业务写操作只能写本逻辑服务数据。
4. 跨逻辑服务访问优先通过接口或事件。
5. 公共模块不得承载具体业务逻辑。

## 后续重新评估条件

出现以下情况时拆分独立服务：

1. 某个模块资源占用明显高于其他模块。
2. 某个模块发布频率明显更高。
3. 某个模块故障影响其他核心链路。
4. 某个模块需要独立扩容。
5. 某个模块需要独立安全隔离。
