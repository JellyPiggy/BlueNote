# ADR-005-第一阶段暂不引入 Kubernetes

状态：Accepted  
日期：2026-06-03

## 背景

BlueNote 按企业项目标准设计，但第一阶段部署资源有限。目标服务器为 2 核 4G 云服务器，并会利用本地旧电脑承载部分非核心服务。

Kubernetes 能提供容器编排、服务发现、滚动发布、弹性伸缩、自愈等能力，但也会带来明显的资源和运维成本。

## 备选方案

| 方案 | 说明 |
|---|---|
| 第一阶段直接上 Kubernetes | 架构能力强，但资源和复杂度高 |
| 第一阶段不用 Kubernetes，使用 Docker Compose | 更贴合当前资源和运维能力 |
| 使用轻量 K3s | 比标准 Kubernetes 轻，但仍有学习和维护成本 |

## 决策

BlueNote 第一阶段暂不引入 Kubernetes。

当前采用：

```text
Docker Compose
```

后续保留迁移到 Kubernetes 或 K3s 的可能。

## 决策原因

1. 2 核 4G 资源不适合同时承载完整 Kubernetes 管理面和多个业务服务。
2. 当前服务规模还没有达到必须使用 Kubernetes 的程度。
3. 第一阶段更重要的是服务边界、数据一致性、监控、备份和可恢复。
4. Docker Compose 已能满足当前单机部署、测试环境和本地环境需要。
5. 暂不使用 Kubernetes 可以降低调试和排障复杂度。

## 影响

正向影响：

1. 部署和排查更直接。
2. 资源占用更低。
3. 更容易在云服务器和本地旧电脑上落地。
4. 可以把精力放在业务架构和基础治理上。

代价：

1. 自动扩缩容能力弱。
2. 多实例调度需要手工处理。
3. 服务自愈和滚动发布能力有限。
4. 后续迁移 Kubernetes 时需要补充部署清单和配置管理。

## 后续重新评估条件

出现以下情况时重新评估：

1. 服务数量和实例数量持续增加。
2. 需要多机自动调度。
3. 需要标准滚动发布、灰度发布和自动回滚。
4. 单机 Compose 运维成本超过收益。
5. 有更充足服务器资源或使用托管 Kubernetes。
