# ADR-004-部署方式选择 Docker Compose

状态：Accepted  
日期：2026-06-03

## 背景

BlueNote 第一阶段计划将一部分内容部署在 2 核 4G 云服务器上，一部分部署在本地旧电脑上。本地旧电脑通过内网穿透承担辅助角色。

当前目标不是构建复杂集群，而是在有限资源下保证项目可运行、可维护、可恢复。

## 备选方案

| 方案 | 说明 |
|---|---|
| 直接宿主机部署 | 简单，但环境不一致，迁移和回滚困难 |
| Docker Compose | 适合单机或少量机器，部署清晰，运维成本低 |
| Kubernetes | 能力强，但资源和运维复杂度高 |
| 云厂商 PaaS | 运维简单，但成本和可控性依赖平台 |

## 决策

BlueNote 第一阶段部署方式选择：

```text
Docker + Docker Compose
```

用于部署：

1. Java 应用服务。
2. MySQL。
3. Redis。
4. RocketMQ。
5. Nacos。
6. Prometheus、Grafana、Loki 等辅助组件。

## 决策原因

1. Docker Compose 更适合当前单机和少量机器部署。
2. 相比宿主机直接部署，环境一致性和迁移能力更好。
3. 相比 Kubernetes，资源占用和学习成本明显更低。
4. Compose 文件便于纳入版本管理。
5. 可以清晰管理本地、测试、生产环境差异。

## 影响

正向影响：

1. 部署结构清晰。
2. 环境可复现。
3. 回滚更容易。
4. 适合 2 核 4G 和本地旧电脑的资源条件。

代价：

1. 不具备 Kubernetes 的自动调度和自愈能力。
2. 多机部署和服务扩缩容需要手动规划。
3. 配置、密钥、网络隔离需要自己约束。

## 后续重新评估条件

出现以下情况时重新评估：

1. 服务实例数量明显增加。
2. 需要自动扩缩容。
3. 多台服务器部署复杂度明显上升。
4. 发布频率和环境数量增加，Compose 运维成本变高。
5. 云服务器资源升级并具备运行 Kubernetes 的条件。
