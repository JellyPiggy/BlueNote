# BlueNote 数据存储方案

版本：v0.1  
状态：草案  
更新时间：2026-06-03

## 1. 文档目标

本文档定义 BlueNote 的数据存储总体策略，包括 MySQL、Redis、对象存储、数据归属、ID 生成、事务一致性、缓存设计、备份恢复和后续演进方向。

本文不展开每个服务的详细表结构。各服务表结构会在服务详细设计文档中单独设计。

## 2. 总体原则

### 2.1 MySQL 是事实来源

核心业务状态以 MySQL 为最终事实来源。

包括：

1. 用户资料。
2. 登录凭证。
3. 笔记主体。
4. 关注关系。
5. 点赞收藏明细。
6. 评论内容。
7. IM 消息。
8. 神券活动、订单状态、用户券和支付记录。

Redis、RocketMQ、对象存储都不能单独决定核心业务状态。

### 2.2 Redis 是加速层和临时状态层

Redis 用于：

1. 缓存。
2. 高频计数。
3. Feed 收件箱。
4. 排行榜。
5. 登录态辅助校验。
6. 幂等去重。
7. 神券秒杀库存、限购集合和秒杀 token。
8. 分布式锁。
9. IM 在线状态。

Redis 数据必须有过期策略、重建策略或可丢失说明。

### 2.3 对象存储保存文件本体

图片、视频、头像等文件不进入 MySQL。

MySQL 只保存：

1. 文件 ID。
2. bucket。
3. objectKey。
4. 文件大小。
5. MIME 类型。
6. 所属用户。
7. 业务绑定关系。

第一阶段文件本体保存在 MinIO；后续可迁移到云 OSS/COS/R2 + CDN。

### 2.4 按服务划分数据所有权

每个服务拥有自己的 schema，不跨服务直接 join。

跨服务数据访问方式：

1. 同步接口查询。
2. 异步事件冗余。
3. 离线校准任务。

### 2.5 避免强分布式事务

第一阶段不引入 Seata 等分布式事务框架。

采用：

1. 本地事务。
2. RocketMQ 事务消息或 Outbox 表。
3. 消费端幂等。
4. 补偿任务。
5. 定期数据校验。

## 3. 存储组件选型

| 类型 | 技术 | 用途 |
|---|---|---|
| 关系数据库 | MySQL 8.x | 核心业务数据 |
| 缓存 | Redis 7.x | 缓存、计数、Feed、排行榜、在线状态 |
| 对象存储 | MinIO（第一阶段）/ 云 OSS（后续演进） | 图片、视频、头像等文件 |
| 消息队列 | RocketMQ 5.x | 数据变更事件和异步一致性 |
| 备份存储 | 本地旧电脑 + 云端备份 | MySQL 备份、对象备份 |

## 4. MySQL 方案

### 4.1 部署策略

第一阶段：

1. MySQL 单机部署。
2. 开启 binlog。
3. 每日全量备份。
4. 关键业务上线后保留一段时间 binlog。
5. 资源允许时做主从复制。

后续演进：

1. 云 RDS。
2. 主从复制。
3. 读写分离。
4. 分库分表。

当前不做分库分表，但 schema 边界要为未来迁移保留空间。

### 4.2 Schema 划分

建议按逻辑服务拆 schema：

| Schema | 归属服务 | 核心数据 |
|---|---|---|
| bluenote_auth | bluenote-auth | 登录凭证、设备会话 |
| bluenote_user | bluenote-user | 用户资料、用户状态 |
| bluenote_file | bluenote-file | 文件元数据、上传记录 |
| bluenote_note | bluenote-note | 笔记主体、媒体绑定、点赞收藏明细 |
| bluenote_counter | bluenote-counter | 计数快照、计数流水 |
| bluenote_relation | bluenote-relation | 关注关系、关系流水 |
| bluenote_feed | bluenote-feed | Feed 快照、Feed 重建状态 |
| bluenote_rank | bluenote-rank | 榜单快照、热度分、成长分和来源贡献记录 |
| bluenote_comment | bluenote-comment | 评论、评论点赞 |
| bluenote_push | bluenote-push | 设备、推送偏好、投递请求、投递日志 |
| bluenote_notification | bluenote-notification | 通知、聚合、未读数、已读状态、outbox |
| bluenote_im | bluenote-im | 会话、消息、会话消息链、用户消息链、未读、ACK |
| bluenote_order | bluenote-order | 神券模板、神券活动、秒杀请求、订单、支付记录、用户券、库存流水、状态流转 |

规则：

1. 服务只写自己的 schema。
2. 禁止跨 schema join。
3. 禁止其他服务直接修改非本服务表。
4. 需要冗余数据时通过事件同步。
5. 需要聚合查询时优先由网关/BFF 或专门读模型处理。

### 4.3 表设计基础规范

所有核心表建议包含：

| 字段 | 说明 |
|---|---|
| id | 主键 |
| created_at | 创建时间 |
| updated_at | 更新时间 |
| deleted | 逻辑删除标记，按场景决定 |
| version | 乐观锁版本，按场景决定 |

时间字段建议：

1. 数据库存储使用 `datetime(3)` 或等价精度。
2. 统一以北京时间展示。
3. 服务内部避免直接依赖服务器本地时区。
4. API 时间格式在接口规范中统一。

金额字段建议：

1. 使用整数分。
2. 不使用 float/double。
3. 金额计算放在服务端。

状态字段建议：

1. 使用明确枚举值。
2. 不使用含义模糊的魔法数字。
3. 状态机类业务必须记录状态流转。

### 4.4 ID 策略

建议使用全局趋势递增 ID。

可选方案：

1. MyBatis-Plus `ASSIGN_ID`，基于雪花算法。
2. 独立 ID 服务，后续需要时再拆。
3. 数据库自增 ID，仅适合单库单表且不关心全局唯一的场景。

第一阶段建议：

1. 后端主键使用 `BIGINT`。
2. 对外 API 返回 ID 时使用字符串，避免移动端或前端 JS 精度问题。
3. 不在 ID 中暴露业务含义。
4. 订单号可以单独设计业务编号，不直接等同数据库主键。

### 4.5 索引原则

索引设计原则：

1. 按查询场景建索引，不为了字段存在而建索引。
2. 高频分页查询必须有合适组合索引。
3. 状态 + 时间类查询常用组合索引。
4. 唯一业务约束必须用唯一索引保护。
5. 避免低区分度字段单独建索引。
6. 避免过多索引影响写入。

典型唯一约束：

| 场景 | 唯一约束 |
|---|---|
| 手机号登录 | phone |
| 第三方账号绑定 | provider + open_id |
| 笔记点赞 | user_id + note_id |
| 笔记收藏 | user_id + note_id |
| 关注关系 | follower_id + followee_id |
| 评论点赞 | user_id + comment_id |
| 神券一人一券 | user_id + activity_id |
| 订单幂等 | user_id + client_request_id |
| 支付回调幂等 | channel + channel_trade_no |
| MQ 消费幂等 | consumer_group + event_id |

### 4.6 分页策略

移动端列表禁止无约束深分页。

建议：

1. 时间线类列表使用游标分页。
2. 关注列表、粉丝列表使用游标分页或基于 ID 的翻页。
3. 管理类列表可使用 page/size，但必须限制最大页码和最大 size。
4. 排行榜使用固定 Top N 或游标。

深分页风险：

1. MySQL 扫描成本高。
2. 数据变化时分页结果不稳定。
3. 容易造成慢查询。

### 4.7 逻辑删除策略

建议按业务语义决定是否逻辑删除。

适合逻辑删除：

1. 用户资料。
2. 笔记。
3. 评论。
4. 通知。
5. 文件元数据。

不一定适合逻辑删除：

1. 登录失败流水。
2. MQ 消费幂等记录。
3. 计数流水。
4. 订单状态流转。

删除原则：

1. 用户可见删除和物理删除分开。
2. 核心行为明细不随便物理删除。
3. 合规要求的数据删除后续单独设计。

## 5. Redis 方案

### 5.1 使用场景

| 场景 | 数据结构 | 说明 |
|---|---|---|
| 热点笔记缓存 | String / Hash | 笔记摘要、详情缓存 |
| 用户资料缓存 | String / Hash | 用户轻量信息 |
| 高频计数 | String / Hash | 点赞数、收藏数、评论数等 |
| Feed 收件箱 | ZSet | score 使用发布时间或分发时间 |
| 排行榜 | ZSet | score 使用热度分 |
| 神券秒杀 | String / Set / Hash | 秒杀库存、用户限购、请求映射、token |
| 登录态辅助 | String / Hash | refresh token 状态、设备会话 |
| 幂等去重 | String / Set | eventId、requestId |
| 分布式锁 | String | 使用 SET NX PX |
| IM 在线状态 | Hash / Set | userId 到连接节点 |

### 5.2 Key 命名规范

建议格式：

```text
bluenote:{env}:{service}:{domain}:{id}
```

示例：

```text
bluenote:prod:user:profile:{userId}
bluenote:prod:note:detail:{noteId}
bluenote:prod:counter:note:{noteId}
bluenote:prod:feed:inbox:{userId}
bluenote:prod:rank:hot:daily
bluenote:prod:im:online:{userId}
bluenote:prod:order:seckill:stock:{activityId}
bluenote:prod:idempotent:mq:{consumerGroup}:{eventId}
```

规则：

1. 必须带环境前缀。
2. 必须带服务或业务域。
3. 大 key 必须有容量上限。
4. 临时 key 必须有 TTL。
5. 禁止使用过短且容易冲突的 key。

### 5.3 TTL 策略

| 数据 | TTL 建议 |
|---|---|
| 用户资料缓存 | 30 分钟到 6 小时 |
| 笔记详情缓存 | 5 分钟到 1 小时 |
| 热点笔记缓存 | 1 小时到 24 小时 |
| Access Token 黑名单 | 不超过 token 剩余有效期 |
| Refresh Token 状态 | 和 refresh token 生命周期一致 |
| 幂等记录 | 按消息重试窗口，通常 1 到 7 天 |
| Feed 收件箱 | 可长期保留，但限制长度 |
| 排行榜 | 按榜单周期保留 |
| IM 在线状态 | 短 TTL + 心跳刷新 |

### 5.4 缓存模式

常用模式：

1. Cache Aside：应用先查缓存，未命中查数据库，再写缓存。
2. Write Through：少量强一致读模型可同步写缓存。
3. 异步刷新：通过 MQ 事件刷新或失效缓存。

建议默认使用 Cache Aside。

更新策略：

1. 写数据库成功后删除缓存。
2. 高频计数可先写 Redis，再异步落库。
3. Feed 和排行榜以 Redis 为在线读模型，但要有重建依据。

### 5.5 缓存问题处理

缓存穿透：

1. 参数校验。
2. 空值缓存。
3. 布隆过滤器后续评估。

缓存击穿：

1. 热点 key 加互斥锁。
2. 提前刷新。
3. 后台异步刷新。

缓存雪崩：

1. TTL 加随机抖动。
2. 分批预热。
3. 限流和降级。

大 key：

1. Feed 收件箱限制长度。
2. 排行榜限制 Top N。
3. Set/ZSet 拆分或分桶。
4. 定期扫描和告警。

热 key：

1. 本地缓存短期兜底。
2. 拆分计数 key。
3. 限流。
4. 后续引入多级缓存。

## 6. 对象存储方案

### 6.1 存储对象

对象存储保存：

1. 用户头像。
2. 笔记图片。
3. 笔记视频。
4. 评论图片，后续扩展。
5. IM 图片消息。

### 6.2 第一阶段选型

可选：

1. 第一阶段开发、测试、正式环境统一使用 MinIO。
2. 正式环境 MinIO 部署在 2 核 4G 云服务器上，文件数据目录必须持久化，并定期备份到本地旧电脑。
3. 后续当文件数量、访问量、带宽或可用性要求提高后，再迁移到云 OSS/COS/R2 + CDN。

长期演进更推荐云对象存储，原因：

1. 减少云服务器磁盘压力。
2. 更适合公网访问。
3. 更容易接 CDN。
4. 可用性通常高于自建单机 MinIO。

### 6.3 上传流程

建议流程：

```text
移动端
  -> 文件服务申请上传凭证
  -> 移动端直传对象存储
  -> 移动端通知文件服务上传完成
  -> 文件服务校验对象存在
  -> 文件服务保存文件元数据
  -> 业务服务绑定 fileId
```

优点：

1. 文件流量不经过业务服务。
2. 降低云服务器带宽和内存压力。
3. 上传过程更容易扩展。

### 6.4 文件安全

要求：

1. 限制文件类型。
2. 限制文件大小。
3. 限制单次上传数量。
4. 限制用户上传频率。
5. 对文件名做服务端生成，不信任客户端文件名。
6. 私有资源使用签名 URL。
7. 后续接入内容审核。

### 6.5 文件生命周期

文件状态建议：

1. uploading：已申请上传，未确认。
2. uploaded：上传完成。
3. bound：已被业务对象绑定。
4. abandoned：上传后未绑定，待清理。
5. deleted：逻辑删除。

清理策略：

1. 定时清理长时间未绑定文件。
2. 业务删除后先逻辑删除，再延迟物理删除。
3. 物理删除失败需要重试。

## 7. 数据一致性方案

### 7.1 本地事务

单服务内部写操作使用 MySQL 本地事务。

例如：

1. 创建笔记和笔记媒体绑定。
2. 创建评论和评论层级关系。
3. 创建关注关系和关系流水。
4. 创建神券订单、扣减活动库存、发放用户券和记录订单状态流转。

### 7.2 跨服务最终一致

跨服务变化通过 RocketMQ 事件实现最终一致。

示例：

| 源服务 | 事件 | 目标服务 |
|---|---|---|
| 笔记服务 | NotePublished | Feed、排行榜、通知 |
| 评论服务 | CommentCreated | 计数、排行榜、通知 |
| 关系服务 | UserFollowed | 计数、Feed、通知 |
| 订单服务 | CouponSeckillAccepted | 订单服务内部异步下单消费者 |
| 订单服务 | OrderTimeoutCheck | 订单服务内部关单消费者 |
| 订单服务 | CouponIssued、OrderClosed | 通知、运营后续；需要提醒时另发 PushSendRequested |

### 7.3 事务消息和 Outbox

对于关键事件，建议二选一：

1. RocketMQ 事务消息。
2. Outbox 表 + 后台投递任务。

选择建议：

| 场景 | 建议 |
|---|---|
| 神券订单创建、支付、发券、关单 | RocketMQ 事务消息或 Outbox |
| 笔记发布 | Outbox 或事务消息 |
| 关注关系变更 | Outbox 或普通消息 + 补偿 |
| 点赞收藏 | 普通消息 + 幂等 + 定期校准 |

第一阶段可以统一先使用 Outbox 模式，简单可控；订单等关键链路后续可使用 RocketMQ 事务消息。

### 7.4 幂等记录

消费端必须记录处理结果。

建议表：

```text
mq_consume_record
```

核心字段：

1. consumer_group
2. event_id
3. event_type
4. status
5. retry_count
6. consumed_at

唯一约束：

```text
consumer_group + event_id
```

## 8. 数据重建能力

必须支持重建的数据：

| 数据 | 重建来源 |
|---|---|
| 笔记计数 | 点赞收藏明细、评论表、计数快照 |
| 用户粉丝数 | 关注关系表 |
| 用户关注数 | 关注关系表 |
| Feed 收件箱 | 关注关系 + 作者笔记 |
| 排行榜 | 笔记 + 计数 + 评论 |
| IM 未读数 | 消息 ACK 状态 |
| 文件绑定状态 | 文件元数据 + 业务表绑定 |

要求：

1. 重建任务可重复执行。
2. 重建过程不能破坏在线数据。
3. 大数据量时分批处理。
4. 重建结果要有校验。

## 9. 备份与恢复

### 9.1 MySQL

最低要求：

1. 每日全量备份。
2. 开启 binlog。
3. 备份文件异地保存。
4. 备份脚本纳入运维目录。
5. 每月至少做一次恢复演练。

### 9.2 Redis

建议：

1. 开启 RDB 或 AOF。
2. 明确哪些数据可丢失。
3. 提供计数、Feed、排行榜重建脚本。
4. 设置 maxmemory，避免挤爆宿主机。

### 9.3 对象存储

建议：

1. 文件元数据在 MySQL。
2. 文件本体定期校验。
3. 重要 bucket 开启备份。
4. 云 OSS 设置生命周期规则。

## 10. 数据安全

### 10.1 敏感字段

敏感字段包括：

1. 手机号。
2. 密码哈希。
3. Refresh Token 哈希。
4. 第三方登录 openId。
5. 支付流水。
6. 用户设备信息。
7. IP 地址。

要求：

1. 密码不可逆哈希。
2. Refresh Token 只存哈希或可撤销标识。
3. 手机号展示脱敏。
4. 日志不得打印敏感字段。
5. 数据库账号按环境隔离。

### 10.2 数据权限

服务层必须校验数据权限。

示例：

1. 用户只能编辑自己的资料。
2. 作者只能编辑自己的笔记。
3. 用户只能删除自己的评论，或作者按规则管理笔记下评论。
4. 用户只能查看自己有权限的 IM 会话。
5. 用户只能查看自己的订单。

## 11. 容量规划

第一阶段不需要精确大规模容量模型，但要按增长点设计。

高增长数据：

1. 笔记。
2. 图片和视频。
3. 评论。
4. 点赞收藏明细。
5. 关注关系。
6. IM 消息。
7. 通知。
8. MQ 消费记录。

容量控制策略：

1. 列表查询必须分页。
2. Feed 收件箱限制长度。
3. 排行榜只保留固定周期。
4. 通知可归档。
5. IM 消息后续做冷热分离。
6. 日志单独设置保留周期。

| MySQL 单机 | 主从复制、云 RDS、读写分离 |
