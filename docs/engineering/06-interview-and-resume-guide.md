# BlueNote 简历与面试讲解指南

版本：v0.1
状态：GitHub 展示与后端实习面试辅助文档
更新时间：2026-06-13

## 1. 文档目标

本文帮助读者把 BlueNote 讲成一个可信的后端实习项目。

一个项目能不能在面试里加分，不只取决于功能数量，还取决于你能不能讲清楚：

1. 项目解决了什么业务问题。
2. 你负责了哪些模块。
3. 为什么这么设计。
4. 数据怎么存。
5. 缓存怎么用。
6. MQ 为什么需要、怎么保证幂等。
7. 异常和补偿怎么处理。
8. 当前方案有哪些边界和后续演进。

BlueNote 不建议包装成“生产级平台”，更适合包装成“个人可运行、模块完整、工程边界清楚的社区类后端项目”。

## 2. 30 秒项目介绍

适合面试开场：

```text
BlueNote 是我做的一个社区类后端项目，业务上覆盖用户登录、图片笔记发布、评论、点赞收藏、关注关系、关注页 Feed、通知、IM 单聊、排行榜和神券秒杀订单。后端使用 Java 21、Spring Boot、Spring Cloud Gateway、MySQL、Redis、RocketMQ 和 MinIO，移动端是 uni-app H5。

项目的重点不是简单 CRUD，而是按真实业务链路做服务边界、契约、数据归属、事件驱动和缓存读模型。比如计数通过 RocketMQ 事件异步聚合，Redis 做在线计数，MySQL 保留 counter_snapshot 和重建能力；订单模块用秒杀 token、Redis Lua 预扣、异步下单、MOCK 支付和超时关单跑通完整购买逻辑。
```

如果时间更短，可以说：

```text
这是一个 Java 后端社区项目，重点展示契约优先、多模块服务边界、RocketMQ 事件驱动、Redis 可重建读模型和订单状态机。它可以本地运行，也有移动端 H5 页面和主链路冒烟脚本。
```

## 3. 简历写法

### 3.1 推荐写法

```text
BlueNote 社区类后端项目
- 使用 Java 21 + Spring Boot 3.5 + Spring Cloud Gateway 搭建社区类多模块后端，按 gateway、member、content、social、order 进行物理应用拆分，逻辑上覆盖 auth、user、file、note、comment、relation、counter、feed、rank、notification、push、im、order 等服务。
- 采用契约优先方式维护 API、错误码、MySQL DDL、Redis Key、RocketMQ 事件和权限矩阵，移动端 H5 与后端按契约联调。
- 实现笔记发布、MinIO 预签名图片上传、评论、点赞收藏、关注关系、关注页 Feed、站内通知、WebSocket 实时投递、IM 单聊、排行榜和神券订单等核心链路。
- 使用 RocketMQ + outbox 实现计数、Feed、排行榜、通知、Push、IM 和订单事件流转，通过消费记录、业务唯一约束和状态机 CAS 保证重复投递下的幂等。
- 使用 Redis 承载在线计数、Feed 收件箱、排行榜 ZSet、通知未读数、设备在线路由和秒杀库存，并保留 MySQL 快照、重建任务、回源和对账修复能力。
- 在订单模块中实现秒杀 token、Redis Lua 原子预扣、异步下单、MOCK 支付、超时关单、库存回补、Redis 库存重建和库存对账修复。
```

### 3.2 更简洁的写法

```text
BlueNote 社区类后端项目：基于 Java 21、Spring Boot、MySQL、Redis、RocketMQ 和 MinIO，实现登录、用户资料、图片笔记、评论互动、关注 Feed、计数、通知、IM、排行榜和神券订单。项目采用契约优先和多模块服务边界设计，使用 outbox + MQ 驱动计数、Feed、榜单、通知和订单链路，Redis 读模型均保留 MySQL 事实和重建路径。
```

### 3.3 不建议写法

不要写：

```text
实现高并发小红书平台。
支撑百万用户访问。
完整实现生产级支付系统。
完整实现推荐系统。
完整接入真实厂商 Push。
使用分布式事务保证强一致。
```

原因：

1. 这些说法超过当前项目实际边界。
2. 面试官追问压测数据、支付回调、厂商凭证、推荐策略、生产部署时会很难自洽。
3. 诚实表达“采用了生产常见设计思想，但当前是个人可运行项目”更可信。

## 4. 项目整体讲法

建议按这条顺序讲：

1. **项目背景**：移动端社区项目，核心是内容发布、社交互动、消息触达和活动订单。
2. **后端结构**：gateway/member/content/social/order 五个物理应用，逻辑服务边界更多。
3. **契约优先**：API、DDL、Redis、MQ 和权限都在 `docs/contracts`。
4. **核心链路**：登录、发笔记、关注 Feed、互动计数、通知、IM、订单、榜单。
5. **技术重点**：RocketMQ outbox、消费幂等、Redis 读模型、MySQL 快照、状态机。
6. **当前边界**：真实支付、厂商 Push、推荐搜索、生产监控和测试覆盖是后续演进。

一个比较自然的 2 分钟版本：

```text
这个项目我按社区产品的核心链路来做。移动端通过 gateway 进入后端，gateway 负责 JWT 校验和用户上下文透传。后端物理上拆成 member、content、social、order 等应用，逻辑上有 auth、user、file、note、comment、relation、counter、feed、notification、push、im、rank、order 等服务。

内容链路里，图片使用 MinIO 预签名直传，file 服务保存文件元数据和上传状态，note 发布时校验文件归属并绑定媒体。互动链路里，点赞、收藏、评论先写行为事实，再通过 RocketMQ 事件给 counter 聚合计数，counter 同时维护 Redis 在线计数和 MySQL 快照，并发布 CounterChanged 给 rank 更新榜单。

Feed 是关注页收件箱模型，普通作者发布笔记后通过 fanout 任务投递到粉丝收件箱，读路径优先 Redis，失败后可以降级 MySQL。订单模块里，用户先拿秒杀 token，再用 Redis Lua 原子校验库存和限购，预扣成功后异步下单，付费券走 MOCK 支付，超时未支付会 CAS 关单并回补库存。

我对这个项目的定位是个人可运行的后端实习项目，重点是服务边界、契约、异步一致性、缓存重建和状态机。真实支付、厂商 Push、推荐系统和生产监控目前作为后续演进，没有包装成已完成能力。
```

## 5. 模块讲解模板

讲任何一个模块都可以套这个结构：

```text
这个模块解决什么问题？
数据事实在哪里？
对外 API 和内部 API 分别是什么？
是否使用 Redis？Redis 丢了怎么恢复？
是否使用 MQ？重复消费怎么处理？
写操作如何保证幂等？
下游失败时怎么降级或补偿？
当前实现还有哪些边界？
```

这比背技术名词更重要。

## 6. Auth / User 讲法

可以这样讲：

```text
auth 负责账号、密码、Token 和设备会话，user 负责用户展示资料。注册时 auth 写账号、密码哈希和会话，同时创建默认用户资料。后续请求由 gateway 校验 Access Token，并向下游服务注入 X-User-Id、X-Device-Id、X-Session-Id 和 X-Trace-Id。业务服务不直接解析移动端 Token，而是读取网关注入的用户上下文。
```

面试追问点：

| 问题 | 回答方向 |
|---|---|
| 为什么要 Access Token + Refresh Token？ | Access Token 短有效期降低泄露风险，Refresh Token 支持长会话和轮换 |
| 为什么用户资料和登录凭证分开？ | 安全凭证和展示资料变化频率、权限和数据敏感度不同 |
| 下游服务为什么不自己解析 Token？ | 统一在网关鉴权，减少重复逻辑，下游只依赖可信上下文 |
| 修改头像为什么要调用 file 服务？ | user 不拥有文件事实，需要校验文件归属、状态、场景和类型 |

## 7. File / Note / Comment 讲法

可以这样讲：

```text
文件上传采用预签名直传。移动端先向 file 服务申请上传凭证，file 保存文件元数据和上传会话，移动端直传 MinIO 后再调用确认接口。发布笔记时 note 服务通过内部接口校验 fileId 是否属于当前用户、是否上传完成、是否符合 NOTE_IMAGE 场景，然后写 note、note_media、note_version 和 outbox。
```

面试追问点：

| 问题 | 回答方向 |
|---|---|
| 为什么图片不直接传给后端？ | 降低后端带宽和内存压力，让对象存储承担文件流量 |
| 为什么要上传确认？ | 避免业务引用未完成上传或非法上传的文件 |
| 点赞收藏为什么要明细表？ | 判断用户是否已操作、支持取消、支持重建计数和审计 |
| 评论为什么要状态删除？ | 保留层级关系和审计，避免子回复结构被破坏 |

## 8. Counter 讲法

可以这样讲：

```text
counter 是计数聚合服务，负责笔记点赞数、收藏数、评论数，用户关注数、粉丝数、作品数、获赞数，以及评论点赞和回复数。它消费 note、interaction、comment、relation 事件，生成 delta，更新 MySQL counter_snapshot，并最佳努力更新 Redis 在线计数。查询时优先 Redis，再 MySQL 快照，最后回源到 note/comment/relation 服务并回填。
```

面试追问点：

| 问题 | 回答方向 |
|---|---|
| 为什么不直接在 note 表上加 like_count？ | 高频互动会造成写热点，也会让 note 强耦合所有互动变化 |
| Redis 计数丢了怎么办？ | 从 MySQL 快照读，或回源到行为明细服务重建 |
| MQ 重复消费怎么办？ | eventId + consumerGroup 消费记录、delta 幂等、业务唯一约束 |
| 计数能不能强一致？ | 展示计数通常接受最终一致；关键行为事实仍在主写服务 |

## 9. Feed 讲法

可以这样讲：

```text
关注页 Feed 使用收件箱读模型。作者发布公开笔记后，feed 消费 NotePublished，写 feed_note_index，创建 fanout 任务，把笔记投递到粉丝的 feed_inbox_item 和 Redis 收件箱。用户拉取关注 Feed 时优先读 Redis，Redis 为空或异常时读 MySQL 收件箱，再聚合笔记摘要、作者摘要和计数。
```

面试追问点：

| 问题 | 回答方向 |
|---|---|
| 为什么关注 Feed 不直接实时查所有关注作者的笔记？ | 关注列表大时查询复杂，分页和排序成本高，收件箱读模型更适合高频读 |
| 大 V 怎么办？ | 当前 foundation 先做普通作者写扩散，后续可按粉丝量做推拉结合 |
| 取关后怎么办？ | 消费 UserUnfollowed 清理或标记对应作者的收件箱项 |
| 笔记删除或私密怎么办？ | 消费 NoteDeleted/VisibilityChanged，Feed 读路径也会校验可见性 |

## 10. Notification / Push 讲法

可以这样讲：

```text
notification 负责站内通知读模型和未读数，消费点赞、收藏、评论、关注和订单事件。点赞收藏可以按接收人、通知类型和笔记聚合，评论、回复、关注和订单通知保留明细。通知创建后会写 PushSendRequested，push 服务消费后根据设备、偏好和在线状态决定是否通过 WebSocket 投递。
```

面试追问点：

| 问题 | 回答方向 |
|---|---|
| 通知和 Push 为什么分开？ | 通知是站内记录，Push 是投递通道，职责不同 |
| 用户关闭推送怎么办？ | push 检查偏好并记录过滤结果，不影响站内通知事实 |
| 离线 Push 做了吗？ | 当前完成设备、偏好、WebSocket 在线投递和日志，真实厂商通道是扩展点 |
| 推送失败会不会丢业务？ | 不会，业务事实和站内通知已落库，客户端可拉取补偿 |

## 11. IM 讲法

可以这样讲：

```text
IM 当前实现单聊文字消息。发送时先校验会话和成员关系，再保存 im_message、conversation_message 和用户消息链，更新会话最后消息和未读数，然后写 ImMessageSent 和 PushSendRequested。接收方在线时由 push 通过 WebSocket 下发，客户端也可以通过消息列表接口拉取。
```

面试追问点：

| 问题 | 回答方向 |
|---|---|
| 为什么消息要先入库？ | 保证离线可拉取和投递失败可恢复 |
| 如何防止重复发送？ | 使用 clientMsgId 和业务唯一约束做发送幂等 |
| 未读数怎么处理？ | 会话成员维度维护未读，已读接口更新 read position |
| 群聊做了吗？ | 当前是单聊 foundation，群聊、撤回、富媒体是后续扩展 |

## 12. Order 讲法

可以这样讲：

```text
订单模块聚焦神券秒杀。用户先查询活动并获取秒杀 token，抢券时 Redis Lua 原子校验 token、库存和用户限购，预扣成功后写 CouponSeckillAccepted 事件，订单消费者异步扣 MySQL 库存并创建订单。免费券直接发券，付费券进入 WAIT_PAY，MOCK 支付成功后发券，超时未支付会 CAS 关单并回补库存。
```

面试追问点：

| 问题 | 回答方向 |
|---|---|
| 为什么用 Redis Lua？ | 原子完成库存扣减和限购判断，避免并发下拆成多条命令出错 |
| 如何避免超卖？ | Redis 预扣削峰，MySQL 乐观扣减和唯一约束兜底，库存对账修复 |
| 重复抢券怎么办？ | Redis 用户参与标记 + MySQL 唯一约束 + 幂等请求 |
| 支付和关单并发怎么办？ | 订单状态机 CAS，只有 WAIT_PAY 且满足条件的状态变更能成功 |
| 为什么是 MOCK 支付？ | 个人项目验证购买状态机和发券逻辑，真实支付作为后续渠道适配 |

## 13. Rank 讲法

可以这样讲：

```text
排行榜当前有本周热门笔记榜和年度创作者成长榜。rank 消费 NotePublished 建立笔记候选索引，消费 CounterChanged 根据点赞、评论、收藏净增量更新分数。Redis ZSet 负责在线排名，MySQL 保存成员分数、贡献记录、变更日志和快照，支持 Redis 丢失后的重建。
```

面试追问点：

| 问题 | 回答方向 |
|---|---|
| 为什么榜单消费 CounterChanged？ | 榜单关心计数变化，不需要耦合所有互动明细事件 |
| Redis ZSet 丢了怎么办？ | 从 MySQL rank_member_score 和快照重建 |
| 笔记删除后怎么处理？ | 暂停贡献、分数扣回或归零，并从在线榜移除 |
| 榜单需要强一致吗？ | 不需要，榜单通常可以接受短暂延迟 |

## 14. 高频通用问题

### 14.1 为什么不做成单体？

可以回答：

```text
项目是个人项目，所以物理部署上没有拆十几个进程；但逻辑上按服务边界设计。这样既能控制运行成本，又能展示真实工程中的数据归属、接口边界和事件流转。如果以后某个模块压力变大，可以优先把 IM、Feed、Counter、Order 这些模块独立进程部署。
```

### 14.2 为什么不用分布式事务？

可以回答：

```text
这个项目避免强分布式事务，采用本地事务 + outbox + MQ + 幂等消费 + 补偿任务。比如笔记发布先在 note 服务本地事务写笔记和 outbox，dispatcher 再发送事件；counter、feed、notification 各自消费并维护读模型。这样复杂度比 Seata 低，也更适合事件驱动读模型。
```

### 14.3 Redis 挂了怎么办？

可以回答：

```text
Redis 在项目里主要是缓存和在线读模型，不是唯一事实来源。计数可以从 counter_snapshot 或来源服务回源，Feed 可以从 MySQL 收件箱或关注关系重建，排行榜可以从 MySQL 分数事实重建，订单库存可以从 MySQL 活动库存和库存流水重建。Redis 异常时部分展示会降级，但核心事实不会丢。
```

### 14.4 MQ 重复消费怎么办？

可以回答：

```text
消费者不能假设消息只投递一次。项目里用 eventId + consumerGroup 记录消费状态，同时结合业务唯一约束和状态机 CAS。比如订单创建有 requestId 和 userId/activityId 约束，计数有消费记录和 delta，通知有聚合唯一键，重复消息不会重复产生业务效果。
```

### 14.5 outbox 解决什么问题？

可以回答：

```text
outbox 解决本地业务事务和消息发送之间的一致性问题。业务表和 outbox 在同一个本地事务里写入，如果 MQ 暂时不可用，dispatcher 后续还能扫描 outbox 重试发送。它不能替代消费者幂等，因为消息仍可能重复发送或重复消费。
```

### 14.6 项目最大短板是什么？

建议诚实回答：

```text
当前最大短板是自动化测试和生产运维还不够完整。项目已经有后端编译、移动端 typecheck/build 和主链路冒烟脚本，但服务级集成测试、OpenAPI 与契约自动 diff、MQ 死信告警、监控面板和压测还需要继续补。这个项目目前定位是实习项目展示和学习，不是生产上线系统。
```

## 15. 面试中不要踩的坑

| 坑 | 更好的说法 |
|---|---|
| 说“完全微服务架构” | 逻辑上按微服务边界设计，物理上为了个人项目合并部署 |
| 说“Redis 保证数据一致” | Redis 是读模型和缓存，MySQL 是事实来源 |
| 说“MQ 保证消息只消费一次” | MQ 可能重复投递，消费者要幂等 |
| 说“订单已经接入真实支付” | 当前是 MOCK 支付，用于跑通订单状态机 |
| 说“Push 已经真实离线送达手机通知栏” | 当前完成设备、偏好、WebSocket 在线投递和投递日志，真实厂商通道待扩展 |
| 说“项目已经生产可用” | 当前是个人可运行 demo，生产化需要补测试、监控、部署、安全和压测 |

## 16. 可以主动强调的亮点

1. **契约优先**：项目不是边写边凑，API、DDL、Redis、MQ 和权限都有文档。
2. **边界清楚**：逻辑服务、物理应用、schema 和事件都有归属。
3. **不是纯 CRUD**：有 outbox、MQ 消费、Redis 读模型、重建任务和状态机。
4. **能跑通链路**：有本地依赖、移动端 H5 和主链路冒烟脚本。
5. **表达诚实**：清楚说明已完成、未完成和后续演进。

## 17. 面试前准备清单
